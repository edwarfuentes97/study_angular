Preguntas que DEBES dominar:

P1: ¿Qué es NgRx y cuáles son sus conceptos fundamentales?

Respuesta esperada: NgRx es una librería de state management para Angular basada en el patrón Redux. Implementa un flujo de datos unidireccional usando:

  • Store: Contenedor inmutable del estado global de la aplicación (single source of truth)
  • Actions: Eventos que describen algo que ocurrió (ej: [User] Login Success)
  • Reducers: Funciones puras que toman el estado actual + una acción y retornan un nuevo estado
  • Selectors: Funciones puras para extraer porciones específicas del estado (memoizadas)
  • Effects: Manejan side effects (llamadas HTTP, navegación, logs) — escuchan acciones y pueden emitir nuevas acciones
Flujo: Component → dispatch(Action) → Reducer → Store → Selector → Component
                                         ↓
                                      Effects → API → dispatch(new Action)

P2: ¿Cuál es la diferencia entre NgRx Store y un simple servicio con BehaviorSubject?

Respuesta esperada: | Característica | Service + BehaviorSubject | NgRx Store | |---|---|---| | Complejidad | Baja | Alta | | Inmutabilidad | No garantizada | Garantizada | | Debugging | Limitado | Redux DevTools, time travel | | Side effects | Mezclados en el servicio | Separados en Effects | | Escalabilidad | Problemática en apps grandes | Excelente | | Testing | Requiere mocking | Reducers son funciones puras | | Boilerplate | Mínimo | Considerable |

Cuándo usar cada uno: - BehaviorSubject: Estado local, apps pequeñas/medianas, features aisladas - NgRx: Apps enterprise, estado compartido entre muchos componentes, debugging complejo necesario

export class ProductEffects {
  loadProducts$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.loadProducts),
      // switchMap cancela peticiones anteriores (útil para búsqueda)
      // exhaustMap ignora nuevas mientras la actual esté activa (útil para submit)

    )
  );

    private productService: ProductService

P9: ¿Cuál es la diferencia entre Observable, Subject, BehaviorSubject y ReplaySubject?

Respuesta esperada: | Tipo | Multicast | Necesita subscribe | Valor inicial | Late subscribers | |---|---|---|---|---| | Observable | No (cold) | Sí | No | Reciben desde inicio | | Subject | Sí (hot) | Sí | No | Solo valores futuros | | BehaviorSubject | Sí (hot) | Sí | Sí (requerido) | Último valor + futuros | | ReplaySubject(n) | Sí (hot) | Sí | No | Últimos n valores + futuros | | AsyncSubject | Sí (hot) | Sí | No | Solo el último al completar |

// BehaviorSubject - ideal para estado
const user$ = new BehaviorSubject<User | null>(null);
user$.next(loggedInUser); // cualquier subscriber recibe el último valor

// ReplaySubject - ideal para eventos que no quieres perder
const notifications$ = new ReplaySubject<Notification>(5); // últimas 5 notificaciones

P10: Explica los operadores de flattening: switchMap, mergeMap, concatMap, exhaustMap

Respuesta esperada:

switchMap  → Cancela la anterior cuando llega una nueva
             Uso: Búsqueda en tiempo real (typeahead), navegación

mergeMap   → Ejecuta todas en paralelo (no cancela)
             Uso: Guardar items independientes, logs, eventos que no se cancelan

concatMap  → Ejecuta secuencialmente (espera que termine la anterior)
             Uso: Operaciones donde el orden importa (transacciones)

exhaustMap → Ignora nuevas mientras la actual está activa
             Uso: Login, submit de formularios (evitar doble click)

// Ejemplo práctico: Búsqueda con typeahead
this.searchControl.valueChanges.pipe(
  debounceTime(300),        // espera 300ms después del último keystroke
  distinctUntilChanged(),   // solo si el valor cambió
  filter(term => term.length >= 3), // mínimo 3 caracteres
  switchMap(term =>         // cancela búsqueda anterior
    this.searchService.search(term).pipe(
      catchError(() => of([]))  // maneja error sin romper el stream
    )
  )
).subscribe(results => this.results = results);

P11: ¿Cómo evitas memory leaks con Observables?

Respuesta esperada:

// Método 1: async pipe (PREFERIDO - se desuscribe automáticamente)
// template: {{ data$ | async }}

// Método 2: takeUntilDestroyed (Angular 16+)
export class MyComponent {
  data$ = this.dataService.getData().pipe(
    takeUntilDestroyed()  // se desuscribe en ngOnDestroy automáticamente
  );
}

// Método 3: DestroyRef (Angular 16+)
export class MyComponent {
  constructor(private destroyRef: DestroyRef) {
    this.dataService.getData()
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(data => this.data = data);
  }
}

// Método 4: Subject + takeUntil (legacy pero común)
private destroy$ = new Subject<void>();

ngOnInit() {
  this.dataService.getData()
    .pipe(takeUntil(this.destroy$))
    .subscribe(data => this.data = data);
}

ngOnDestroy() {
  this.destroy$.next();
  this.destroy$.complete();
}

// Método 5: Subscription explícita
private sub = new Subscription();
ngOnInit() {
  this.sub.add(this.data$.subscribe(...));
}
ngOnDestroy() {
  this.sub.unsubscribe();
}

P13: ¿Qué es shareReplay y cuándo lo usarías?

Respuesta esperada:

// Sin shareReplay: cada subscriber hace una nueva llamada HTTP
const data$ = this.http.get('/api/config');
// subscriber 1 → HTTP call
// subscriber 2 → HTTP call (duplicada!)

// Con shareReplay: comparte la respuesta y la cachea
const data$ = this.http.get('/api/config').pipe(
  shareReplay({ bufferSize: 1, refCount: true })
);
// subscriber 1 → HTTP call
// subscriber 2 → Usa la respuesta cacheada

// refCount: true → cuando todos se desuscriben, limpia el cache
// bufferSize: 1 → cachea solo la última emisión

Uso ideal: datos de configuración, datos que no cambian frecuentemente, evitar llamadas HTTP duplicadas.

P14: ¿Cómo combinarías múltiples Observables?

Respuesta esperada:

// combineLatest: emite cuando CUALQUIERA emite (necesita al menos 1 de cada uno)
combineLatest([user$, permissions$, settings$]).pipe(
  map(([user, perms, settings]) => ({ user, perms, settings }))
);

// forkJoin: emite cuando TODOS completan (como Promise.all)
forkJoin({
  users: this.http.get('/api/users'),
  roles: this.http.get('/api/roles'),
  config: this.http.get('/api/config')
});

// withLatestFrom: toma el último valor de otros streams cuando el principal emite
this.actions$.pipe(
  ofType(saveUser),
  withLatestFrom(this.store.select(selectCurrentUser)),
  mergeMap(([action, user]) => this.userService.save(user))
);

// merge: combina streams en paralelo, emite cualquier valor de cualquiera
merge(click$, keypress$, touch$).subscribe(event => handle(event));

P15: ¿Qué es Change Detection y cómo lo optimizas?

Respuesta esperada: Angular tiene dos estrategias:

Default: Verifica TODO el árbol de componentes en cada evento (click, HTTP response, setTimeout, etc.)

OnPush: Solo verifica el componente cuando: 1. Un @Input() recibe una nueva referencia (inmutabilidad obligatoria) 2. Se dispara un evento del template del componente 3. Se usa async pipe y el Observable emite 4. Se llama manualmente ChangeDetectorRef.markForCheck() o detectChanges()

@Component({
  selector: 'app-product-list',
  changeDetection: ChangeDetectionStrategy.OnPush, // ← Optimización clave
  template: `
    <app-product-card
      *ngFor="let product of products$ | async; trackBy: trackById"
      [product]="product">
    </app-product-card>
  `
})
export class ProductListComponent {
  products$ = this.store.select(selectAllProducts); // Observable con async pipe

  trackById(index: number, item: Product): number {
    return item.id; // Evita re-render innecesario en listas
  }
}

Optimizaciones clave: - Usar OnPush en todos los componentes (compatible con NgRx) - Usar async pipe en lugar de .subscribe() - Usar trackBy en *ngFor - Objetos inmutables (spread operator: { ...obj, prop: newValue }) - runOutsideAngular() para operaciones que no necesitan change detection

P18: ¿Cómo implementas HTTP Interceptors?

Respuesta esperada:

// Interceptor funcional (Angular 15+, recomendado)
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getToken();

  if (token) {
    const cloned = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` }
    });
    return next(cloned);
  }
  return next(req);
};

// Interceptor para manejo global de errores
export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req).pipe(
    retry({ count: 2, delay: 1000 }),
    catchError((error: HttpErrorResponse) => {
      if (error.status === 401) {
        inject(AuthService).logout();
        inject(Router).navigate(['/login']);
      }
      return throwError(() => error);
    })
  );
};

// Registro (standalone)
bootstrapApplication(AppComponent, {
  providers: [
    provideHttpClient(
      withInterceptors([authInterceptor, errorInterceptor])
    )

export const authGuard: CanActivateFn = (route, state) => {

export abstract class BaseApiService<T> {

  delete(id: number | string): Observable<void> {
    return this.http.delete<void>(`${environment.apiUrl}/${this.endpoint}/${id}`);
  }
}

P33: ¿Cómo manejas errores HTTP de forma global?

// Interceptor de errores global
export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const notificationService = inject(NotificationService);
  const router = inject(Router);

  return next(req).pipe(
    catchError((error: HttpErrorResponse) => {
      switch (error.status) {
        case 401:
          inject(AuthService).logout();
          router.navigate(['/login']);
          break;
        case 403:
          notificationService.error('No tiene permisos para esta acción');
          break;
        case 404:
          notificationService.error('Recurso no encontrado');
          break;
        case 500:
          notificationService.error('Error interno del servidor');
          break;
        default:
          notificationService.error('Error de conexión');
      }
      return throwError(() => error);
    })
  );
};

@Component({
  selector: 'app-search',
  template: `
    <input [formControl]="searchControl" placeholder="Buscar suppliers...">
    <ul *ngIf="results$ | async as results">
      <li *ngFor="let result of results; trackBy: trackById"
          (click)="onSelect(result)">
        {{ result.name }}
      </li>
    </ul>
  `,
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class SearchComponent {
  searchControl = new FormControl('');

  results$ = this.searchControl.valueChanges.pipe(
    debounceTime(300),
    distinctUntilChanged(),
    filter((term): term is string => !!term && term.length >= 2),
    switchMap(term => this.searchService.search(term).pipe(
      catchError(() => of([]))
    )),
    shareReplay(1)
  );

  constructor(private searchService: SearchService) {}

  trackById(index: number, item: any) { return item.id; }
  onSelect(item: any) { /* navigate or emit */ }
}

Ejercicio 3: "Implementa un formulario multi-step con validación"

// Wizard con FormGroup por step
steps = [
  this.fb.group({ name: ['', Validators.required], email: ['', Validators.email] }),
  this.fb.group({ address: ['', Validators.required], city: [''] }),
  this.fb.group({ cardNumber: ['', [Validators.required, this.luhnValidator]] })
];
currentStep = 0;

get currentFormGroup() { return this.steps[this.currentStep]; }

nextStep() {
  if (this.currentFormGroup.valid) {
    this.currentStep++;
  } else {
    this.currentFormGroup.markAllAsTouched();
  }
}

submit() {
  const formData = this.steps.reduce((acc, step) => ({ ...acc, ...step.value }), {});
  this.store.dispatch(submitForm({ data: formData }));
}
