🔵 Observable vs Promise

Característica Promise Observable
Ejecución Eager — se ejecuta al crearse Lazy — solo al suscribirse
Valores Un solo valor Múltiples valores en el tiempo
Cancelación No se puede cancelar unsubscribe() cancela
Operadores .then(), .catch() 100+ operadores composables
Multicast Siempre compartida Unicast por defecto, multicast con Subject 
// Promise — eager, un solo valor
const promise = new Promise(resolve => {{ '{' }}
  console.log('Ejecutado inmediatamente');
  resolve(42);
{{ '}' }});

// Observable — lazy, múltiples valores
const obs$ = new Observable(subscriber => {{ '{' }}
  console.log('Ejecutado al suscribirse');
  subscriber.next(1);
  subscriber.next(2);
  subscriber.next(3);
  subscriber.complete();
{{ '}' }});

Conversión entre Promise y Observable

import { firstValueFrom, lastValueFrom, defer } from 'rxjs';

// Observable → Promise (moderno; reemplaza al deprecado .toPromise())
const user = await firstValueFrom(this.http.get<User>('/api/me'));

// Promise → Observable de forma LAZY (se ejecuta al suscribirse)
const data$ = defer(() => fetch('/api/data').then(r => r.json()));
// ⚠️ from(promise) NO es lazy: la Promise ya está en ejecución

⚠️ Errores comunes

  • Usar from(promise) creyendo que es lazy — la Promise se ejecutó al crearse. Usa defer(() => promesa()) para diferir la ejecución.
  • Hacer await firstValueFrom(stream$) sobre un stream que quizá nunca emite (p. ej. un Subject) — el await se queda colgado. Añade timeout() o valor por defecto.
  • Suscribirse varias veces a un cold Observable sin darse cuenta: cada subscribe() repite el trabajo (p. ej. otra petición HTTP).

🎤 Preguntas de entrevista

P: ¿Tres diferencias clave entre Promise y Observable?
R: Lazy vs eager, múltiples valores en el tiempo vs uno solo, y cancelable con unsubscribe() vs no cancelable.

P: ¿Cómo cancelas una petición HTTP en Angular?
R: HttpClient devuelve Observables: al desuscribirse (manual, vía switchMap, o al destruirse un async pipe) la petición XHR se aborta de verdad.

📈 Marble Diagrams

Los marble diagrams son la notación estándar para describir streams en el tiempo. Dominarlos permite leer la documentación oficial, razonar sobre operadores y comunicar comportamiento asíncrono en entrevistas y code reviews. 

Leyenda:
  --a--b--c-->   el tiempo avanza a la derecha; a, b, c son emisiones (next)
  |              complete — el stream terminó con éxito
  #              error — el stream terminó con fallo
  (ab|)          emisiones síncronas agrupadas: emite a, b y completa a la vez

Ejemplo — map(x => x * 10):
  Source:  --1----2----3--|
  Output:  -10---20---30--|

Ejemplo — debounceTime(30):
  Source:  --a-b-c------d----|
  Output:  ---------c------d-|

Esta notación es ejecutable en tests con TestScheduler (marble testing), con tiempo virtual:

import { TestScheduler } from 'rxjs/testing';

const scheduler = new TestScheduler((actual, expected) => {
  expect(actual).toEqual(expected);
});

scheduler.run(({ cold, expectObservable }) => {
  const source$ = cold('--a--b--c--|', { a: 1, b: 2, c: 3 });
  const result$ = source$.pipe(map(x => x * 10));
  expectObservable(result$).toBe('--x--y--z--|', { x: 10, y: 20, z: 30 });
});

⚠️ Errores comunes

  • Confundir | (complete) con # (error): catchError solo reacciona ante #, no ante un complete "vacío".
  • Olvidar que en marble testing el tiempo es virtual: debounceTime(300) se simula instantáneamente, no espera 300 ms reales.

🎤 Preguntas de entrevista

P: ¿Qué significa --a--(bc)--|?
R: Emite a, luego b y c en el mismo instante, y después completa.

P: ¿Por qué marble testing y no setTimeout en los tests?
R: TestScheduler usa tiempo virtual: tests deterministas e instantáneos, sin flakiness ni esperas reales.

🌡️ Cold vs Hot Observables

Un Observable cold crea su productor de datos dentro de cada suscripción: cada subscribe() arranca una ejecución nueva e independiente (unicast). Un Observable hot comparte un productor que vive fuera: los suscriptores se conectan a una emisión en curso y se pierden lo anterior (multicast).

Aspecto Cold ❄️ Hot 🔥
Productor Se crea al suscribirse Existe independientemente
Suscriptores Cada uno su propia ejecución (unicast) Comparten las emisiones (multicast)
Late subscriber Recibe todo desde el inicio Se pierde lo ya emitido
Ejemplos HttpClient, of, timer, interval Subject, fromEvent, WebSocket, router events 
// COLD: cada suscripción genera SU PROPIO valor
const cold$ = new Observable<number>(sub => sub.next(Math.random()));
cold$.subscribe(v => console.log('A:', v)); // A: 0.183
cold$.subscribe(v => console.log('B:', v)); // B: 0.771 (¡distinto!)

// HOT: el productor está fuera; todos ven lo mismo
const subject$ = new Subject<number>();
subject$.subscribe(v => console.log('A:', v));
subject$.subscribe(v => console.log('B:', v));
subject$.next(Math.random()); // A y B reciben el MISMO valor

// Cold → Hot: multicasting
const shared$ = this.http.get<Data>('/api/data').pipe(shareReplay(1));

⚠️ Errores comunes

  • Suscribirse dos veces al mismo HttpClient.get() (p. ej. un tap de logging y un async pipe) → 2 peticiones HTTP reales.
  • Suscribirse tarde a un hot Observable esperando recibir el "estado actual" — para eso existe BehaviorSubject o shareReplay(1).
  • Creer que un Subject guarda valores: sin replay, todo lo emitido antes de suscribirse se pierde.

🎤 Preguntas de entrevista

P: ¿HttpClient devuelve un Observable cold o hot?
R: Cold: cada suscripción dispara una petición nueva; por eso se cachea con shareReplay.

P: ¿Cómo conviertes un cold Observable en hot?
R: Con multicasting: share(), shareReplay() o canalizándolo por un Subject.

📡 Subject Types

Los Subjects son Observables multicast que también son Observers (puedes emitir valores con .next()).

Tipo Late Subscriber recibe Uso típico
Subject Nada — solo valores futuros Event bus, notificaciones
BehaviorSubject Último valor emitido (requiere valor inicial) Estado actual, config, auth status
ReplaySubject(n) Últimos N valores Cache de mensajes, historial
AsyncSubject Solo el último valor, al completar HTTP responses, cálculos finales 
// BehaviorSubject — siempre tiene valor actual
const user$ = new BehaviorSubject<User | null>(null);
user$.next({{ '{' }} name: 'Ana' {{ '}' }});

// Late subscriber recibe 'Ana' inmediatamente
user$.subscribe(u => console.log(u)); // {{ '{' }} name: 'Ana' {{ '}' }}

// ReplaySubject — guarda últimos N valores
const messages$ = new ReplaySubject<string>(3);
messages$.next('msg1');
messages$.next('msg2');
messages$.next('msg3');
messages$.next('msg4');

// Late subscriber recibe: msg2, msg3, msg4
messages$.subscribe(m => console.log(m));

Caso real: servicio de autenticación con BehaviorSubject

@Injectable({ providedIn: 'root' })
export class AuthService {
  // Estado privado mutable, API pública de SOLO lectura
  private readonly userSubject = new BehaviorSubject<User | null>(null);
  readonly user$ = this.userSubject.asObservable();
  readonly isLoggedIn$ = this.user$.pipe(map(u => u !== null));

  login(credentials: Credentials) {
    return this.http.post<User>('/api/login', credentials).pipe(
      tap(user => this.userSubject.next(user))
    );
  }

  logout() {
    this.userSubject.next(null);
  }
}

Casos de uso reales por tipo

  • Subject — event bus entre componentes hermanos, notificaciones toast, envío de mensajes por WebSocket.
  • BehaviorSubject — estado con "valor actual": usuario logueado, tema, filtros de tabla, carrito. Es la base de los state services y permite lectura síncrona con .value.
  • ReplaySubject(n) — historial: últimos N mensajes de chat, últimos eventos de navegación para un "volver atrás" inteligente.
  • AsyncSubject — solo el resultado final de un proceso largo; raro en la práctica (una petición de HttpClient ya se comporta parecido).

⚠️ Errores comunes

  • Exponer el Subject directamente: cualquier consumidor puede hacer next(). Expón asObservable() y métodos de mutación controlados.
  • Abusar de .value del BehaviorSubject en vez de componer streams — rompe la reactividad y esconde dependencias.
  • Olvidar que un Subject tras complete() o error() queda muerto: nunca vuelve a emitir, aunque llames next().
  • Crear un Subject donde bastaba un Observable derivado (map, combineLatest) — estado duplicado que acaba desincronizado.

🎤 Preguntas de entrevista

P: ¿Diferencia entre Subject y BehaviorSubject?
R: BehaviorSubject exige valor inicial, recuerda el último valor y lo entrega de inmediato a nuevos suscriptores; Subject no recuerda nada.

P: ¿ReplaySubject(1) es lo mismo que BehaviorSubject?
R: Parecido, pero no: ReplaySubject no requiere valor inicial, no expone .value síncrono y sigue re-emitiendo su buffer incluso después de complete().

P: ¿En qué sentido un Subject es a la vez Observable y Observer?
R: Implementa ambas interfaces: puedes suscribirte a él (Observable) y puedes empujarle valores con next/error/complete (Observer), lo que lo hace ideal como puente multicast.

🔀 Operadores de Flattening

Transforman un Observable que emite Observables internos en un stream plano. La diferencia está en cómo manejan la concurrencia:

Operador Comportamiento Caso de uso
switchMap Cancela el inner anterior al recibir nuevo valor Búsqueda typeahead, autocomplete
mergeMap Ejecuta todos en paralelo Descargas paralelas, logs
concatMap Ejecuta en secuencia, espera que termine cada uno Guardar formularios, orden importa
exhaustMap Ignora nuevos valores mientras hay uno en progreso Login button, submit de formulario 
// switchMap — autocomplete: cancela búsqueda anterior
searchInput$.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.api.search(term))
).subscribe(results => this.results = results);

// exhaustMap — login: ignora clicks mientras se procesa
loginClick$.pipe(
  exhaustMap(() => this.auth.login(credentials))
).subscribe(user => this.router.navigate(['/home']));

🧠 Regla mental para elegir

  • switchMap → "solo me importa lo último": búsquedas, autocomplete, parámetros de ruta. Ideal para lecturas.
  • concatMap → "el orden importa": guardar cambios, colas de escritura. Ideal para escrituras secuenciales.
  • mergeMap → "todo en paralelo, el orden no importa": subir varios archivos, borrados en lote.
  • exhaustMap → "ignora hasta que termine lo actual": login, submit — evita el doble envío.

⚠️ Errores comunes

  • Usar switchMap para escrituras (POST/PUT/DELETE): si llega otro valor, la petición en vuelo se cancela en el cliente pero puede haber llegado al servidor — estado inconsistente. Usa concatMap o exhaustMap.
  • mergeMap sin límite de concurrencia sobre listas grandes → cientos de peticiones simultáneas. Usa el segundo argumento: mergeMap(fn, 4).
  • concatMap con un inner que nunca completa (p. ej. un Subject) → la cola se congela para siempre.
  • Anidar subscribe() dentro de subscribe() en vez de aplanar — pierdes cancelación, composición y control de errores.

🎤 Preguntas de entrevista

P: ¿Qué operador usarías en un typeahead y por qué?
R: switchMap: cancela la petición anterior y evita la race condition en la que una respuesta lenta y obsoleta pisa a la más reciente.

P: ¿Diferencia entre exhaustMap y throttleTime para evitar doble click?
R: throttleTime ignora por un tiempo fijo; exhaustMap ignora mientras la operación interna siga activa — se adapta a la duración real del trabajo.

🔗 Combining Operators

Combinan múltiples streams en uno solo:

  • combineLatest: Emite cuando cualquiera de los streams emite (después de que todos hayan emitido al menos una vez). Ideal para derivar estado combinado.
  • forkJoin: Espera a que todos completen y emite los últimos valores como array. Perfecto para llamadas HTTP paralelas.
  • withLatestFrom: Cuando el source emite, toma el valor más reciente de otro stream. Útil para enriquecer eventos con datos actuales.
// combineLatest — actualiza cuando cualquiera cambia
combineLatest([user$, config$]).pipe(
  map(([user, config]) => ({{ '{' }} ...user, theme: config.theme {{ '}' }}))
);

// forkJoin — espera todas las llamadas HTTP
forkJoin([
  this.http.get('/api/users'),
  this.http.get('/api/products'),
  this.http.get('/api/orders')
]).subscribe(([users, products, orders]) => {{ '{' }}
  // Los tres disponibles aquí
{{ '}' }});

// withLatestFrom — enriquece click con datos actuales
saveClick$.pipe(
  withLatestFrom(formData$),
  switchMap(([_, data]) => this.api.save(data))
);

🚨 Memory Leaks

Las suscripciones que no se cancelan siguen ejecutándose después de destruir el componente. Especialmente peligroso con interval(), fromEvent(), y WebSockets.

4 Patrones para evitarlos:

// 1. takeUntilDestroyed() — ✅ Recomendado en Angular 20
import {{ '{' }} takeUntilDestroyed {{ '}' }} from '@angular/core/rxjs-interop';

// En el constructor o campo del componente
interval(1000).pipe(
  takeUntilDestroyed()   // Se cancela automáticamente al destruir
).subscribe(n => console.log(n));

// 2. DestroyRef + callback manual
const destroyRef = inject(DestroyRef);
const sub = interval(1000).subscribe(n => console.log(n));
destroyRef.onDestroy(() => sub.unsubscribe());

// 3. async pipe en template — se cancela automáticamente
// <div>{{ '{{' }} data$ | async {{ '}}' }}</div>

// 4. toSignal() — Angular signals, se limpia solo
const count = toSignal(interval(1000), {{ '{' }} initialValue: 0 {{ '}' }});

Regla: Si no usas async pipe, toSignal(), o takeUntilDestroyed(), necesitas cancelar manualmente.

⚠️ Errores comunes

  • Creer que HttpClient "no necesita cleanup": completa solo, sí, pero si el componente se destruye antes de la respuesta el callback aún se ejecuta sobre un componente muerto.
  • Llamar takeUntilDestroyed() sin argumento fuera del injection context (dentro de ngOnInit o de un handler) → error en runtime. Pásale un DestroyRef inyectado.
  • Con el patrón clásico: olvidar el destroy$.next() en ngOnDestroy, o poner takeUntil antes del último operador del pipe.
  • fromEvent(window, 'scroll') sin cleanup: el listener retiene el componente entero en su closure — uno de los leaks más grandes y silenciosos.
  • Guardar suscripciones en variables sueltas "para luego" y nunca llamar unsubscribe(); si lo haces manual, usa un Subscription contenedor con .add().

🎤 Preguntas de entrevista

P: ¿Qué suscripciones necesitan unsubscribe manual?
R: Las de fuentes que no completan (interval, fromEvent, Subjects, store) suscritas en TypeScript sin async pipe, toSignal ni takeUntilDestroyed.

P: ¿Cómo detectarías un memory leak de RxJS en producción?
R: Síntomas: logs o peticiones que continúan tras destruir el componente, memoria creciente al navegar ida y vuelta. Diagnóstico: heap snapshots en DevTools buscando componentes retenidos por closures de suscripciones.

P: ¿Ventaja de takeUntilDestroyed sobre el patrón destroy$?
R: Elimina el boilerplate (Subject + ngOnDestroy) y los olvidos: se engancha a DestroyRef, así el cleanup queda garantizado por el framework.

🔧 Operadores Pipeables en Profundidad

¿Qué es un operador pipeable?

Es una función pura que recibe un Observable y retorna un nuevo Observable sin mutar el original. Se encadenan dentro de .pipe(), lo que permite composición funcional legible y tree-shakeable. 

import {{ '{' }} map, filter {{ '}' }} from 'rxjs/operators';

// Cada operador es una función: Observable → Observable
source$.pipe(
  filter(x => x > 0),   // Observable → Observable (filtra negativos)
  map(x => x * 10)      // Observable → Observable (multiplica)
);

// Equivale a:
const filtered$ = filter(x => x > 0)(source$);
const mapped$   = map(x => x * 10)(filtered$);

Categorías de operadores

Categoría Operadores Uso
Transformación map, switchMap, mergeMap, concatMap, exhaustMap, scan, reduce Transformar valores emitidos
Filtrado filter, take, takeUntil, takeWhile, first, last, skip, distinctUntilChanged, debounceTime, throttleTime Controlar qué valores pasan
Combinación combineLatest, forkJoin, withLatestFrom, merge, concat, zip, race Combinar múltiples streams
Creación of, from, interval, timer, fromEvent, EMPTY, NEVER, throwError Crear Observables
Utilidad tap, delay, finalize, timeout, retry, retryWhen, catchError Control y debugging
Multicasting share, shareReplay, publish, refCount Compartir suscripciones

🔄 Operadores de Transformación — Deep Dive

1. map — Transformación 1:1

Transforma cada valor emitido aplicando una función. Nunca cambia la cantidad de emisiones.

of(1, 2, 3).pipe(
  map(x => x * 2)
).subscribe(v => console.log(v));
// Output: 2, 4, 6

2. switchMap — ⭐ EL MÁS IMPORTANTE

Al recibir un nuevo valor, cancela la suscripción interna anterior y crea una nueva. Es el operador por defecto para búsquedas, autocomplete y navegación. 

Diagrama — switchMap cancela inner al recibir nuevo outer:

Source:  --A--------B--------C-->
Inner:   --1-2-3    --4-5    --6-7-8-->
Output:  --1-2------4-5------6-7-8-->
          ↑          ↑
          A's stream  B's stream cancelled
          cancelled   when C arrives
          when B arrives

Código completo de búsqueda con manejo de errores:

this.searchInput.valueChanges.pipe(
  debounceTime(300),            // espera 300ms de silencio
  distinctUntilChanged(),       // ignora si el valor no cambió
  filter(term => term.length >= 2), // mínimo 2 caracteres
  switchMap(term => this.api.search(term).pipe(
    catchError(() => of([]))    // error NO mata el stream externo
  ))
).subscribe(results => this.results = results);

3. mergeMap (flatMap) — Paralelo sin cancelar

Ejecuta TODAS las suscripciones internas en paralelo. Ninguna se cancela. Acepta un parámetro de concurrencia: mergeMap(fn, 3) — máximo 3 simultáneas. 

Diagrama — mergeMap mantiene todas las suscripciones:

Source:  --A--------B--------C-->
Inner:   --1-2-3    --4-5-6  --7-8-->
Output:  --1-2-3----4-5-6----7-8-->
          ↑          ↑        ↑
          All inner streams run to completion
// Descargar múltiples archivos en paralelo (máx 3 simultáneos)
fileIds$.pipe(
  mergeMap(id => this.http.get(`/api/files/${{ '{' }}id{{ '}' }}`), 3)
).subscribe(file => this.downloaded.push(file));

4. concatMap — Secuencial estricto

Espera a que la suscripción interna complete antes de procesar el siguiente valor. ⚠️ PELIGRO: si el inner Observable nunca completa, la cola se bloquea para siempre. 

Diagrama — concatMap espera completar antes de siguiente:

Source:  --A----B----C-->
Inner:   --1-2| --3-4| --5-6|
Output:  --1-2----3-4----5-6-->
          ↑      ↑
          B waits for A to complete
                 C waits for B to complete
// Guardar cambios en orden estricto
changes$.pipe(
  concatMap(change => this.api.save(change))
).subscribe(() => console.log('Guardado en orden'));

5. exhaustMap — Ignora mientras está ocupado

Mientras hay una suscripción interna activa, ignora completamente los nuevos valores del source. Perfecto para evitar doble-click en formularios y botones de login. 

Diagrama — exhaustMap descarta B y C:

Source:  --A--B--C--------D-->
Inner:   --1-2-3|         --4-5|
Output:  --1-2-3----------4-5-->
          ↑  ↑  ↑         ↑
          A  B  C ignored  D processed
          processing       (inner finished)
          active
// Login: ignora clicks adicionales mientras se procesa
this.loginBtn.click$.pipe(
  exhaustMap(() => this.auth.login(this.credentials))
).subscribe(user => this.router.navigate(['/home']));

6. scan — Acumulador progresivo

Como Array.reduce(), pero emite cada valor acumulado en vez de solo el final. Ideal para contadores, estado acumulado y running totals. 

// Contador de clicks
clicks$.pipe(
  scan((count) => count + 1, 0)
).subscribe(n => console.log(n));
// Output: 1, 2, 3, 4...

// Acumular lista de items
newItem$.pipe(
  scan((items, item) => [...items, item], [] as Item[])
).subscribe(allItems => this.items = allItems);

scan vs reduce

reduce acumula igual que scan, pero solo emite el resultado final cuando la fuente completa. Con streams infinitos (clicks, WebSocket, Subjects) reduce no emite jamás — usa scan. 

of(1, 2, 3, 4).pipe(scan((acc, n) => acc + n, 0));
// Emite: 1, 3, 6, 10 (cada paso intermedio)

of(1, 2, 3, 4).pipe(reduce((acc, n) => acc + n, 0));
// Emite: 10 (solo al completar la fuente)

7. Tabla de decisión: ¿Cuál usar?

Situación Operador Por qué
Búsqueda / autocomplete switchMap Cancela peticiones obsoletas
Login / Submit formulario exhaustMap Evita doble envío
Guardar cambios en orden concatMap Mantiene secuencia estricta
Descargas paralelas mergeMap Máxima velocidad, sin esperas
Acumular estado scan Emite progresivamente

⚠️ Errores comunes

  • Devolver un valor plano dentro de switchMap — el callback debe retornar un Observable, Promise o iterable.
  • Poner catchError fuera del inner Observable: el primer error mata el stream principal y, p. ej., el typeahead deja de responder para siempre.
  • Mutar el acumulador en scan — retorna siempre un objeto/array nuevo para no romper distinctUntilChanged ni change detection OnPush.
  • Usar reduce sobre un stream que nunca completa: no verás ninguna emisión.

🎤 Preguntas de entrevista

P: ¿Diferencia entre map y switchMap?
R: map transforma cada valor 1:1 de forma síncrona; switchMap proyecta cada valor a un Observable interno, se suscribe a él y cancela el anterior.

P: ¿Qué pasa si usas mergeMap en un buscador?
R: Las respuestas pueden llegar desordenadas y una búsqueda vieja puede sobrescribir a la última — la race condition clásica que switchMap evita cancelando.

P: ¿Cómo implementarías un contador de "no leídos" con RxJS?
R: notifications$.pipe(scan(acc => acc + 1, 0)), reseteando con un merge de eventos de lectura que emitan un valor de reset.

🔍 Operadores de Filtrado — Deep Dive

1. filter — Predicado básico

Solo deja pasar valores que cumplan la condición. Equivale a Array.filter() pero para streams.

numbers$.pipe(
  filter(n => n % 2 === 0)  // solo pares
)

2. debounceTime(ms) — Espera silencio

Espera X milisegundos de silencio antes de emitir el último valor. Si llega un nuevo valor antes de que pase el tiempo, reinicia el temporizador. Ideal para inputs de búsqueda.

3. throttleTime(ms) — Limita frecuencia

Emite el primer valor, luego ignora todo durante X milisegundos. Ideal para eventos de scroll y resize. 

Comparación debounceTime vs throttleTime:

Input:    --A-B-C------D-E----->
                 300ms gap

debounce: ----------C--------E->  (espera silencio, emite último)
throttle: --A--------D-------->   (emite primero, ignora el resto)

4. distinctUntilChanged — Solo si cambió

Solo emite si el valor es diferente al anterior. ⚠️ Cuidado con objetos: compara por referencia. Usa un comparador custom para objetos: 

// Primitivos — funciona directo
search$.pipe(distinctUntilChanged())

// Objetos — necesita comparador custom
user$.pipe(
  distinctUntilChanged((prev, curr) => prev.id === curr.id)
)

5. take(n) — Solo N valores

Emite los primeros n valores y luego completa el Observable. take(1) es muy común: "dame el primer valor y desuscríbete automáticamente". 

// take(1) — equivale a un "one-shot"
this.store.select(selectUser).pipe(
  take(1)  // obtiene valor actual y se desuscribe
).subscribe(user => console.log(user));

6. takeUntil(notifier$) — Patrón de cleanup clásico

Emite valores hasta que el notifier$ emite. Es el patrón tradicional para limpiar suscripciones:

private destroy$ = new Subject<void>();

ngOnInit() {{ '{' }}
  this.data$.pipe(
    takeUntil(this.destroy$)
  ).subscribe(data => this.data = data);

  this.events$.pipe(
    takeUntil(this.destroy$)
  ).subscribe(event => this.handle(event));
{{ '}' }}

ngOnDestroy() {{ '{' }}
  this.destroy$.next();
  this.destroy$.complete();
{{ '}' }}

7. takeUntilDestroyed() — Angular 16+ ✅ Recomendado

Reemplaza el patrón manual de takeUntil + Subject. Más limpio y menos propenso a errores:

import {{ '{' }} takeUntilDestroyed {{ '}' }} from '@angular/core/rxjs-interop';

// ✅ En campo de clase o constructor — inyecta DestroyRef automáticamente
data$ = this.store.select(selectData).pipe(takeUntilDestroyed());

// ✅ En ngOnInit — necesita DestroyRef explícito
private destroyRef = inject(DestroyRef);

ngOnInit() {{ '{' }}
  this.data$.pipe(
    takeUntilDestroyed(this.destroyRef)
  ).subscribe(d => this.process(d));
{{ '}' }}

8. first() y skip(n)

first() — Emite el primer valor que cumpla la condición (o error si no hay ninguno). skip(n) — Ignora los primeros n valores y emite a partir del siguiente. 

// first con condición — primer usuario admin
users$.pipe(
  first(user => user.role === 'admin')
).subscribe(admin => console.log(admin));
// ⚠️ Si no hay admin, lanza EmptyError

// skip — ignorar estado inicial
formValue$.pipe(
  skip(1)  // ignora el valor inicial del formulario
).subscribe(val => this.markDirty());

⚠️ Errores comunes

  • Poner debounceTime DESPUÉS de switchMap: debe ir antes para frenar las peticiones, no para retrasar las respuestas.
  • distinctUntilChanged() con objetos compara por referencia — pasa un comparador custom o compara claves primitivas.
  • first() sobre un stream que completa sin emitir lanza EmptyError; usa take(1) si el silencio es un caso válido.
  • Colocar takeUntil(destroy$) antes de un switchMap o shareReplay: la suscripción interna puede sobrevivir. Regla: takeUntil siempre al final del pipe.
  • Usar debounceTime donde tocaba throttleTime en scroll: con eventos continuos, debounce no emite nada hasta que el usuario para.

🎤 Preguntas de entrevista

P: ¿debounceTime vs throttleTime?
R: Debounce espera X ms de silencio y emite el último valor (inputs de búsqueda); throttle emite el primero y bloquea durante X ms (scroll, resize, clicks repetidos).

P: ¿Por qué takeUntil debe ir al final del pipe?
R: Los operadores posteriores crean nuevas suscripciones internas que el notifier ya no corta; ponerlo al final garantiza que toda la cadena se cierre.

P: ¿take(1) vs first()?
R: Ambos completan tras el primer valor, pero first() lanza EmptyError si la fuente completa sin emitir (y acepta predicado); take(1) completa en silencio.

🔗 Operadores de Combinación — Deep Dive

1. combineLatest — Último de cada fuente, reactivo

Emite un array con el último valor de CADA fuente cada vez que cualquiera emite. Solo empieza a emitir cuando TODOS han emitido al menos una vez. 

Diagrama — combineLatest:

a$:  --1----2-----------3-->
b$:  ----X------Y----------->
Out: ---[1,X]-[2,X]-[2,Y]-[3,Y]-->
     ↑         ↑     ↑     ↑
     Ambos     a$    b$    a$ emitió
     emitieron emitió emitió
// Combinar filtros de una tabla
combineLatest([
  this.searchTerm$,
  this.categoryFilter$,
  this.sortOrder$
]).pipe(
  debounceTime(100),  // evitar re-renders excesivos
  switchMap(([term, cat, sort]) =>
    this.api.getProducts({{ '{' }} search: term, category: cat, sort {{ '}' }})
  )
).subscribe(products => this.products = products);

2. forkJoin — Esperar que TODOS completen (como Promise.all)

Espera a que todos los Observables completen, luego emite un array (o objeto) con el último valor de cada uno. ⚠️ PELIGRO: Si un Observable nunca completa, forkJoin nunca emite. 

// Cargar datos iniciales en paralelo (con nombre de propiedades)
forkJoin({{ '{' }}
  users:    this.http.get<User[]>('/api/users'),
  products: this.http.get<Product[]>('/api/products'),
  config:   this.http.get<Config>('/api/config')
{{ '}' }}).subscribe(({{ '{' }} users, products, config {{ '}' }}) => {{ '{' }}
  this.users    = users;
  this.products = products;
  this.config   = config;
{{ '}' }});
// ✅ HttpClient completa después de emitir → forkJoin funciona perfecto
// ❌ No usar con BehaviorSubject (nunca completa automáticamente)

3. withLatestFrom — Enriquecer evento con datos

Cuando el source emite, toma el último valor de otro Observable. El segundo Observable NO trigger emisiones por sí solo. 

// Click de guardar + datos actuales del formulario
this.saveClick$.pipe(
  withLatestFrom(this.store.select(selectFormData)),
  map(([_, formData]) => formData),
  switchMap(data => this.api.save(data))
).subscribe(() => this.showSuccess());

4. merge, concat, zip, race

// merge — combina en un solo stream (el primero que emita pasa)
merge(click$, keypress$, gesture$).subscribe(event => handle(event));

// concat — en secuencia: primero todos de a$, luego todos de b$
concat(cachedData$, freshData$).subscribe(data => render(data));

// zip — parejas 1:1 (espera que ambos tengan nuevo valor)
zip(nombres$, apellidos$).pipe(
  map(([nombre, apellido]) => `${{ '{' }}nombre{{ '}' }} ${{ '{' }}apellido{{ '}' }}`)
);

// race — el primero en emitir gana, los otros se cancelan
race(httpResponse$, timeout$).subscribe(result => handle(result));

5. startWith — Valor inicial inmediato

Emite un valor síncrono antes que los de la fuente. Imprescindible con combineLatest (que exige que todas las fuentes hayan emitido) y para pintar estado inicial en el template sin esperar al primer evento. 

// Sin startWith, combineLatest no arranca hasta que
// el usuario toque TODOS los filtros:
combineLatest([
  this.search$.pipe(startWith('')),
  this.category$.pipe(startWith('all')),
  this.sort$.pipe(startWith('name'))
]).pipe(
  switchMap(([term, cat, sort]) => this.api.getProducts(term, cat, sort))
).subscribe(products => this.products = products);

Tabla de decisión: ¿Cuál combinador usar?

Necesito... Operador Comportamiento
Último valor de cada fuente, reactivo combineLatest Emite cuando cualquiera emite
Esperar que TODOS completen forkJoin Como Promise.all()
Enriquecer evento con dato actual withLatestFrom Solo source trigger emisión
Mezclar streams independientes merge El primero que emita pasa
Secuencia ordenada de streams concat Primero a$, luego b$
Parear valores 1:1 zip Espera par de ambos
El más rápido gana race Cancela los demás
Valor inicial antes de la fuente startWith Emisión síncrona inmediata

⚠️ Errores comunes

  • combineLatest no emite hasta que TODAS las fuentes hayan emitido — añade startWith a las opcionales.
  • forkJoin con fuentes que no completan (Subjects, selectores de store) → nunca emite. Aplica take(1) a cada una.
  • En forkJoin, si UNA fuente falla, todo el conjunto falla y pierdes las demás respuestas — captura por fuente: catchError(() => of(null)).
  • "Glitches" de combineLatest: si dos fuentes derivan del mismo origen, emite estados intermedios inconsistentes. Mitiga con debounceTime(0).
  • Confundir zip con combineLatest: zip empareja estrictamente 1:1 y bufferiza — puede acumular memoria si una fuente emite mucho más rápido que la otra.

🎤 Preguntas de entrevista

P: ¿combineLatest vs withLatestFrom?
R: En combineLatest cualquier fuente dispara una emisión; en withLatestFrom solo el source dispara y las demás aportan su último valor de forma pasiva.

P: ¿Equivalentes RxJS de Promise.all y Promise.race?
R: forkJoin y race. Y combineLatest no tiene equivalente en Promises: reacciona en el tiempo.

P: ¿Por qué tu forkJoin "no emite nada"?
R: Alguna fuente no completa (típicamente un Subject o un selector de store). forkJoin solo emite cuando todas completan.

🛡️ Manejo de Errores

En RxJS un error es terminal: cierra la suscripción y ningún operador vuelve a ejecutarse en ese stream. Por eso el manejo de errores no es un detalle estético — decide si tu stream sobrevive al primer fallo.

1. catchError — capturar y reemplazar

Intercepta el error y debe retornar un Observable de reemplazo (o relanzar):

this.http.get<Product[]>('/api/products').pipe(
  catchError(err => {
    if (err.status === 404) return of([]);   // valor por defecto
    if (err.status === 401) return EMPTY;    // completar sin emitir
    return throwError(() => err);            // relanzar hacia arriba
  })
)

2. EMPTY vs throwError vs of(fallback)

Estrategia Efecto Cuándo
of(fallback) Emite un valor por defecto y completa La UI debe seguir funcionando (lista vacía, cache)
EMPTY Completa sin emitir nada Silenciar el error: no hay nada que mostrar
throwError(() => err) Propaga el error aguas arriba Que lo maneje un interceptor, un effect o un handler global

3. ¿DÓNDE poner catchError? — La pregunta clave

// ❌ FUERA del switchMap: el primer error "completa" el stream
// externo y el buscador deja de responder PARA SIEMPRE
input$.pipe(
  switchMap(term => this.api.search(term)),
  catchError(() => of([]))
)

// ✅ DENTRO del switchMap: solo se reemplaza el inner fallido;
// el stream externo sigue vivo para la siguiente búsqueda
input$.pipe(
  switchMap(term => this.api.search(term).pipe(
    catchError(() => of([]))
  ))
)

4. retry con delay — backoff exponencial (RxJS 7.4+)

La API moderna de retry con objeto de configuración reemplaza al deprecado retryWhen:

this.http.get<Data>('/api/data').pipe(
  retry({
    count: 3,
    delay: (error, retryCount) => {
      // No reintentar errores de cliente (4xx)
      if (error.status >= 400 && error.status < 500) {
        return throwError(() => error);
      }
      return timer(Math.pow(2, retryCount) * 1000); // 2s, 4s, 8s
    }
  }),
  catchError(() => of(fallbackData))
)

5. finalize — cleanup garantizado

Se ejecuta al terminar el stream por cualquier vía: complete, error o unsubscribe. Es el finally de RxJS:

this.loading.set(true);
this.api.load().pipe(
  finalize(() => this.loading.set(false))  // spinner SIEMPRE se apaga
).subscribe(data => this.data.set(data));

⚠️ Errores comunes

  • catchError fuera del flattening operator en streams de larga vida — el stream muere al primer fallo (ver punto 3).
  • Olvidar que catchError debe RETORNAR un Observable: retornar undefined produce un segundo error.
  • retry sin filtrar el tipo de error: reintentar un 401 tres veces solo bloquea la cuenta más rápido.
  • Usar retryWhen en código nuevo — está deprecado; usa retry({ count, delay }).
  • Apagar spinners en el next en vez de en finalize: si hay error, el spinner se queda girando.

🎤 Preguntas de entrevista

P: ¿Qué le pasa a un Observable después de emitir un error?
R: Termina definitivamente: no habrá más next ni complete y la suscripción se cierra. Por eso los streams de UI deben aislar errores en el inner Observable.

P: ¿EMPTY vs NEVER vs throwError?
R: EMPTY completa de inmediato sin emitir; NEVER no emite ni completa jamás; throwError emite un error inmediatamente.

P: ¿Por qué en un typeahead el catchError va dentro del switchMap?
R: Para que el fallo de UNA petición no complete el stream externo del input: solo se sustituye el inner fallido y el buscador sigue funcionando.

📡 Multicasting y shareReplay

El problema: Cold vs Hot Observables

Tipo Comportamiento Ejemplo
Cold Cada suscripción ejecuta el productor desde cero HttpClient.get() — N suscripciones = N HTTP calls
Hot Todos los suscriptores comparten la misma fuente WebSocket, fromEvent(), Subject

share() — Convertir cold a hot

Comparte la suscripción entre todos los suscriptores activos. Cuando el último se desuscribe, la fuente se desconecta.

shareReplay(1) — Compartir + cachear

Comparte la suscripción y cachea el último valor para nuevos suscriptores tardíos. Es el operador más usado para cachear respuestas HTTP. 

// ❌ Sin shareReplay: 3 suscripciones = 3 HTTP calls
products$ = this.http.get<Product[]>('/api/products');

// ✅ Con shareReplay: 1 HTTP call, todos reciben el mismo resultado
products$ = this.http.get<Product[]>('/api/products').pipe(
  shareReplay({{ '{' }} bufferSize: 1, refCount: true {{ '}' }})
);

// template A:  {{ '{{' }} products$ | async {{ '}}' }}  → trigger HTTP
// template B:  {{ '{{' }} products$ | async {{ '}}' }}  → usa cache
// template C:  {{ '{{' }} products$ | async {{ '}}' }}  → usa cache

refCount: true vs false

  • refCount: true — cuando no hay suscriptores, desconecta la fuente. ✅ Recomendado para evitar memory leaks.
  • refCount: false (default) — cachea para siempre incluso sin suscriptores. ⚠️ Posible memory leak.

Patrón de caché con invalidación manual

private refresh$ = new Subject<void>();

products$ = this.refresh$.pipe(
  startWith(undefined),                               // carga inicial
  switchMap(() => this.http.get<Product[]>('/api/products')),
  shareReplay(1)
);

// Cuando se necesita invalidar el caché:
invalidateCache() {{ '{' }}
  this.refresh$.next();  // trigger nueva petición HTTP
{{ '}' }}

⚠️ Errores comunes

  • shareReplay(1) a secas equivale a refCount: false: sobre fuentes infinitas mantiene la suscripción viva para siempre → memory leak. Prefiere shareReplay({ bufferSize: 1, refCount: true }).
  • shareReplay cachea también el ERROR: los nuevos suscriptores reciben el error cacheado y nunca se reintenta. Maneja el error antes o re-crea el stream al invalidar.
  • Poner shareReplay en medio del pipe, antes de operadores por-suscriptor (takeUntil, map con estado): comparte lo que no debía compartirse. En servicios va al final del pipe.
  • Asumir que el async pipe evita peticiones duplicadas: cada | async es una suscripción independiente; sin multicasting, cada una dispara su HTTP.

🎤 Preguntas de entrevista

P: ¿share vs shareReplay?
R: Ambos multicastan una fuente; shareReplay además re-emite los últimos N valores a suscriptores tardíos. Con share, quien llega tarde no recibe nada hasta la siguiente emisión.

P: Tienes 3 async pipes sobre el mismo products$ y ves 3 peticiones HTTP. ¿Solución?
R: shareReplay({ bufferSize: 1, refCount: true }) en el servicio, o un único @if (products$ | async; as products) en el template.

P: ¿Qué hace exactamente refCount: true?
R: Cuenta suscriptores activos: al llegar el primero conecta con la fuente y al irse el último se desuscribe de ella, evitando que quede viva sin consumidores.

🏗️ Patrones RxJS en Angular

1. HTTP con retry y backoff exponencial

Reintentar peticiones fallidas con espera exponencial (1s, 2s, 4s...):

this.http.get(url).pipe(
  retry({{ '{' }}
    count: 3,
    delay: (error, retryCount) => timer(Math.pow(2, retryCount) * 1000)
  {{ '}' }}),
  catchError(err => {{ '{' }}
    console.error('Failed after 3 retries', err);
    return of(null);
  {{ '}' }})
)

2. Polling — Consultas periódicas

const polling$ = timer(0, 5000).pipe(  // inmediato + cada 5s
  switchMap(() => this.http.get('/api/status')),
  retry(3),
  share(),
  takeUntilDestroyed()
);

// Todos los suscriptores comparten el mismo polling

3. Prevención de race conditions

// switchMap cancela la petición anterior automáticamente
// Si el usuario navega rápido: /user/1 → /user/2 → /user/3
// Solo la última petición (/user/3) llega al subscribe
this.route.params.pipe(
  switchMap(params => this.api.getUser(params['id']))
).subscribe(user => this.user = user);

4. Loading state pattern

Manejar estados de carga, datos y error en un solo stream:

interface State<T> {{ '{' }}
  loading: boolean;
  data: T | null;
  error: boolean;
{{ '}' }}

const data$ = this.trigger$.pipe(
  switchMap(query => concat(
    of({{ '{' }} loading: true, data: null, error: false {{ '}' }}),
    this.api.search(query).pipe(
      map(data => ({{ '{' }} loading: false, data, error: false {{ '}' }})),
      catchError(() => of({{ '{' }} loading: false, data: null, error: true {{ '}' }}))
    )
  )),
  shareReplay(1)
);

// En template:
// @if (data$ | async; as state) {{ '{' }}
//   @if (state.loading) {{ '{' }} <spinner /> {{ '}' }}
//   @if (state.error) {{ '{' }} <error-msg /> {{ '}' }}
//   @if (state.data) {{ '{' }} <results [data]="state.data" /> {{ '}' }}
// {{ '}' }}

5. Estrategias de unsubscribe — Comparación

Estrategia Código Cuándo usar
takeUntil(destroy$) Subject + ngOnDestroy Enfoque clásico, pre-Angular 16
takeUntilDestroyed() inject(DestroyRef) ✅ Angular 16+ — preferido
async pipe template | async Suscripciones solo en template
toSignal() toSignal(obs$) Componentes basados en Signals
take(1) One-shot Cuando solo necesitas un valor

⚠️ Errores comunes

  • Polling con interval + mergeMap: si el servidor va lento, las peticiones se apilan. Usa timer(0, n) + switchMap (o exhaustMap si prefieres no cancelar).
  • Implementar el loading state con dos Subjects sueltos (loading$, error$) que se desincronizan — modela el estado como UN solo stream (patrón 4).
  • Reintentar con retry plano dentro de un polling: multiplica peticiones a un backend caído; combina backoff + límite.

P: El usuario navega rápido entre /user/1 → /user/2 → /user/3 y a veces ve datos del usuario equivocado. ¿Causa y solución?

P: ¿Cómo harías polling cada 5s que se pause al destruir el componente?
