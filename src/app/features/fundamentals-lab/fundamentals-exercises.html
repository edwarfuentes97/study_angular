1️⃣ Ejercicio: Directiva Custom — appTooltipCustom

Crea una directiva de atributo appTooltipCustom que muestre un tooltip al hacer hover sobre un elemento. Debe aceptar un @Input con el texto del tooltip. El tooltip debe aparecer como un pequeño popup encima del elemento.

✅ Criterios de aceptación

  • La directiva se llama [appTooltipCustom] y acepta un string como input
  • Al hacer mouseenter, se muestra un elemento tooltip posicionado sobre el host
  • Al hacer mouseleave, el tooltip desaparece
  • El tooltip tiene estilos propios (background oscuro, texto blanco, border-radius)
  • Usa inject(ElementRef) y Renderer2 para crear el tooltip dinámicamente
  • Bonus: Posiciona el tooltip arriba del elemento usando getBoundingClientRect()
💡 Tip: Usa Renderer2.createElement() y Renderer2.appendChild() para crear el tooltip. Guárdalo en una variable privada para poder removerlo en mouseleave. Usa position: absolute y calcula el top con getBoundingClientRect().

2️⃣ Ejercicio: Pipe Custom — timeAgo

Crea un pipe timeAgo que convierta un Date a texto relativo: "hace 5 min", "hace 2 horas", "hace 3 días". El pipe debe ser pure.

✅ Criterios de aceptación

  • El pipe se llama timeAgo y recibe un Date o string de fecha
  • Retorna texto en español: "hace X segundos", "hace X minutos", "hace X horas", "hace X días"
  • Maneja correctamente: segundos (<60s), minutos (<60m), horas (<24h), días
  • El pipe es pure (default)
  • Incluye un comentario explicando: ¿por qué NO debería ser impure? (rendimiento vs precisión)
  • Muestra en template al menos 3 fechas diferentes con el pipe aplicado
💡 Tip: Un pipe impure se re-ejecutaría en cada ciclo de Change Detection (potencialmente cientos de veces por segundo), lo cual es innecesario para mostrar "hace 5 min". Si necesitas actualización en tiempo real, usa un interval() + signal para cambiar la referencia del Date.

3️⃣ Ejercicio: Lifecycle Hooks — Mutación vs Reemplazo

Crea un componente padre-hijo. El padre pasa un objeto por @Input(). Implementa ngOnChanges en el hijo y documenta: ¿se detecta el cambio si mutas el objeto vs lo reemplazas?

✅ Criterios de aceptación

  • Componente padre con un objeto user = { name: 'Ana', age: 25 }
  • Componente hijo recibe @Input() user e implementa ngOnChanges
  • Botón "Mutar" que hace this.user.name = 'Nuevo' (misma referencia)
  • Botón "Reemplazar" que hace this.user = { ...this.user, name: 'Nuevo' } (nueva referencia)
  • Log visible mostrando si ngOnChanges se ejecutó o no en cada caso
  • Comentario con la conclusión: mutar NO trigger ngOnChanges, reemplazar SÍ
💡 Tip: ngOnChanges usa === para comparar — solo detecta cambios de referencia, no mutaciones internas. Por eso Angular promueve inmutabilidad. Con OnPush change detection, esto es aún más crítico.

4️⃣ Ejercicio: Content Projection — Multi-Slot Card

Crea un componente <app-card> con multi-slot content projection: header, body y footer, cada uno usando <ng-content select="...">.

✅ Criterios de aceptación

  • Componente <app-card> con 3 slots: .card-header, default (body), [card-footer]
  • Cada slot tiene estilos diferentes (header con background, footer con borde top)
  • Contenido que no matchea ningún selector va al slot default
  • Ejemplo de uso con los 3 slots poblados
  • Ejemplo de uso con solo 1-2 slots (los vacíos no deben generar espacio)
  • Bonus: Usa @if con una referencia para ocultar el header si no hay contenido proyectado
💡 Tip: Para detectar si hay contenido proyectado, puedes usar @ContentChild o verificar ngAfterContentInit. La alternativa moderna es usar contentChild() signal query.

5️⃣ Ejercicio: ViewEncapsulation — Las 3 Versiones

Crea 3 versiones del mismo componente, cada una con una ViewEncapsulation diferente: Emulated, ShadowDom y None. Aplica el mismo CSS a los 3 y observa las diferencias en el DOM inspector.

✅ Criterios de aceptación

  • 3 componentes: StyledEmulated, StyledShadow, StyledNone
  • Los 3 tienen exactamente el mismo template y CSS: p { color: red; font-weight: bold; }
  • Se renderizan lado a lado en la misma página
  • Documenta qué ves en el DOM inspector para cada uno
  • Explica por qué None afecta TODOS los <p> de la app
  • Explica cuándo usarías ShadowDom (Web Components, widgets embebidos)
💡 Tip: Abre Chrome DevTools → Elements y observa cómo Angular genera atributos _nghost y _ngcontent para Emulated, usa #shadow-root para ShadowDom, y no hace nada especial para None.

6️⃣ Ejercicio: DI Avanzado — InjectionToken con Jerarquía

Crea un InjectionToken<string> para API_URL. Provéelo a nivel raíz con un valor y a nivel de componente con otro valor. Verifica qué valor recibe cada componente.

✅ Criterios de aceptación

  • Crea API_URL = new InjectionToken<string>('API URL')
  • Provéelo en app.config.ts con valor 'https://api.global.com'
  • Crea un componente que lo provee localmente con 'https://api.local.com'
  • Componente padre muestra el valor global
  • Componente hijo (con provider local) muestra el valor local
  • Documenta: "El ElementInjector del hijo tiene prioridad sobre el ModuleInjector"
  • Bonus: Usa useFactory para crear el valor dinámicamente basado en el environment
💡 Tip: Angular resuelve DI subiendo por el árbol de injectors. Si encuentra el provider en el ElementInjector del componente, lo usa. Si no, sube al ModuleInjector. Esto permite override per-componente.
