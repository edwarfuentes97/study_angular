🟢 Angular Fundamentals — Conceptos Esenciales

Todo lo que un Senior Angular developer debe dominar. Estos temas aparecen constantemente en entrevistas técnicas.

A. Directivas — Los 3 Tipos

Las Directivas son clases que modifican elementos del DOM. Todo en Angular es una directiva bajo el capó — incluso los componentes.

1. Componentes (Directivas con Template)

El tipo más común. @Component hereda de @Directive y añade un template HTML.

@Component({
  selector: 'app-hero',
  template: '<h1>{{ name }}</h1>'
})
export class HeroComponent {
  name = 'Angular';
}

2. Directivas Estructurales — Modifican el DOM

Añaden, eliminan o manipulan elementos del DOM. Se reconocen por el prefijo * (sintaxis de microsintaxis) o el nuevo control flow.

  • Clásicas: *ngIf, *ngFor, *ngSwitch
  • Nuevo Control Flow (Angular 17+): @if, @for, @switch — built-in, mejor rendimiento, no requiere import

¿Cómo funciona internamente? Angular convierte *ngIf en un <ng-template> + usa ViewContainerRef para insertar/remover la vista:

// Esto:
<div *ngIf="show">Hola</div>

// Angular lo convierte internamente a:
<ng-template [ngIf]="show">
  <div>Hola</div>
</ng-template>

3. Directivas de Atributo — Modifican Apariencia/Comportamiento

No cambian la estructura del DOM, sino la apariencia o el comportamiento de un elemento existente.

  • ngClass — añade/remueve clases CSS dinámicamente
  • ngStyle — aplica estilos inline dinámicamente
  • Custom attribute directives (ej: appHighlight)

Directiva Custom — Attribute Directive

@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective {
  @Input() appHighlight = 'yellow';
  private el = inject(ElementRef);

  @HostListener('mouseenter') onEnter() {
    this.el.nativeElement.style.backgroundColor = this.appHighlight;
  }
  @HostListener('mouseleave') onLeave() {
    this.el.nativeElement.style.backgroundColor = '';
  }
}

// Uso: <p [appHighlight]="'cyan'">Hover me</p>

Directiva Custom — Structural Directive

@Directive({
  selector: '[appUnless]'
})
export class UnlessDirective {
  private templateRef = inject(TemplateRef<any>);
  private vcRef = inject(ViewContainerRef);
  private hasView = false;

  @Input() set appUnless(condition: boolean) {
    if (!condition && !this.hasView) {
      this.vcRef.createEmbeddedView(this.templateRef);
      this.hasView = true;
    } else if (condition && this.hasView) {
      this.vcRef.clear();
      this.hasView = false;
    }
  }
}

// Uso: <div *appUnless="isLoading">Contenido</div>

B. Pipes — Pure vs Impure

Los Pipes transforman datos en el template. Sintaxis: {{ value | pipeName:arg1:arg2 }}

Built-in Pipes

PipeUsoEjemplo
dateFormatear fechas{{ today | date:'dd/MM/yyyy' }}
currencyFormato moneda{{ price | currency:'USD' }}
percentFormato porcentaje{{ 0.85 | percent }} → 85%
decimalFormato numérico{{ 3.14159 | number:'1.2-2' }} → 3.14
uppercaseMayúsculas{{ 'hola' | uppercase }} → HOLA
lowercaseMinúsculas{{ 'HOLA' | lowercase }} → hola
titlecaseCapitalizar{{ 'hola mundo' | titlecase }} → Hola Mundo
jsonDebug objetos{{ obj | json }}
asyncSuscribir Observable/Promise{{ data$ | async }}
sliceSubcadena/subarray{{ items | slice:0:5 }}
keyvalueIterar objetos@for (item of obj | keyvalue; track item.key)

Pure Pipes (default) — Mejor Rendimiento

Solo se re-ejecutan cuando el valor de referencia del input cambia. Angular usa === para comparar.

  • ✅ Rendimiento óptimo — se ejecuta pocas veces
  • ⚠️ NO detecta cambios dentro de arrays/objetos mutados
  • Para arrays: crear nuevo array ([...arr, newItem]) para triggear el pipe
@Pipe({ name: 'filterBy' }) // pure: true por defecto
export class FilterByPipe implements PipeTransform {
  transform(items: string[], search: string): string[] {
    if (!search) return items;
    return items.filter(i => i.toLowerCase().includes(search.toLowerCase()));
  }
}

Impure Pipes (pure: false) — Cuidado con el Rendimiento

Se re-ejecutan en CADA ciclo de Change Detection. Necesarias cuando el array se muta in-place, pero son costosas.

@Pipe({ name: 'sortBy', pure: false })
export class SortByPipe implements PipeTransform {
  transform(items: any[], field: string): any[] {
    return [...items].sort((a, b) => a[field] > b[field] ? 1 : -1);
  }
}

⚡ Cuándo usar cada una

AspectoPureImpure
EjecuciónSolo si referencia cambiaCada ciclo de CD
Rendimiento✅ Excelente⚠️ Puede ser costoso
Detecta mutaciones❌ No✅ Sí
Uso idealTransformaciones simplesArrays mutados, filtros dinámicos

C. Ciclo de Vida de Componentes (Lifecycle Hooks)

Angular llama estos métodos en momentos específicos del ciclo de vida del componente.

Orden de Ejecución

constructor→ ngOnChanges→ ngOnInit→ ngDoCheck→ ngAfterContentInit→ ngAfterContentChecked→ ngAfterViewInit→ ngAfterViewChecked→ ngOnDestroy

constructor

Se ejecuta cuando Angular instancia la clase. Solo para inyección de dependencias. NO hacer lógica aquí — los @Input() aún no tienen valor.

ngOnChanges(changes: SimpleChanges)

Se ejecuta antes de ngOnInit y cada vez que un @Input() cambia. Recibe un objeto con valores previos y actuales.

ngOnChanges(changes: SimpleChanges) {
  if (changes['userId']) {
    console.log('Prev:', changes['userId'].previousValue);
    console.log('Curr:', changes['userId'].currentValue);
    console.log('First:', changes['userId'].firstChange);
  }
}

ngOnInit

Se ejecuta una sola vez después del primer ngOnChanges. Ideal para:

  • Llamadas HTTP iniciales
  • Setup de suscripciones
  • Lógica que depende de @Input() valores

ngDoCheck

Se ejecuta en cada ciclo de Change Detection. Peligroso para rendimiento. Solo usar para detección de cambios custom que Angular no detecta (ej: mutaciones profundas de objetos).

ngAfterContentInit / ngAfterContentChecked

Se ejecutan después de que Angular proyecta el contenido externo (<ng-content>) en el componente. Init se ejecuta una vez; Checked en cada CD.

ngAfterViewInit / ngAfterViewChecked

Se ejecutan después de que Angular inicializa las vistas hijas del componente. Aquí es seguro acceder a @ViewChild.

@ViewChild('myInput') inputRef!: ElementRef;

ngAfterViewInit() {
  // Ahora inputRef está disponible
  this.inputRef.nativeElement.focus();
}

ngOnDestroy

Cleanup obligatorio. Se ejecuta cuando el componente se destruye.

  • Cancelar suscripciones: subscription.unsubscribe()
  • Remover event listeners: removeEventListener()
  • Limpiar timers: clearInterval(), clearTimeout()

Alternativas Modernas (Angular 16+)

// DestroyRef — alternativa funcional a ngOnDestroy
export class MyComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    const sub = this.data$.subscribe(d => this.process(d));
    this.destroyRef.onDestroy(() => sub.unsubscribe());
  }
}

// afterNextRender — ejecutar código después del primer render (SSR-safe)
export class ChartComponent {
  constructor() {
    afterNextRender(() => {
      // Acceso seguro al DOM — solo se ejecuta en el browser
      this.initChart();
    });
  }
}

// afterRender — ejecutar después de CADA render
export class AnimComponent {
  constructor() {
    afterRender(() => {
      this.updateAnimation();
    });
  }
}

D. Content Projection (ng-content)

Mecanismo para pasar HTML del componente padre al hijo. Similar a "slots" en Vue/Web Components.

Single Slot

// card.component.html
<div class="card">
  <ng-content />
</div>

// Uso:
<app-card>
  <p>Este contenido se proyecta dentro del card</p>
</app-card>

Multi-Slot Projection

// panel.component.html
<div class="panel">
  <header><ng-content select=".panel-header" /></header>
  <main><ng-content /></main>
  <footer><ng-content select="[footer]" /></footer>
</div>

// Uso:
<app-panel>
  <div class="panel-header">Título</div>
  <p>Contenido principal (slot por defecto)</p>
  <div footer>Pie de página</div>
</app-panel>

ng-container vs ng-template

ElementoSe renderizaUso principal
<ng-container> ❌ No genera elemento DOM Agrupar directivas sin div extra. Ej: <ng-container *ngIf="x">
<ng-template> ❌ No se renderiza por defecto Definir templates reutilizables. Se instancia con *ngIf else, ngTemplateOutlet, o ViewContainerRef

ngTemplateOutlet — Renderizar Templates Dinámicamente

// Definir template
<ng-template #greetTpl let-name="name">
  <p>Hola, {{ name }}!</p>
</ng-template>

// Renderizar dinámicamente
<ng-container
  [ngTemplateOutlet]="greetTpl"
  [ngTemplateOutletContext]="{ name: 'Angular' }"
/>

// Uso avanzado: componente que acepta template custom
@Component({
  selector: 'app-list',
  template: \`
    @for (item of items; track item) {
      <ng-container
        [ngTemplateOutlet]="itemTpl"
        [ngTemplateOutletContext]="{ $implicit: item }"
      />
    }
  \`
})
export class ListComponent {
  @Input() items: any[] = [];
  @ContentChild(TemplateRef) itemTpl!: TemplateRef<any>;
}

Conditional Projection

// Proyección condicional: solo mostrar si hay contenido
<div class="card">
  @if (showHeader) {
    <header><ng-content select=".header" /></header>
  }
  <ng-content />
</div>

E. ViewEncapsulation (Shadow DOM)

Controla cómo Angular encapsula los estilos CSS de cada componente.

ModoMecanismoCSS AisladoCuándo usar
Emulated (default) Angular añade atributos _nghost-xxx y _ngcontent-xxx ✅ Sí (emulado) 99% de los casos — es el default por buena razón
ShadowDom Usa Shadow DOM nativo del navegador ✅ Sí (nativo) Web Components, aislamiento total. No todos los componentes lo soportan bien
None Sin encapsulación — estilos globales ❌ No Estilos globales de librería, overrides. Usar con cuidado 
import { ViewEncapsulation } from '@angular/core';

@Component({
  selector: 'app-styled',
  template: '<p>Styled component</p>',
  styles: ['p { color: red; }'],
  encapsulation: ViewEncapsulation.Emulated // default
})
export class StyledComponent {}

// Emulated genera en el DOM:
// <p _ngcontent-abc="">Styled component</p>
// CSS: p[_ngcontent-abc] { color: red; }

// ShadowDom genera:
// #shadow-root
//   <style>p { color: red; }</style>
//   <p>Styled component</p>

// None genera:
// <p>Styled component</p>
// CSS global: p { color: red; } (¡afecta TODOS los <p>!)

::ng-deep (Deprecated pero aún usado)

Permite que estilos de un componente padre penetren la encapsulación de componentes hijos. Deprecated pero sin reemplazo oficial aún.

// Rompe la encapsulación del componente hijo
:host ::ng-deep .child-class {
  color: blue;
}

// Alternativa: usar ViewEncapsulation.None en el hijo
// o CSS custom properties (variables) que sí atraviesan Shadow DOM

F. Dependency Injection (DI)

El sistema de DI de Angular es jerárquico. Resuelve dependencias subiendo por el árbol de injectors.

Injector Tree

Angular tiene dos árboles de injectors paralelos:

  • ModuleInjector — registros de providedIn y providers de módulos
  • ElementInjector — registros de providers en @Component/@Directive

Angular busca primero en ElementInjector, luego sube al ModuleInjector.

Formas de registrar un servicio

MétodoScopeTree-shakeable
providedIn: 'root'Singleton global✅ Sí
providedIn: 'any'Instancia por lazy module✅ Sí
providers: [] en @ComponentInstancia por componente❌ No
providers: [] en @NgModuleSingleton en ese module❌ No

Provider Types

// useClass — reemplazar implementación
providers: [{ provide: LoggerService, useClass: BetterLoggerService }]

// useValue — proveer un valor estático
providers: [{ provide: API_URL, useValue: 'https://api.example.com' }]

// useFactory — lógica condicional para crear el servicio
providers: [{
  provide: LoggerService,
  useFactory: (env: Environment) => env.production
    ? new ProdLogger() : new DevLogger(),
  deps: [Environment]
}]

// useExisting — alias a otro provider
providers: [{ provide: OldService, useExisting: NewService }]

InjectionToken — Para valores no-clase

// Crear token
export const API_URL = new InjectionToken<string>('API Base URL');

// Proveer
providers: [{ provide: API_URL, useValue: 'https://api.example.com' }]

// Inyectar
export class ApiService {
  private url = inject(API_URL);
  // o en constructor: constructor(@Inject(API_URL) private url: string)
}

Multi Providers

// Múltiples valores para el mismo token
export const HTTP_INTERCEPTORS = new InjectionToken<HttpInterceptorFn[]>('interceptors');

providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
]
// Inyecta un ARRAY con ambas instancias

inject() vs Constructor Injection

// Moderno (Angular 14+) — preferido
export class UserService {
  private http = inject(HttpClient);
  private router = inject(Router);
}

// Clásico — aún válido
export class UserService {
  constructor(
    private http: HttpClient,
    private router: Router
  ) {}
}

// inject() ventajas:
// ✅ Funciona en funciones (guards, interceptors, resolvers)
// ✅ No necesita constructor
// ✅ Más fácil de refactorizar
// ✅ Funciona con herencia sin super() complicado

G. Template Reference Variables & ViewChild

Acceder a elementos del DOM o instancias de componentes hijos desde la clase.

Template Reference Variables (#ref)

// En el template — acceso directo al elemento o componente
<input #nameInput type="text">
<button (click)="greet(nameInput.value)">Saludar</button>

// Si es un componente, #ref apunta a la instancia del componente
<app-timer #timer />
<button (click)="timer.start()">Start</button>

@ViewChild / @ViewChildren

// Acceder desde la clase del componente
@ViewChild('nameInput') inputRef!: ElementRef<HTMLInputElement>;
@ViewChild(TimerComponent) timer!: TimerComponent;
@ViewChildren(ItemComponent) items!: QueryList<ItemComponent>;

ngAfterViewInit() {
  this.inputRef.nativeElement.focus();
  this.timer.start();
  this.items.forEach(item => item.highlight());
}

ElementRef vs TemplateRef vs ViewContainerRef

ClaseQué representaUso principal
ElementRef Wrapper del elemento DOM nativo Acceso a nativeElement. ⚠️ No SSR-safe
TemplateRef Referencia a un <ng-template> Crear vistas embebidas dinámicamente
ViewContainerRef Punto de anclaje en el DOM Insertar/remover vistas y componentes dinámicamente

Signal-based Queries (Angular 17+)

// Alternativas modernas basadas en signals
export class MyComponent {
  // ViewChild → viewChild()
  nameInput = viewChild<ElementRef>('nameInput');
  timer = viewChild.required(TimerComponent);

  // ViewChildren → viewChildren()
  items = viewChildren(ItemComponent);

  // ContentChild → contentChild()
  header = contentChild<ElementRef>('.header');

  // ContentChildren → contentChildren()
  tabs = contentChildren(TabComponent);

  doSomething() {
    // Acceso via signal — siempre actualizado
    this.nameInput()?.nativeElement.focus();
    this.timer().start();
    this.items().forEach(i => i.highlight());
  }
}

H. Routing Avanzado

Route Params, Query Params & Fragments

// Definir ruta con parámetro
{ path: 'user/:id', component: UserComponent }

// Navegar
this.router.navigate(['/user', 42], {
  queryParams: { tab: 'profile' },
  fragment: 'bio'
});
// URL: /user/42?tab=profile#bio

ActivatedRoute — snapshot vs observable

export class UserComponent {
  private route = inject(ActivatedRoute);

  // Snapshot — valor estático (NO se actualiza si la ruta cambia sin destruir el componente)
  ngOnInit() {
    const id = this.route.snapshot.params['id'];
    const tab = this.route.snapshot.queryParams['tab'];
  }

  // Observable — reactivo (SE actualiza cuando cambian los params)
  ngOnInit() {
    this.route.params.subscribe(params => {
      this.loadUser(params['id']); // Se re-ejecuta al navegar /user/1 → /user/2
    });
  }
}
// ⚡ Usar observable cuando el mismo componente puede recibir diferentes params
// (ej: lista de usuarios con navegación entre ellos)

Resolvers — Pre-cargar datos

// Functional resolver (Angular 15+)
export const userResolver: ResolveFn<User> = (route) => {
  const userService = inject(UserService);
  return userService.getById(route.params['id']);
};

// En la ruta
{
  path: 'user/:id',
  component: UserDetailComponent,
  resolve: { user: userResolver }
}

// En el componente — datos ya disponibles
export class UserDetailComponent {
  private route = inject(ActivatedRoute);
  user = this.route.snapshot.data['user'] as User;
}

Router Events

export class AppComponent {
  private router = inject(Router);

  constructor() {
    this.router.events.pipe(
      filter(e => e instanceof NavigationStart)
    ).subscribe(event => {
      console.log('Navegando a:', event.url);
    });
  }
}
// Eventos: NavigationStart → NavigationEnd / NavigationCancel / NavigationError

Nested Routes & Route Data

{
  path: 'admin',
  component: AdminLayoutComponent,
  data: { role: 'admin' }, // datos estáticos
  children: [
    { path: '', component: AdminDashboard },
    { path: 'users', component: AdminUsers },
    { path: 'settings', component: AdminSettings },
  ]
}

Functional Guards (Angular 15+)

// Guard como función — reemplaza CanActivate class
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isLoggedIn()) return true;
  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url }
  });
};

// Uso en rutas
{ path: 'admin', canActivate: [authGuard], component: AdminComponent }

I. Modules vs Standalone

NgModule (Clásico — Angular 2-13)

@NgModule({
  declarations: [AppComponent, HeaderComponent], // componentes que PERTENECEN a este módulo
  imports: [BrowserModule, FormsModule],          // otros módulos necesarios
  exports: [HeaderComponent],                      // lo que otros módulos pueden usar
  providers: [AuthService],                        // servicios (scope: este módulo)
  bootstrap: [AppComponent]                        // componente raíz (solo en AppModule)
})
export class AppModule {}

// Bootstrap clásico
platformBrowserDynamic().bootstrapModule(AppModule);

Standalone (Angular 14+ — Default en Angular 20)

// En Angular 20, standalone es el default — no necesitas declararlo
@Component({
  selector: 'app-header',
  imports: [CommonModule, RouterModule], // importa directamente lo que necesita
  template: '<nav>...</nav>'
})
export class HeaderComponent {}

// Bootstrap standalone
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes),
    provideHttpClient(withInterceptors([authInterceptor])),
    provideAnimations(),
  ]
});

Migración NgModule → Standalone

AspectoNgModuleStandalone
Declaracióndeclarations en móduloComponente se auto-contiene
DependenciasMódulo importa otros módulosComponente importa lo que necesita
Providersproviders en móduloprovidedIn: 'root' o providers en componente
BootstrapplatformBrowserDynamic().bootstrapModule()bootstrapApplication()
Lazy loadingloadChildren: () => ModuleloadComponent o loadChildren: () => Routes
Tree shaking⚠️ Módulo entero✅ Componente individual

¿Por qué Angular 20 ya no necesita NgModule?

  • Standalone es el default — standalone: true ya no es necesario declarar
  • Mejor tree shaking — solo se incluye lo que se importa
  • Menos boilerplate — no más archivos .module.ts
  • Lazy loading más simple — loadComponent directamente
  • Providers funcionales: provideRouter(), provideHttpClient(), etc.

J. Decoradores Importantes

@Input() / @Output() — Comunicación Padre-Hijo

// Hijo recibe datos del padre
@Input() title = '';
@Input({ required: true }) userId!: string; // obligatorio
@Input({ transform: booleanAttribute }) disabled = false; // transforma "true" string → true boolean
@Input({ alias: 'label' }) buttonLabel = ''; // nombre diferente en el template

// Hijo emite eventos al padre
@Output() saved = new EventEmitter<User>();
this.saved.emit(user);

// En el padre:
// <app-child [title]="'Hola'" [userId]="id" (saved)="onSave($event)" />

Model Inputs — Two-Way Binding con Signals (Angular 17+)

// Hijo — model input (señal de lectura/escritura)
export class ToggleComponent {
  checked = model(false); // crea un WritableSignal

  toggle() {
    this.checked.update(v => !v);
  }
}

// Padre — two-way binding con banana-in-a-box
// <app-toggle [(checked)]="isActive" />
// isActive se actualiza automáticamente cuando el hijo cambia checked

@HostListener / @HostBinding

@Directive({ selector: '[appDropdown]' })
export class DropdownDirective {
  // Bindea una propiedad del elemento host
  @HostBinding('class.open') isOpen = false;

  // Escucha eventos del elemento host
  @HostListener('click') toggleOpen() {
    this.isOpen = !this.isOpen;
  }

  @HostListener('document:click', ['$event.target'])
  onDocumentClick(target: HTMLElement) {
    // Cerrar si click fuera del elemento
  }
}

host: property — Alternativa Moderna

// Preferido en Angular moderno — sin decoradores
@Component({
  selector: 'app-button',
  host: {
    '[class.active]': 'isActive',
    '[attr.aria-pressed]': 'isActive',
    '(click)': 'onClick()',
    'role': 'button',
    '[tabindex]': '0',
  },
  template: '<ng-content />'
})
export class ButtonComponent {
  isActive = false;
  onClick() { this.isActive = !this.isActive; }
}

@ContentChild / @ContentChildren

// Acceder al contenido proyectado (ng-content)
@ContentChild('headerRef') header!: ElementRef;
@ContentChildren(TabComponent) tabs!: QueryList<TabComponent>;

ngAfterContentInit() {
  // header y tabs están disponibles aquí
  console.log('Tabs:', this.tabs.length);
  this.tabs.changes.subscribe(() => {
    console.log('Tabs cambiaron:', this.tabs.length);
  });
}

// Signal alternatives (Angular 17+):
header = contentChild<ElementRef>('headerRef');
tabs = contentChildren(TabComponent);
