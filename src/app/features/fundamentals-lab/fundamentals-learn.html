🟢 Angular Fundamentals — Conceptos Esenciales

Todo lo que un Senior Angular developer debe dominar. Estos temas aparecen constantemente en entrevistas técnicas.

A. Directivas — Los 3 Tipos

Las Directivas son clases que modifican elementos del DOM. Todo en Angular es una directiva bajo el capó — incluso los componentes.

1. Componentes (Directivas con Template)

El tipo más común. @Component hereda de @Directive y añade un template HTML.

@Component({
  selector: 'app-hero',
  template: '<h1>{{ name }}</h1>'
})
export class HeroComponent {
  name = 'Angular';
}

2. Directivas Estructurales — Modifican el DOM

Añaden, eliminan o manipulan elementos del DOM. Se reconocen por el prefijo * (sintaxis de microsintaxis) o el nuevo control flow.

  • Clásicas: *ngIf, *ngFor, *ngSwitch
  • Nuevo Control Flow (Angular 17+): @if, @for, @switch — built-in, mejor rendimiento, no requiere import

¿Cómo funciona internamente? Angular convierte *ngIf en un <ng-template> + usa ViewContainerRef para insertar/remover la vista:

// Esto:
<div *ngIf="show">Hola</div>

// Angular lo convierte internamente a:
<ng-template [ngIf]="show">
  <div>Hola</div>
</ng-template>

3. Directivas de Atributo — Modifican Apariencia/Comportamiento

No cambian la estructura del DOM, sino la apariencia o el comportamiento de un elemento existente.

  • ngClass — añade/remueve clases CSS dinámicamente
  • ngStyle — aplica estilos inline dinámicamente
  • Custom attribute directives (ej: appHighlight)

Directiva Custom — Attribute Directive

@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective {
  @Input() appHighlight = 'yellow';
  private el = inject(ElementRef);

  @HostListener('mouseenter') onEnter() {
    this.el.nativeElement.style.backgroundColor = this.appHighlight;
  }
  @HostListener('mouseleave') onLeave() {
    this.el.nativeElement.style.backgroundColor = '';
  }
}

// Uso: <p [appHighlight]="'cyan'">Hover me</p>

Directiva Custom — Structural Directive

@Directive({
  selector: '[appUnless]'
})
export class UnlessDirective {
  private templateRef = inject(TemplateRef<any>);
  private vcRef = inject(ViewContainerRef);
  private hasView = false;

  @Input() set appUnless(condition: boolean) {
    if (!condition && !this.hasView) {
      this.vcRef.createEmbeddedView(this.templateRef);
      this.hasView = true;
    } else if (condition && this.hasView) {
      this.vcRef.clear();
      this.hasView = false;
    }
  }
}

// Uso: <div *appUnless="isLoading">Contenido</div>

B. Pipes — Pure vs Impure

Los Pipes transforman datos en el template. Sintaxis: {{ value | pipeName:arg1:arg2 }}

Built-in Pipes

PipeUsoEjemplo
dateFormatear fechas{{ today | date:'dd/MM/yyyy' }}
currencyFormato moneda{{ price | currency:'USD' }}
percentFormato porcentaje{{ 0.85 | percent }} → 85%
decimalFormato numérico{{ 3.14159 | number:'1.2-2' }} → 3.14
uppercaseMayúsculas{{ 'hola' | uppercase }} → HOLA
lowercaseMinúsculas{{ 'HOLA' | lowercase }} → hola
titlecaseCapitalizar{{ 'hola mundo' | titlecase }} → Hola Mundo
jsonDebug objetos{{ obj | json }}
asyncSuscribir Observable/Promise{{ data$ | async }}
sliceSubcadena/subarray{{ items | slice:0:5 }}
keyvalueIterar objetos@for (item of obj | keyvalue; track item.key)

Pure Pipes (default) — Mejor Rendimiento

Solo se re-ejecutan cuando el valor de referencia del input cambia. Angular usa === para comparar.

  • ✅ Rendimiento óptimo — se ejecuta pocas veces
  • ⚠️ NO detecta cambios dentro de arrays/objetos mutados
  • Para arrays: crear nuevo array ([...arr, newItem]) para triggear el pipe
@Pipe({ name: 'filterBy' }) // pure: true por defecto
export class FilterByPipe implements PipeTransform {
  transform(items: string[], search: string): string[] {
    if (!search) return items;
    return items.filter(i => i.toLowerCase().includes(search.toLowerCase()));
  }
}

Impure Pipes (pure: false) — Cuidado con el Rendimiento

Se re-ejecutan en CADA ciclo de Change Detection. Necesarias cuando el array se muta in-place, pero son costosas.

@Pipe({ name: 'sortBy', pure: false })
export class SortByPipe implements PipeTransform {
  transform(items: any[], field: string): any[] {
    return [...items].sort((a, b) => a[field] > b[field] ? 1 : -1);
  }
}

⚡ Cuándo usar cada una

AspectoPureImpure
EjecuciónSolo si referencia cambiaCada ciclo de CD
Rendimiento✅ Excelente⚠️ Puede ser costoso
Detecta mutaciones❌ No✅ Sí
Uso idealTransformaciones simplesArrays mutados, filtros dinámicos

C. Ciclo de Vida de Componentes (Lifecycle Hooks)

Angular llama estos métodos en momentos específicos del ciclo de vida del componente.

Orden de Ejecución

constructor→ ngOnChanges→ ngOnInit→ ngDoCheck→ ngAfterContentInit→ ngAfterContentChecked→ ngAfterViewInit→ ngAfterViewChecked→ ngOnDestroy

constructor

Se ejecuta cuando Angular instancia la clase. Solo para inyección de dependencias. NO hacer lógica aquí — los @Input() aún no tienen valor.

ngOnChanges(changes: SimpleChanges)

Se ejecuta antes de ngOnInit y cada vez que un @Input() cambia. Recibe un objeto con valores previos y actuales.

ngOnChanges(changes: SimpleChanges) {
  if (changes['userId']) {
    console.log('Prev:', changes['userId'].previousValue);
    console.log('Curr:', changes['userId'].currentValue);
    console.log('First:', changes['userId'].firstChange);
  }
}

ngOnInit

Se ejecuta una sola vez después del primer ngOnChanges. Ideal para:

  • Llamadas HTTP iniciales
  • Setup de suscripciones
  • Lógica que depende de @Input() valores

ngDoCheck

Se ejecuta en cada ciclo de Change Detection. Peligroso para rendimiento. Solo usar para detección de cambios custom que Angular no detecta (ej: mutaciones profundas de objetos).

ngAfterContentInit / ngAfterContentChecked

Se ejecutan después de que Angular proyecta el contenido externo (<ng-content>) en el componente. Init se ejecuta una vez; Checked en cada CD.

ngAfterViewInit / ngAfterViewChecked

Se ejecutan después de que Angular inicializa las vistas hijas del componente. Aquí es seguro acceder a @ViewChild.

@ViewChild('myInput') inputRef!: ElementRef;

ngAfterViewInit() {
  // Ahora inputRef está disponible
  this.inputRef.nativeElement.focus();
}

ngOnDestroy

Cleanup obligatorio. Se ejecuta cuando el componente se destruye.

  • Cancelar suscripciones: subscription.unsubscribe()
  • Remover event listeners: removeEventListener()
  • Limpiar timers: clearInterval(), clearTimeout()

Alternativas Modernas (Angular 16+)

// DestroyRef — alternativa funcional a ngOnDestroy
export class MyComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    const sub = this.data$.subscribe(d => this.process(d));
    this.destroyRef.onDestroy(() => sub.unsubscribe());
  }
}

// afterNextRender — ejecutar código después del primer render (SSR-safe)
export class ChartComponent {
  constructor() {
    afterNextRender(() => {
      // Acceso seguro al DOM — solo se ejecuta en el browser
      this.initChart();
    });
  }
}

// afterRender — ejecutar después de CADA render
export class AnimComponent {
  constructor() {
    afterRender(() => {
      this.updateAnimation();
    });
  }
}

D. Content Projection (ng-content)

Mecanismo para pasar HTML del componente padre al hijo. Similar a "slots" en Vue/Web Components.

Single Slot

// card.component.html
<div class="card">
  <ng-content />
</div>

// Uso:
<app-card>
  <p>Este contenido se proyecta dentro del card</p>
</app-card>

Multi-Slot Projection

// panel.component.html
<div class="panel">
  <header><ng-content select=".panel-header" /></header>
  <main><ng-content /></main>
  <footer><ng-content select="[footer]" /></footer>
</div>

// Uso:
<app-panel>
  <div class="panel-header">Título</div>
  <p>Contenido principal (slot por defecto)</p>
  <div footer>Pie de página</div>
</app-panel>

ng-container vs ng-template

ElementoSe renderizaUso principal
<ng-container> ❌ No genera elemento DOM Agrupar directivas sin div extra. Ej: <ng-container *ngIf="x">
<ng-template> ❌ No se renderiza por defecto Definir templates reutilizables. Se instancia con *ngIf else, ngTemplateOutlet, o ViewContainerRef

ngTemplateOutlet — Renderizar Templates Dinámicamente

// Definir template
<ng-template #greetTpl let-name="name">
  <p>Hola, {{ name }}!</p>
</ng-template>

// Renderizar dinámicamente
<ng-container
  [ngTemplateOutlet]="greetTpl"
  [ngTemplateOutletContext]="{ name: 'Angular' }"
/>

// Uso avanzado: componente que acepta template custom
@Component({
  selector: 'app-list',
  template: \`
    @for (item of items; track item) {
      <ng-container
        [ngTemplateOutlet]="itemTpl"
        [ngTemplateOutletContext]="{ $implicit: item }"
      />
    }
  \`
})
export class ListComponent {
  @Input() items: any[] = [];
  @ContentChild(TemplateRef) itemTpl!: TemplateRef<any>;
}

Conditional Projection

// Proyección condicional: solo mostrar si hay contenido
<div class="card">
  @if (showHeader) {
    <header><ng-content select=".header" /></header>
  }
  <ng-content />
</div>

E. ViewEncapsulation (Shadow DOM)

Controla cómo Angular encapsula los estilos CSS de cada componente.

ModoMecanismoCSS AisladoCuándo usar
Emulated (default) Angular añade atributos _nghost-xxx y _ngcontent-xxx ✅ Sí (emulado) 99% de los casos — es el default por buena razón
ShadowDom Usa Shadow DOM nativo del navegador ✅ Sí (nativo) Web Components, aislamiento total. No todos los componentes lo soportan bien
None Sin encapsulación — estilos globales ❌ No Estilos globales de librería, overrides. Usar con cuidado 
import { ViewEncapsulation } from '@angular/core';

@Component({
  selector: 'app-styled',
  template: '<p>Styled component</p>',
  styles: ['p { color: red; }'],
  encapsulation: ViewEncapsulation.Emulated // default
})
export class StyledComponent {}

// Emulated genera en el DOM:
// <p _ngcontent-abc="">Styled component</p>
// CSS: p[_ngcontent-abc] { color: red; }

// ShadowDom genera:
// #shadow-root
//   <style>p { color: red; }</style>
//   <p>Styled component</p>

// None genera:
// <p>Styled component</p>
// CSS global: p { color: red; } (¡afecta TODOS los <p>!)

::ng-deep (Deprecated pero aún usado)

Permite que estilos de un componente padre penetren la encapsulación de componentes hijos. Deprecated pero sin reemplazo oficial aún.

// Rompe la encapsulación del componente hijo
:host ::ng-deep .child-class {
  color: blue;
}

// Alternativa: usar ViewEncapsulation.None en el hijo
// o CSS custom properties (variables) que sí atraviesan Shadow DOM

F. Dependency Injection (DI) — En Profundidad

El sistema de DI de Angular es jerárquico, potente y a veces confuso. Dominar DI es clave para entrevistas senior.

¿Cómo funciona DI internamente?

Cuando Angular necesita crear un componente o servicio, mira sus dependencias (parámetros del constructor o inject()). Para cada dependencia, busca un provider registrado subiendo por el árbol de injectors:

  1. ElementInjector del componente actual (providers en @Component)
  2. ElementInjector del componente padre, abuelo, etc.
  3. ModuleInjector del módulo que declaró el componente
  4. ModuleInjector raíz (root) — donde viven los providedIn: 'root'
  5. PlatformInjector — nivel plataforma (raro, para multi-app)
  6. NullInjector — si llega aquí, lanza error (a menos que sea @Optional)
// Visualización del árbol de injectors:
//
// NullInjector (throws error)
//   └─ PlatformInjector
//       └─ RootModuleInjector (providedIn: 'root')
//           ├─ LazyModuleInjector A (providers del lazy module)
//           │   └─ ElementInjector ComponentA (providers en @Component)
//           │       └─ ElementInjector ChildComponent
//           └─ LazyModuleInjector B
//               └─ ElementInjector ComponentB

providedIn — Las opciones y cuándo usar cada una

ValorComportamientoTree-shakeableCuándo usar
providedIn: 'root' Singleton global — UNA instancia para toda la app ✅ Sí Opción por defecto. Auth, HTTP, config, state
providedIn: 'any' Una instancia por cada lazy-loaded module + una para eager modules ✅ Sí Servicios que necesitan estado aislado por feature
providedIn: 'platform' Compartido entre múltiples apps Angular en la misma página ✅ Sí Micro-frontends con shared state
providers: [] en @Component Nueva instancia por CADA instancia del componente ❌ No Estado por componente (ej: FormStateService)
providers: [] en ruta Singleton para esa ruta y sus hijos ❌ No Feature-level services (NgRx provideState)
providers: [] en @NgModule Singleton en ese module injector ❌ No Legacy — evitar en apps nuevas

⚠️ El problema del Singleton en Lazy Modules

Si un servicio se registra en providers de un lazy-loaded module, Angular crea un nuevo injector hijo, y ese servicio es un singleton SOLO dentro de ese module, NO el mismo que el del root. Esto causa bugs sutiles: 

// ❌ PROBLEMA: AuthService registrado en SharedModule
@NgModule({
  providers: [AuthService]  // ← cada lazy module que importe SharedModule
})                           //   obtiene su PROPIA instancia de AuthService
export class SharedModule {}

// ✅ SOLUCIÓN 1: providedIn: 'root' (preferido)
@Injectable({ providedIn: 'root' })
export class AuthService {}

// ✅ SOLUCIÓN 2: forRoot() pattern (legacy)
@NgModule({})
export class SharedModule {
  static forRoot(): ModuleWithProviders<SharedModule> {
    return {
      ngModule: SharedModule,
      providers: [AuthService]  // solo se registra en root
    };
  }
}

Provider Types — Todos los tipos en detalle

// === useClass — reemplazar implementación ===
// Útil para testing, o estrategias diferentes por entorno
providers: [
  { provide: LoggerService, useClass: environment.production 
    ? ProdLoggerService 
    : DevLoggerService }
]

// === useValue — proveer un valor estático ===
// Para configuración, tokens, constantes
providers: [
  { provide: API_URL, useValue: 'https://api.example.com' },
  { provide: IS_PRODUCTION, useValue: true },
  { provide: APP_CONFIG, useValue: { theme: 'dark', lang: 'es' } }
]

// === useFactory — lógica condicional ===
// Cuando necesitas lógica para crear el servicio
// deps: servicios que la factory necesita
providers: [{
  provide: StorageService,
  useFactory: (platform: Platform) => {
    if (platform.isBrowser) {
      return new LocalStorageService();
    } else {
      return new InMemoryStorageService();  // SSR
    }
  },
  deps: [Platform]
}]

// === useExisting — alias ===
// Redirige un token a otro provider existente
providers: [
  NewLoggerService,
  { provide: OldLoggerService, useExisting: NewLoggerService }
  // Cuando alguien pida OldLogger, recibe la instancia de NewLogger
]

InjectionToken — Para valores no-clase

Las clases sirven como su propio token de DI, pero para valores primitivos (strings, números, objetos de config) necesitas un InjectionToken: 

// Crear token tipado
export const API_URL = new InjectionToken<string>('API Base URL');
export const APP_CONFIG = new InjectionToken<AppConfig>('App Configuration');
export const FEATURE_FLAGS = new InjectionToken<FeatureFlags>('Feature Flags');

// Token con factory (se auto-provee, tree-shakeable)
export const WINDOW = new InjectionToken<Window>('Window Reference', {
  providedIn: 'root',
  factory: () => window  // valor por defecto
});

export const IS_BROWSER = new InjectionToken<boolean>('Is Browser', {
  providedIn: 'root',
  factory: () => typeof window !== 'undefined'
});

// Proveer en config
export const appConfig: ApplicationConfig = {
  providers: [
    { provide: API_URL, useValue: 'https://api.worldly.com' },
    { provide: APP_CONFIG, useValue: { theme: 'dark', maxRetries: 3 } },
  ]
};

// Inyectar
export class ApiService {
  private url = inject(API_URL);
  private config = inject(APP_CONFIG);
  private isBrowser = inject(IS_BROWSER);
}

Multi Providers — Arrays de servicios

// Cuando VARIOS providers deben registrarse bajo el mismo token
// Angular los inyecta como un ARRAY

export const APP_INITIALIZER = new InjectionToken<(() => void)[]>('App Init');

providers: [
  { provide: APP_INITIALIZER, useFactory: () => () => loadConfig(), multi: true },

// Resultado: [loadConfig, loadTranslations, initAnalytics]

  { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: ErrorInterceptor, multi: true },
]

inject() vs Constructor — Comparación completa

// ✅ MODERNO: inject() function (Angular 14+, preferido)
export class UserService {
  private http = inject(HttpClient);
  private router = inject(Router);
  private config = inject(APP_CONFIG);
  private logger = inject(LoggerService, { optional: true }); // no lanza error si no existe
}

// ✅ CLÁSICO: Constructor injection (aún válido)
export class UserService {
  constructor(
    private http: HttpClient,
    private router: Router,
    @Inject(APP_CONFIG) private config: AppConfig,
    @Optional() private logger: LoggerService // decorador para optional
  ) {}
}

// inject() funciona en más contextos:
// ✅ Functional guards
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn() || router.createUrlTree(['/login']);
};

// ✅ Functional interceptors
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).getToken();
  return next(req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }));
};

// ✅ Factory functions
export function createApiService(): ApiService {
  const http = inject(HttpClient);
  const url = inject(API_URL);
  return new ApiService(http, url);
}

Decoradores de resolución avanzados

// @Optional / { optional: true } — no lanza error si no se encuentra
private logger = inject(LoggerService, { optional: true });

// @Self / { self: true } — solo busca en el propio ElementInjector
private service = inject(MyService, { self: true });
// Útil para: asegurar que el servicio es una instancia local

// @SkipSelf / { skipSelf: true } — salta el propio injector, busca en padre
private parentService = inject(MyService, { skipSelf: true });
// Útil para: acceder al servicio del componente padre

// @Host / { host: true } — busca solo hasta el host component
private hostService = inject(MyService, { host: true });

// Combinaciones:
// "Quiero el del padre, pero si no existe, no pasa nada"
private parentOpt = inject(MyService, { skipSelf: true, optional: true });

Patrón: Component-level providers para estado local

// Servicio que maneja estado de un formulario
@Injectable() // NO providedIn: 'root' — se crea por componente
export class FormStateService {
  private state = signal<FormData>({ name: '', email: '' });
  
  readonly formState = this.state.asReadonly();
  
  updateField(field: string, value: string) {
    this.state.update(s => ({ ...s, [field]: value }));
  }
  
  reset() { this.state.set({ name: '', email: '' }); }
}

@Component({
  providers: [FormStateService], // ← nueva instancia POR CADA componente
  template: `...`
})
export class UserFormComponent {
  private formState = inject(FormStateService);
  // Cada instancia de UserFormComponent tiene su propio FormStateService
}

F2. Core Module vs Shared Module vs Feature Module

Aunque Angular 20 usa Standalone por defecto, estos conceptos arquitectónicos siguen siendo preguntas frecuentes en entrevistas y definen cómo organizar providers y código.

Core Module (ahora: carpeta core/)

Contiene servicios singleton que se usan en TODA la app. Se importa UNA sola vez en el root.

  • Auth service, API services, config service
  • HTTP interceptors
  • Route guards
  • Modelos de dominio globales
  • Regla: todo en core/ tiene providedIn: 'root' o se registra en app.config
// Patrón legacy: CoreModule con guard anti-reimportación
@NgModule({
  providers: [AuthService, ConfigService]
})
export class CoreModule {
  constructor(@Optional() @SkipSelf() parent: CoreModule) {
    if (parent) {
      throw new Error('CoreModule ya está cargado. Solo importar en AppModule.');
    }
  }
}

// Patrón moderno (Angular 20): NO necesita módulo
// Todo es providedIn: 'root' y se organiza por carpeta
// src/app/core/
//   services/auth.service.ts     → @Injectable({ providedIn: 'root' })
//   interceptors/auth.interceptor.ts → HttpInterceptorFn
//   guards/auth.guard.ts         → CanActivateFn

Shared Module (ahora: carpeta shared/)

Contiene componentes, directivas y pipes reutilizables y sin estado. Se puede importar en cualquier feature.

  • Componentes UI: Button, Card, Modal, Spinner, Table
  • Directivas: Highlight, Tooltip, Autosize
  • Pipes: FormatDate, FilterBy, TruncateText
  • Regla: NUNCA incluir servicios con estado en shared/
  • Regla: shared/ NO importa de core/ ni de features/
// Patrón legacy: SharedModule que re-exporta todo
@NgModule({
  declarations: [ButtonComponent, SpinnerComponent, FilterPipe],
  imports: [CommonModule, FormsModule],
  exports: [
    ButtonComponent, SpinnerComponent, FilterPipe,
    CommonModule, FormsModule  // re-exportar módulos comunes
  ]
  // ⚠️ NO poner providers aquí — causa múltiples instancias en lazy modules
})
export class SharedModule {}

// Patrón moderno: Standalone + barrel export
// shared/components/index.ts
export { ButtonComponent } from './button.component';
export { SpinnerComponent } from './spinner.component';

// Usar en cualquier componente:
import { ButtonComponent } from '../../shared/components';

@Component({
  imports: [ButtonComponent],  // import directo, sin módulo
  template: `<app-button label="Save" />`
})

Feature Module (ahora: carpeta features/)

Cada feature es autocontenida: sus componentes, servicios, rutas y estado. Se carga con lazy loading. 

// Patrón legacy: Feature NgModule
@NgModule({
  declarations: [ProductListComponent, ProductCardComponent],
  imports: [CommonModule, SharedModule, ProductRoutingModule],
  providers: [ProductService]
})
export class ProductModule {}

// Patrón moderno: Feature routes (sin módulo)
// features/products/products.routes.ts
export const PRODUCT_ROUTES: Routes = [{
  path: '',
  providers: [
    provideState('products', productReducer),  // state scoped a esta feature
    provideEffects(ProductEffects),
  ],
  loadComponent: () => import('./product-list').then(m => m.ProductListComponent),
  children: [...]
}];

// En app.routes.ts:
{ path: 'products', loadChildren: () => 
  import('./features/products/products.routes').then(m => m.PRODUCT_ROUTES) }

forRoot() / forChild() Pattern (legacy pero preguntado en entrevistas)

// Problema: SharedModule con providers se importa en múltiples lazy modules
// Cada lazy module crea su propia instancia del servicio (no singleton!)

// Solución: forRoot() pattern
@NgModule({
  declarations: [SpinnerComponent],
  exports: [SpinnerComponent]
})
export class SharedModule {
  // forRoot() — solo se llama en AppModule, registra servicios
  static forRoot(): ModuleWithProviders<SharedModule> {
    return {
      ngModule: SharedModule,
      providers: [NotificationService, ConfigService]
    };
  }
  
  // forChild() — se llama en feature modules, no registra servicios
  static forChild(): ModuleWithProviders<SharedModule> {
    return { ngModule: SharedModule };
  }
}

H. HttpClient & Llamadas API

Configuración

Angular usa HttpClient del paquete @angular/common/http. Se registra con provideHttpClient() en la configuración de la app: 

// app.config.ts
import { provideHttpClient, withInterceptors, withFetch } from '@angular/common/http';

export const appConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([authInterceptor, loggingInterceptor]),
      withFetch()  // usa fetch() API en vez de XMLHttpRequest
    ),
  ]
};

Métodos HTTP principales

HttpClient retorna Observables — son lazy, no se ejecutan hasta .subscribe() o | async:

import { HttpClient, HttpParams, HttpHeaders } from '@angular/common/http';

@Injectable({ providedIn: 'root' })
export class ProductService {
  private http = inject(HttpClient);
  private apiUrl = '/api/products';

  // GET — obtener lista
  getAll(): Observable<Product[]> {
    return this.http.get<Product[]>(this.apiUrl);
  }

  // GET con params — búsqueda/paginación
  search(term: string, page = 1): Observable<Product[]> {
    const params = new HttpParams()
      .set('q', term)
      .set('page', page)
      .set('limit', 20);
    return this.http.get<Product[]>(this.apiUrl, { params });
  }

  // GET por ID
  getById(id: string): Observable<Product> {
    return this.http.get<Product>(`${this.apiUrl}/${id}`);
  }

  // POST — crear
  create(product: Partial<Product>): Observable<Product> {
    return this.http.post<Product>(this.apiUrl, product);
  }

  // PUT — reemplazar completo
  update(id: string, product: Product): Observable<Product> {
    return this.http.put<Product>(`${this.apiUrl}/${id}`, product);
  }

  // PATCH — actualización parcial
  patch(id: string, changes: Partial<Product>): Observable<Product> {
    return this.http.patch<Product>(`${this.apiUrl}/${id}`, changes);
  }

  // DELETE
  delete(id: string): Observable<void> {
    return this.http.delete<void>(`${this.apiUrl}/${id}`);
  }
}

Manejo de errores

Los errores HTTP llegan como HttpErrorResponse. Se manejan con catchError de RxJS:

import { catchError, retry, throwError } from 'rxjs';
import { HttpErrorResponse } from '@angular/common/http';

getProducts(): Observable<Product[]> {
  return this.http.get<Product[]>(this.apiUrl).pipe(
    retry(2),  // reintentar 2 veces antes de fallar
    catchError((error: HttpErrorResponse) => {
      if (error.status === 0) {
        // Error de red (sin conexión)
        console.error('Network error:', error.error);
      } else if (error.status === 401) {
        // No autorizado — redirigir a login
        this.router.navigate(['/login']);
      } else if (error.status === 404) {
        // No encontrado
        console.error('Not found');
      } else {
        // Error del servidor (500, etc)
        console.error(`Server error ${error.status}:`, error.message);
      }
      return throwError(() => error);
    })
  );
}

Interceptors (funcionales, Angular 15+)

Los interceptors procesan TODAS las peticiones/respuestas HTTP en un pipeline:

// auth.interceptor.ts
import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).getToken();
  
  if (token) {
    req = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` }
    });
  }
  
  return next(req);
};

// logging.interceptor.ts — medir tiempo de respuesta
export const loggingInterceptor: HttpInterceptorFn = (req, next) => {
  const start = Date.now();
  return next(req).pipe(
    tap({
      next: () => console.log(`${req.method} ${req.url} — ${Date.now() - start}ms`),
      error: (err) => console.error(`${req.method} ${req.url} FAILED — ${err.status}`)
    })
  );
};

// error-handler.interceptor.ts — manejo global de errores
export const errorHandlerInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req).pipe(
    catchError((error: HttpErrorResponse) => {
      if (error.status === 401) {
        inject(Router).navigate(['/login']);
      }
      return throwError(() => error);
    })
  );
};

ActivatedRoute — snapshot vs observable

export class UserComponent {
  private route = inject(ActivatedRoute);

  // Snapshot — valor estático (NO se actualiza si la ruta cambia sin destruir el componente)
  ngOnInit() {
    const id = this.route.snapshot.params['id'];
    const tab = this.route.snapshot.queryParams['tab'];
  }

  // Observable — reactivo (SE actualiza cuando cambian los params)
  ngOnInit() {
    this.route.params.subscribe(params => {
      this.loadUser(params['id']); // Se re-ejecuta al navegar /user/1 → /user/2
    });
  }
}
// ⚡ Usar observable cuando el mismo componente puede recibir diferentes params

¿Por qué Angular 20 ya no necesita NgModule?

  • Standalone es el default — standalone: true ya no es necesario declarar
  • Mejor tree shaking — solo se incluye lo que se importa
  • Menos boilerplate — no más archivos .module.ts
  • Lazy loading más simple — loadComponent directamente
  • Providers funcionales: provideRouter(), provideHttpClient(), etc.
