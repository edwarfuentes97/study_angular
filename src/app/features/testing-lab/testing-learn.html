0. Pirámide de Testing

La pirámide de testing describe cómo distribuir el esfuerzo: muchos tests unitarios (rápidos, baratos, aislados), una capa media de tests de integración y pocos tests E2E (lentos, frágiles, pero los más cercanos al usuario real). En Angular cada nivel tiene herramientas y objetivos distintos. 

        /\
       /E2E\        ~10% — Playwright / Cypress
      /-----\       Flujos completos de usuario contra la app real
     / Integ.\      ~20% — TestBed + template + DI
    /---------\     Componente + template + servicios reales o parciales
   /   Unit    \    ~70% — Jasmine puro, sin DOM
  /-------------\   Pipes, reducers, selectors, validators, lógica de servicios

Qué testear en cada nivel:

  • Unit: funciones puras (reducers NgRx, selectors, pipes, validators), lógica de servicios con dependencias mockeadas. Sin DOM ni TestBed cuando sea posible.
  • Integración: componente + su template (render condicional, bindings, eventos), interacción componente ↔ servicio, formularios reactivos con el DOM.
  • E2E: los happy paths críticos del negocio (login, checkout, CRUD principal). No dupliques aquí lo que ya cubren los tests unitarios.

Variante moderna: el testing trophy prioriza tests de integración porque dan más confianza por línea de test. En Angular esto significa preferir tests con TestBed que ejercitan el template sobre tests que solo llaman métodos de la clase.

⚠️ Errores comunes

  • Invertir la pirámide: cientos de tests E2E lentos y casi ningún unitario → suite frágil que nadie mantiene.
  • Testear la misma lógica en los tres niveles: duplicación sin ganancia de confianza.
  • Medir calidad solo por cantidad de tests en vez de por comportamiento cubierto.

🎤 Preguntas de entrevista

  • ¿Por qué la base de la pirámide son los tests unitarios? — Porque son rápidos, deterministas y baratos de mantener; permiten feedback inmediato y localizan fallos con precisión.
  • ¿Cuándo elegirías un test de integración sobre uno unitario? — Cuando el valor está en la colaboración entre piezas (componente + template + DI), no en una función aislada.
  • ¿Qué NO deberías testear con E2E? — Casos borde y combinaciones exhaustivas; eso pertenece al nivel unitario. E2E cubre flujos críticos de usuario.

1. TestBed & ComponentFixture

TestBed es el entorno de testing de Angular: crea un módulo de prueba dinámico donde declaras imports y providers, y del que obtienes componentes reales renderizados. ComponentFixture es el "mango" del componente: da acceso a la instancia (componentInstance), al DOM (nativeElement), al árbol de debug (debugElement) y al ciclo de change detection (detectChanges()). 

describe('MyComponent', () => {
  let fixture: ComponentFixture<MyComponent>;
  let component: MyComponent;

  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [MyComponent],          // standalone: se importa, no se declara
      providers: [{ provide: ApiService, useValue: apiSpy }],
    }).compileComponents();

    fixture = TestBed.createComponent(MyComponent);
    component = fixture.componentInstance;
    fixture.detectChanges();           // primer render (ngOnInit + bindings)
  });

  it('should render title', () => {
    const el = fixture.nativeElement.querySelector('h1');
    expect(el.textContent).toContain('Mi Título');
  });
});

detectChanges vs autoDetectChanges

En tests, la change detection NO corre sola: cada vez que cambias estado debes llamar fixture.detectChanges() para sincronizar el DOM. Con fixture.autoDetectChanges() Angular detecta cambios automáticamente tras eventos y microtareas (útil en tests con signals y zoneless), a costa de menos control sobre cuándo se renderiza.

DebugElement vs nativeElement + By.css

nativeElement es el DOM real (HTMLElement): úsalo para leer texto y atributos. debugElement es la abstracción de Angular: permite consultar con By.css() / By.directive(), acceder al injector local y disparar eventos sin DOM real con triggerEventHandler. 

import { By } from '@angular/platform-browser';

// Consulta tipada por CSS
const btn = fixture.debugElement.query(By.css('[data-testid="increment-btn"]'));

// Dispara el (click) del template SIN necesitar un evento DOM real
btn.triggerEventHandler('click', null);
fixture.detectChanges();

// Consulta por directiva aplicada
const tooltip = fixture.debugElement.query(By.directive(TooltipDirective));

// Injector local del elemento: acceder a servicios provistos en el componente
const service = btn.injector.get(MyScopedService);

⚠️ Errores comunes

  • Olvidar fixture.detectChanges() tras cambiar estado → el DOM sigue mostrando el valor anterior y el test falla "misteriosamente".
  • Llamar detectChanges() en el beforeEach y luego sorprenderse de que ngOnInit ya se ejecutó antes de configurar el spy del test.
  • Usar nativeElement.click() esperando que también corra change detection: dispara el handler, pero el DOM no se actualiza hasta el próximo detectChanges().
  • Declarar el componente standalone en declarations en vez de imports → error de compilación del TestBed.

🎤 Preguntas de entrevista

  • ¿Qué hace compileComponents()? — Compila templates y estilos externos de forma asíncrona. Con el CLI (que precompila) suele ser opcional, pero es la convención segura.
  • ¿Diferencia entre debugElement y nativeElement? — nativeElement es el DOM crudo; debugElement envuelve el elemento con utilidades de Angular (query con predicados, injector, triggerEventHandler) y funciona incluso en renderers sin DOM.
  • ¿Por qué la change detection no es automática en tests? — Para que el test controle exactamente cuándo se renderiza y pueda asertar estados intermedios (loading, error, etc.).

2. fakeAsync & tick

fakeAsync ejecuta el test dentro de una zona con tiempo virtual: los setTimeout, intervals y promesas no corren de verdad, sino que TÚ avanzas el reloj. El test queda 100% síncrono y determinista — sin sleeps reales ni flakiness. 

it('should show loading and then products after delay', fakeAsync(() => {
  mockApiService.getProducts.and.returnValue(of(mockProducts).pipe(delay(800)));

  component.loadProducts();
  expect(component.loading()).toBeTrue();   // estado intermedio asertable

  tick(800);                                // avanza 800ms virtuales
  expect(component.loading()).toBeFalse();
  expect(component.products()).toEqual(mockProducts);
}));

La caja de herramientas async

  • tick(ms) — avanza el reloj virtual exactamente ms milisegundos. Ejecuta timers vencidos y microtareas.
  • flush() — avanza el reloj hasta vaciar TODOS los timers pendientes (útil cuando no conoces el delay exacto). Devuelve los ms avanzados.
  • flushMicrotasks() — resuelve solo microtareas (Promises) sin avanzar timers.
  • discardPeriodicTasks() — descarta intervals pendientes (ej. setInterval) para que el test no falle al salir de la zona.
  • waitForAsync + fixture.whenStable() — espera asíncrona REAL: el test espera a que la zona esté estable. Necesario cuando hay async que fakeAsync no puede controlar (ej. XHR real, APIs que usan tiempo nativo).
// flush: no importa cuántos timers anidados haya
it('completes all pending timers', fakeAsync(() => {
  component.startPolling();      // setTimeout dentro de setTimeout...
  flush();                       // vacía toda la cola de timers
  expect(component.done()).toBeTrue();
}));

// flushMicrotasks: solo Promises, el reloj no avanza
it('resolves promises without advancing timers', fakeAsync(() => {
  let value = 0;
  Promise.resolve(42).then(v => (value = v));
  flushMicrotasks();
  expect(value).toBe(42);
}));

// waitForAsync: async real, se usa con whenStable
it('waits for real async', waitForAsync(() => {
  fixture.whenStable().then(() => {
    fixture.detectChanges();
    expect(component.ready()).toBeTrue();
  });
}));

Regla práctica: usa fakeAsync por defecto (control total, test síncrono); waitForAsync solo cuando el código usa async que la zona fake no intercepta; y nunca mezcles ambos en el mismo test.

⚠️ Errores comunes

  • Error "1 timer(s) still in the queue" al terminar el test: quedó un timer sin consumir. Solución: flush() o discardPeriodicTasks() antes de salir.
  • Llamar tick() fuera de una zona fakeAsync → excepción inmediata.
  • Usar delay(0) creyendo que es síncrono: sigue siendo un timer, necesita tick(0).
  • Hacer llamadas HTTP reales dentro de fakeAsync: XHR no es "patcheable" por la zona fake; mockea el servicio o usa HttpTestingController.

🎤 Preguntas de entrevista

  • ¿Diferencia entre tick() y flush()? — tick avanza una cantidad exacta de tiempo; flush avanza lo necesario hasta vaciar la cola de timers.
  • ¿Cuándo usarías waitForAsync en lugar de fakeAsync? — Cuando hay asincronía real que la zona fake no controla (compilación, APIs nativas, librerías con timers propios).
  • ¿Qué son macrotareas y microtareas en este contexto? — Macrotareas: setTimeout/setInterval (las avanza tick/flush). Microtareas: Promises (las resuelve flushMicrotasks o cualquier tick).

3. Jasmine Spies

Los spies interceptan llamadas a funciones: registran cuántas veces se llamó, con qué argumentos, y opcionalmente reemplazan la implementación. Son la base del mocking en Angular + Jasmine.

Crear spies

// 1) spyOn: espía un método de un objeto EXISTENTE (lo reemplaza por defecto)
spyOn(service, 'getAll').and.returnValue(of(mockProducts));

// 2) createSpy: función suelta espiada
const callback = jasmine.createSpy('callback');

// 3) createSpyObj: objeto completo con métodos espiados — ideal para mocks de servicios
const apiSpy = jasmine.createSpyObj<MockApiService>('MockApiService',
  ['getProducts', 'getProduct', 'createProduct']);
apiSpy.getProducts.and.returnValue(of([]));  // tipado: SpyObj<T>

Controlar el comportamiento (.and)

spy.and.returnValue(of(data));        // siempre devuelve lo mismo
spy.and.returnValues(1, 2, 3);        // valores distintos por llamada
spy.and.callFake((id) => of(db[id])); // lógica personalizada
spy.and.callThrough();                // espía PERO ejecuta la impl. real
spy.and.throwError('boom');           // lanza excepción
spy.and.resolveTo(data);              // para métodos que devuelven Promise
spy.and.stub();                       // vuelve al comportamiento por defecto (nada)

Verificar llamadas (calls API)

expect(spy).toHaveBeenCalled();
expect(spy).toHaveBeenCalledTimes(2);
expect(spy).toHaveBeenCalledWith('42', jasmine.any(Object));
expect(spy).toHaveBeenCalledOnceWith('42');   // exactamente 1 vez y con esos args

spy.calls.count();                 // número de llamadas
spy.calls.mostRecent().args;       // args de la última llamada
spy.calls.first().args;            // args de la primera
spy.calls.argsFor(1);              // args de la llamada n
spy.calls.allArgs();               // args de todas
spy.calls.reset();                 // limpia el historial (útil entre fases del test)

Matchers útiles de Jasmine

expect(a).toBe(b);                 // igualdad por referencia (===)
expect(obj).toEqual({ id: 1 });    // igualdad estructural profunda
expect(list).toContain('item');
expect(fn).toThrowError(/not found/);
expect(0.1 + 0.2).toBeCloseTo(0.3, 5);
expect(value).toBeNull();  expect(value).toBeUndefined();

// Matchers asimétricos: aserciones parciales
expect(spy).toHaveBeenCalledWith(jasmine.objectContaining({ id: '1' }));
expect(result).toEqual(jasmine.arrayContaining([item1]));
expect(text).toEqual(jasmine.stringMatching(/^Error/));

⚠️ Errores comunes

  • Olvidar que spyOn reemplaza la implementación: si necesitas la real, añade .and.callThrough().
  • Configurar el spy DESPUÉS de que el componente ya llamó al método (típico con ngOnInit disparado por el detectChanges() del beforeEach).
  • Usar toBe con objetos: compara referencias, no contenido. Para objetos usa toEqual.
  • Un createSpyObj sin returnValue devuelve undefined → undefined.subscribe is not a function. Da siempre un valor por defecto.

🎤 Preguntas de entrevista

  • ¿Diferencia entre spyOn y createSpyObj? — spyOn parchea un método de una instancia real existente; createSpyObj fabrica un doble completo sin implementación real, ideal para sustituir el servicio entero vía DI.
  • ¿Qué hace callThrough? — Registra la llamada pero delega en la implementación original: observas sin alterar el comportamiento.
  • ¿Para qué sirven los matchers asimétricos? — Para asertar solo la parte relevante de un argumento (ej. objectContaining) sin acoplar el test a todo el objeto.

4. MockStore (NgRx)

provideMockStore sustituye el Store real: los componentes reciben un store falso donde defines el estado y el valor de cada selector, sin reducers ni effects reales. Así el test del componente queda aislado de la lógica de NgRx. 

import { MockStore, provideMockStore } from '@ngrx/store/testing';

let store: MockStore;

beforeEach(() => {
  TestBed.configureTestingModule({
    imports: [ProductListComponent],
    providers: [
      provideMockStore({
        initialState: { products: { entities: {}, loading: false } },
        selectors: [
          { selector: selectAllProducts, value: mockProducts },
          { selector: selectLoading, value: false },
        ],
      }),
    ],
  });
  store = TestBed.inject(MockStore);
});

Cambiar el estado durante el test

it('should show spinner while loading', () => {
  // 1) redefine el selector
  store.overrideSelector(selectLoading, true);
  // 2) fuerza a los subscribers a re-emitir
  store.refreshState();
  fixture.detectChanges();

  expect(fixture.nativeElement.querySelector('.spinner')).toBeTruthy();
});

it('should dispatch load action on init', () => {
  const dispatchSpy = spyOn(store, 'dispatch');
  component.ngOnInit();
  expect(dispatchSpy).toHaveBeenCalledWith(ProductActions.loadProducts());
});

afterEach(() => {
  store.resetSelectors();  // evita que overrides contaminen otros specs
});

También existe store.setState(nuevoEstado) para reemplazar el estado completo — útil si el componente usa store.select(fn) con funciones en vez de selectors memoizados.

⚠️ Errores comunes

  • Llamar overrideSelector y olvidar refreshState(): los observables ya suscritos no re-emiten y el DOM no cambia.
  • No llamar resetSelectors() en afterEach: los selectors memoizados quedan sobreescritos y contaminan los siguientes tests del archivo.
  • Testear reducers o effects a través del MockStore: MockStore no los ejecuta; testéalos por separado (ver card 10).
  • Asertar el dispatch con un objeto literal en vez de la action creator: acopla el test al string del type.

🎤 Preguntas de entrevista

  • ¿Por qué usar MockStore en vez del Store real? — Aísla el componente: el test no depende de reducers/effects, defines directamente los datos que el componente ve, y no necesitas providers de toda la feature.
  • ¿Qué hace overrideSelector internamente? — Sustituye el resultado memoizado del selector, por eso hace falta refreshState para propagar la nueva emisión.
  • ¿Cómo verificas que un componente despacha la acción correcta? — spyOn(store, 'dispatch') y asertar con toHaveBeenCalledWith(actionCreator(props)).

5. HttpTestingController

Permite testear servicios que usan HttpClient sin servidor: cada request queda "en vuelo" hasta que el test la intercepta y responde. Controlas la respuesta, el status y los errores, y verificas que no queden requests inesperadas. 

import { provideHttpClient } from '@angular/common/http';
import { provideHttpClientTesting, HttpTestingController }
  from '@angular/common/http/testing';

let service: ProductService;
let httpMock: HttpTestingController;

beforeEach(() => {
  TestBed.configureTestingModule({
    providers: [
      provideHttpClient(),         // el orden importa: primero el real...
      provideHttpClientTesting(),  // ...luego el testing backend que lo intercepta
      ProductService,
    ],
  });
  service = TestBed.inject(ProductService);
  httpMock = TestBed.inject(HttpTestingController);
});

afterEach(() => httpMock.verify());  // falla si quedaron requests sin responder

it('should fetch products', () => {
  let result: Product[] = [];
  service.getProducts().subscribe(p => (result = p));

  const req = httpMock.expectOne('/api/products');
  expect(req.request.method).toBe('GET');
  req.flush(mockProducts);           // responde la request pendiente

  expect(result.length).toBe(2);
});

expectOne, match, flush con status, errores

// expectOne con predicado: método + URL + params
const req = httpMock.expectOne(
  r => r.method === 'POST' && r.url === '/api/products'
);
expect(req.request.body).toEqual(newProduct);
expect(req.request.headers.get('Authorization')).toBe('Bearer token');

// match: varias requests a la misma URL (expectOne fallaría)
const reqs = httpMock.match('/api/products');
expect(reqs.length).toBe(2);
reqs.forEach(r => r.flush([]));

// flush con status personalizado
req.flush({ message: 'Created' }, { status: 201, statusText: 'Created' });

// Error HTTP (el servidor respondió con error)
req.flush('Not found', { status: 404, statusText: 'Not Found' });

// Error de red (la request nunca llegó)
req.error(new ProgressEvent('network error'));

⚠️ Errores comunes

  • Olvidar suscribirse al observable del servicio: HttpClient es lazy, sin subscribe no hay request y expectOne falla con "no matching request".
  • No llamar httpMock.verify() en afterEach: requests inesperadas pasan desapercibidas.
  • expectOne('/api/products') no matchea si el servicio añade query params: usa r => r.url === ... (url sin params) o incluye los params exactos.
  • Asertar dentro del subscribe sin capturar el valor fuera: si flush nunca ocurre, el expect nunca corre y el test pasa en falso.

🎤 Preguntas de entrevista

  • ¿Diferencia entre expectOne y match? — expectOne exige exactamente una request que coincida (falla con 0 o >1); match devuelve todas las coincidencias sin asertar cantidad.
  • ¿Cómo simulas un error 500 vs un fallo de red? — flush(body, {status: 500}) para respuesta de error del servidor; req.error(new ProgressEvent(...)) para fallo de red/transporte.
  • ¿Por qué verify() va en afterEach? — Garantiza que cada test declaró TODAS las requests que su código disparó; detecta llamadas HTTP accidentales.

6. Marble Testing

Los diagramas marble representan flujos RxJS como strings ASCII, permitiendo testear operadores y streams con tiempo virtual y precisión de frame. Con TestScheduler el tiempo no pasa de verdad: se calcula.

Sintaxis marble

' -a--b|'   //  - = 1 frame (1ms virtual en run mode)
            //  a,b = emisiones (next)
            //  | = complete       # = error
            //  () = emisiones síncronas agrupadas: '(ab|)' emite a, b y completa
            //  ^ = punto de suscripción (hot)   ! = unsubscribe
            //  '100ms a' = sintaxis de tiempo explícito
import { TestScheduler } from 'rxjs/testing';

let scheduler: TestScheduler;

beforeEach(() => {
  scheduler = new TestScheduler((actual, expected) => {
    expect(actual).toEqual(expected);   // conecta con Jasmine
  });
});

it('should debounce and map values', () => {
  scheduler.run(({ cold, hot, expectObservable, expectSubscriptions }) => {
    const source = cold('-a--b--c---|', { a: 1, b: 2, c: 3 });
    const expected =    '-x--y--z---|';

    const result = source.pipe(map(v => v * 10));

    expectObservable(result).toBe(expected, { x: 10, y: 20, z: 30 });
    expectSubscriptions(source.subscriptions).toBe('^----------!');
  });
});

cold vs hot

  • cold('--a|') — el observable empieza a emitir cuando alguien se suscribe (como HttpClient). Cada subscriber recibe la secuencia completa.
  • hot('--^-a|') — el stream ya está "vivo"; ^ marca cuándo se suscribe el test. Los valores anteriores a ^ se pierden (como eventos de usuario o un Subject).

Dentro de scheduler.run(), operadores basados en tiempo (debounceTime, delay, interval) usan el reloj virtual automáticamente — un debounceTime(300) se testea en microsegundos.

⚠️ Errores comunes

  • Crear el TestScheduler sin la función de aserción → los tests "pasan" aunque los marbles no coincidan.
  • Contar mal los frames: cada carácter es 1 frame, y la emisión también consume un frame. '-a-b': a en frame 1, b en frame 3.
  • Usar cold cuando el comportamiento depende de suscripción tardía (multicast): necesitas hot con ^.
  • Reutilizar el mismo TestScheduler entre tests sin recrearlo en beforeEach: el tiempo virtual queda contaminado.

🎤 Preguntas de entrevista

  • ¿Qué ventaja da marble testing sobre subscribe + done? — Verifica la secuencia COMPLETA con su timing exacto (valores, orden, tiempo, complete/error) de forma síncrona y sin flakiness.
  • ¿Diferencia entre cold y hot en un test? — cold emite desde el principio para cada subscriber; hot emite independientemente de los subscribers y ^ define cuándo entra el test.
  • ¿Cómo testeas un debounceTime(300) sin esperar 300ms? — Dentro de scheduler.run() el operador usa el scheduler virtual: el tiempo se simula, no transcurre.

7. Testing de Componentes con Signals e Inputs

Los componentes modernos usan signal() para estado local e input() para datos de entrada. Los signals se leen como funciones (component.counter()) y los inputs se establecen desde el test con fixture.componentRef.setInput() — nunca por asignación directa. 

// Componente bajo test
export class ProductCardComponent {
  readonly product = input.required<Product>();   // InputSignal: solo lectura
  readonly highlighted = input(false);
  readonly discounted = computed(() => this.product().price < 50);
}

// Test
it('should compute discount from input', () => {
  const fixture = TestBed.createComponent(ProductCardComponent);

  // ✅ la única forma correcta de setear un input signal
  fixture.componentRef.setInput('product', { id: '1', name: 'Tee', price: 29.99 });
  fixture.componentRef.setInput('highlighted', true);
  fixture.detectChanges();

  expect(fixture.componentInstance.discounted()).toBeTrue();
  const el = fixture.nativeElement.querySelector('.badge');
  expect(el.textContent).toContain('Oferta');
});

setInput funciona igual para @Input() clásicos y para input() signals, y además marca el componente para check (importante con OnPush). Para probar estado local basta con llamar los métodos públicos y leer el signal, como hace este lab: 

it('should increment the counter', () => {
  component.increment();
  expect(component.counter()).toBe(1);   // leer un signal = invocarlo
});

it('should display the counter value in the template', () => {
  component.increment();
  fixture.detectChanges();               // sincroniza signal → DOM
  const el = fixture.nativeElement.querySelector('[data-testid="counter-value"]');
  expect(el.textContent?.trim()).toBe('1');
});

Alternativa para output() y proyección de contenido: el host component pattern (ver card 11) — un componente wrapper definido en el spec que usa el componente real en su template.

⚠️ Errores comunes

  • Asignar directamente component.product = value con input signals → error en runtime: InputSignal no es escribible.
  • Olvidar los paréntesis al leer: expect(component.counter).toBe(1) compara la función, no el valor.
  • No llamar detectChanges() tras setInput: el computed se recalcula lazy al leerlo, pero el DOM no se actualiza solo.
  • Testear un input.required sin setearlo → error NG0950: required input is accessed before a value is set.

🎤 Preguntas de entrevista

  • ¿Por qué setInput y no asignación directa? — Respeta el ciclo de inputs de Angular (transforms, marcado OnPush, ngOnChanges en inputs clásicos) y es la única vía para InputSignal, que es read-only.
  • ¿Cómo testeas un computed()? — Seteas sus dependencias (signals/inputs) y lees el computed: al ser derivación pura no necesita DOM.
  • ¿Los signals necesitan fakeAsync? — No: escribir y leer signals es síncrono. Solo la actualización del DOM requiere detectChanges.

8. Testing de Servicios

Dos estrategias: aislada (instanciar la clase a mano, sin Angular) y con TestBed (usar el injector real). La aislada es más rápida y simple; TestBed es necesario cuando el servicio usa inject() en field initializers o depende del árbol de DI. 

// A) Aislado — constructor injection, sin TestBed
it('should compute totals (isolated)', () => {
  const logger = jasmine.createSpyObj<LoggerService>('Logger', ['log']);
  const service = new CartService(logger);   // new directo

  service.add({ id: '1', price: 10 });
  expect(service.total()).toBe(10);
  expect(logger.log).toHaveBeenCalled();
});
// B) Con TestBed — necesario si el servicio usa inject()
describe('ProductService', () => {
  let service: ProductService;
  let apiSpy: jasmine.SpyObj<MockApiService>;

  beforeEach(() => {
    apiSpy = jasmine.createSpyObj<MockApiService>('MockApiService', ['getProducts']);

    TestBed.configureTestingModule({
      providers: [
        ProductService,
        { provide: MockApiService, useValue: apiSpy },
      ],
    });
    service = TestBed.inject(ProductService);  // resuelve TODO el árbol de deps
  });

  it('should filter sustainable products', () => {
    apiSpy.getProducts.and.returnValue(of(mockProducts));

    let result: Product[] = [];
    service.getSustainable(80).subscribe(p => (result = p));

    expect(result.every(p => p.sustainabilityScore >= 80)).toBeTrue();
  });
});

Los servicios providedIn: 'root' no necesitan declararse en providers — TestBed los resuelve — pero SÍ debes sobreescribir sus dependencias si quieres aislarlos. Para código que usa inject() fuera de un contexto de construcción, existe TestBed.runInInjectionContext(() => ...).

⚠️ Errores comunes

  • Instanciar con new un servicio que usa inject() → NG0203: inject() must be called from an injection context.
  • Testear el servicio a través del componente: si falla, no sabes qué capa rompió. Testéalos por separado.
  • Compartir estado entre tests con servicios singleton: TestBed se resetea entre specs, pero variables module-scope del spec no.
  • Mockear TODO, incluidas funciones puras del propio servicio: termina testeando los mocks, no el servicio.

🎤 Preguntas de entrevista

  • ¿Cuándo usar TestBed.inject vs new? — new para clases con constructor injection y lógica pura (más rápido); TestBed cuando hay inject(), providers jerárquicos o interceptors involucrados.
  • ¿Cómo testeas un servicio providedIn: 'root' con dependencias? — Lo obtienes con TestBed.inject y sustituyes sus dependencias con { provide: Dep, useValue: spy }.
  • ¿Qué es un injection context? — El ámbito donde inject() es válido: constructores/inicializadores de clases DI, factories de providers, o runInInjectionContext.

9. Mocking de Dependencias

El injector de Angular hace el mocking trivial: en el TestBed sustituyes cualquier token por un doble. Las tres formas principales son useValue, useClass y useFactory, más spyOn sobre la instancia real cuando solo quieres interceptar un método. 

// useValue: objeto/spy concreto — el caso más común
{ provide: MockApiService, useValue: jasmine.createSpyObj('Api', ['getProducts']) }

// useClass: implementación fake completa con estado propio
class FakeAuthService {
  private loggedIn = false;
  login() { this.loggedIn = true; return of(true); }
  isAuthenticated() { return this.loggedIn; }
}
{ provide: AuthService, useClass: FakeAuthService }

// useFactory: cuando el mock depende de otro token
{ provide: ConfigService,
  useFactory: (env: EnvService) => new FakeConfig(env.name),
  deps: [EnvService] }

// spyOn sobre la instancia REAL: mínima intervención
const service = TestBed.inject(ProductService);
spyOn(service, 'getAll').and.returnValue(of(mockProducts));

¿Spy parcial o mock completo?

  • spyOn (parcial): el resto del servicio sigue siendo real. Bueno para interceptar un solo método; peligroso si el constructor real tiene side effects (HTTP, timers).
  • createSpyObj / useValue (completo): aislamiento total y sin side effects; requiere definir el comportamiento de cada método usado.
  • useClass (fake con lógica): ideal cuando el doble necesita estado coherente entre llamadas (ej. un fake de storage o auth).

Herramientas extra: TestBed.overrideProvider() para reemplazar providers después de configurar (gana incluso sobre providers del propio componente) y TestBed.overrideComponent() para tocar template/imports del componente bajo test. Evita NO_ERRORS_SCHEMA: silencia errores de template y esconde typos en bindings.

⚠️ Errores comunes

  • Mutar el objeto de useValue entre tests sin restaurarlo: los specs se contaminan entre sí. Recréalo en beforeEach.
  • Mockear lo que no posees en exceso (ej. HttpClient a mano): usa las utilidades oficiales (HttpTestingController, provideMockStore).
  • Que el mock se desvíe del contrato real y el test pase mientras producción falla: tipa los dobles (jasmine.SpyObj<T>) para que TypeScript vigile las firmas.
  • Providers a nivel de componente (providers: [] en el decorador) que ignoran tu mock del TestBed: usa overrideComponent u overrideProvider.

🎤 Preguntas de entrevista

  • ¿Cuándo useValue vs useClass? — useValue para dobles simples/spies preconstruidos; useClass cuando el fake necesita comportamiento y estado propios instanciados por el injector.
  • ¿Por qué tipar los spies con SpyObj<T>? — El compilador verifica que el mock respeta la firma real: si el servicio cambia, el test no compila en vez de fallar en runtime (o peor, pasar).
  • ¿Qué riesgo tiene NO_ERRORS_SCHEMA? — Silencia elementos y bindings desconocidos: un typo en un input deja de ser error y el test da falsa confianza.

10. Testing de NgRx: Reducers, Selectors y Effects

Cada pieza de NgRx se testea a su nivel: reducers y selectors son funciones puras (tests unitarios triviales, sin TestBed); los effects son streams (se testean con provideMockActions y marbles o suscripción directa).

Reducers: estado + acción = nuevo estado

describe('productReducer', () => {
  it('should set loading on loadProducts', () => {
    const state = productReducer(initialState, ProductActions.loadProducts());
    expect(state.loading).toBeTrue();
    expect(state.error).toBeNull();
  });

  it('should populate products on success', () => {
    const state = productReducer(
      { ...initialState, loading: true },
      ProductActions.loadProductsSuccess({ products: mockProducts })
    );
    expect(state.products).toEqual(mockProducts);
    expect(state.loading).toBeFalse();
  });

  it('should return same state for unknown action', () => {
    const state = productReducer(initialState, { type: 'NOOP' } as any);
    expect(state).toBe(initialState);   // misma referencia = inmutabilidad OK
  });
});

Selectors: usa projector()

it('should select products above sustainability threshold', () => {
  // projector salta la memoización y el estado global:
  // pasas directamente las ENTRADAS del selector
  const result = selectHighSustainability.projector(mockProducts, 80);
  expect(result.length).toBe(1);
  expect(result[0].sustainabilityScore).toBeGreaterThanOrEqual(80);
});

Effects: provideMockActions

import { provideMockActions } from '@ngrx/effects/testing';

describe('ProductEffects', () => {
  let actions$: Observable<Action>;
  let effects: ProductEffects;
  let apiSpy: jasmine.SpyObj<MockApiService>;

  beforeEach(() => {
    apiSpy = jasmine.createSpyObj('Api', ['getProducts']);
    TestBed.configureTestingModule({
      providers: [
        ProductEffects,
        provideMockActions(() => actions$),
        { provide: MockApiService, useValue: apiSpy },
      ],
    });
    effects = TestBed.inject(ProductEffects);
  });

  it('should dispatch success on API ok', (done) => {
    apiSpy.getProducts.and.returnValue(of(mockProducts));
    actions$ = of(ProductActions.loadProducts());

    effects.loadProducts$.subscribe(action => {
      expect(action).toEqual(
        ProductActions.loadProductsSuccess({ products: mockProducts })
      );
      done();
    });
  });

  it('should dispatch failure on API error (marble)', () => {
    scheduler.run(({ cold, expectObservable }) => {
      actions$ = cold('-a', { a: ProductActions.loadProducts() });
      apiSpy.getProducts.and.returnValue(cold('-#', {}, new Error('boom')));

      expectObservable(effects.loadProducts$).toBe('--b', {
        b: ProductActions.loadProductsFailure({ error: 'boom' }),
      });
    });
  });
});

⚠️ Errores comunes

  • Testear reducers vía el Store completo: son funciones puras, llámalas directamente.
  • Olvidar el caso "acción desconocida devuelve el mismo estado" — detecta mutaciones accidentales.
  • En effects con done: si el effect nunca emite, el test se cuelga hasta timeout en vez de fallar claro. El marble falla al instante.
  • Usar el selector completo (con estado global gigante) en vez de projector(): acopla el test a la forma de TODO el estado.

🎤 Preguntas de entrevista

  • ¿Por qué los reducers son tan fáciles de testear? — Son funciones puras y síncronas: mismas entradas → misma salida, sin mocks ni DI.
  • ¿Qué hace projector()? — Ejecuta la función de proyección del selector con las entradas que le pases, saltándose memoización y selectors padre.
  • ¿Cómo testeas que un effect NO despacha nada (dispatch: false)? — Suscríbete y verifica el side effect (spy llamado) en lugar de la emisión, o aserta el marble vacío ''.

11. Testing de Pipes y Directivas

Un pipe puro es una clase con un método transform: se testea con new, sin TestBed. Una directiva no vive sola — necesita un elemento donde aplicarse — así que se testea con el host component pattern: un componente de prueba definido en el spec cuyo template usa la directiva.

Pipes: unit test puro

describe('SustainabilityLabelPipe', () => {
  const pipe = new SustainabilityLabelPipe();

  it('should label high scores', () => {
    expect(pipe.transform(85)).toBe('🌱 Eco líder');
  });

  it('should handle edge values', () => {
    expect(pipe.transform(0)).toBe('Sin datos');
    expect(pipe.transform(null)).toBe('Sin datos');
  });
});

Directivas: host component pattern

// Componente host definido SOLO en el spec
@Component({
  imports: [HighlightDirective],
  template: `<p appHighlight="gold" data-testid="target">Texto</p>`,
})
class HostComponent {}

describe('HighlightDirective', () => {
  let fixture: ComponentFixture<HostComponent>;

  beforeEach(async () => {
    await TestBed.configureTestingModule({ imports: [HostComponent] })
      .compileComponents();
    fixture = TestBed.createComponent(HostComponent);
    fixture.detectChanges();
  });

  it('should highlight on mouseenter', () => {
    const p = fixture.debugElement.query(By.css('[data-testid="target"]'));

    p.triggerEventHandler('mouseenter', null);
    fixture.detectChanges();

    expect(p.nativeElement.style.backgroundColor).toBe('gold');
  });

  it('should expose the directive instance', () => {
    const dirEl = fixture.debugElement.query(By.directive(HighlightDirective));
    const directive = dirEl.injector.get(HighlightDirective);
    expect(directive).toBeTruthy();
  });
});

El mismo patrón sirve para testear componentes con output(), proyección de contenido (ng-content) o interacción padre-hijo realista.

⚠️ Errores comunes

  • Levantar TestBed para un pipe puro: sobreingeniería; new Pipe() basta.
  • Testear la directiva llamando sus métodos directamente sin DOM: no verifica el binding real con el elemento host (HostListener/HostBinding).
  • Olvidar importar la directiva en el imports del host component → la directiva simplemente no se aplica y los asserts fallan sin error de compilación (con esquemas laxos).
  • Pipes impuros testeados como puros: verifica también el comportamiento por referencia/mutación.

🎤 Preguntas de entrevista

  • ¿Por qué las directivas necesitan un host component? — Su comportamiento depende del elemento al que se aplican (listeners, bindings de atributos); sin DOM host no hay nada que observar.
  • ¿Cómo obtienes la instancia de una directiva en el test? — debugElement.query(By.directive(X)).injector.get(X).
  • ¿Qué diferencia hay al testear un pipe puro vs impuro? — El puro solo depende de sus argumentos (test trivial); el impuro puede depender de estado externo y se recalcula cada CD: hay que testear ambos disparadores.

12. Testing de Formularios Reactivos

Los formularios reactivos viven en la clase, así que gran parte se testea sin DOM: setValue/patchValue, estados de validez y errores. El DOM entra cuando quieres verificar que el usuario ve los mensajes y que los inputs están enlazados de verdad. 

describe('ProductFormComponent', () => {
  it('should be invalid when empty', () => {
    expect(component.form.valid).toBeFalse();
    expect(component.form.get('name')?.hasError('required')).toBeTrue();
  });

  it('should validate price range', () => {
    const price = component.form.get('price')!;
    price.setValue(-5);
    expect(price.hasError('min')).toBeTrue();

    price.setValue(29.99);
    expect(price.valid).toBeTrue();
  });

  it('should submit valid form with payload', () => {
    spyOn(component.saved, 'emit');
    component.form.setValue({ name: 'Tee', price: 29.99, category: 'Textiles' });

    component.onSubmit();

    expect(component.saved.emit).toHaveBeenCalledWith(
      jasmine.objectContaining({ name: 'Tee' })
    );
  });
});

Interacción con el DOM real

it('should show error message after user touches the field', () => {
  const input: HTMLInputElement =
    fixture.nativeElement.querySelector('[data-testid="name-input"]');

  // Simula escritura real del usuario
  input.value = '';
  input.dispatchEvent(new Event('input'));   // actualiza el FormControl
  input.dispatchEvent(new Event('blur'));    // marca touched
  fixture.detectChanges();

  const error = fixture.nativeElement.querySelector('[data-testid="name-error"]');
  expect(error.textContent).toContain('El nombre es obligatorio');
});

Validators custom y async

// Validator síncrono: función pura, sin TestBed
it('should reject forbidden words', () => {
  const control = new FormControl('spam');
  expect(forbiddenWordValidator(control)).toEqual({ forbidden: true });
});

// Validator async: fakeAsync + tick
it('should validate unique name against API', fakeAsync(() => {
  apiSpy.search.and.returnValue(of([existing]).pipe(delay(300)));
  const name = component.form.get('name')!;

  name.setValue('Organic Cotton T-Shirt');
  expect(name.pending).toBeTrue();      // validación en curso

  tick(300);
  expect(name.hasError('nameTaken')).toBeTrue();
}));

⚠️ Errores comunes

  • Setear input.value sin despachar el evento input: el FormControl nunca se entera.
  • Usar setValue con objeto incompleto → lanza error; para parciales usa patchValue.
  • Asertar mensajes de error sin marcar el control como touched: la mayoría de templates los ocultan hasta touched && invalid.
  • Olvidar que los async validators dejan el control en estado pending (ni valid ni invalid) hasta resolverse.

🎤 Preguntas de entrevista

  • ¿setValue vs patchValue en tests? — setValue exige la forma completa del grupo (falla rápido ante drift del form); patchValue acepta parciales.
  • ¿Cómo testeas un async validator? — Mockea la dependencia, usa fakeAsync, aserta pending antes del tick y el error después.
  • ¿Qué testeas en la clase y qué en el DOM? — Reglas de validación y payload en la clase; visibilidad de errores, binding de inputs y accesibilidad en el DOM.

13. Coverage y Umbrales

El code coverage mide qué porcentaje del código ejecutan tus tests, en cuatro métricas: statements, branches (cada rama de if/switch/ternario), functions y lines. Es un detector de huecos, no una medida de calidad: 100% de coverage con asserts débiles no garantiza nada. 

# Genera el reporte en /coverage (HTML navegable + lcov)
ng test --code-coverage --watch=false

# karma.conf.js — falla el build si se baja del umbral
coverageReporter: {
  dir: require('path').join(__dirname, './coverage'),
  subdir: '.',
  reporters: [{ type: 'html' }, { type: 'text-summary' }, { type: 'lcov' }],
  check: {
    global: {
      statements: 80,
      branches: 75,
      functions: 80,
      lines: 80,
    },
  },
}

Umbral razonable: 75–85% global es un buen equilibrio en apps Angular. Sube el listón en código crítico (reducers, servicios de dominio, validators: 90%+) y relájalo en código declarativo de bajo riesgo (configuración de rutas, wiring de módulos). Las branches son la métrica más honesta: revela los else y casos de error que nadie testeó.

Para medir la calidad de los tests (no solo su alcance) existe el mutation testing (Stryker): introduce bugs artificiales y comprueba si algún test los detecta.

⚠️ Errores comunes

  • Perseguir 100%: lleva a tests triviales que ejecutan código sin asertar comportamiento ("assertion-free coverage").
  • Mirar solo lines y no branches: el happy path cubre líneas, pero los errores viven en las ramas.
  • Subir el umbral de golpe en un proyecto legacy: bloquea todo el equipo. Sube gradualmente y por carpeta.
  • Excluir archivos "difíciles" del coverage para maquillar el número.

🎤 Preguntas de entrevista

  • ¿Es 100% de coverage un buen objetivo? — No: coste marginal altísimo y falsa seguridad. Mejor 80% con asserts fuertes y 90%+ en dominio crítico.
  • ¿Diferencia entre line coverage y branch coverage? — Line: la línea se ejecutó; branch: CADA camino de decisión se ejecutó. Un if sin else puede dar 100% lines y 50% branches.
  • ¿Qué aporta mutation testing sobre coverage? — Verifica que los tests DETECTAN cambios de comportamiento, no solo que ejecutan el código.

14. Buenas Prácticas de Testing

Los tests son código de producción: se leen mucho más de lo que se escriben. Estas prácticas mantienen la suite útil, legible y resistente a refactors.

Patrón AAA: Arrange, Act, Assert

it('should show error banner when the API fails', () => {
  // Arrange — prepara el mundo
  mockApiService.getProducts.and.returnValue(
    throwError(() => new Error('Network error'))
  );

  // Act — UNA acción bajo test
  component.loadProducts();
  fixture.detectChanges();

  // Assert — verifica el comportamiento observable
  const banner = fixture.nativeElement.querySelector('[data-testid="error-message"]');
  expect(banner.textContent).toContain('Network error');
});

Comportamiento, no implementación

// ❌ MAL — acoplado a detalles internos: se rompe con cualquier refactor
expect(component['sortInternal']).toHaveBeenCalled();
expect((component as any)._cache.size).toBe(3);

// ✅ BIEN — aserta lo que el usuario/consumidor observa
expect(component.products()).toEqual(sortedProducts);
expect(rows[0].textContent).toContain('Organic Cotton T-Shirt');

Page Object en unit tests

Igual que en E2E, encapsula los selectores del template en un objeto: si el HTML cambia, actualizas UN lugar en vez de cincuenta tests. 

class DemoPage {
  constructor(private fixture: ComponentFixture<TestingDemoComponent>) {}

  get counterValue() {
    return this.fixture.nativeElement
      .querySelector('[data-testid="counter-value"]')?.textContent?.trim();
  }
  get productRows(): NodeListOf<HTMLElement> {
    return this.fixture.nativeElement.querySelectorAll('[data-testid="product-row"]');
  }
  clickIncrement() {
    this.fixture.nativeElement.querySelector('[data-testid="increment-btn"]').click();
    this.fixture.detectChanges();
  }
}

it('should increment via UI', () => {
  const page = new DemoPage(fixture);
  page.clickIncrement();
  expect(page.counterValue).toBe('1');
});

Checklist profesional

  • Un comportamiento por test: nombres descriptivos ("should show error banner when API fails"), un motivo de fallo por spec.
  • Selectores estables: data-testid en vez de clases CSS o estructura del DOM.
  • Deterministas: nada de fechas reales (jasmine.clock().mockDate()), random sin semilla ni timers reales.
  • Sin lógica en los tests: ni ifs ni loops complejos; si los necesitas, el test está mal diseñado.
  • Independientes: cada test debe pasar solo y en cualquier orden; estado nuevo en cada beforeEach.
  • Rápidos: tiempo virtual (fakeAsync/TestScheduler), mocks para I/O, TestBed mínimo.

⚠️ Errores comunes

  • Tests que espían métodos privados o leen estado interno: cualquier refactor los rompe aunque el comportamiento sea idéntico.
  • Un solo it gigante con 20 asserts: cuando falla, no sabes qué se rompió.
  • Copiar-pegar el beforeEach entre archivos con variaciones sutiles: extrae helpers de setup.
  • Tests flaky tolerados ("relanza el CI"): erosionan la confianza en toda la suite. Un flaky se arregla o se borra.

🎤 Preguntas de entrevista

  • ¿Qué significa "testear comportamiento y no implementación"? — Asertar salidas observables (DOM, valores retornados, llamadas a colaboradores externos) en vez de detalles internos; así los refactors no rompen tests.
  • ¿Qué es el patrón AAA? — Estructurar cada test en Arrange (setup), Act (una acción) y Assert (verificación), separados visualmente para máxima legibilidad.
  • ¿Cómo manejas un test flaky? — Se cuarentena y diagnostica ya: casi siempre es async mal controlado, estado compartido o dependencia de orden. Nunca se normaliza el re-run.
