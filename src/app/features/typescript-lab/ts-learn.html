🔷 1. Generics

Los generics permiten crear componentes reutilizables que trabajan con cualquier tipo. En lugar de usar any, los generics preservan la información de tipos, dando autocompletado y verificación en tiempo de compilación. 

class DataService<T> {
  private items: T[] = [];

  add(item: T): void {
    this.items.push(item);
  }

  getAll(): T[] {
    return [...this.items];
  }

  getById(id: string): T | undefined {
    return this.items.find(
      (item) => (item as any).id === id
    );
  }
}

// Uso con constraint
interface HasId { id: string; }

class BetterService<T extends HasId> {
  private items: T[] = [];

  getById(id: string): T | undefined {
    return this.items.find((item) => item.id === id);
  }
}

// Generic con tipo por defecto
class Cache<T = string> {
  private store = new Map<string, T>();

  set(key: string, value: T): void {
    this.store.set(key, value);
  }

  get(key: string): T | undefined {
    return this.store.get(key);
  }
}

const stringCache = new Cache();        // Cache<string>
const numberCache = new Cache<number>(); // Cache<number>

Generics en funciones, interfaces e inferencia

Los generics no solo aplican a clases: funciones e interfaces también los usan. TypeScript infiere el parámetro de tipo a partir de los argumentos, por lo que rara vez hay que escribirlo explícitamente. 

// Generic function — T se infiere del argumento
function firstOrNull<T>(items: T[]): T | null {
  return items.length > 0 ? items[0] : null;
}

const n = firstOrNull([1, 2, 3]);       // T = number → number | null
const s = firstOrNull(['a', 'b']);      // T = string → string | null

// Múltiples parámetros de tipo relacionados
function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] {
  return items.map((item) => item[key]);
}

const prices = pluck(products, 'price'); // number[]
// pluck(products, 'colour');            // ❌ error: 'colour' no es keyof Product

// Generic interface — típico en APIs
interface ApiResult<T> {
  data: T;
  total: number;
  page: number;
}

function fetchPage<T>(url: string): Promise<ApiResult<T>> {
  return fetch(url).then((r) => r.json());
}

✅ Mejores prácticas

  • Usa constraints (extends) en lugar de castear a any dentro del cuerpo: T extends HasId hace que item.id sea seguro sin cast.
  • Deja que TypeScript infiera: firstOrNull([1, 2]) es mejor que firstOrNull<number>([1, 2]). Especifica el tipo solo cuando la inferencia falla (por ejemplo, arrays vacíos).
  • Nombres descriptivos cuando hay varios parámetros: <TItem, TKey> comunica más que <T, U>.
  • No hagas genérico lo que no lo necesita: si el parámetro de tipo aparece una sola vez en la firma, probablemente sobra.

⚠️ Errores comunes

  • Usar any para "salir del paso": (item as any).id compila pero pierde toda la seguridad. La solución correcta es el constraint T extends HasId.
  • Confundir <T> con un cast: function f<T>() declara un parámetro de tipo; value as T es una aserción. Son mecanismos distintos.
  • Constraint demasiado amplio: T extends object no permite acceder a ninguna propiedad concreta. Define una interfaz mínima con lo que realmente necesitas.

🎤 Preguntas de entrevista

  • ¿Qué diferencia hay entre generics y any? R: any desactiva el chequeo de tipos; un generic preserva la relación entre entrada y salida (identity<T>(x: T): T devuelve el mismo tipo que recibe).
  • ¿Para qué sirve K extends keyof T? R: Restringe K a las claves de T, permitiendo funciones type-safe de acceso a propiedades donde el retorno es T[K] (indexed access type).
  • ¿Cuándo usarías un default en un generic? R: Cuando hay un tipo habitual y quieres evitar que el consumidor lo escriba siempre: Cache<T = string>. Similar a los parámetros por defecto en funciones.

🔷 2. Type Guards

Los type guards permiten estrechar (narrow) el tipo de una variable dentro de un bloque condicional. TypeScript usa el keyword is, instanceof, in y typeof para inferir tipos más específicos automáticamente. 

interface Product {
  id: string;
  name: string;
  price: number;
}

// Type guard con 'is' keyword
function isProduct(value: unknown): value is Product {
  return (
    typeof value === 'object' &&
    value !== null &&
    'id' in value &&
    'name' in value &&
    'price' in value
  );
}

// Uso seguro
function processData(data: unknown): string {
  if (isProduct(data)) {
    // TS sabe que data es Product aquí
    return \`${data.name}: $${data.price}\`;
  }
  return 'Dato no válido';
}

// instanceof guard
class ApiError extends Error {
  constructor(public code: number, message: string) {
    super(message);
  }
}

function handleError(err: unknown): string {
  if (err instanceof ApiError) {
    return \`Error ${err.code}: ${err.message}\`;
  }
  if (err instanceof Error) {
    return err.message;
  }
  return 'Error desconocido';
}

// 'in' operator guard
interface Cat { meow(): void; }
interface Dog { bark(): void; }

function makeSound(animal: Cat | Dog): void {
  if ('meow' in animal) {
    animal.meow(); // TS sabe que es Cat
  } else {
    animal.bark(); // TS sabe que es Dog
  }
}

Assertion functions

Una assertion function usa asserts en la firma: si la función retorna sin lanzar excepción, TypeScript asume que la condición es cierta para el resto del scope. Es ideal para validar precondiciones al inicio de una función. 

// asserts value is T — narrowing por excepción
function assertIsProduct(value: unknown): asserts value is Product {
  if (!isProduct(value)) {
    throw new Error('El valor no es un Product válido');
  }
}

// asserts condition — versión genérica (como Node assert)
function assert(condition: unknown, msg?: string): asserts condition {
  if (!condition) throw new Error(msg ?? 'Assertion failed');
}

function checkout(data: unknown, user: User | null): void {
  assertIsProduct(data);   // a partir de aquí data: Product
  assert(user, 'Se requiere sesión'); // user: User (sin null)

  console.log(\`${user.name} compra ${data.name}\`);
}

✅ Mejores prácticas

  • Valida en los bordes de la app: respuestas HTTP, JSON.parse, localStorage. Tipa esas entradas como unknown y pásalas por un guard antes de usarlas.
  • Un predicado is debe verificar de verdad: el compilador confía ciegamente en tu implementación. Si el cuerpo devuelve true sin comprobar nada, has creado un cast disfrazado.
  • Prefiere typeof/in inline para casos simples; reserva los predicados user-defined para validaciones que se repiten.

⚠️ Errores comunes

  • typeof null === 'object': siempre añade value !== null al comprobar objetos, o el guard aceptará null.
  • typeof solo distingue primitivos: typeof x === 'object' no diferencia entre Product y User; para eso necesitas in, un discriminante o un predicado.
  • instanceof no funciona con datos deserializados: un objeto de JSON.parse nunca es instanceof de tu clase porque no pasó por el constructor. Usa guards estructurales.
  • Assertion functions con arrow functions: TypeScript exige una anotación de tipo explícita en la variable (const assert: (c: unknown) => asserts c = ...) o el asserts no se aplica.

🎤 Preguntas de entrevista

  • ¿Qué es un user-defined type guard? R: Una función cuyo retorno se anota como value is Tipo. Si devuelve true, el compilador estrecha el tipo en la rama correspondiente.
  • ¿Diferencia entre value is T y asserts value is T? R: El primero estrecha dentro del if según el booleano devuelto; el segundo estrecha todo el código posterior a la llamada, porque lanzar excepción es la única salida negativa.
  • ¿Por qué instanceof puede fallar entre iframes o realms? R: Cada realm tiene sus propios constructores globales; un Array creado en otro iframe no es instanceof Array del actual. Por eso existe Array.isArray.

🔷 3. Discriminated Unions

Las discriminated unions usan un campo literal común (discriminante) para que TypeScript pueda estrechar el tipo en cada rama de un switch o if. Es el patrón ideal para modelar estados de una aplicación: cada estado declara solo los datos que existen en él, haciendo imposibles los estados inválidos (make illegal states unrepresentable). 

type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string };

function handleState<T>(state: AsyncState<T>): string | T {
  switch (state.status) {
    case 'idle':
      return 'Esperando...';
    case 'loading':
      return 'Cargando...';
    case 'success':
      return state.data;    // TS sabe que data existe
    case 'error':
      return state.message; // TS sabe que message existe
  }
}

// Verificación exhaustiva con never
function assertNever(value: never): never {
  throw new Error(\`Valor inesperado: ${value}\`);
}

function render<T>(state: AsyncState<T>): string {
  switch (state.status) {
    case 'idle':    return '⏸ Idle';
    case 'loading': return '⏳ Loading...';
    case 'success': return '✅ Done';
    case 'error':   return \`❌ ${state.message}\`;
    default:        return assertNever(state);
  }
}

// Ejemplo práctico: acciones de un reducer
type Action =
  | { type: 'ADD_ITEM'; payload: string }
  | { type: 'REMOVE_ITEM'; payload: number }
  | { type: 'CLEAR' };

function reducer(state: string[], action: Action): string[] {
  switch (action.type) {
    case 'ADD_ITEM':
      return [...state, action.payload];
    case 'REMOVE_ITEM':
      return state.filter((_, i) => i !== action.payload);
    case 'CLEAR':
      return [];
  }
}

Patrón state machine

Comparado con la alternativa de flags booleanos (isLoading, hasError, data?), la union discriminada elimina combinaciones absurdas como isLoading && hasError. El compilador se convierte en la documentación de la máquina de estados: si añades un estado nuevo, el assertNever del default hará fallar la compilación en cada switch que falte por actualizar.

✅ Mejores prácticas

  • Discriminante literal y consistente: usa siempre el mismo nombre (status, type, kind) con valores string literales, nunca booleanos ni tipos amplios.
  • Exhaustive check en cada consumidor: el default: assertNever(state) convierte "olvidé un caso" en un error de compilación en vez de un bug en producción.
  • Modela con la union, no con opcionales: data?: T; error?: string permite 4 combinaciones, de las que 2 son inválidas. La union solo permite las válidas.

⚠️ Errores comunes

  • Discriminante tipado como string: si declaras status: string en vez del literal 'loading', el narrowing no funciona. Cuidado al construir objetos sin anotar el tipo (usa satisfies o as const).
  • Desestructurar antes del switch: const { status, data } = state rompe la relación entre variables (en versiones antiguas de TS) y suele fallar porque data no existe en todas las ramas. Estrecha primero, luego accede.
  • Switch sin never check y sin noImplicitReturns: un caso olvidado devuelve undefined silenciosamente.

🎤 Preguntas de entrevista

  • ¿Qué necesita una union para ser "discriminada"? R: Que todos los miembros compartan una propiedad con un tipo literal único por miembro. TypeScript usa esa propiedad para estrechar.
  • ¿Cómo garantizas exhaustividad en un switch? R: Asignando el valor a never en el default (función assertNever). Si queda algún miembro sin manejar, no es asignable a never y falla la compilación.
  • ¿Dónde se usa este patrón en Angular/NgRx? R: Las actions de NgRx son una discriminated union por type; los estados de carga (AsyncState) y los eventos de HttpClient (HttpEventType) siguen la misma idea.

🔷 4. Utility Types

TypeScript incluye utility types integrados que transforman tipos existentes. Son esenciales para derivar tipos sin duplicar definiciones, manteniendo el código DRY y type-safe. 

interface Product {
  id: string;
  name: string;
  price: number;
  category: string;
}

// Omit — quita propiedades
type ProductCreate = Omit<Product, 'id'>;
// { name: string; price: number; category: string }

// Pick — selecciona propiedades
type ProductSummary = Pick<Product, 'name' | 'price'>;
// { name: string; price: number }

// Record — mapea claves a un tipo
type ProductIndex = Record<string, Product>;
// { [key: string]: Product }

// Partial — todas las propiedades opcionales
type ProductUpdate = Partial<Product>;
// { id?: string; name?: string; price?: number; category?: string }

// Required — todas obligatorias
type StrictProduct = Required<Partial<Product>>;

// Extract — extrae tipos de una union
type StringFields = Extract<keyof Product, string>;
// 'id' | 'name' | 'price' | 'category'

// Exclude — excluye tipos de una union
type NonIdFields = Exclude<keyof Product, 'id'>;
// 'name' | 'price' | 'category'

// ReturnType — obtiene el tipo de retorno de una función
function createProduct(): Product {
  return { id: '1', name: 'Item', price: 10, category: 'General' };
}
type Created = ReturnType<typeof createProduct>;
// Product

// Combinaciones avanzadas
type ProductPreview = Readonly<Pick<Product, 'name' | 'price'>>;
type ProductDraft = Partial<Omit<Product, 'id'>> & { id?: never };

Más utility types: NonNullable, Parameters, Awaited, Readonly

// NonNullable — quita null y undefined de un tipo
type MaybeUser = User | null | undefined;
type DefiniteUser = NonNullable<MaybeUser>; // User

// Parameters — tupla con los tipos de los argumentos
function updatePrice(id: string, price: number, notify?: boolean) { /* ... */ }
type UpdateArgs = Parameters<typeof updatePrice>;
// [id: string, price: number, notify?: boolean]

// Útil para wrappers que reenvían argumentos:
function withLogging(...args: Parameters<typeof updatePrice>) {
  console.log('updatePrice', args);
  return updatePrice(...args);
}

// Awaited — desenvuelve promesas (recursivo)
type Data = Awaited<Promise<Product[]>>;          // Product[]
type Nested = Awaited<Promise<Promise<number>>>; // number
type FromFn = Awaited<ReturnType<typeof fetchProducts>>;

// Readonly — inmutabilidad superficial en compile-time
type FrozenProduct = Readonly<Product>;
const p: FrozenProduct = { id: '1', name: 'A', price: 10, category: 'x' };
// p.price = 20; // ❌ error: price es readonly

¿Cuándo usar cada uno?

  • Partial — payloads de update (PATCH) y opciones de configuración con defaults.
  • Required — normalizar una config después de aplicar defaults.
  • Pick / Omit — DTOs derivados del modelo: formularios de creación (Omit<T, 'id'>), vistas resumidas (Pick).
  • Record — diccionarios indexados y mapas de configuración por clave conocida (Record<Status, string>).
  • Exclude / Extract — filtrar uniones (frecuente sobre keyof T).
  • NonNullable — resultados de find/APIs tras validar que no son null.
  • ReturnType / Parameters / Awaited — derivar tipos de funciones existentes sin exportar interfaces extra (wrappers, mocks de tests, servicios HTTP).

⚠️ Errores comunes

  • Partial es superficial: Partial<Order> hace opcional customer, pero si está presente debe estar completo. Para anidados necesitas un DeepPartial propio (mapped type recursivo).
  • Omit no valida las claves: Omit<Product, 'pricee'> (typo) compila sin aviso porque usa Exclude<keyof T, K> sin constraint. Pick sí falla con claves inexistentes.
  • Record<string, T> promete demasiado: el acceso dict['loQueSea'] se tipa como T aunque la clave no exista. Activa noUncheckedIndexedAccess o usa Map.
  • Readonly solo existe en compile-time: no congela el objeto en runtime; para eso está Object.freeze.

🎤 Preguntas de entrevista

  • ¿Cómo se implementa Partial<T> internamente? R: Es un mapped type: { [K in keyof T]?: T[K] }. Casi todos los utility types son mapped types o conditional types de una línea.
  • ¿Diferencia entre Omit y Exclude? R: Exclude opera sobre uniones (quita miembros); Omit opera sobre objetos (quita propiedades) y de hecho se define como Pick<T, Exclude<keyof T, K>>.
  • ¿Por qué Awaited y no ReturnType para una función async? R: ReturnType de una función async es Promise<T>; Awaited desenvuelve la promesa (recursivamente) para llegar a T.

🔷 5. Sistema de Tipos Estructural

TypeScript usa structural typing ("duck typing"): dos tipos son compatibles si tienen la misma forma, sin importar su nombre ni si se declararon con implements. Esto contrasta con lenguajes nominales como Java o C#, donde la compatibilidad depende de la declaración explícita. 

interface Point { x: number; y: number; }

class Coordinate {
  constructor(public x: number, public y: number) {}
}

// ✅ Compatible: Coordinate tiene la forma de Point
const p: Point = new Coordinate(1, 2);

// ✅ Un objeto con MÁS propiedades también es asignable...
const p3d = { x: 1, y: 2, z: 3 };
const p2: Point = p3d; // OK (asignación desde variable)

// ❌ ...pero los literales frescos sufren "excess property check"
// const p3: Point = { x: 1, y: 2, z: 3 };
// Error: 'z' no existe en Point

// Consecuencia: "branded types" para simular tipos nominales
type UserId = string & { readonly __brand: 'UserId' };
type OrderId = string & { readonly __brand: 'OrderId' };

function getUser(id: UserId) { /* ... */ }

const orderId = 'ord-1' as OrderId;
// getUser(orderId); // ❌ error: los brands no coinciden
getUser('u-1' as UserId); // ✅

✅ Mejores prácticas

  • Aprovecha la compatibilidad estructural para tests y mocks: cualquier objeto con la forma correcta sirve, sin herencia ni frameworks de mocking.
  • Usa branded types cuando dos conceptos comparten primitivo (ids, monedas, unidades) y confundirlos sería un bug grave.
  • Interfaces mínimas en parámetros: pide solo la forma que usas ({ id: string } en vez de todo Product); cualquier superset será asignable.

⚠️ Errores comunes

  • Asumir tipado nominal: dos interfaces con la misma forma son intercambiables aunque semánticamente sean distintas (Euros vs Dolares como number).
  • Confundir excess property check con incompatibilidad: solo aplica a literales de objeto "frescos" asignados directamente; la misma forma pasada por una variable intermedia compila.
  • Interfaces vacías: interface Marker {} es compatible con casi todo (estructuralmente no exige nada), por lo que no sirve como marcador.

🎤 Preguntas de entrevista

  • ¿Qué significa que TypeScript sea estructural? R: La compatibilidad se decide por la forma (miembros y tipos), no por el nombre del tipo ni por declaraciones de herencia.
  • ¿Qué es el excess property check? R: Una comprobación extra que aplica solo a object literals asignados directamente a un tipo: propiedades desconocidas producen error, para cazar typos.
  • ¿Cómo simulas tipado nominal? R: Con branded/opaque types: intersección con una propiedad fantasma (& { __brand: 'X' }) que no existe en runtime pero diferencia los tipos en compile-time.

🔷 6. Mapped Types

Un mapped type itera sobre las claves de un tipo ([K in keyof T]) y produce un tipo nuevo transformando cada propiedad. Con los modificadores +/- se añaden o quitan readonly y ?, y con as se renombran claves (key remapping). 

interface Product {
  id: string;
  name: string;
  price: number;
}

// keyof — union de claves
type ProductKeys = keyof Product; // 'id' | 'name' | 'price'

// Mapped type básico (así se implementa Readonly)
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

// Modificadores -readonly y -? (quitar)
type Mutable<T> = {
  -readonly [K in keyof T]: T[K];
};
type Concrete<T> = {
  [K in keyof T]-?: T[K]; // así funciona Required
};

// Transformar el tipo de cada valor
type Nullable<T> = {
  [K in keyof T]: T[K] | null;
};
type ProductForm = Nullable<Product>;
// { id: string | null; name: string | null; price: number | null }

// Key remapping con 'as' + template literals
type Getters<T> = {
  [K in keyof T as \`get${Capitalize<string & K>}\`]: () => T[K];
};
type ProductGetters = Getters<Product>;
// { getId: () => string; getName: () => string; getPrice: () => number }

// Filtrar propiedades: remapear a never las elimina
type OnlyStrings<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};
type ProductStrings = OnlyStrings<Product>;
// { id: string; name: string }

✅ Mejores prácticas

  • Deriva, no dupliques: si un tipo "form" o "DTO" es una transformación mecánica del modelo, exprésalo como mapped type para que se actualice solo al cambiar el modelo.
  • Combínalos con keyof genérico: K in keyof T mantiene la relación clave→valor (T[K]), a diferencia de Record, que aplana todo al mismo tipo.
  • Documenta los mapped types complejos con un comentario que muestre el resultado para un tipo de ejemplo, como en los snippets de arriba.

⚠️ Errores comunes

  • Olvidar string & K en template literals: keyof T incluye symbol, que no es interpolable; hay que intersectar con string.
  • Usar mapped types sobre uniones esperando iterar miembros: [K in keyof (A | B)] solo ve las claves comunes. Para distribuir sobre una unión se usan conditional types.
  • Confundir in (mapped type) con el operador runtime in: mismo keyword, contextos distintos (tipos vs expresiones).

🎤 Preguntas de entrevista

  • ¿Qué hace keyof? R: Produce la unión de las claves de un tipo como literales: keyof Product = 'id' | 'name' | 'price'. Es la base de mapped types y de funciones type-safe de acceso a propiedades.
  • ¿Cómo eliminarías una propiedad con key remapping? R: Remapeándola a never: [K in keyof T as K extends 'secret' ? never : K].
  • ¿Qué significan -? y -readonly? R: Quitan el modificador correspondiente al mapear: así están implementados Required (quita ?) y utilidades tipo Mutable (quita readonly).

🔷 7. Conditional Types e infer

Un conditional type elige entre dos tipos con la sintaxis T extends U ? X : Y. Con infer se "captura" una parte del tipo comparado (el retorno de una función, el elemento de un array, el payload de una promesa). Además, sobre uniones los conditional types son distributivos: se evalúan miembro a miembro. 

// Conditional básico
type IsString<T> = T extends string ? true : false;
type A = IsString<'hola'>; // true
type B = IsString<42>;     // false

// infer — capturar tipos internos
type ElementOf<T> = T extends (infer E)[] ? E : never;
type Num = ElementOf<number[]>; // number

type MyReturnType<T> =
  T extends (...args: any[]) => infer R ? R : never;

type UnwrapPromise<T> = T extends Promise<infer V> ? V : T;
type V = UnwrapPromise<Promise<Product>>; // Product
type W = UnwrapPromise<string>;            // string (rama else)

// Distributividad: se aplica a cada miembro de la union
type ToArray<T> = T extends any ? T[] : never;
type Arrays = ToArray<string | number>;
// string[] | number[]  (¡no (string | number)[]!)

// Así funciona Exclude:
type MyExclude<T, U> = T extends U ? never : T;
type SinId = MyExclude<'id' | 'name' | 'price', 'id'>;
// 'name' | 'price' — never desaparece de las uniones

// Desactivar la distribución envolviendo en tupla
type ToArrayNonDist<T> = [T] extends [any] ? T[] : never;
type OneArray = ToArrayNonDist<string | number>;
// (string | number)[]

✅ Mejores prácticas

  • Empieza por los utility types integrados (ReturnType, Awaited, Exclude...): casi todos son conditional types de una línea; escribe los tuyos solo si no existe uno.
  • Devuelve never en la rama imposible: en uniones, never se absorbe, lo que convierte el conditional en un filtro.
  • Nombra los infer: infer TElement se lee mejor que infer E en tipos largos.

⚠️ Errores comunes

  • Olvidar la distributividad: con un "naked type parameter" a la izquierda de extends, la unión se procesa miembro a miembro. Si no lo quieres, envuelve: [T] extends [U].
  • never como entrada: un conditional distributivo sobre never devuelve never directamente (unión vacía), sin evaluar ninguna rama. Sorprende en tipos recursivos.
  • Usar infer fuera de un conditional: solo es válido dentro de la cláusula extends de un conditional type.

🎤 Preguntas de entrevista

  • ¿Qué es infer? R: Un keyword que declara una variable de tipo dentro del patrón de un conditional type, capturando la parte que encaje: T extends Promise<infer V> ? V : never.
  • ¿Qué es un conditional type distributivo? R: Cuando el operando izquierdo es un parámetro de tipo desnudo, el conditional se aplica a cada miembro de la unión y los resultados se re-unen. Es la base de Exclude/Extract.
  • Implementa Extract. R: type Extract<T, U> = T extends U ? T : never; — lo contrario de Exclude.

🔷 8. Template Literal Types

Los template literal types construyen tipos string combinando literales y uniones, con la misma sintaxis que los template strings. Al combinar uniones se genera el producto cartesiano de todas las combinaciones. Junto con Uppercase, Lowercase, Capitalize y Uncapitalize permiten tipar convenciones de nombres. 

type Size = 'sm' | 'md' | 'lg';
type Variant = 'primary' | 'danger';

// Producto cartesiano: 6 combinaciones
type ButtonClass = \`btn-${Variant}-${Size}\`;
// 'btn-primary-sm' | 'btn-primary-md' | ... | 'btn-danger-lg'

// Tipar rutas de API
type Entity = 'products' | 'users';
type ApiRoute = \`/api/${Entity}\` | \`/api/${Entity}/${string}\`;

const ok: ApiRoute = '/api/products/42'; // ✅
// const bad: ApiRoute = '/api/orders';  // ❌

// Eventos tipados a partir de propiedades
type ChangeEvents<T> = {
  [K in keyof T as \`${string & K}Changed\`]: (value: T[K]) => void;
};
type ProductEvents = ChangeEvents<{ name: string; price: number }>;
// { nameChanged: (v: string) => void; priceChanged: (v: number) => void }

// Parsear strings con infer + template literals
type RouteParam<T> =
  T extends \`${string}:${infer P}\` ? P : never;
type Param = RouteParam<'/products/:productId'>; // 'productId'

// Intrinsics de manipulación
type Method = 'get' | 'post';
type HandlerName = \`on${Capitalize<Method>}\`; // 'onGet' | 'onPost'

✅ Mejores prácticas

  • Úsalos para contratos de strings estructurados: rutas, claves de i18n, nombres de eventos, clases CSS con convención. El autocompletado y la validación gratis compensan la complejidad.
  • ${string} como comodín: permite "cualquier string aquí" dentro de un patrón, evitando enumerar infinitos valores.
  • Vigila el tamaño de las combinaciones: uniones grandes multiplicadas explotan (límites del compilador y lentitud del editor).

⚠️ Errores comunes

  • Esperar validación en runtime: los template literal types solo restringen valores conocidos en compile-time; un string que llega de una API sigue necesitando validación.
  • Modelar gramáticas complejas: parsear URLs completas o SQL con tipos recursivos es un ejercicio interesante pero suele ser inmantenible en producción.

🎤 Preguntas de entrevista

  • ¿Qué produce \`${A}-${B}\` si A y B son uniones? R: La unión de todas las combinaciones (producto cartesiano).
  • ¿Cómo extraerías el parámetro de una ruta tipo '/user/:id'? R: Con un conditional type que combine template literal e infer: T extends \`${string}:${infer P}\` ? P : never.

🔷 9. unknown vs any vs never

Tres tipos especiales que marcan los extremos del sistema: any desactiva el chequeo (todo vale, en ambas direcciones), unknown es el "top type" seguro (todo es asignable a él, pero él no es asignable a nada sin narrowing) y never es el "bottom type": no tiene valores posibles y representa código inalcanzable. 

// any — contagioso y peligroso
let a: any = JSON.parse('{"x":1}');
a.foo.bar.baz;        // compila... y explota en runtime
const n: number = a;  // compila sin queja

// unknown — obliga a validar antes de usar
let u: unknown = JSON.parse('{"x":1}');
// u.x;               // ❌ error: primero hay que estrechar
if (typeof u === 'object' && u !== null && 'x' in u) {
  console.log((u as { x: number }).x); // ✅ tras validar
}

// never — funciones que no retornan y ramas imposibles
function fail(msg: string): never {
  throw new Error(msg);
}

type Status = 'on' | 'off';
function toggle(s: Status) {
  if (s === 'on') return 'off';
  if (s === 'off') return 'on';
  const imposible: never = s; // ✅ s ya no tiene miembros posibles
  return imposible;
}

// never se absorbe en uniones y anula intersecciones
type U = string | never;  // string
type I = string & never;  // never

// Firma segura para errores de catch (TS 4.4+: useUnknownInCatchVariables)
try {
  riskyOperation();
} catch (err: unknown) {
  if (err instanceof Error) console.error(err.message);
}

✅ Mejores prácticas

  • unknown en los bordes: APIs externas, JSON.parse, parámetros de librerías genéricas y variables de catch. Obliga a validar exactamente una vez.
  • never como herramienta de diseño: exhaustive checks, prohibir propiedades ({ id?: never }) y filtrar en mapped/conditional types.
  • any solo como último recurso documentado (migraciones, tipos de librerías rotos), idealmente encapsulado en una función pequeña con firma bien tipada.

⚠️ Errores comunes

  • any se propaga: una expresión any convierte en any casi todo lo que toca, desactivando el chequeo silenciosamente aguas abajo.
  • Doble cast as unknown as T: es un any disfrazado; si lo necesitas, deja un comentario del porqué o escribe un type guard.
  • Confundir never con void: void es "retorna sin valor útil"; never es "no retorna jamás" (lanza o bucle infinito).

🎤 Preguntas de entrevista

  • ¿Por qué unknown es más seguro que any? R: Ambos aceptan cualquier valor, pero unknown no permite ninguna operación hasta estrecharlo; any permite todas sin comprobar nada.
  • ¿Cuándo aparece never por inferencia? R: En ramas donde el narrowing agotó todos los miembros de una unión, en Exclude total, y como retorno de funciones que siempre lanzan.
  • ¿Qué es un top type y un bottom type? R: Top (unknown): supertipo de todos, todo es asignable a él. Bottom (never): subtipo de todos, asignable a cualquier cosa, pero nada es asignable a él.

🔷 10. Type Narrowing y Control Flow Analysis

TypeScript re-analiza el tipo de cada variable en cada punto del código siguiendo el flujo de control: if, return tempranos, throw, asignaciones, comparaciones de igualdad y truthiness. Entender este análisis explica por qué a veces "TS ya sabe" el tipo y por qué a veces lo "olvida". 

function format(value: string | number | null) {
  if (value === null) return '—';   // elimina null
  // value: string | number

  if (typeof value === 'number') {
    return value.toFixed(2);        // value: number
  }
  return value.toUpperCase();       // value: string
}

// Narrowing por igualdad entre variables
function pick(a: string | number, b: string | boolean) {
  if (a === b) {
    // Ambos solo pueden ser el tipo común: string
    a.toUpperCase();
    b.toUpperCase();
  }
}

// Truthiness: cuidado con '' y 0
function greet(name?: string) {
  if (name) {
    return \`Hola ${name}\`; // excluye undefined... y también ''
  }
  return 'Hola anónimo';
}

// Asignación re-estrecha el tipo declarado
let x: string | number = 'hola';
x.toUpperCase(); // x: string (por la asignación)
x = 42;
x.toFixed();     // x: number

// Límite clásico: callbacks pierden el narrowing
function process(id: string | null) {
  if (id !== null) {
    [1, 2, 3].forEach(() => {
      // id vuelve a ser string | null dentro del callback (si id es mutable)
      // porque TS no sabe cuándo se ejecutará
    });
  }
}
// Solución: const local → const safeId = id;

✅ Mejores prácticas

  • Early returns: descartar casos inválidos al principio (if (!user) return;) deja el resto de la función con tipos limpios y sin anidamiento.
  • Prefiere const: el narrowing sobre const se conserva dentro de closures; sobre let se descarta.
  • Compara con explícitos !== null / !== undefined cuando 0, '' o false sean valores válidos.

⚠️ Errores comunes

  • Narrowing perdido en callbacks: TS invalida el estrechamiento de variables mutables dentro de funciones anidadas. Copia a una const local antes del callback.
  • Narrowing sobre propiedades: if (obj.prop) estrecha obj.prop, pero cualquier llamada a función intermedia puede invalidarlo (la función podría mutar el objeto). Extrae la propiedad a una variable local.
  • Abusar de ! (non-null assertion): user!.name silencia al compilador sin comprobar nada; si te equivocas, es un TypeError en runtime.

🎤 Preguntas de entrevista

  • ¿Qué mecanismos de narrowing conoce TypeScript? R: typeof, instanceof, in, comparaciones de igualdad, truthiness, discriminantes literales, predicados is, asserts y asignaciones.
  • ¿Por qué se pierde el narrowing dentro de un callback? R: El compilador no puede garantizar cuándo se ejecuta el callback ni que la variable mutable no haya cambiado entre medias, así que revierte al tipo declarado.
  • ¿Qué diferencia hay entre ?. y !? R: ?. comprueba en runtime y produce undefined si falta; ! solo afirma al compilador que no es null, sin ninguna comprobación real.

🔷 11. satisfies Operator

satisfies (TS 4.9+) valida que una expresión cumple un tipo sin ensancharla a ese tipo: el valor conserva su tipo inferido más específico. Resuelve el dilema clásico entre "anotar el tipo" (pierdes literales) y "no anotar" (pierdes validación). 

type Theme = Record<string, string | number[]>;

// ❌ Con anotación: valida, pero pierde los tipos específicos
const theme1: Theme = {
  primary: '#1976d2',
  spacing: [4, 8, 16],
};
// theme1.primary → string | number[] ... hay que estrechar 😕

// ❌ Sin anotación: tipos específicos, pero sin validación
const theme2 = {
  primary: '#1976d2',
  spacng: [4, 8, 16],   // typo sin detectar
};

// ✅ satisfies: valida contra Theme Y conserva la inferencia
const theme = {
  primary: '#1976d2',
  spacing: [4, 8, 16],
} satisfies Theme;

theme.primary.toUpperCase(); // ✅ TS sabe que es string
theme.spacing.map((n) => n); // ✅ TS sabe que es number[]
// theme.spacng ...           // ❌ typo detectado por Theme

// Caso típico Angular: configuración de rutas
const routes = [
  { path: '', component: HomeComponent },
  { path: 'products', component: ProductsComponent },
] satisfies Routes;

// Con as const para máxima precisión
const config = {
  env: 'production',
  retries: 3,
} as const satisfies { env: string; retries: number };
// config.env: 'production' (literal), validado contra el contrato

✅ Mejores prácticas

  • Objetos de configuración: temas, mapas de rutas, diccionarios de handlers... cualquier literal grande que deba cumplir un contrato pero cuyo detalle quieras conservar.
  • as const satisfies T: combinación potente — literales inmutables validados contra un contrato.
  • Prefiérelo sobre as: satisfies comprueba de verdad (falla si no encaja); as fuerza y puede mentir.

⚠️ Errores comunes

  • Confundirlo con una anotación: const x: T = ... cambia el tipo de x a T; satisfies no cambia el tipo, solo valida.
  • Usarlo donde hace falta el tipo amplio: si una variable debe poder reasignarse a otros valores del contrato, necesitas la anotación, no satisfies.

🎤 Preguntas de entrevista

  • ¿Qué aporta satisfies frente a la anotación de tipo? R: Ambos validan, pero la anotación ensancha el valor al tipo anotado; satisfies conserva el tipo inferido (más estrecho), manteniendo literales y claves concretas.
  • ¿Diferencia entre satisfies y as? R: as es una aserción (puede ocultar errores); satisfies es una comprobación real que falla en compile-time si el valor no cumple el tipo.

🔷 12. Const Assertions (as const)

as const le pide al compilador la inferencia más literal posible: strings y números quedan como tipos literales, los objetos se vuelven readonly y los arrays se convierten en tuplas readonly. Es la forma idiomática de declarar constantes de las que luego se derivan tipos. 

// Sin as const: TS ensancha a tipos generales
const config1 = { env: 'prod', port: 3000 };
// { env: string; port: number }

// Con as const: literales + readonly
const config = { env: 'prod', port: 3000 } as const;
// { readonly env: 'prod'; readonly port: 3000 }

// Patrón estrella: derivar una union de un array
const STATUSES = ['idle', 'loading', 'success', 'error'] as const;
type Status = typeof STATUSES[number];
// 'idle' | 'loading' | 'success' | 'error'

// Una sola fuente de verdad: el array sirve en runtime
// (validaciones, @for en templates) y el tipo se deriva gratis
function isStatus(v: string): v is Status {
  return (STATUSES as readonly string[]).includes(v);
}

// Objeto como enum: claves y valores tipados
const HTTP_STATUS = {
  OK: 200,
  NOT_FOUND: 404,
  SERVER_ERROR: 500,
} as const;
type HttpStatusCode = typeof HTTP_STATUS[keyof typeof HTTP_STATUS];
// 200 | 404 | 500

// Tuplas exactas para retornos múltiples
function useCounter() {
  let count = 0;
  const inc = () => count++;
  return [count, inc] as const; // readonly [number, () => void]
}
const [value, increment] = useCounter(); // tipos correctos por posición

✅ Mejores prácticas

  • Array as const + typeof arr[number]: una lista de valores con su union derivada, sin duplicar nada. Ideal para opciones de selects, estados y catálogos pequeños.
  • Constantes de configuración: as const documenta la inmutabilidad y habilita narrowing preciso donde se usen.
  • Combínalo con satisfies cuando además haya un contrato que cumplir.

⚠️ Errores comunes

  • Readonly solo en compile-time: as const no congela el objeto en runtime (Object.freeze sí).
  • Aplicarlo a valores dinámicos: solo tiene sentido sobre literales; sobre una variable calculada no puede inferir literales.
  • Tuplas readonly vs arrays: una función que espera string[] no acepta readonly string[]; ajusta las firmas para aceptar readonly cuando no mutes.

🎤 Preguntas de entrevista

  • ¿Qué hace exactamente as const? R: Inferencia literal (no widening), propiedades readonly y arrays como tuplas readonly, aplicado recursivamente.
  • ¿Qué significa typeof STATUSES[number]? R: Indexed access sobre el tipo del array con índice number: la unión de los tipos de todos sus elementos.

🔷 13. Enums vs Union Types

Los enum de TypeScript generan código en runtime y tienen semánticas sorprendentes; la comunidad (y la documentación oficial) tiende a preferir uniones de literales u objetos as const, que ofrecen la misma seguridad sin coste ni sorpresas. De hecho, el flag --erasableSyntaxOnly (TS 5.8) prohíbe los enums clásicos por no ser sintaxis borrable. 

// ❌ enum numérico: inseguro y con reverse mapping
enum Level { Low, Medium, High }
const l: Level = 99 as Level;  // números arbitrarios cuelan (pre TS 5.0 sin cast)
console.log(Level[0]);          // 'Low' — genera objeto doble en runtime

// ⚠️ enum de strings: mejor, pero es nominal
enum Status { Active = 'active', Inactive = 'inactive' }
// acceptsStatus('active');    // ❌ hay que escribir Status.Active
// no acepta el literal equivalente

// ✅ Union de literales: cero runtime, estructural
type OrderStatus = 'pending' | 'paid' | 'shipped';
function setStatus(s: OrderStatus) { /* ... */ }
setStatus('paid');              // ✅ literal directo, con autocompletado

// ✅ Objeto as const cuando necesitas los valores en runtime
const OrderStatus = {
  Pending: 'pending',
  Paid: 'paid',
  Shipped: 'shipped',
} as const;
type OrderStatus = typeof OrderStatus[keyof typeof OrderStatus];
// El type y el objeto pueden compartir nombre (declaration merging valor/tipo)

Object.values(OrderStatus);     // iterable en runtime, como un enum
setStatus(OrderStatus.Paid);    // estilo enum
setStatus('shipped');           // o literal directo — ambos valen

✅ Mejores prácticas

  • Por defecto, union de literales: gratis en runtime, estructural, serializable tal cual (coincide con lo que envía el backend).
  • Objeto as const si además necesitas iterar los valores o un namespace con autocompletado (OrderStatus.Paid).
  • Si usas enums, que sean de strings: los numéricos heredan los problemas de reverse mapping y valores fuera de rango. Evita const enum en librerías (problemas con isolatedModules).

⚠️ Errores comunes

  • Comparar enums de strings con literales: status === 'active' falla con enum nominal; con uniones funciona de forma natural.
  • Enums en templates Angular: hay que exponer el enum como propiedad del componente para usarlo en el HTML; con uniones se comparan literales directamente.
  • Olvidar que los enums generan JS: aumentan el bundle y no se tree-shakean bien si se usan parcialmente.

🎤 Preguntas de entrevista

  • ¿Por qué se suelen preferir uniones sobre enums? R: No generan código, son estructurales (aceptan el literal directamente), serializan igual que se declaran y funcionan mejor con narrowing y APIs externas.
  • ¿Cuándo tiene sentido un enum? R: Interoperabilidad con código existente que ya los usa, o preferencia de equipo por el namespace Enum.Member; el objeto as const cubre ese caso sin los inconvenientes.
  • ¿Qué es el reverse mapping? R: Los enums numéricos generan un objeto con mapeo doble (Level[0] === 'Low' y Level.Low === 0); los de strings no lo tienen.

🔷 14. Interfaces vs Types y Declaration Merging

interface y type se solapan en el 90% de los casos, pero difieren en puntos clave: las interfaces admiten declaration merging (varias declaraciones con el mismo nombre se fusionan) y extends; los type aliases pueden nombrar cualquier tipo (uniones, tuplas, mapped, conditional, primitivos) y no se fusionan. 

// interface: extends + merging
interface Animal { name: string; }
interface Dog extends Animal { bark(): void; }

// Declaration merging: ambas declaraciones se fusionan
interface Window { analytics: AnalyticsClient; }
// Ahora window.analytics está tipado globalmente
// (así se "parchean" librerías y objetos globales)

// type: puede nombrar lo que interface no puede
type Id = string | number;                  // union
type Pair = [x: number, y: number];        // tupla nombrada
type Callback = (e: Event) => void;        // función (también con interface)
type ReadonlyProduct = Readonly<Product>;  // mapped / utility
type Key = \`item-${number}\`;         // template literal

// Ambos se combinan sin problema
type WithTimestamps = { createdAt: Date; updatedAt: Date };
interface ProductEntity extends Animal { }   // interface extends interface
type FullProduct = Product & WithTimestamps; // intersección con type

// ❌ type no se fusiona:
// type User = { a: 1 };
// type User = { b: 2 };  // Error: duplicate identifier

// Módulo augmentation (merging aplicado a librerías)
declare module '@angular/core' {
  // añadir tipos a un módulo existente
}

✅ Mejores prácticas

  • Elige uno como convención de equipo y sé consistente. Una guía habitual: interface para formas de objetos públicos y contratos extensibles; type para uniones, tuplas, funciones y tipos derivados.
  • Usa merging deliberadamente: es la herramienta correcta para aumentar tipos de terceros (Window, módulos de npm), pero dentro de tu propio código resulta confuso — prefiere extends.
  • Los errores de interfaces suelen ser más legibles (se muestra el nombre, no la expansión completa), útil en APIs públicas.

⚠️ Errores comunes

  • Merging accidental: dos interface User en el mismo scope se fusionan en silencio; con type obtendrías un error de duplicado. En archivos sin import/export (scripts globales) esto sorprende especialmente.
  • Intentar "extender" una union con interface: interface X extends A | B no es válido; las uniones solo se manejan con type.
  • Augmentar un módulo sin importar nada de él: el declare module se interpreta como declaración nueva en vez de augmentation. Debe haber al menos un import del módulo.

🎤 Preguntas de entrevista

  • ¿Diferencias entre interface y type? R: Interface: merging, extends, solo formas de objeto/función. Type: cualquier tipo (uniones, mapped, conditional), sin merging. En rendimiento y capacidades básicas son equivalentes.
  • ¿Qué es declaration merging y un uso real? R: La fusión de múltiples declaraciones del mismo nombre. Uso típico: augmentar Window o los tipos de una librería (por ejemplo, añadir matchers custom a Jest/Vitest).

🔷 15. Strict Mode y tsconfig

"strict": true activa la familia de flags estrictos y es la base de cualquier proyecto serio (Angular CLI lo activa por defecto). Cada flag elimina una clase de bugs en compile-time. Conviene conocer qué hace cada uno y qué flags extra valen la pena. 

// tsconfig.json (extracto del proyecto)
{
  "compilerOptions": {
    "strict": true,                    // activa todos los flags strict*
    // Incluye, entre otros:
    // strictNullChecks — null/undefined dejan de ser asignables a todo
    // noImplicitAny — parámetros sin tipo inferible son error
    // strictFunctionTypes — parámetros de funciones contravariantes
    // strictPropertyInitialization — propiedades de clase inicializadas
    // useUnknownInCatchVariables — catch (e: unknown)

    "noImplicitReturns": true,         // toda rama debe retornar
    "noImplicitOverride": true,        // override explícito al sobrescribir
    "noUncheckedIndexedAccess": true,  // index signatures devuelven T | undefined
    "noFallthroughCasesInSwitch": true
  }
}

// strictNullChecks en acción
function findProduct(id: string): Product | undefined {
  return products.find((p) => p.id === id);
}
const prod = findProduct('1');
// prod.name;              // ❌ posiblemente undefined
if (prod) prod.name;       // ✅ narrowing
prod?.name;                // ✅ optional chaining
prod ?? DEFAULT_PRODUCT;   // ✅ nullish coalescing

// noUncheckedIndexedAccess
const dict: Record<string, number> = { a: 1 };
const v = dict['b'];       // v: number | undefined (¡honesto!)
const arr = [1, 2, 3];
const first = arr[0];      // number | undefined — obliga a comprobar

✅ Mejores prácticas

  • Nunca desactives strict en proyectos nuevos. En migraciones, actívalo flag a flag y usa // @ts-expect-error (que caduca: falla si el error desaparece) en vez de // @ts-ignore.
  • Añade noUncheckedIndexedAccess: no viene en strict y es de los flags con mejor ratio coste/beneficio — hace honesto el acceso por índice.
  • En Angular, activa también strictTemplates (en angularCompilerOptions): lleva el chequeo estricto a los bindings de los templates.

⚠️ Errores comunes

  • Silenciar strictNullChecks con !: sembrar non-null assertions devuelve los undefined is not an object que el flag venía a evitar.
  • strictPropertyInitialization vs inyección: en Angular clásico se resolvía con ! en @Input; hoy el patrón preferido es inject(), inicializadores o inputs con input.required<T>().
  • Creer que strict valida en runtime: los flags solo afectan a la compilación; los datos externos siguen necesitando validación (guards, zod, etc.).

🎤 Preguntas de entrevista

  • ¿Qué hace strictNullChecks? R: Saca null y undefined del dominio de los demás tipos: hay que declararlos (T | null) y estrecharlos antes de usar el valor. Elimina la mayor fuente de errores de runtime en JS.
  • ¿Qué aporta noUncheckedIndexedAccess? R: El acceso por índice a arrays e index signatures pasa a tiparse T | undefined, reflejando que la clave puede no existir.
  • ¿@ts-ignore o @ts-expect-error? R: @ts-expect-error: si el error subyacente se arregla, la directiva pasa a ser un error, evitando supresiones zombis.
📂 tsconfig.json — strict, noImplicitReturns, noImplicitOverride, strictTemplates

🔷 16. Decoradores (contexto Angular)

Un decorador es una función que anota una clase o sus miembros añadiendo metadata o comportamiento. Angular los usa como sistema de metadata: @Component, @Injectable, @Pipe, @Directive... El compilador (AOT) lee esa metadata y genera el código de creación de vistas e inyección. Nota: los decoradores "experimentales" de TypeScript (experimentalDecorators) no son idénticos a los decoradores estándar de TC39 (TS 5.0+); Angular usa su propio procesamiento en compile-time. 

// Decorador de clase: recibe el constructor
function Logged<T extends new (...args: any[]) => object>(Base: T) {
  return class extends Base {
    constructor(...args: any[]) {
      super(...args);
      console.log(\`Instancia de ${Base.name} creada\`);
    }
  };
}

@Logged
class CartService { /* ... */ }

// En Angular: decoradores como metadata declarativa
@Component({
  selector: 'app-price-tag',
  template: \`<span>{{ price | currency }}</span>\`,
})
export class PriceTagComponent {
  @Input({ required: true }) price!: number;
  @Output() selected = new EventEmitter<void>();

  // Tendencia moderna: menos decoradores, más funciones tipadas
  private readonly http = inject(HttpClient);      // vs constructor + @Inject
  quantity = input.required<number>();             // vs @Input
  changed = output<number>();                      // vs @Output
}

@Injectable({ providedIn: 'root' })
export class ProductService {
  private readonly api = inject(ApiClient);
}

✅ Mejores prácticas

  • En Angular moderno, prefiere las APIs funcionales: inject(), input(), output(), functional guards/interceptors. Tipan mejor (genéricos reales) y reducen la dependencia de decoradores de miembro.
  • Los decoradores de clase de Angular no se negocian: @Component, @Injectable y compañía son el contrato con el compilador AOT; mantenlos con metadata estática y analizable (sin expresiones dinámicas).
  • No abuses de decoradores custom para lógica de negocio: son difíciles de tipar y de testear; una función explícita suele ser más clara.

⚠️ Errores comunes

  • Confundir decoradores experimentales con los estándar: firma y capacidades difieren (los TC39 usan objetos context). Código escrito para uno puede no funcionar con el otro flag.
  • Olvidar @Injectable en servicios con dependencias: sin decorador no se emite la metadata de DI y la inyección por constructor falla. Con inject() el problema desaparece.
  • Lógica pesada dentro del decorador: se ejecuta al cargar el módulo (import time), no al instanciar; puede penalizar el arranque y dificultar el testing.

🎤 Preguntas de entrevista

  • ¿Qué es un decorador y cómo los usa Angular? R: Una función que anota clases/miembros. Angular los usa como metadata declarativa que el compilador AOT transforma en código de renderizado e inyección de dependencias.
  • ¿Por qué Angular se mueve hacia inject() e input()? R: Mejor inferencia de tipos, composición en funciones reutilizables (fuera del constructor), menos dependencia de metadata de parámetros y preparación para los decoradores estándar.
  • ¿Cuándo se ejecuta un decorador? R: Una vez, cuando se evalúa la declaración de la clase (al importar el módulo), no en cada instanciación.
