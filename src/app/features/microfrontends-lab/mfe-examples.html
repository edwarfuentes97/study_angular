💡 1 · Native Federation — configurar un remote (Angular 17+)

El remote expone rutas o componentes para que el shell los cargue en runtime. Instalación: ng add @angular-architects/native-federation --type remote --port 4201

federation.config.js (remote "catalogo")

const { withNativeFederation, shareAll } = require('@angular-architects/native-federation/config');

module.exports = withNativeFederation({
  name: 'catalogo',

  exposes: {
    // clave pública → archivo interno que se expone
    './routes': './src/app/catalogo/catalogo.routes.ts',
  },

  shared: {
    // comparte todas las deps con singleton para evitar
    // dos instancias de Angular/RxJS en memoria
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
  },

  skip: [
    'rxjs/ajax', 'rxjs/fetch', 'rxjs/testing', 'rxjs/webSocket', // no usados
  ],
});
strictVersion: true falla rápido y explícito si el shell y el remote piden versiones incompatibles de una dependencia singleton — mejor un error claro al cargar que un bug silencioso en runtime.

💡 2 · Native Federation — el shell carga el remote

main.ts del shell — inicializar la federación ANTES de arrancar Angular

import { initFederation } from '@angular-architects/native-federation';

// el manifest se resuelve por entorno (dev/staging/prod)
initFederation('/assets/federation.manifest.json')
  .catch(err => console.error(err))
  .then(() => import('./bootstrap'))   // bootstrap.ts hace el bootstrapApplication real
  .catch(err => console.error(err));

federation.manifest.json — un JSON por entorno, sin recompilar

{
  "catalogo": "http://localhost:4201/remoteEntry.json",
  "carrito":  "http://localhost:4202/remoteEntry.json"
}

app.routes.ts del shell — ruta lazy hacia el remote

import { loadRemoteModule } from '@angular-architects/native-federation';

export const routes: Routes = [
  {
    path: 'catalogo',
    loadChildren: () =>
      loadRemoteModule('catalogo', './routes').then(m => m.CATALOGO_ROUTES),
  },
];
En producción solo cambias el manifest (por ejemplo con un envsubst en el pipeline): el mismo build del shell apunta a los remotes de cada entorno.

💡 3 · Module Federation clásico (webpack 5)

El equivalente con webpack (proyectos Angular ≤16 o mixtos React/Vue). Mismos conceptos: remotes en el host, exposes en el remote, shared en ambos.

webpack.config.js — HOST

const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'shell',
      remotes: {
        // alias → nombreRemote@URL del remoteEntry
        catalogo: 'catalogo@http://localhost:4201/remoteEntry.js',
      },
      shared: {
        '@angular/core':   { singleton: true, strictVersion: true },
        '@angular/router': { singleton: true, strictVersion: true },
        rxjs:              { singleton: true },
      },
    }),
  ],
};

webpack.config.js — REMOTE

new ModuleFederationPlugin({
  name: 'catalogo',
  filename: 'remoteEntry.js',
  exposes: {
    './Routes': './src/app/catalogo/catalogo.routes.ts',
  },
  shared: { /* misma config de singletons que el host */ },
})

💡 4 · Angular Elements — MFE como Web Component

Empaquetar un componente como custom element crea una frontera framework-agnóstica: el shell (aunque sea React, Vue o HTML plano) solo conoce la etiqueta. 

// main.ts del microfrontend
import { createApplication } from '@angular/platform-browser';
import { createCustomElement } from '@angular/elements';
import { MiniCartComponent } from './app/mini-cart.component';

(async () => {
  const app = await createApplication({ providers: [] });
  const MiniCartElement = createCustomElement(MiniCartComponent, {
    injector: app.injector,
  });
  customElements.define('mfe-mini-cart', MiniCartElement);
})();

Consumo desde cualquier shell

<script src="https://cdn.miapp.com/mini-cart/main.js"></script>

<mfe-mini-cart user-id="42"></mfe-mini-cart>
Los @Input() se vuelven atributos/propiedades del elemento y los @Output() se emiten como CustomEvent — ese es el contrato público del MFE.

💡 5 · Comunicación desacoplada — CustomEvent como event bus

Los MFEs no deben importarse entre sí ni compartir un Store: se comunican con eventos tipados sobre window (o BroadcastChannel entre pestañas). 

// contrato compartido y versionado (npm package ligero o d.ts duplicado)
export interface CartItemAddedV1 {
  productId: string;
  quantity: number;
}

// MFE Catálogo — publica
window.dispatchEvent(new CustomEvent<CartItemAddedV1>('cart:item-added:v1', {
  detail: { productId: 'p-42', quantity: 1 },
}));

// MFE Carrito — se suscribe (y limpia al destruirse)
const handler = (e: Event) => {
  const detail = (e as CustomEvent<CartItemAddedV1>).detail;
  this.cart.add(detail.productId, detail.quantity);
};
window.addEventListener('cart:item-added:v1', handler);
inject(DestroyRef).onDestroy(() =>
  window.removeEventListener('cart:item-added:v1', handler));
El sufijo :v1 en el nombre del evento permite evolucionar el contrato sin romper a los consumidores: publicas :v2 y mantienes :v1 un tiempo.
