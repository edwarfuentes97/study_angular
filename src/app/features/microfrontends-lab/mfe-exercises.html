1 · Shell + remote con Native Federation

Crea dos apps Angular (shell en el puerto 4200 y catalogo en el 4201) con ng add @angular-architects/native-federation. El remote debe exponer sus rutas y el shell cargarlas de forma lazy en /catalogo.

  • El remote expone './routes' en federation.config.js
  • El shell usa loadRemoteModule() en una ruta lazy
  • Ambas apps arrancan por separado y el shell muestra el catálogo
Arranca primero el remote: el shell falla al resolver el manifest si el remoteEntry.json no responde
Ver guía de solución
  • Remote: ng add @angular-architects/native-federation --project catalogo --type remote --port 4201
  • En federation.config.js del remote: exposes: { './routes': './src/app/catalogo.routes.ts' }
  • Shell: en app.routes.ts, loadChildren: () => loadRemoteModule('catalogo', './routes').then(m => m.CATALOGO_ROUTES)
  • Registra el remote en assets/federation.manifest.json del shell

2 · Dependencias compartidas y versiones

Con el setup del ejercicio 1, experimenta con shared: marca Angular y RxJS como singleton, luego provoca un conflicto instalando una versión distinta de una librería en el remote y observa qué pasa con strictVersion: true vs false.

  • @angular/* y rxjs configurados como singleton en ambos lados
  • Reproduces el error de versión incompatible con strictVersion: true
  • Documentas qué estrategia usarías en producción y por qué
Mira la pestaña Network: con shared bien configurado, Angular se descarga UNA vez aunque cargues varios remotes
Ver guía de solución
  • Con strictVersion: true la carga del remote lanza un error explícito al detectar la versión incompatible
  • Con false puede cargar dos copias (bundle más pesado y bugs sutiles con dos runtimes)
  • En producción: singleton + strictVersion para el framework; versiones alineadas vía renovate/nx en todos los MFEs

3 · Comunicación por eventos con contrato versionado

Implementa el flujo "agregar al carrito": el MFE catálogo publica un CustomEvent('cart:item-added:v1') y el MFE carrito (u otro componente del shell) se suscribe, actualiza un contador y limpia el listener al destruirse.

  • Interfaz TypeScript compartida para el detail del evento
  • El publicador no importa nada del suscriptor (cero acoplamiento)
  • El listener se remueve con DestroyRef.onDestroy
Si necesitas comunicar pestañas distintas del navegador, cambia window.dispatchEvent por BroadcastChannel
Ver guía de solución
  • Define CartItemAddedV1 en un paquete de contratos (o un .d.ts duplicado conscientemente)
  • Publica con window.dispatchEvent(new CustomEvent('cart:item-added:v1', { detail }))
  • Suscribe en el constructor del servicio del carrito y registra la limpieza con inject(DestroyRef).onDestroy(...)

4 · Widget como Angular Element

Convierte un componente (por ejemplo un mini-carrito con contador) en un Web Component con createCustomElement y consúmelo desde una página HTML plana sin Angular.

  • customElements.define registra la etiqueta mfe-mini-cart
  • Un @Input() se controla vía atributo HTML
  • Un @Output() se recibe como CustomEvent con addEventListener
Usa createApplication() (standalone) en vez de un NgModule: es el enfoque moderno para Elements
Ver guía de solución
  • Instala @angular/elements y registra el elemento en main.ts (ver pestaña Ejemplos 💡)
  • Compila con un solo bundle (esbuild) y cárgalo con una etiqueta <script> en un index.html plano
  • Prueba document.querySelector('mfe-mini-cart').setAttribute('user-id', '42')

5 · Routing shell ↔ remote y deep linking

Asegura que las rutas internas del remote funcionen con deep linking: recargar /catalogo/producto/42 directamente en el navegador debe restaurar la vista exacta.

  • El remote define rutas hijas relativas (sin hardcodear el prefijo /catalogo)
  • Recargar una URL profunda funciona (fallback del servidor a index.html)
  • La navegación entre MFEs no recarga la página completa
El remote nunca debe conocer su prefijo: el shell decide dónde montarlo — así puedes moverlo de /catalogo a /tienda sin tocar el remote
Ver guía de solución
  • En el remote exporta rutas con paths relativos: [{ path: '', component: Lista }, { path: 'producto/:id', component: Detalle }]
  • El prefijo lo aporta la ruta lazy del shell (path: 'catalogo')
  • Configura el servidor (nginx/Pages) con fallback a index.html para rutas desconocidas

6 · Pipeline de deploy independiente

Diseña (en un diagrama o YAML) el CI/CD para shell + 2 remotes: cada repo despliega su propio bundle a un CDN y el shell resuelve las URLs vía manifest por entorno, sin recompilarse.

  • Cada MFE tiene pipeline propio (build, test, deploy a su carpeta del CDN)
  • El manifest de staging/prod apunta a versiones inmutables (hash o semver)
  • Hay estrategia de rollback: volver a la versión anterior editando solo el manifest
Publica cada versión del remote en una ruta inmutable (p. ej. /catalogo/1.4.2/remoteEntry.json) y haz que el manifest sea lo único mutable
Ver guía de solución
  • Remote: pipeline sube dist/ a cdn.com/catalogo/<version>/ y actualiza el manifest del entorno
  • Shell: se construye una sola vez; en cada entorno lee su federation.manifest.json
  • Rollback = editar el manifest para apuntar a la versión anterior (segundos, sin build)
