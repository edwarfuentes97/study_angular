1 · Shell + remote con Native Federation (desde 0)

Crea dos apps Angular (shell en el puerto 4200 y catalogo en el 4201). El remote debe exponer sus rutas y el shell cargarlas de forma lazy en /catalogo.

  • El remote expone './routes' en federation.config.js
  • El shell usa loadRemoteModule() en una ruta lazy
  • Ambas apps arrancan por separado y el shell muestra el catálogo
Arranca primero el remote: el shell falla al resolver el manifest si el remoteEntry.json no responde
Ver solución paso a paso
  1. Crea el workspace con las dos aplicaciones 
    npm i -g @angular/cli
ng new mfe-workspace --create-application=false
cd mfe-workspace
ng generate application shell --routing --style=scss
ng generate application catalogo --routing --style=scss
  2. Instala Native Federation en cada app con su rol 
    ng add @angular-architects/native-federation --project shell --type dynamic-host --port 4200
ng add @angular-architects/native-federation --project catalogo --type remote --port 4201
    El schematic crea federation.config.js por proyecto, divide el arranque en main.ts + bootstrap.ts y cambia el builder en angular.json.
  3. Crea el feature del remote con sus rutas 
    ng generate component catalogo-home --project catalogo
    // projects/catalogo/src/app/catalogo.routes.ts
import { Routes } from '@angular/router';
import { CatalogoHome } from './catalogo-home/catalogo-home';

export const CATALOGO_ROUTES: Routes = [
  { path: '', component: CatalogoHome },
];
  4. Expón las rutas en el federation.config.js del remote 
    // projects/catalogo/federation.config.js
const { withNativeFederation, shareAll } = require('@angular-architects/native-federation/config');

module.exports = withNativeFederation({
  name: 'catalogo',
  exposes: {
    './routes': './projects/catalogo/src/app/catalogo.routes.ts',
  },
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
  },
});
  5. Registra el remote en el manifest del shell 
    // projects/shell/src/assets/federation.manifest.json
{
  "catalogo": "http://localhost:4201/remoteEntry.json"
}
    El main.ts del shell (generado por el schematic) ya carga este manifest con initFederation('/assets/federation.manifest.json') antes de importar bootstrap.ts — revísalo para entender el orden de arranque.
  6. Crea la ruta lazy en el shell 
    // projects/shell/src/app/app.routes.ts
import { Routes } from '@angular/router';
import { loadRemoteModule } from '@angular-architects/native-federation';

export const routes: Routes = [
  {
    path: 'catalogo',
    loadChildren: () =>
      loadRemoteModule('catalogo', './routes').then(m => m.CATALOGO_ROUTES),
  },
];
    Y agrega un enlace en el template del shell: 
    <a routerLink="/catalogo">Catálogo</a>
<router-outlet />
  7. Arranca ambas apps (remote primero) en dos terminales 
    ng serve catalogo   # terminal 1 → http://localhost:4201
ng serve shell      # terminal 2 → http://localhost:4200
    abre http://localhost:4201/remoteEntry.json (debe responder el metadata del remote) y luego http://localhost:4200/catalogo — el shell debe renderizar el componente del catálogo. En la pestaña Network verás que el código del catálogo se descarga desde el puerto 4201.

2 · Dependencias compartidas y versiones

Con el setup del ejercicio 1, experimenta con shared: marca Angular y RxJS como singleton, provoca un conflicto de versiones y observa qué pasa con strictVersion: true vs false.

  • @angular/* y rxjs configurados como singleton en ambos lados
  • Reproduces el error de versión incompatible con strictVersion: true
  • Documentas qué estrategia usarías en producción y por qué
Mira la pestaña Network: con shared bien configurado, Angular se descarga UNA vez aunque cargues varios remotes
Ver solución paso a paso
  1. Observa la línea base

    Con ambas apps corriendo, abre DevTools → Network en el shell, filtra por JS y entra a /catalogo. Anota cuántos bundles llegan desde el puerto 4201: solo debe llegar el código del feature, no otra copia de Angular (eso es el shared con shareAll({ singleton: true }) funcionando).

  2. Provoca el conflicto de versiones

    El truco más simple sin segundo workspace: edita a mano la versión que el remote declara compartir y reconstruye. En projects/catalogo/package.json (o el package.json raíz si es workspace único) baja una dependencia compartida, por ejemplo:

    // package.json del remote (solo para el experimento)
"rxjs": "~7.5.0"   // el shell sigue en ~7.8.0
    npm install
ng serve catalogo
  3. Observa el fallo con strictVersion: true

    Recarga el shell y entra a /catalogo. En consola verás un error del tipo:

    Unsatisfied version 7.5.0 of shared singleton module rxjs (required ~7.8.0)
    la carga del remote falla de forma explícita y temprana — este es el comportamiento que quieres en producción: un error claro en vez de dos runtimes conviviendo.
  4. Relaja la validación y compara 
    // federation.config.js (ambos lados, solo para comparar)
...shareAll({ singleton: true, strictVersion: false, requiredVersion: 'auto' })

    Reconstruye y recarga: ahora carga, pero con warning en consola y riesgo real — si la API de la librería cambió entre versiones, el bug aparece en runtime, lejos de la causa.

  5. Restaura y documenta la política

    Vuelve a ~7.8.0 y a strictVersion: true. Política recomendada para producción:

    • singleton + strictVersion: true para @angular/*, rxjs y librerías con estado global (router, store).
    • Versiones alineadas entre MFEs con tooling (renovate, Nx, un solo lockfile en monorepo).
    • Librerías utilitarias sin estado (date-fns, lodash-es) pueden no compartirse: duplicar unos KB es más barato que acoplar upgrades.

3 · Comunicación por eventos con contrato versionado

Implementa el flujo "agregar al carrito": el MFE catálogo publica un CustomEvent('cart:item-added:v1') y el shell (o el MFE carrito) se suscribe, actualiza un contador y limpia el listener al destruirse.

  • Interfaz TypeScript compartida para el detail del evento
  • El publicador no importa nada del suscriptor (cero acoplamiento)
  • El listener se remueve con DestroyRef.onDestroy
Si necesitas comunicar pestañas distintas del navegador, cambia window.dispatchEvent por BroadcastChannel
Ver solución paso a paso
  1. Define el contrato del evento

    En un paquete de contratos compartido — o, si aún no lo tienes, un archivo duplicado conscientemente en cada MFE (el contrato es el nombre + shape, no el archivo):

    // shared-contracts/cart-events.ts
export const CART_ITEM_ADDED_V1 = 'cart:item-added:v1';

export interface CartItemAddedV1 {
  productId: string;
  quantity: number;
}
  2. Publica el evento desde el catálogo 
    // projects/catalogo/src/app/catalogo-home/catalogo-home.ts
import { Component } from '@angular/core';
import { CART_ITEM_ADDED_V1, CartItemAddedV1 } from '../cart-events';

@Component({
  selector: 'app-catalogo-home',
  template: `
    <h2>Catálogo</h2>
    <button (click)="addToCart('p-42')">Agregar producto 42</button>
  `,
})
export class CatalogoHome {
  addToCart(productId: string) {
    const detail: CartItemAddedV1 = { productId, quantity: 1 };
    window.dispatchEvent(new CustomEvent(CART_ITEM_ADDED_V1, { detail }));
  }
}
  3. Suscríbete en el shell con limpieza automática 
    // projects/shell/src/app/cart-badge.ts
import { Component, DestroyRef, NgZone, inject, signal } from '@angular/core';
import { CART_ITEM_ADDED_V1, CartItemAddedV1 } from './cart-events';

@Component({
  selector: 'app-cart-badge',
  template: `<span>🛒 @if (count()) { ({{ count() }}) }</span>`,
})
export class CartBadge {
  count = signal(0);
  private zone = inject(NgZone);

  constructor() {
    const handler = (e: Event) => {
      const detail = (e as CustomEvent<CartItemAddedV1>).detail;
      // el evento nace fuera de Angular: re-entra a la zona para que la UI se actualice
      this.zone.run(() => this.count.update(c => c + detail.quantity));
    };
    window.addEventListener(CART_ITEM_ADDED_V1, handler);
    inject(DestroyRef).onDestroy(() =>
      window.removeEventListener(CART_ITEM_ADDED_V1, handler));
  }
}
  4. Monta el badge en el layout del shell 
    <header>
  <a routerLink="/catalogo">Catálogo</a>
  <app-cart-badge />
</header>
<router-outlet />
    haz clic en "Agregar producto 42" dentro del MFE catálogo: el contador del shell sube sin que el catálogo importe nada del shell. Navega fuera y de vuelta: no se acumulan listeners duplicados (gracias a DestroyRef).
  5. Evoluciona el contrato sin romper consumidores

    Si mañana necesitas enviar también el precio, NO cambies el shape de :v1: publica cart:item-added:v2 con el campo nuevo y emite ambos eventos durante la transición. Cuando todos los consumidores migren a v2, retira v1.

4 · Widget como Angular Element

Convierte un componente (un mini-carrito con contador) en un Web Component con createCustomElement y consúmelo desde una página HTML plana sin Angular.

  • customElements.define registra la etiqueta mfe-mini-cart
  • Un input se controla vía atributo HTML
  • Un output se recibe como CustomEvent con addEventListener
Usa createApplication() (standalone) en vez de un NgModule: es el enfoque moderno para Elements
Ver solución paso a paso
  1. Crea el proyecto e instala @angular/elements 
    ng new mini-cart-element --routing=false --style=css
cd mini-cart-element
npm i @angular/elements
  2. Crea el componente con input y output clásicos

    Para Elements usa @Input()/@Output(): se mapean a atributos del custom element y a CustomEvents del DOM.

    // src/app/mini-cart.ts
import { Component, EventEmitter, Input, Output, signal } from '@angular/core';

@Component({
  selector: 'app-mini-cart',
  template: `
    <div class="cart">
      🛒 {{ count() }} — usuario: {{ userId }}
      <button (click)="add()">+</button>
    </div>
  `,
  styles: [`.cart { font-family: sans-serif; border: 1px solid #ccc; padding: 8px; border-radius: 6px; }`],
})
export class MiniCart {
  @Input() userId = '';
  @Output() itemAdded = new EventEmitter<number>();
  count = signal(0);

  add() {
    this.count.update(c => c + 1);
    this.itemAdded.emit(this.count());
  }
}
  3. Reemplaza el bootstrap por el registro del elemento 
    // src/main.ts
import { createApplication } from '@angular/platform-browser';
import { createCustomElement } from '@angular/elements';
import { MiniCart } from './app/mini-cart';

(async () => {
  const app = await createApplication({ providers: [] });
  const element = createCustomElement(MiniCart, { injector: app.injector });
  customElements.define('mfe-mini-cart', element);
})();
  4. Compila a un solo bundle 
    ng build --output-hashing=none

    En dist/mini-cart-element/browser/ quedan main.js y polyfills.js con nombres estables (sin hash), listos para servirse desde un CDN.

  5. Consúmelo desde HTML plano 
    <!-- demo.html (fuera de Angular) -->
<!DOCTYPE html>
<html>
<body>
  <h1>Página sin Angular</h1>

  <mfe-mini-cart user-id="42"></mfe-mini-cart>

  <script src="dist/mini-cart-element/browser/polyfills.js" type="module"></script>
  <script src="dist/mini-cart-element/browser/main.js" type="module"></script>
  <script>
    const cart = document.querySelector('mfe-mini-cart');
    cart.addEventListener('itemAdded', (e) => {
      console.log('items en el carrito:', e.detail);
    });
  </script>
</body>
</html>

    Sírvelo con cualquier servidor estático:

    npx http-server . -p 8080
# abre http://localhost:8080/demo.html
    el widget renderiza, el atributo kebab-case user-id llena el @Input() camelCase userId, y cada clic en "+" dispara el CustomEvent itemAdded que ves en la consola del navegador.

5 · Routing shell ↔ remote y deep linking

Asegura que las rutas internas del remote funcionen con deep linking: recargar /catalogo/producto/42 directamente en el navegador debe restaurar la vista exacta.

  • El remote define rutas hijas relativas (sin hardcodear el prefijo /catalogo)
  • Recargar una URL profunda funciona (fallback del servidor a index.html)
  • La navegación entre MFEs no recarga la página completa
El remote nunca debe conocer su prefijo: el shell decide dónde montarlo — así puedes moverlo de /catalogo a /tienda sin tocar el remote
Ver solución paso a paso
  1. Agrega una ruta con parámetro al remote (siempre relativa) 
    ng generate component producto-detalle --project catalogo
    // projects/catalogo/src/app/catalogo.routes.ts
export const CATALOGO_ROUTES: Routes = [
  { path: '', component: CatalogoHome },
  { path: 'producto/:id', component: ProductoDetalle },  // SIN prefijo /catalogo
];
  2. Lee el parámetro con input binding del router 
    // projects/catalogo/src/app/producto-detalle/producto-detalle.ts
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-producto-detalle',
  template: `<h2>Producto {{ id() }}</h2>`,
})
export class ProductoDetalle {
  id = input.required<string>();   // llega desde :id
}

    Activa el binding en la config del shell (quien crea el router):

    // projects/shell/src/app/app.config.ts
provideRouter(routes, withComponentInputBinding())
  3. Navega dentro del remote con rutas relativas 
    <!-- catalogo-home.html -->
<a [routerLink]="['producto', '42']">Ver producto 42</a>

    Al ser relativa, la URL final es /catalogo/producto/42 porque el shell montó el remote bajo catalogo — si mañana lo monta en tienda, nada del remote cambia.

  4. Prueba el deep linking en dev
    pega http://localhost:4200/catalogo/producto/42 directo en el navegador y recarga. ng serve ya hace fallback a index.html, así que debe restaurar la vista exacta: initFederation carga el manifest → el router resuelve /catalogo → loadRemoteModule trae las rutas → se renderiza producto/42.
  5. Configura el fallback en el servidor real

    En producción el servidor debe devolver index.html para cualquier ruta desconocida:

    # nginx
location / {
  try_files $uri $uri/ /index.html;
}

    En GitHub Pages el equivalente es el truco del 404.html que redirige al index (este mismo repo lo usa en public/404.html).

  6. Verifica que navegar entre MFEs es SPA pura
    con DevTools → Network abierto y "Preserve log" activado, navega dashboard → catálogo → producto → atrás. No debe aparecer ningún document request nuevo: solo chunks JS lazy. Si ves recargas completas, algún enlace usa href en vez de routerLink.

6 · Pipeline de deploy independiente

Diseña el CI/CD para shell + 2 remotes: cada repo despliega su propio bundle a un CDN y el shell resuelve las URLs vía manifest por entorno, sin recompilarse.

  • Cada MFE tiene pipeline propio (build, test, deploy a su carpeta del CDN)
  • El manifest de staging/prod apunta a versiones inmutables (hash o semver)
  • Hay estrategia de rollback: volver a la versión anterior editando solo el manifest
Publica cada versión del remote en una ruta inmutable (p. ej. /catalogo/1.4.2/remoteEntry.json) y haz que el manifest sea lo único mutable
Ver solución paso a paso
  1. Define el layout del CDN (rutas inmutables por versión) 
    cdn.miapp.com/
├── shell/                        # se sirve como sitio raíz
├── catalogo/1.4.1/remoteEntry.json
├── catalogo/1.4.2/remoteEntry.json   ← versión nueva, conviven ambas
├── carrito/2.0.0/remoteEntry.json
└── config/
    ├── staging.manifest.json     # lo ÚNICO mutable
    └── prod.manifest.json
  2. Pipeline de cada remote (uno por repo/equipo) 
    # .github/workflows/deploy-catalogo.yml
name: Deploy catalogo
on:
  push:
    tags: ['v*']          # deploy por tag semver → versión inmutable
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 'lts/*' }
      - run: npm ci
      - run: npm test
      - run: npx ng build catalogo
      - name: Upload versión inmutable al CDN
        run: |
          VERSION=${GITHUB_REF_NAME#v}
          aws s3 sync dist/catalogo/browser \
            s3://cdn-miapp/catalogo/$VERSION/ --cache-control "public,max-age=31536000,immutable"
  3. Promoción a un entorno = editar solo el manifest 
    # config/staging.manifest.json (staging prueba la 1.4.2)
{
  "catalogo": "https://cdn.miapp.com/catalogo/1.4.2/remoteEntry.json",
  "carrito":  "https://cdn.miapp.com/carrito/2.0.0/remoteEntry.json"
}

# config/prod.manifest.json (prod sigue en 1.4.1 hasta validar)
{
  "catalogo": "https://cdn.miapp.com/catalogo/1.4.1/remoteEntry.json",
  "carrito":  "https://cdn.miapp.com/carrito/2.0.0/remoteEntry.json"
}

    La promoción puede ser un job manual (workflow_dispatch) que sobreescribe el manifest del entorno con la versión aprobada — con max-age corto o no-cache en ese archivo.

  4. El shell se construye UNA vez y lee el manifest de su entorno 
    // main.ts del shell
const env = (window as any).ENV ?? 'prod';   // inyectado por el server o meta tag
initFederation(`https://cdn.miapp.com/config/${env}.manifest.json`)
  .then(() => import('./bootstrap'));

    Así el mismo artefacto del shell corre en staging y prod: cero rebuilds para promover remotes.

  5. Rollback en segundos 
    # catalogo 1.4.2 rompió prod → revertir = apuntar el manifest a la anterior
"catalogo": "https://cdn.miapp.com/catalogo/1.4.1/remoteEntry.json"
    el rollback no compila ni despliega nada: es un cambio de un JSON. Las versiones anteriores siguen en el CDN porque las rutas son inmutables. Mide el tiempo total: debe ser menor a 1 minuto incluyendo la invalidación de caché del manifest.
  6. Guardas de calidad antes de promover
    • Smoke E2E del shell en staging apuntando a la versión candidata (contract testing cross-MFE).
    • El manifest de prod solo lo modifica el pipeline de promoción (nunca a mano) — queda auditado en git.
    • Alerta si un remote en prod tiene más de N versiones de distancia con main.
