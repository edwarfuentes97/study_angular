1 · ¿Qué son los microfrontends?

Los microfrontends (MFE) extienden la idea de los microservicios al frontend: en lugar de una única SPA monolítica, la aplicación se compone de piezas independientes, cada una desarrollada, testeada y desplegada por un equipo autónomo. Un shell (también llamado host o contenedor) orquesta la carga de cada pieza en runtime.

La motivación principal no es técnica, es organizacional: cuando 5+ equipos trabajan sobre el mismo repositorio frontend, los merges se vuelven dolorosos, los releases se bloquean entre sí y un bug de un equipo detiene el deploy de todos. Los MFE alinean la arquitectura con la estructura de equipos (Ley de Conway) y permiten deploys independientes. 

┌──────────────────────────────────────────────────────┐
│                    Shell (host)                      │
│  header · navegación · auth · orquestación de MFEs   │
│                                                      │
│  ┌────────────────┐ ┌───────────────┐ ┌───────────┐  │
│  │  MFE Catálogo  │ │  MFE Carrito  │ │ MFE Pagos │  │
│  │  Equipo A      │ │  Equipo B     │ │ Equipo C  │  │
│  │  deploy propio │ │  deploy propio│ │ deploy    │  │
│  └────────────────┘ └───────────────┘ └───────────┘  │
└──────────────────────────────────────────────────────┘
Cada MFE: repo/carpeta propia · pipeline propio · release propio

Monolito frontend vs microfrontends

AspectoSPA monolíticaMicrofrontends
DeployTodo o nada, releases coordinadosIndependiente por equipo
OwnershipDifuso, código compartido por todosClaro: un equipo, un dominio
Complejidad operativaBaja (un build, un pipeline)Alta (N pipelines, versionado, manifest)
Consistencia UXNatural (mismo código)Requiere design system compartido
Peso del bundleOptimizable globalmenteRiesgo de duplicar dependencias
OnboardingUn solo proyecto que entenderCada MFE es pequeño y enfocado

Cuándo SÍ usar microfrontends

  • Varios equipos (3+) con dominios de negocio bien separados (catálogo, checkout, cuenta…).
  • Necesitas ciclos de release independientes: un equipo despliega a diario, otro cada mes.
  • Migración incremental: convivir con un legacy (AngularJS, jQuery) mientras migras por partes.
  • Distintas partes de la app tienen ciclos de vida o stacks distintos y eso está justificado.

Cuándo NO (trade-offs honestos)

  • Un solo equipo o una app pequeña/mediana: los MFE añaden complejidad operacional (infra, versionado, debugging distribuido) sin devolver nada a cambio.
  • Si la UX exige transiciones y estado íntimamente compartidos entre secciones: la frontera de MFE se vuelve artificial y acabarás acoplándolos.
  • Presupuesto de performance estricto: la duplicación de dependencias y la carga remota tienen coste real en KB y latencia.
  • Sin design system maduro, la consistencia visual se degrada rápido entre equipos.
⚠️ Errores comunes
  • Adoptar MFE "porque lo hace Netflix" sin tener el problema organizacional que resuelven.
  • Cortar los MFE por capas técnicas (header-MFE, footer-MFE) en vez de por dominios de negocio.
  • Convertir el shell en un mini-monolito con lógica de negocio de todos los equipos.
  • Ignorar el coste de mantener N pipelines, N repos y N versiones de Angular sincronizadas.

2 · Estrategias de integración

Hay varias formas de componer microfrontends en una sola experiencia. La decisión clave es cuándo ocurre la composición: en build, en el servidor o en el navegador.

1. Build-time — paquetes npm

Cada "MFE" se publica como paquete npm y el shell los instala como dependencias. Es lo más simple, pero no es un microfrontend real: cualquier cambio exige rebuild y redeploy del shell, perdiendo el deploy independiente. 

// package.json del shell — composición en build-time
{
  "dependencies": {
    "@acme/mfe-catalog": "^2.1.0",   // ← nueva versión ⇒ rebuild del shell
    "@acme/mfe-checkout": "^1.4.0"
  }
}

2. Run-time en el cliente — Module Federation / import maps

El shell descarga los remotes en el navegador, en runtime. Es el enfoque dominante hoy: Module Federation (webpack/rspack), Native Federation (esbuild) o import maps nativos del navegador con módulos ES. 

<!-- index.html — import map nativo del navegador -->
<script type="importmap">
{
  "imports": {
    "catalog/": "https://cdn.acme.com/catalog/v3/",
    "checkout/": "https://cdn.acme.com/checkout/v7/"
  }
}
</script>

<script type="module">
  // El navegador resuelve la URL usando el import map
  const { mount } = await import('catalog/bootstrap.js');
  mount(document.querySelector('#catalog-slot'));
</script>

3. iframes

Aislamiento total (CSS, JS, incluso versiones de framework) y seguridad fuerte. El precio: comunicación incómoda (postMessage), routing y deep-linking difíciles, problemas de tamaño/foco/accesibilidad. Útil para integrar apps de terceros o legacy intocable.

4. Server-side composition

El HTML se ensambla en el servidor o en el edge (SSI, ESI, Podium, Tailor, module federation en SSR). Excelente para performance percibida y SEO; requiere infraestructura de composición.

Tabla comparativa

⚠️ Errores comunes
  • Llamar "microfrontends" a librerías npm: sin deploy independiente no hay autonomía real.
  • Descartar iframes por moda: para integrar un tercero no confiable siguen siendo la opción correcta.
  • Mezclar estrategias sin criterio (unos MFE por npm, otros federados) y duplicar Angular dos veces.

3 · Module Federation (webpack)

Module Federation (webpack 5) permite que una app cargue código de otra app compilada y desplegada por separado, compartiendo dependencias en runtime. Dos roles: el host (shell) consume, y el remote expone. Un mismo build puede ser host y remote a la vez (federación bidireccional).

  • remotes — qué apps remotas consume el host y dónde está su remoteEntry.js.
  • exposes — qué módulos publica un remote hacia el exterior.
  • shared — dependencias que host y remotes intentan compartir en runtime.

webpack.config.js del remote (mfe-catalog)

const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
  output: {
    // URL pública desde la que se servirá este remote
    publicPath: 'https://cdn.acme.com/catalog/',
    uniqueName: 'catalog',            // evita colisiones de webpack runtime
  },
  plugins: [
    new ModuleFederationPlugin({
      name: 'catalog',                // nombre global del contenedor remoto
      filename: 'remoteEntry.js',     // manifiesto que descargará el host

      // ── Qué expone este MFE ─────────────────────────────
      exposes: {
        // alias público            → archivo interno
        './Routes':   './src/app/catalog/catalog.routes.ts',
        './Widget':   './src/app/widget/best-sellers.component.ts',
      },

      // ── Dependencias compartidas ────────────────────────
      shared: {
        '@angular/core': {
          singleton: true,            // UNA sola instancia en toda la página
          strictVersion: true,        // error si las versiones no son compatibles
          requiredVersion: '^20.0.0', // rango que este remote necesita
        },
        '@angular/common':  { singleton: true, strictVersion: true, requiredVersion: '^20.0.0' },
        '@angular/router':  { singleton: true, strictVersion: true, requiredVersion: '^20.0.0' },
        rxjs:               { singleton: true, requiredVersion: '^7.8.0' },
      },
    }),
  ],
};

webpack.config.js del host (shell)

new ModuleFederationPlugin({
  name: 'shell',
  remotes: {
    // alias local → nombreGlobal@URL del remoteEntry
    catalog:  'catalog@https://cdn.acme.com/catalog/remoteEntry.js',
    checkout: 'checkout@https://cdn.acme.com/checkout/remoteEntry.js',
  },
  shared: { /* mismas dependencias y flags que los remotes */ },
});

// En el código del shell — import dinámico federado:
const routes = await import('catalog/Routes');   // resuelto en runtime

Cómo funciona shared

  • singleton: true — obligatorio para Angular y RxJS: dos instancias de @angular/core rompen la DI y la detección de cambios.
  • strictVersion: true — lanza error en runtime si el rango no se satisface; sin él, solo un warning (y bugs sutiles después).
  • requiredVersion — rango semver que cada parte declara necesitar; el runtime negocia la versión más alta compatible.
  • eager: true — incluye la dependencia en el bundle inicial (evita el bootstrap asíncrono, pero pierde el compartido dinámico). Úsalo con cuidado.
⚠️ Errores comunes
  • Olvidar singleton: true en Angular/RxJS ⇒ errores tipo NG0203 inject() must be called from an injection context por dos runtimes de Angular.
  • No usar bootstrap asíncrono (import('./bootstrap') en main.ts): webpack necesita ese salto para negociar las dependencias compartidas.
  • Hardcodear las URLs de remotes en el build ⇒ imposible promover el mismo artefacto entre entornos.
  • Exponer componentes internos sueltos en vez de una API estable (rutas o un módulo de entrada).

4 · Native Federation (@angular-architects/native-federation)

Desde Angular 17 el builder por defecto usa esbuild, no webpack, así que el plugin clásico de Module Federation no aplica. Native Federation reimplementa la misma idea sobre estándares del navegador: módulos ES + import maps. Es la opción recomendada para MFE con Angular moderno y es agnóstica del bundler. 

# Instalar y activar en cada app (shell y remotes)
npm i @angular-architects/native-federation
ng g @angular-architects/native-federation:init \
  --project mfe-catalog --port 4201 --type remote

ng g @angular-architects/native-federation:init \
  --project shell --port 4200 --type dynamic-host

federation.config.js del remote

const { withNativeFederation, shareAll } =
  require('@angular-architects/native-federation/config');

module.exports = withNativeFederation({
  name: 'mfe-catalog',

  exposes: {
    // El remote expone sus rutas como punto de entrada estable
    './routes': './src/app/catalog/catalog.routes.ts',
  },

  shared: {
    // Comparte todo lo de package.json como singleton
    ...shareAll({
      singleton: true,
      strictVersion: true,
      requiredVersion: 'auto',   // toma la versión del package.json
    }),
  },

  // Paquetes que NO deben federarse (solo tipos, server-only…)
  skip: ['rxjs/ajax', 'rxjs/testing'],
});

Bootstrap del shell: initFederation + manifest

// main.ts del shell — inicializa la federación ANTES de arrancar Angular
import { initFederation } from '@angular-architects/native-federation';

initFederation('/assets/federation.manifest.json')
  .catch(err => console.error(err))
  .then(() => import('./bootstrap'))   // ← bootstrap diferido obligatorio
  .catch(err => console.error(err));

// bootstrap.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { appConfig } from './app/app.config';
import { App } from './app/app';

bootstrapApplication(App, appConfig);
// assets/federation.manifest.json — un archivo POR ENTORNO
// El shell lo carga en runtime: mismo build, distintas URLs
{
  "mfe-catalog":  "https://cdn.acme.com/catalog/remoteEntry.json",
  "mfe-checkout": "https://cdn.acme.com/checkout/remoteEntry.json"
}

Rutas lazy con loadRemoteModule

// app.routes.ts del shell
import { loadRemoteModule } from '@angular-architects/native-federation';

export const routes: Routes = [
  {
    path: 'catalog',
    loadChildren: () =>
      loadRemoteModule('mfe-catalog', './routes')   // (remote, expose)
        .then(m => m.CATALOG_ROUTES),
  },
  {
    path: 'checkout',
    loadChildren: () =>
      loadRemoteModule('mfe-checkout', './routes')
        .then(m => m.CHECKOUT_ROUTES),
  },
];
⚠️ Errores comunes
  • Bootstrapear Angular directamente en main.ts: el import map debe instalarse antes de que se evalúe cualquier módulo de Angular.
  • Apuntar el manifest a remoteEntry.js (webpack) en vez de remoteEntry.json (native federation): no son intercambiables.
  • Olvidar CORS en el CDN de los remotes: el shell los carga cross-origin.
  • Exponer decenas de componentes en vez de un único ./routes: cada expose es superficie de contrato que hay que mantener.

5 · single-spa

single-spa es un meta-framework que orquesta múltiples SPAs (Angular, React, Vue…) en una misma página. Un root config registra cada aplicación con una activity function que decide, según la URL, cuándo montarla o desmontarla. Cada app implementa el ciclo de vida bootstrap / mount / unmount.

Root config

// root-config.ts — el orquestador (no tiene UI propia)
import { registerApplication, start } from 'single-spa';

registerApplication({
  name: '@acme/catalog',
  app: () => System.import('@acme/catalog'),      // carga vía import map
  activeWhen: ['/catalog'],                        // activity function (prefijo)
});

registerApplication({
  name: '@acme/checkout',
  app: () => System.import('@acme/checkout'),
  // activity function explícita: control total sobre cuándo montar
  activeWhen: (location) =>
    location.pathname.startsWith('/checkout') &&
    !location.pathname.startsWith('/checkout/admin'),
});

start({ urlRerouteOnly: true });

App Angular con single-spa-angular

// main.single-spa.ts — expone los lifecycles en vez de bootstrapear
import { singleSpaAngular } from 'single-spa-angular';

const lifecycles = singleSpaAngular({
  bootstrapFunction: (singleSpaProps) =>
    bootstrapApplication(App, appConfig),
  template: '<app-catalog-root />',
});

export const { bootstrap, mount, unmount } = lifecycles;

Parcels

Un parcel es un fragmento de UI montable manualmente (no ligado a rutas): ideal para incrustar un widget de un framework dentro de otro, por ejemplo un mapa React dentro de una página Angular. 

import { mountRootParcel } from 'single-spa';

const parcel = mountRootParcel(
  () => System.import('@acme/react-map-widget'),
  { domElement: document.getElementById('map-slot')!, zoom: 12 }
);
// después: parcel.unmount()

¿single-spa o Module Federation?

Criteriosingle-spaModule/Native Federation
Orquestación de ciclo de vida✅ Núcleo del framework❌ La haces tú (o el Router)
Multi-framework✅ Pensado para ello⚠️ Posible, pero incómodo
Compartir dependencias⚠️ Manual (import maps)✅ Negociación automática
Integración con Angular Router⚠️ Requiere cuidado✅ Rutas lazy naturales

Regla práctica: todo Angular ⇒ Native Federation; mezcla real de frameworks o migración larga ⇒ single-spa (a menudo combinado con federation para cargar los bundles).

⚠️ Errores comunes
  • Meter UI y lógica de negocio en el root config: debe ser mínimo y aburrido.
  • Activity functions solapadas que montan dos apps sobre el mismo nodo DOM.
  • No limpiar en unmount (listeners, intervals, estilos globales) ⇒ fugas al navegar.
  • Usar single-spa cuando todos los MFE son Angular: pagas la complejidad multi-framework gratis.

6 · Web Components y Angular Elements

Los custom elements son la frontera framework-agnóstica por excelencia: un MFE se publica como <acme-catalog> y cualquier host (Angular, React, HTML plano) lo consume como una etiqueta más. Angular Elements (@angular/elements) empaqueta un componente Angular como custom element con createCustomElement. 

// main.ts del MFE publicado como Web Component
import { createApplication } from '@angular/platform-browser';
import { createCustomElement } from '@angular/elements';
import { CatalogWidgetComponent } from './app/catalog-widget.component';

(async () => {
  // App Angular "sin componente raíz": solo provee el injector
  const app = await createApplication({
    providers: [provideHttpClient()],
  });

  const CatalogElement = createCustomElement(CatalogWidgetComponent, {
    injector: app.injector,
  });

  // A partir de aquí el navegador gestiona el ciclo de vida
  customElements.define('acme-catalog', CatalogElement);
})();
<!-- Cualquier host lo consume sin saber que es Angular -->
<acme-catalog category="books" limit="10"></acme-catalog>

<script>
  const el = document.querySelector('acme-catalog');
  // @Input → atributo/propiedad · @Output → CustomEvent
  el.addEventListener('addToCart', (e) => {
    console.log('producto:', e.detail);
  });
</script>

Shadow DOM y estilos

  • encapsulation: ViewEncapsulation.ShadowDom — aislamiento real de CSS: ni tus estilos salen ni los del host entran. Ideal para MFE embebidos en páginas ajenas.
  • Con Shadow DOM, los estilos globales del design system no penetran: comparte tokens vía CSS custom properties (sí atraviesan la frontera).
  • La encapsulación Emulated (default) NO aísla de estilos globales agresivos del host; solo evita colisiones de tus propios selectores.
  • Cuidado con overlays (dialogs, tooltips) que se montan en body: quedan fuera del shadow root y pierden los estilos.
⚠️ Errores comunes
  • Definir dos veces el mismo tag ⇒ NotSupportedError: protege con if (!customElements.get('acme-catalog')).
  • Cargar Angular completo por cada custom element en la misma página en vez de compartir runtime.
  • Pasar objetos complejos como atributos HTML (solo strings): usa propiedades.
  • Asumir que Shadow DOM te da aislamiento de JavaScript: solo aísla DOM y CSS.

7 · Comunicación entre microfrontends

Regla de oro: cuanto menos, mejor. Si dos MFE necesitan hablar constantemente, la frontera está mal trazada. Para lo inevitable (usuario autenticado, carrito, tema), usa mecanismos de plataforma desacoplados y tipados por contrato, nunca imports directos entre MFEs.

1. Custom events — pub/sub por el DOM

// Contrato compartido (paquete npm ligero SOLO de tipos)
export interface CartItemAddedV1 {
  version: 1;                       // versionado explícito del evento
  productId: string;
  qty: number;
}
export const CART_ITEM_ADDED = 'acme:cart:item-added';

// MFE Catálogo — publica
window.dispatchEvent(new CustomEvent<CartItemAddedV1>(CART_ITEM_ADDED, {
  detail: { version: 1, productId: 'p-42', qty: 1 },
}));

// MFE Carrito — suscribe (y limpia al destruirse)
const onAdd = (e: Event) => {
  const detail = (e as CustomEvent<CartItemAddedV1>).detail;
  if (detail.version !== 1) return;   // tolerante a versiones futuras
  this.cart.add(detail.productId, detail.qty);
};
window.addEventListener(CART_ITEM_ADDED, onAdd);
// ngOnDestroy: window.removeEventListener(CART_ITEM_ADDED, onAdd);

2. BroadcastChannel — entre pestañas y contextos

// Sincronizar logout entre pestañas y MFEs
const channel = new BroadcastChannel('acme:auth');

channel.postMessage({ type: 'LOGOUT', at: Date.now() });

channel.onmessage = (msg) => {
  if (msg.data.type === 'LOGOUT') this.router.navigate(['/login']);
};
// cerrar al destruir: channel.close();

3. Estado compartido mínimo en el shell

// El shell expone una API pequeña y estable (no su Store interno)
export interface PlatformContext {
  user$: Observable<User | null>;      // solo lectura
  locale$: Observable<string>;
  navigate(url: string): void;
}
// Se entrega al MFE al montarlo (props / DI / window.__platform__)

Por qué NO compartir el Store

  • Compartir un Store NgRx entre MFEs acopla sus releases: cambiar el shape del estado exige coordinar todos los equipos — justo lo que los MFE querían evitar.
  • Los eventos son contratos pequeños y versionables; un store es una estructura interna gigante sin fronteras.
  • Contrato explícito: nombres con namespace (acme:cart:*), payload versionado (version: 1), documentado y con tests de contrato.
  • Evolución: agrega campos de forma retrocompatible; para breaking changes, publica V2 y mantén V1 un tiempo (deprecación gradual).
⚠️ Errores comunes
  • Importar servicios de otro MFE "porque total, está federado": acoplamiento invisible en runtime.
  • Eventos sin namespace ni versión: colisiones y breaking changes silenciosos.
  • No desuscribirse de window/BroadcastChannel en ngOnDestroy ⇒ fugas y handlers duplicados al remontar el MFE.
  • Usar localStorage como bus de eventos (solo dispara storage en OTRAS pestañas).

8 · Routing en shell y remotes

El patrón estándar: el shell posee el router y asigna un prefijo de URL por MFE (/catalog/**, /checkout/**). Cada remote define sus rutas relativas a ese prefijo, sin saber dónde será montado. Así el deep linking (F5 en /catalog/products/42) funciona: el shell resuelve el prefijo, carga el remote y este resuelve el resto. 

// shell/app.routes.ts — un prefijo por MFE
export const routes: Routes = [
  {
    path: 'catalog',
    loadChildren: () =>
      loadRemoteModule('mfe-catalog', './routes').then(m => m.CATALOG_ROUTES),
  {,
  { path: '', pathMatch: 'full', redirectTo: 'catalog' },
  { path: '**', component: NotFoundComponent },   // 404 del shell, al final
];

// mfe-catalog/catalog.routes.ts — SIEMPRE rutas relativas
export const CATALOG_ROUTES: Routes = [
  { path: '', component: ProductListComponent },
  { path: 'products/:id', component: ProductDetailComponent },
  // NO usar rutas absolutas ni redirectTo: '/algo' aquí
];

Sincronización cuando el remote tiene router propio

Con single-spa o Web Components, cada MFE puede llevar su propio router. Entonces hay que sincronizar: los routers de los MFE escuchan popstate, y las navegaciones programáticas deben notificar al resto (single-spa reenvía eventos de routing; con Angular puedes emitir un CustomEvent de navegación que el shell traduce). Con federation + un único Angular Router del shell, este problema desaparece — otra razón para preferirlo en stacks 100 % Angular.

  • El remote en standalone (desarrollo aislado) monta sus mismas rutas bajo '' y así se prueba sin el shell.
  • Nunca hagas router.navigate(['/checkout']) desde un MFE hacia rutas de otro: pide la navegación al shell (evento o PlatformContext.navigate).
  • Configura el servidor/CDN para servir index.html del shell en cualquier ruta (fallback SPA), o el deep linking devolverá 404.
⚠️ Errores comunes
  • Rutas absolutas dentro del remote ⇒ funciona standalone, rompe montado bajo prefijo.
  • Wildcard ** del shell declarado antes de los prefijos de MFE ⇒ se traga todo.
  • Dos routers (shell y remote) peleando por la misma URL sin un dueño claro.
  • Guards del shell que asumen conocimiento de rutas internas del remote.

9 · Dependencias compartidas y versionado

El punto más delicado de los MFE en runtime. Angular y RxJS deben ser singleton: dos copias de @angular/core significan dos sistemas de DI, dos zonas de change detection y errores imposibles de depurar. Otras librerías (date-fns, lodash) pueden duplicarse sin romper nada — solo pesan más. 

// Qué significa cada flag al negociar versiones en runtime
shared: {
  '@angular/core': {
    singleton: true,          // solo UNA instancia viva en la página
    strictVersion: true,      // incompatibilidad ⇒ error inmediato y visible
    requiredVersion: '^20.1.0'
  }
}

// Escenario: shell trae @angular/core 20.1.0 (singleton)
//  · remote pide ^20.0.5  → OK, reutiliza 20.1.0 (rango compatible)
//  · remote pide ^21.0.0  → strictVersion: ERROR al cargar ese remote
//                           sin strictVersion: warning + usa 20.1.0 (💣 riesgo)

Estrategias frente a versiones incompatibles

  • Contrato de plataforma: el shell publica qué versiones singleton ofrece (Angular 20.x, RxJS 7.x); los MFE se comprometen a ese rango. Es la opción sana.
  • Upgrade en tren (version train): ventanas acordadas donde todos los MFE suben la major de Angular; los singletons impiden mezclarlas indefinidamente.
  • Aislamiento total para el MFE rezagado: empaquetarlo como Web Component / iframe con SU propio Angular (paga bundle doble, gana independencia temporal).
  • Librerías no-singleton: deja que cada MFE traiga la suya y monitoriza el peso; a veces duplicar 30 KB es más barato que coordinar equipos.
  • Alinea también los polyfills y zone.js (o zoneless en todos): cargarlos dos veces produce bugs globales.
  • Automatiza la verificación: un job de CI que compara requiredVersion de todos los MFE contra el shell antes de desplegar.
⚠️ Errores comunes
  • Desactivar strictVersion "para que arranque": conviertes un error visible en corrupción silenciosa en producción.
  • Compartir librerías con estado de módulo (registries, plugin systems) sin singleton: cada MFE ve un registry distinto.
  • Subir Angular major en un remote sin plan de tren: su deploy queda bloqueado meses.
  • Olvidar que los secondary entry points (@angular/common/http) también deben compartirse.

  • El shell está en Angular 20 y un equipo necesita Angular 21 ya: ¿qué opciones tienes y qué coste tiene cada una?

10 · Deploy independiente y CI/CD

El deploy independiente es la razón de ser de los MFE: si para publicar el catálogo hay que redeployar el shell, solo tienes un monolito distribuido. Cada MFE tiene su pipeline (build → test → publish a CDN) y el shell descubre las versiones vía manifest dinámico por entorno. 

# .github/workflows/mfe-catalog.yml — pipeline SOLO de este MFE
name: deploy-mfe-catalog
on:
  push:
    branches: [main]
    paths: ['apps/mfe-catalog/**']        # solo cambios de este MFE

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx nx build mfe-catalog     # o ng build
      - run: npx nx test mfe-catalog
      # Publica INMUTABLE por versión + actualiza alias del entorno
      - run: |
          aws s3 sync dist/mfe-catalog s3://cdn/catalog/${GITHUB_SHA}/
          aws s3 cp s3://cdn/catalog/${GITHUB_SHA}/ s3://cdn/catalog/staging/ --recursive

Manifest dinámico: promoción y rollback sin rebuild

// federation.manifest.json (staging)
{
  "mfe-catalog":  "https://cdn.acme.com/catalog/staging/remoteEntry.json",
  "mfe-checkout": "https://cdn.acme.com/checkout/staging/remoteEntry.json"
}

// Producción: mismo build del shell, otro manifest.
// ROLLBACK del catálogo = apuntar su entrada a la versión anterior:
{
  "mfe-catalog": "https://cdn.acme.com/catalog/a1b2c3d/remoteEntry.json"
}
// Sin tocar shell ni al resto de equipos. Segundos, no horas.

Prácticas clave

  • Artefactos inmutables versionados por hash/semver; los "alias" de entorno (staging, prod) son punteros — el rollback es mover un puntero.
  • remoteEntry.json/manifest con Cache-Control: no-cache (o TTL muy corto); los chunks con hash pueden cachearse para siempre.
  • Smoke test post-deploy: el shell de staging carga el remote nuevo y corre un E2E mínimo antes de promover a prod.
  • Canary por MFE: el manifest puede servir la v2 del catálogo solo al 5 % de usuarios.
  • Observabilidad por MFE: etiqueta errores (Sentry/OTel) con nombre y versión del remote para saber a qué equipo despertar.
⚠️ Errores comunes
  • URLs de remotes compiladas dentro del bundle del shell ⇒ cada cambio de versión exige redeploy del shell (adiós independencia).
  • Cachear el manifest o el remoteEntry agresivamente ⇒ usuarios atrapados en versiones viejas.
  • Sobrescribir artefactos (deploy mutable) ⇒ imposible hacer rollback fiable.
  • No probar la matriz de compatibilidad shell↔remote antes de promover: cada uno pasa su CI en verde y la combinación explota en prod.

  • ¿Qué políticas de caché aplicarías al manifest, al remoteEntry y a los chunks?

11 · Monorepo vs polyrepo

Los MFE son una decisión de deploy, no de repositorio: puedes tener MFE con deploy independiente dentro de un monorepo (Nx) o repartidos en polyrepo. El monorepo facilita compartir código, tooling y refactors atómicos; el polyrepo maximiza la autonomía (y la fricción para compartir).

CriterioMonorepo (Nx)Polyrepo
Compartir design system✅ Import directo, refactor atómico⚠️ Publicar npm, versionar, esperar upgrades
Autonomía de equipo⚠️ Reglas y CODEOWNERS necesarios✅ Total (repo, tooling, permisos)
Coherencia de versiones✅ Una sola versión de Angular❌ Drift casi garantizado
CI✅ nx affected construye solo lo tocado⚠️ N pipelines aislados, integración tardía
Escala organizativaIdeal hasta decenas de equipos con buena disciplinaIdeal con organizaciones muy distribuidas

Nx: module boundaries como ley

// Cada proyecto se etiqueta por scope (dominio) y type (capa)
// project.json → "tags": ["scope:catalog", "type:feature"]

// eslint.config — @nx/enforce-module-boundaries
{
  "rules": {
    "@nx/enforce-module-boundaries": ["error", {
      "depConstraints": [
        // catálogo solo puede depender de sí mismo y de shared
        { "sourceTag": "scope:catalog",
          "onlyDependOnLibsWithTags": ["scope:catalog", "scope:shared"] },
        { "sourceTag": "scope:checkout",
          "onlyDependOnLibsWithTags": ["scope:checkout", "scope:shared"] },
        // el shell puede orquestar a todos
        { "sourceTag": "scope:shell",
          "onlyDependOnLibsWithTags": ["scope:*"] }
      ]
    }]
  }
}
// Resultado: importar checkout desde catalog ⇒ error de lint en CI
  • Ownership: CODEOWNERS por carpeta — cada equipo aprueba solo su dominio.
  • Design system compartido: en monorepo es una lib scope:shared estable; en polyrepo, un paquete npm con semver disciplinado y changelog.
  • Regla anti-acoplamiento: los MFE nunca se importan entre sí; solo pueden depender de shared — federation es la única puerta en runtime.
⚠️ Errores comunes
  • Monorepo sin boundaries: en 6 meses todos importan de todos y los "MFE" son un monolito con pasos extra.
  • Polyrepo con design system sin releases frecuentes: cada MFE congela una versión distinta y la UX diverge.
  • Confundir monorepo con deploy conjunto: nx affected existe precisamente para desplegar solo lo que cambió.

12 · Testing de microfrontends

La pirámide clásica sigue valiendo dentro de cada MFE (unit + component tests normales). Lo nuevo son las fronteras: ¿el remote sigue exponiendo lo que el shell espera? ¿los eventos mantienen su contrato? ¿la combinación desplegada funciona de extremo a extremo?

1. Contract testing — la frontera como contrato

// contrato: el remote DEBE exponer './routes' con CATALOG_ROUTES
// contract.spec.ts (corre en el CI del remote)
import { CATALOG_ROUTES } from '../src/app/catalog/catalog.routes';

describe('contrato federado de mfe-catalog', () => {
  it('expone rutas montables bajo un prefijo', () => {
    expect(Array.isArray(CATALOG_ROUTES)).toBe(true);
    // ninguna ruta absoluta que rompa el montaje bajo /catalog
    for (const r of CATALOG_ROUTES) {
      expect(r.path?.startsWith('/')).toBeFalsy();
    }
  });

  it('mantiene el contrato del evento cart:item-added v1', () => {
    const evt: CartItemAddedV1 = { version: 1, productId: 'x', qty: 1 };
    expect(evt.version).toBe(1);   // el tipo compartido compila = contrato vivo
  });
});

2. Shell con remotes mockeados

// El shell se testea SIN depender de que los remotes estén desplegados
// test-setup: manifest que apunta a un remote falso local
{
  "mfe-catalog": "http://localhost:4299/fake-catalog/remoteEntry.json"
}

// o stub de loadRemoteModule en tests unitarios del shell:
const routes: Routes = [{
  path: 'catalog',
  loadChildren: () => Promise.resolve(FAKE_CATALOG_ROUTES),  // stub
}];
// Verifica: prefijos, guards, fallback 404, manejo de error de carga

3. E2E cross-MFE — pocos y críticos

// e2e/checkout-journey.spec.ts (Playwright) — cruza 3 MFEs reales
test('@smoke compra end-to-end', async ({ page }) => {
  await page.goto('/catalog');                       // MFE catálogo
  await page.getByTestId('product-42').click();
  await page.getByRole('button', { name: 'Añadir' }).click();

  await page.goto('/cart');                          // MFE carrito
  await expect(page.getByTestId('cart-count')).toHaveText('1');

  await page.getByRole('link', { name: 'Pagar' }).click();  // MFE pagos
  await expect(page).toHaveURL(/\/checkout/);
});
// Corre contra STAGING con las versiones reales candidatas a prod

Estrategia por capas

NivelQué validaDónde corre
Unit / componentLógica interna del MFECI del MFE (rápido, masivo)
Contract testsExposes, eventos, tipos compartidosCI del MFE + CI del shell
Shell + remotes mockOrquestación, routing, errores de cargaCI del shell
E2E cross-MFEFlujos de negocio críticos integradosStaging, pre-promoción (pocos)
  • Testea también el fallo: ¿qué muestra el shell si un remote devuelve 500 o tarda 10 s? (placeholder, retry, degradación elegante).
  • Los contract tests corren en ambos lados: el remote garantiza lo que expone; el shell fija lo que consume (estilo consumer-driven).
⚠️ Errores comunes
  • Confiar solo en E2E completos: lentos, frágiles y llegan tarde — el contrato roto se detecta antes con un contract test de segundos.
  • E2E del shell que dependen de remotes de OTROS equipos en dev: el pipeline de A se pone rojo por un bug de B.
  • No testear el estado de error de carga de un remote (pantalla en blanco en prod).
  • Mockear tanto que nunca se prueba la combinación real: staging con versiones candidatas es obligatorio antes de promover.
