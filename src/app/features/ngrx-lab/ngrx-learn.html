📖 NgRx — Teoría y Conceptos

🔴 ¿Qué es NgRx?

NgRx es la implementación del patrón Redux para Angular. Proporciona un flujo unidireccional de datos predecible mediante cinco piezas clave:

  • Store: Fuente única de verdad — un objeto inmutable que contiene todo el estado de la app.
  • Actions: Eventos que describen qué pasó (nunca cómo). Formato: [Source] Event.
  • Reducers: Funciones puras que reciben el estado actual + una action y retornan un nuevo estado.
  • Selectors: Funciones memoizadas que extraen y computan slices del Store.
  • Effects: Manejan side effects (HTTP, WebSocket, localStorage) fuera del reducer.
// Ejemplo mínimo — Action
export const loadProducts = createAction('[Product Page] Load Products');

// Reducer
export const productReducer = createReducer(
  initialState,
  on(loadProducts, (state) => ({{ '{{' }} ...state, loading: true {{ '}}' }}))
);

// Selector
export const selectAllProducts = createSelector(
  selectProductState,
  (state) => state.products
);

🔄 Flujo de Datos

El ciclo completo de datos en NgRx sigue este camino:

Component → dispatch(Action) → Reducer → Store → Selector → Component

Para operaciones asíncronas, los Effects interceptan la action, realizan el side effect y despachan una nueva action con el resultado:

// Effect — escucha loadProducts, llama al API, despacha success/failure
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() =>
      this.productService.getAll().pipe(
        map(products => ProductActions.loadProductsSuccess({{ '{' }} products {{ '}' }})),
        catchError(error => of(ProductActions.loadProductsFailure({{ '{' }} error {{ '}' }})))
      )
    )
  )
);

🔁 Ciclo de Vida Completo de NgRx

1. El ciclo completo paso a paso

Flujo síncrono (sin side effects):

Usuario hace click
  → Component.dispatch(action)
    → Store recibe action
      → Reducer procesa action → produce nuevo state
        → Store notifica selectors
          → Selectors re-calculan (si el input cambió)
            → Component recibe nuevo valor
              → View se actualiza

Flujo con side effects (HTTP, WebSocket, etc.):

Action despachada (ej: loadProducts)
  → Effect escucha con ofType(loadProducts)
    → Llama al API (HTTP / WebSocket / localStorage)
      → Éxito: Effect despacha loadProductsSuccess({ products })
      → Error: Effect despacha loadProductsFailure({ error })
        → Reducer procesa Success o Failure
          → Nuevo state (products cargados O error guardado)
            → Selectors recalculan → Component se actualiza

2. Actions en detalle

Las actions describen QUÉ pasó, nunca CÓMO reaccionar. Convención de naming: [Source] Event. 

import { createAction, props, createActionGroup } from '@ngrx/store';

// Action simple (sin payload)
export const loadProducts = createAction('[Product Page] Load Products');

// Action con payload
export const addToCart = createAction(
  '[Cart] Add Item',
  props<{ product: Product }>()
);

// Action Groups (NgRx 15+) — agrupa actions relacionadas
export const ProductActions = createActionGroup({
  source: 'Product Page',
  events: {
    'Load Products': emptyProps(),
    'Load Products Success': props<{ products: Product[] }>(),
    'Load Products Failure': props<{ error: string }>(),
    'Delete Product': props<{ id: string }>(),
    'Delete Product Success': props<{ id: string }>(),
  }
});

// Uso:
this.store.dispatch(ProductActions.loadProducts());
this.store.dispatch(ProductActions.deleteProduct({ id: '123' }));

3. Reducer en detalle

Función pura: (state, action) => newState. NUNCA muta el state directamente — siempre usa spread operator para crear un nuevo objeto. 

import { createReducer, on } from '@ngrx/store';
import { createEntityAdapter, EntityState } from '@ngrx/entity';

// Estado inicial — siempre definir un default
export interface ProductState extends EntityState<Product> {
  loading: boolean;
  error: string | null;
}

const adapter = createEntityAdapter<Product>();
const initialState: ProductState = adapter.getInitialState({
  loading: false,
  error: null,
});

export const productReducer = createReducer(
  initialState,

  // Load — activar loading
  on(ProductActions.loadProducts, (state) => ({
    ...state,
    loading: true,
    error: null,
  })),

  // Success — guardar datos con EntityAdapter
  on(ProductActions.loadProductsSuccess, (state, { products }) =>
    adapter.setAll(products, { ...state, loading: false })
  ),

  // Failure — guardar error
  on(ProductActions.loadProductsFailure, (state, { error }) => ({
    ...state,
    loading: false,
    error,
  })),

  // Delete con EntityAdapter
  on(ProductActions.deleteProductSuccess, (state, { id }) =>
    adapter.removeOne(id, state)
  ),
);

// Feature state registration (lazy-loaded)
// En las rutas del feature:
provideState('products', productReducer);

4. Effects en detalle

Effects manejan side effects fuera del reducer. Se crean con createEffect() y escuchan el stream de actions con ofType().

Operadores críticos en Effects:
  • switchMap → para LOAD (cancela petición anterior si llega otra)
  • exhaustMap → para CREATE/LOGIN (ignora clicks duplicados)
  • mergeMap → para DELETE (varias en paralelo)
  • concatMap → para UPDATE (mantiene el orden)
import { createEffect, Actions, ofType } from '@ngrx/effects';

@Injectable()
export class ProductEffects {
  private actions$ = inject(Actions);
  private productService = inject(ProductService);

  // LOAD — switchMap cancela peticiones anteriores
  loadProducts$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.loadProducts),
      switchMap(() =>
        this.productService.getAll().pipe(
          map(products => ProductActions.loadProductsSuccess({ products })),
          catchError(err =>
            of(ProductActions.loadProductsFailure({ error: err.message }))
          )
        )
      )
    )
  );

  // CREATE — exhaustMap ignora clicks duplicados
  createProduct$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.createProduct),
      exhaustMap(({ product }) =>
        this.productService.create(product).pipe(
          map(created => ProductActions.createProductSuccess({ product: created })),
          catchError(err =>
            of(ProductActions.createProductFailure({ error: err.message }))
          )
        )
      )
    )
  );

  // DELETE — mergeMap permite varias en paralelo
  deleteProduct$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.deleteProduct),
      mergeMap(({ id }) =>
        this.productService.delete(id).pipe(
          map(() => ProductActions.deleteProductSuccess({ id })),
          catchError(err =>
            of(ProductActions.deleteProductFailure({ error: err.message }))
          )
        )
      )
    )
  );
}
⚠️ Error handling — SIEMPRE catchError DENTRO del operador de aplanamiento:
// ✅ CORRECTO — el effect sigue vivo después de un error
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() =>
      this.productService.getAll().pipe(
        map(products => ProductActions.loadProductsSuccess({ products })),
        catchError(err => of(ProductActions.loadProductsFailure({ error: err.message })))
        // ☝️ catchError DENTRO del switchMap — el observable interno falla,
        //    pero el observable externo (el effect) sigue escuchando actions
      )
    )
  )
);

// ❌ INCORRECTO — si falla, el effect MUERE y nunca más procesa actions
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() => this.productService.getAll()),
    map(products => ProductActions.loadProductsSuccess({ products })),
    catchError(err => of(ProductActions.loadProductsFailure({ error: err.message })))
    // ☝️ catchError FUERA del switchMap — mata todo el effect stream
  )
);
Non-dispatching effects y lifecycle hooks:
// Non-dispatching effect — para logging, navigation, analytics
logActions$ = createEffect(() =>
  this.actions$.pipe(
    tap(action => console.log('[NgRx]', action.type))
  ),
  { dispatch: false }  // No despacha ninguna action
);

// Navegar después de crear exitosamente
navigateAfterCreate$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.createProductSuccess),
    tap(({ product }) =>
      inject(Router).navigate(['/products', product.id])
    )
  ),
  { dispatch: false }
);

// Lifecycle hooks
export class ProductEffects implements OnInitEffects {
  // Se ejecuta una vez cuando los effects se registran
  ngrxOnInitEffects() {
    return ProductActions.loadProducts(); // Carga inicial automática
  }
}

5. Selectors en detalle

Los selectors son funciones memoizadas que extraen y computan slices del Store. Solo recalculan si su input cambió (igualdad por referencia). 

import { createFeatureSelector, createSelector } from '@ngrx/store';

// 1. Feature selector — accede al slice del state global
const selectProductState = createFeatureSelector<ProductState>('products');

// 2. Selectors derivados — memoizados automáticamente
const { selectAll, selectEntities, selectTotal } = adapter.getSelectors();

export const selectAllProducts = createSelector(
  selectProductState,
  selectAll   // retorna Product[]
);

export const selectLoading = createSelector(
  selectProductState,
  (state) => state.loading
);

export const selectError = createSelector(
  selectProductState,
  (state) => state.error
);

// 3. Composición de selectors — un selector usa otros selectors
export const selectExpensiveProducts = createSelector(
  selectAllProducts,
  (products) => products.filter(p => p.price > 100)
);

export const selectProductCount = createSelector(
  selectAllProducts,
  (products) => products.length
);

// 4. Selector con parámetros (factory selector)
export const selectProductById = (id: string) =>
  createSelector(
    selectAllProducts,
    (products) => products.find(p => p.id === id)
  );

// 5. selectSignal (NgRx 18+) — retorna Signal en vez de Observable
// En el componente:
products = this.store.selectSignal(selectAllProducts);
// products() retorna Product[] — sin async pipe, síncrono

6. Store lifecycle en un componente

@Component({
  selector: 'app-product-list',
  template: \`
    @if (loading()) {
      <app-spinner />
    }

    @for (product of products(); track product.id) {
      <app-product-card
        [product]="product"
        (delete)="onDelete(product.id)" />
    }

    @if (error()) {
      <p class="error">{{ error() }}</p>
    }
  \`
})
export class ProductListComponent implements OnInit {

  // ✅ Approach moderno: Signals (NgRx 18+)
  products = this.store.selectSignal(selectAllProducts);
  loading = this.store.selectSignal(selectLoading);
  error = this.store.selectSignal(selectError);

  // Approach clásico: Observables
  // products$ = this.store.select(selectAllProducts);

  ngOnInit() {
    // Despachar action para cargar datos
    this.store.dispatch(ProductActions.loadProducts());
  }

  onDelete(id: string) {
    this.store.dispatch(ProductActions.deleteProduct({ id }));
  }
}

NgRx DevTools permite inspeccionar el estado, ver cada action despachada, hacer time-travel, y exportar/importar sesiones completas para debugging.

🧩 Patrones Avanzados de NgRx

1. Facade Pattern

Un servicio "facade" encapsula el Store — los componentes nunca inyectan Store directamente. Esto desacopla los componentes de NgRx, facilita testing, y centraliza la lógica de dispatch/select. 

// product.facade.ts — encapsula TODO el acceso al Store
@Injectable({ providedIn: 'root' })
export class ProductFacade {
  private store = inject(Store);

  // Selectors expuestos como Signals
  readonly products = this.store.selectSignal(selectAllProducts);
  readonly loading = this.store.selectSignal(selectLoading);
  readonly error = this.store.selectSignal(selectError);
  readonly selectedProduct = this.store.selectSignal(selectSelectedProduct);

  // Actions encapsuladas como métodos
  loadProducts() {
    this.store.dispatch(ProductActions.loadProducts());
  }

  createProduct(product: CreateProductDto) {
    this.store.dispatch(ProductActions.createProduct({ product }));
  }

  deleteProduct(id: string) {
    this.store.dispatch(ProductActions.deleteProduct({ id }));
  }

  selectProduct(id: string) {
    this.store.dispatch(ProductActions.selectProduct({ id }));
  }
}

// En el componente — limpio y desacoplado de NgRx
@Component({ ... })
export class ProductListComponent {
  private facade = inject(ProductFacade);
  products = this.facade.products;
  loading = this.facade.loading;

  ngOnInit() { this.facade.loadProducts(); }
  onDelete(id: string) { this.facade.deleteProduct(id); }
}

2. Optimistic Updates

Actualizar la UI inmediatamente antes de que el API responda, y revertir si falla. Mejora la UX percibida. 

// Reducer — actualiza INMEDIATAMENTE al despachar deleteProduct
on(ProductActions.deleteProduct, (state, { id }) =>
  adapter.removeOne(id, state)   // ← optimista: borra antes de confirmar
),

// Si falla, re-agregar el producto
on(ProductActions.deleteProductFailure, (state, { product }) =>
  adapter.addOne(product, state) // ← revertir: lo vuelve a agregar
),

// Effect — intenta borrar en el API
deleteProduct$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.deleteProduct),
    mergeMap(({ id }) =>
      this.productService.delete(id).pipe(
        map(() => ProductActions.deleteProductSuccess({ id })),
        catchError(() => {
          // Necesitamos el producto original para revertir
          const product = this.store.selectSignal(selectProductById(id))();
          return of(ProductActions.deleteProductFailure({ product }));
        })
      )
    )
  )
);

3. Entity Pattern — Normalización

¿Por qué no usar arrays? Buscar un item en un array es O(n). Con EntityAdapter, los datos se guardan como { ids: string[], entities: { [id]: Entity } }, haciendo lookups O(1). Elimina duplicados y simplifica updates. 

// ❌ Con array — lento, propenso a duplicados
interface State {
  products: Product[];  // Buscar: products.find(p => p.id === id) → O(n)
}

// ✅ Con EntityAdapter — normalizado, O(1) lookups
interface State extends EntityState<Product> {
  // EntityState provee: ids: string[], entities: { [id: string]: Product }
  loading: boolean;
}

const adapter = createEntityAdapter<Product>();

// Operaciones CRUD limpias:
adapter.addOne(product, state);              // Agregar uno
adapter.addMany(products, state);            // Agregar varios
adapter.setAll(products, state);             // Reemplazar todos
adapter.updateOne({ id, changes }, state);  // Actualizar parcial
adapter.removeOne(id, state);                // Eliminar uno
adapter.upsertOne(product, state);           // Agregar o actualizar

// Selectors generados automáticamente:
const { selectAll, selectEntities, selectIds, selectTotal } =
  adapter.getSelectors();

5. Meta-reducers

Meta-reducers son middleware que interceptan TODAS las actions antes de que lleguen a los reducers. Útiles para logging, freeze de state en dev, y limpiar state al hacer logout. 

import { MetaReducer, ActionReducer } from '@ngrx/store';

// Meta-reducer para logging
function logger(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    console.group(action.type);
    console.log('State antes:', state);
    const nextState = reducer(state, action);
    console.log('State después:', nextState);
    console.groupEnd();
    return nextState;
  };
}

// Meta-reducer para freeze — previene mutaciones accidentales en dev
function freeze(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    const nextState = reducer(state, action);
    Object.freeze(nextState); // Lanza error si alguien intenta mutar
    return nextState;
  };
}

// Meta-reducer para limpiar state al hacer logout
function clearState(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    if (action.type === '[Auth] Logout') {
      return reducer(undefined, action); // Reset a initialState
    }
    return reducer(state, action);
  };
}

// Registrar meta-reducers
const metaReducers: MetaReducer[] = isDevMode()
  ? [logger, freeze]
  : [clearState];

// En app.config.ts:
provideStore({ router: routerReducer }, { metaReducers });

⚖️ ¿Cuándo usar NgRx vs Servicios simples?

Criterio Servicio con BehaviorSubject NgRx Store
Complejidad del estado Bajo — pocos datos compartidos Alto — muchos componentes leen/escriben
Debugging console.log manual Redux DevTools, time-travel
Boilerplate Mínimo Actions + Reducer + Selectors + Effects
Testabilidad Buena Excelente — cada pieza es una función pura
Equipo grande Difícil mantener consistencia Patrón único, convenciones claras
Side effects Dentro del servicio Aislados en Effects

Regla práctica: Si tu app tiene más de 3 features que comparten estado, NgRx vale la pena.

📦 @ngrx/entity — EntityAdapter

EntityAdapter normaliza colecciones en un formato {{ '{' }} ids: [], entities: {{ '{' }}{{ '}' }} {{ '}' }}, eliminando arrays anidados y haciendo lookups O(1) por ID. 

import {{ '{' }} createEntityAdapter, EntityState {{ '}' }} from '@ngrx/entity';

export interface Product {{ '{' }}
  id: string;
  name: string;
  sustainabilityScore: number;
{{ '}' }}

// Crear el adapter
export const productAdapter = createEntityAdapter<Product>();

// Estado inicial normalizado
export interface ProductState extends EntityState<Product> {{ '{' }}
  loading: boolean;
  error: string | null;
  selectedId: string | null;
{{ '}' }}

export const initialState: ProductState = productAdapter.getInitialState({{ '{' }}
  loading: false,
  error: null,
  selectedId: null,
{{ '}' }});

// En el reducer — operaciones CRUD limpias
on(loadProductsSuccess, (state, {{ '{' }} products {{ '}' }}) =>
  productAdapter.setAll(products, {{ '{' }} ...state, loading: false {{ '}' }})
),
on(addProductSuccess, (state, {{ '{' }} product {{ '}' }}) =>
  productAdapter.addOne(product, state)
),

¿Por qué normalizar? Evita duplicados, simplifica updates, y los selectors son más eficientes.
