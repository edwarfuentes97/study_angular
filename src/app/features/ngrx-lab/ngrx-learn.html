📖 NgRx — Teoría y Conceptos

🔴 ¿Qué es NgRx?

NgRx es la implementación del patrón Redux para Angular. Proporciona un flujo unidireccional de datos predecible mediante cinco piezas clave:

  • Store: Fuente única de verdad — un objeto inmutable que contiene todo el estado de la app.
  • Actions: Eventos que describen qué pasó (nunca cómo). Formato: [Source] Event.
  • Reducers: Funciones puras que reciben el estado actual + una action y retornan un nuevo estado.
  • Selectors: Funciones memoizadas que extraen y computan slices del Store.
  • Effects: Manejan side effects (HTTP, WebSocket, localStorage) fuera del reducer.
// Ejemplo mínimo — Action
export const loadProducts = createAction('[Product Page] Load Products');

// Reducer
export const productReducer = createReducer(
  initialState,
  on(loadProducts, (state) => ({{ '{{' }} ...state, loading: true {{ '}}' }}))
);

// Selector
export const selectAllProducts = createSelector(
  selectProductState,
  (state) => state.products
);

🎤 Preguntas de entrevista

P: ¿Qué problema resuelve NgRx?
R: Gestionar estado compartido complejo de forma predecible: una única fuente de verdad, cambios trazables mediante actions, y side effects aislados. Elimina el "¿quién cambió este dato y cuándo?".
P: ¿Cuáles son los tres principios de Redux que hereda NgRx?
R: 1) Single source of truth (un solo Store), 2) el estado es de solo lectura (solo cambia despachando actions), 3) los cambios se hacen con funciones puras (reducers).
P: ¿Qué significa "flujo unidireccional de datos"?
R: Los datos siempre viajan en un solo sentido: Component → dispatch(Action) → Reducer → Store → Selector → Component. Nunca hay escrituras directas al estado desde la vista.

🔄 Flujo de Datos

El ciclo completo de datos en NgRx sigue este camino:

Component → dispatch(Action) → Reducer → Store → Selector → Component

Para operaciones asíncronas, los Effects interceptan la action, realizan el side effect y despachan una nueva action con el resultado:

// Effect — escucha loadProducts, llama al API, despacha success/failure
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() =>
      this.productService.getAll().pipe(
        map(products => ProductActions.loadProductsSuccess({{ '{' }} products {{ '}' }})),
        catchError(error => of(ProductActions.loadProductsFailure({{ '{' }} error {{ '}' }})))
      )
    )
  )
);

🔁 Ciclo de Vida Completo de NgRx

1. El ciclo completo paso a paso

Flujo síncrono (sin side effects):

Usuario hace click
  → Component.dispatch(action)
    → Store recibe action
      → Reducer procesa action → produce nuevo state
        → Store notifica selectors
          → Selectors re-calculan (si el input cambió)
            → Component recibe nuevo valor
              → View se actualiza

Flujo con side effects (HTTP, WebSocket, etc.):

Action despachada (ej: loadProducts)
  → Effect escucha con ofType(loadProducts)
    → Llama al API (HTTP / WebSocket / localStorage)
      → Éxito: Effect despacha loadProductsSuccess({ products })
      → Error: Effect despacha loadProductsFailure({ error })
        → Reducer procesa Success o Failure
          → Nuevo state (products cargados O error guardado)
            → Selectors recalculan → Component se actualiza

2. Actions en detalle

Las actions describen QUÉ pasó, nunca CÓMO reaccionar. Convención de naming: [Source] Event. 

import { createAction, props, createActionGroup } from '@ngrx/store';

// Action simple (sin payload)
export const loadProducts = createAction('[Product Page] Load Products');

// Action con payload
export const addToCart = createAction(
  '[Cart] Add Item',
  props<{ product: Product }>()
);

// Action Groups (NgRx 15+) — agrupa actions relacionadas
export const ProductActions = createActionGroup({
  source: 'Product Page',
  events: {
    'Load Products': emptyProps(),
    'Load Products Success': props<{ products: Product[] }>(),
    'Load Products Failure': props<{ error: string }>(),
    'Delete Product': props<{ id: string }>(),
    'Delete Product Success': props<{ id: string }>(),
  }
});

// Uso:
this.store.dispatch(ProductActions.loadProducts());
this.store.dispatch(ProductActions.deleteProduct({ id: '123' }));

3. Reducer en detalle

Función pura: (state, action) => newState. NUNCA muta el state directamente — siempre usa spread operator para crear un nuevo objeto. 

import { createReducer, on } from '@ngrx/store';
import { createEntityAdapter, EntityState } from '@ngrx/entity';

// Estado inicial — siempre definir un default
export interface ProductState extends EntityState<Product> {
  loading: boolean;
  error: string | null;
}

const adapter = createEntityAdapter<Product>();
const initialState: ProductState = adapter.getInitialState({
  loading: false,
  error: null,
});

export const productReducer = createReducer(
  initialState,

  // Load — activar loading
  on(ProductActions.loadProducts, (state) => ({
    ...state,
    loading: true,
    error: null,
  })),

  // Success — guardar datos con EntityAdapter
  on(ProductActions.loadProductsSuccess, (state, { products }) =>
    adapter.setAll(products, { ...state, loading: false })
  ),

  // Failure — guardar error
  on(ProductActions.loadProductsFailure, (state, { error }) => ({
    ...state,
    loading: false,
    error,
  })),

  // Delete con EntityAdapter
  on(ProductActions.deleteProductSuccess, (state, { id }) =>
    adapter.removeOne(id, state)
  ),
);

// Feature state registration (lazy-loaded)
// En las rutas del feature:
provideState('products', productReducer);

4. Effects en detalle

Effects manejan side effects fuera del reducer. Se crean con createEffect() y escuchan el stream de actions con ofType().

Operadores críticos en Effects:
  • switchMap → para LOAD (cancela petición anterior si llega otra)
  • exhaustMap → para CREATE/LOGIN (ignora clicks duplicados)
  • mergeMap → para DELETE (varias en paralelo)
  • concatMap → para UPDATE (mantiene el orden)
import { createEffect, Actions, ofType } from '@ngrx/effects';

@Injectable()
export class ProductEffects {
  private actions$ = inject(Actions);
  private productService = inject(ProductService);

  // LOAD — switchMap cancela peticiones anteriores
  loadProducts$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.loadProducts),
      switchMap(() =>
        this.productService.getAll().pipe(
          map(products => ProductActions.loadProductsSuccess({ products })),
          catchError(err =>
            of(ProductActions.loadProductsFailure({ error: err.message }))
          )
        )
      )
    )
  );

  // CREATE — exhaustMap ignora clicks duplicados
  createProduct$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.createProduct),
      exhaustMap(({ product }) =>
        this.productService.create(product).pipe(
          map(created => ProductActions.createProductSuccess({ product: created })),
          catchError(err =>
            of(ProductActions.createProductFailure({ error: err.message }))
          )
        )
      )
    )
  );

  // DELETE — mergeMap permite varias en paralelo
  deleteProduct$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.deleteProduct),
      mergeMap(({ id }) =>
        this.productService.delete(id).pipe(
          map(() => ProductActions.deleteProductSuccess({ id })),
          catchError(err =>
            of(ProductActions.deleteProductFailure({ error: err.message }))
          )
        )
      )
    )
  );
}
⚠️ Error handling — SIEMPRE catchError DENTRO del operador de aplanamiento:
// ✅ CORRECTO — el effect sigue vivo después de un error
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() =>
      this.productService.getAll().pipe(
        map(products => ProductActions.loadProductsSuccess({ products })),
        catchError(err => of(ProductActions.loadProductsFailure({ error: err.message })))
        // ☝️ catchError DENTRO del switchMap — el observable interno falla,
        //    pero el observable externo (el effect) sigue escuchando actions
      )
    )
  )
);

// ❌ INCORRECTO — si falla, el effect MUERE y nunca más procesa actions
loadProducts$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    switchMap(() => this.productService.getAll()),
    map(products => ProductActions.loadProductsSuccess({ products })),
    catchError(err => of(ProductActions.loadProductsFailure({ error: err.message })))
    // ☝️ catchError FUERA del switchMap — mata todo el effect stream
  )
);
Non-dispatching effects y lifecycle hooks:
// Non-dispatching effect — para logging, navigation, analytics
logActions$ = createEffect(() =>
  this.actions$.pipe(
    tap(action => console.log('[NgRx]', action.type))
  ),
  { dispatch: false }  // No despacha ninguna action
);

// Navegar después de crear exitosamente
navigateAfterCreate$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.createProductSuccess),
    tap(({ product }) =>
      inject(Router).navigate(['/products', product.id])
    )
  ),
  { dispatch: false }
);

// Lifecycle hooks
export class ProductEffects implements OnInitEffects {
  // Se ejecuta una vez cuando los effects se registran
  ngrxOnInitEffects() {
    return ProductActions.loadProducts(); // Carga inicial automática
  }

5. Selectors en detalle

Los selectors son funciones memoizadas que extraen y computan slices del Store. Solo recalculan si su input cambió (igualdad por referencia). 

import { createFeatureSelector, createSelector } from '@ngrx/store';

// 1. Feature selector — accede al slice del state global
const selectProductState = createFeatureSelector<ProductState>('products');

// 2. Selectors derivados — memoizados automáticamente
const { selectAll, selectEntities, selectTotal } = adapter.getSelectors();

export const selectAllProducts = createSelector(
  selectProductState,
  selectAll   // retorna Product[]
);

export const selectLoading = createSelector(
  selectProductState,
  (state) => state.loading
);

export const selectError = createSelector(
  selectProductState,
  (state) => state.error
);

// 3. Composición de selectors — un selector usa otros selectors
export const selectExpensiveProducts = createSelector(
  selectAllProducts,
  (products) => products.filter(p => p.price > 100)
);

export const selectProductCount = createSelector(
  selectAllProducts,
  (products) => products.length
);

// 4. Selector con parámetros (factory selector)
export const selectProductById = (id: string) =>
  createSelector(
    selectAllProducts,
    (products) => products.find(p => p.id === id)
  );

// 5. selectSignal (NgRx 18+) — retorna Signal en vez de Observable
// En el componente:
products = this.store.selectSignal(selectAllProducts);
// products() retorna Product[] — sin async pipe, síncrono

🎬 Actions en Profundidad

Una action es un evento, no un comando. Describe algo que ya pasó ("el usuario hizo click en guardar", "el API respondió con error"), y son los reducers y effects quienes deciden cómo reaccionar. Esta distinción — event-driven, no command-driven — es la clave para que NgRx escale.

createAction + props

import { createAction, props } from '@ngrx/store';

// Sin payload — el tipo string DEBE ser único en toda la app
export const loadProducts = createAction('[Products Page] Load Products');

// Con payload tipado — props<T>() define la forma del payload
export const createProduct = createAction(
  '[Products Page] Create Product',
  props<{ product: Omit<Product, 'id'> }>()
);

// Payload derivado — tercera forma: función creadora
export const searchProducts = createAction(
  '[Products Page] Search',
  (term: string) => ({ term: term.trim().toLowerCase() })
);

createActionGroup (NgRx 15+)

Agrupa actions relacionadas bajo un mismo source y genera los nombres de método automáticamente (Title Case → camelCase). Reduce boilerplate drásticamente: 

import { createActionGroup, emptyProps, props } from '@ngrx/store';

export const ProductsApiActions = createActionGroup({
  source: 'Products API',
  events: {
    // 'Load Success' → ProductsApiActions.loadSuccess
    'Load Success': props<{ products: Product[] }>(),
    'Load Failure': props<{ error: string }>(),
    'Delete Success': props<{ id: string }>(),
    'Noop': emptyProps(),   // action sin payload dentro del grupo
  },
});

// Uso — tipos generados: '[Products API] Load Success'
this.store.dispatch(ProductsApiActions.loadSuccess({ products }));

Buenas prácticas de naming

  • [Source] Event: el source dice DÓNDE se originó ([Products Page], [Products API], [Auth Guard]).
  • Actions de UI y de API separadas: la página despacha Load Products; el effect despacha Load Products Success/Failure.
  • Nombra el evento, no el efecto deseado: [Login Page] Login Button Clicked es mejor que [Auth] Set User.
  • Varios reducers/effects pueden reaccionar a la MISMA action — eso es deseable (event-driven).

⚠️ Errores comunes

  • Actions genéricas tipo setter ([App] Set Products) reutilizadas desde 10 sitios: pierdes trazabilidad total en DevTools. Cada origen debería tener su propia action.
  • Tipos duplicados: dos actions con el mismo string type — el reducer responde a ambas sin que lo notes. El type es la identidad de la action.
  • Payloads no serializables (instancias de clase, funciones, Map): rompen DevTools, time-travel e hidratación.
  • Despachar una action "para que el effect haga algo" sin que describa un evento real — señal de que estás usando NgRx como un bus de comandos.

🎤 Preguntas de entrevista

P: ¿Por qué las actions deben ser eventos y no comandos?
R: Un evento describe qué pasó y permite que múltiples consumidores (reducers, effects) reaccionen de forma independiente. Un comando acopla al emisor con un único receptor y convierte el Store en un bus de RPC.
P: ¿Qué ventaja tiene createActionGroup sobre createAction?
R: Agrupa actions con el mismo source, genera automáticamente los creators con nombres camelCase y elimina repetición de strings. Menos boilerplate y menos typos.
P: ¿Pueden dos reducers responder a la misma action?
R: Sí, y es un patrón recomendado: por ejemplo, [Auth] Logout puede limpiar el slice de auth, el de carrito y el de perfil a la vez.

🧮 Reducers e Inmutabilidad

Un reducer es una función pura: (state, action) => newState. Puro significa: sin side effects, sin mutar argumentos, y el mismo input produce SIEMPRE el mismo output. NgRx (y el OnPush de Angular) dependen de la igualdad por referencia: si el estado cambió, la referencia del objeto debe ser nueva.

createReducer + on

export const productReducer = createReducer(
  initialState,   // ← obligatorio: el estado por defecto

  // Un on puede escuchar VARIAS actions con el mismo handler
  on(
    ProductActions.loadProducts,
    ProductActions.createProduct,
    ProductActions.deleteProduct,
    (state) => ({ ...state, loading: true, error: null })
  ),

  // Handler con payload destructurado
  on(ProductActions.loadProductsFailure, (state, { error }) => ({
    ...state,
    loading: false,
    error,
  })),
);

Actualizaciones inmutables anidadas

// ❌ MUTACIÓN — misma referencia, nadie se entera del cambio
on(updateAddress, (state, { city }) => {
  state.user.address.city = city;   // ¡NO! muta el state original
  return state;
});

// ✅ INMUTABLE — copia cada nivel que cambia
on(updateAddress, (state, { city }) => ({
  ...state,
  user: {
    ...state.user,
    address: { ...state.user.address, city },
  },
}));

// ✅ Arrays: nunca push/splice — crea uno nuevo
on(addTag, (state, { tag }) => ({
  ...state,
  tags: [...state.tags, tag],
}));
on(removeTag, (state, { tag }) => ({
  ...state,
  tags: state.tags.filter(t => t !== tag),
}));

Mejores prácticas

  • Reducers "tontos": solo transforman datos. La lógica de negocio va en effects; la lógica de derivación, en selectors.
  • No guardes datos derivados (totales, filtrados): calcúlalos en selectors. El Store guarda el estado mínimo.
  • Estado serializable: primitivos, objetos planos y arrays. Nada de Date (usa ISO string), Map, Set ni instancias de clase.
  • En desarrollo activa provideStore(reducers, { runtimeChecks: { strictStateImmutability: true, strictActionImmutability: true } }) para detectar mutaciones al instante.

⚠️ Errores comunes

  • Mutar y devolver el mismo objeto: los selectors memoizados no recalculan y los componentes OnPush no se re-renderizan. El bug clásico de "el dato cambió pero la UI no".
  • Side effects en el reducer: Date.now(), Math.random(), llamadas HTTP o localStorage rompen la pureza — el timestamp/id se genera en la action creator o en un effect.
  • Spread superficial en objetos anidados: { ...state } copia solo el primer nivel; si mutas state.user.address sigues mutando el original.
  • Lógica condicional compleja por action type dentro de un solo handler — divide en varios on().

🎤 Preguntas de entrevista

P: ¿Qué pasa exactamente si mutas el state en un reducer?
R: La referencia no cambia, así que los selectors memoizados devuelven el valor cacheado, OnPush no dispara change detection y el time-travel de DevTools muestra estados corruptos. Con strictStateImmutability activado, NgRx lanza un error.
P: ¿Por qué los reducers deben ser funciones puras?
R: Garantizan predictibilidad (mismo input → mismo output), permiten testear sin mocks, y hacen posible el time-travel debugging: re-ejecutar la secuencia de actions reconstruye el estado exacto.
P: ¿Dónde generarías un id o timestamp para una entidad nueva?
R: Nunca en el reducer. En la función creadora de la action, en un effect, o idealmente en el backend.

🌊 Effects — Patrones de Concurrencia

El operador de aplanamiento (flattening operator) que elijas define qué pasa cuando llegan varias actions del mismo tipo seguidas. Elegir mal no es un detalle: puede cancelar un POST a mitad de camino o duplicar un pago.

Operador Si llega otra action mientras la anterior está en vuelo… Caso de uso
switchMap Cancela la petición anterior y arranca la nueva GET / búsqueda / autocomplete — solo importa el último resultado
concatMap Encola: espera a que termine la anterior, ejecuta en orden UPDATE / escrituras donde el orden importa
mergeMap Ejecuta todas en paralelo, sin orden garantizado DELETE de items independientes, operaciones sin dependencia
exhaustMap IGNORA las nuevas hasta que termine la actual LOGIN / submit / refresh — anti doble-click

Regla mnemotécnica: lecturas → switchMap; escrituras ordenadas → concatMap; escrituras independientes → mergeMap; acciones que no deben repetirse → exhaustMap. Si dudas con escrituras, concatMap es lo más seguro.

Leer state dentro de un effect — concatLatestFrom

import { concatLatestFrom } from '@ngrx/operators';

// Evitar recargar si ya hay datos en el Store
loadIfEmpty$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProducts),
    concatLatestFrom(() => this.store.select(selectProductTotal)),
    filter(([, total]) => total === 0),   // solo si el store está vacío
    switchMap(() =>
      this.api.getProducts().pipe(
        map((products) => ProductActions.loadProductsSuccess({ products })),
        catchError((err) => of(ProductActions.loadProductsFailure({ error: err.message })))
      )
    )
  )
);
// concatLatestFrom es lazy: solo evalúa el selector cuando llega la action
// (withLatestFrom lo evaluaría siempre, aunque no haya actions)

Reintentos con backoff

switchMap(() =>
  this.api.getProducts().pipe(
    retry({ count: 3, delay: 1000 }),   // 3 reintentos, 1s entre cada uno
    map((products) => ProductActions.loadProductsSuccess({ products })),
    catchError((err) => of(ProductActions.loadProductsFailure({ error: err.message })))
  )
)

⚠️ Errores comunes

  • catchError FUERA del operador de aplanamiento: completa el stream del effect y NUNCA vuelve a procesar actions. Siempre dentro del pipe interno.
  • switchMap en escrituras: si el usuario dispara dos POST rápidos, el primero se cancela en el cliente… pero el servidor quizá ya lo procesó. Resultado: estado inconsistente.
  • Loops infinitos: un effect que despacha (directa o indirectamente) la misma action que escucha. DevTools lo delata: la misma action repetida sin parar.
  • subscribe() manual dentro del effect: rompe la cadena declarativa y pierde el manejo de errores/cancelación. El effect debe devolver un Observable de actions.
  • Despachar la misma action Success desde varios effects sin source distinto — imposible saber quién la emitió.

🎤 Preguntas de entrevista

P: ¿Cuándo usarías exhaustMap y por qué?
R: En login o cualquier submit: mientras la petición está en vuelo, los clicks repetidos se ignoran. Es la protección anti doble-submit más limpia.
P: ¿Por qué "muere" un effect y cómo lo evitas?
R: Si un error alcanza el stream externo del effect, RxJS lo completa y deja de escuchar actions para siempre. Se evita poniendo catchError dentro del observable interno (dentro del switchMap/mergeMap) y devolviendo una action de failure.
P: ¿Cómo lees un valor del Store dentro de un effect?
R: Con concatLatestFrom(() => this.store.select(...)) de @ngrx/operators — es lazy y solo evalúa el selector cuando llega la action.
P: ¿Qué es un non-dispatching effect?
R: Un effect con { dispatch: false } que ejecuta un side effect terminal (navegación, analytics, toast, logging) sin emitir una nueva action.

⚡ Functional Effects (NgRx moderno)

Desde NgRx 15 los effects pueden ser funciones en vez de clases. Las dependencias se obtienen con inject() como parámetros por defecto, lo que los hace tree-shakeables y triviales de testear (puedes pasar mocks como argumentos). 

// product.effects.ts — sin clase, sin @Injectable
import { createEffect, Actions, ofType } from '@ngrx/effects';
import { inject } from '@angular/core';

export const loadProducts = createEffect(
  (actions$ = inject(Actions), api = inject(MockApiService)) =>
    actions$.pipe(
      ofType(ProductActions.loadProducts),
      switchMap(() =>
        api.getProducts().pipe(
          map((products) => ProductActions.loadProductsSuccess({ products })),
          catchError((err) =>
            of(ProductActions.loadProductsFailure({ error: err.message }))
          )
        )
      )
    ),
  { functional: true }   // ← imprescindible
);

// Non-dispatching functional effect
export const logActions = createEffect(
  (actions$ = inject(Actions)) =>
    actions$.pipe(tap((a) => console.log('[NgRx]', a.type))),
  { functional: true, dispatch: false }
);

// Registro — se pasa un objeto con los effects (o import * as)
import * as productEffects from './store/product.effects';
provideEffects(productEffects);

🎯 Selectors — Memoización y Composición

createSelector memoiza con caché de tamaño 1: guarda los últimos inputs y el último resultado. Si los inputs son los mismos por referencia, devuelve el resultado cacheado sin ejecutar la función proyectora. Por eso la inmutabilidad del reducer es crítica: referencia nueva = recálculo; misma referencia = caché. 

// Composición: cada selector construye sobre otros
export const selectProductState = createFeatureSelector<ProductState>('products');

export const selectAllProducts = createSelector(selectProductState, selectAll);
export const selectLoading = createSelector(selectProductState, (s) => s.loading);

// Derivar, no almacenar: el filtrado vive AQUÍ, no en el Store
export const selectHighSustainability = createSelector(
  selectAllProducts,
  (products) => products.filter((p) => p.sustainabilityScore >= 80)
);

// View model selector: un solo selector para toda la vista
// → el componente hace UN select en vez de cinco
export const selectProductListVm = createSelector(
  selectAllProducts,
  selectLoading,
  selectError,
  (products, loading, error) => ({ products, loading, error })
);

Selectores parametrizados

// Factory selector — crea una instancia memoizada POR LLAMADA
export const selectProductById = (id: string) =>
  createSelector(selectProductEntities, (entities) => entities[id] ?? null);

// ⚠️ Cada llamada a selectProductById('1') crea un selector NUEVO
// (memoización desde cero). Crea la instancia UNA vez por componente:
export class ProductDetailComponent {
  private store = inject(Store);
  private route = inject(ActivatedRoute);

  // ✅ el selector se crea UNA sola vez y se reutiliza
  private readonly byId = selectProductById(this.route.snapshot.params['id']);
  product = this.store.selectSignal(this.byId);
}

// Alternativa sin factory: seleccionar el mapa de entities
// e indexar con un computed — memoización intacta, cero instancias extra
export class ProductDetailComponent {
  private store = inject(Store);
  id = input.required<string>();

  private entities = this.store.selectSignal(selectProductEntities);
  product = computed(() => this.entities()[this.id()] ?? null);
}

Testing con projector

// .projector() ejecuta SOLO la función proyectora, sin Store ni memoización
it('filtra productos sostenibles', () => {
  const products = [
    { id: '1', sustainabilityScore: 90 },
    { id: '2', sustainabilityScore: 40 },
  ] as Product[];

  const result = selectHighSustainability.projector(products);
  expect(result.length).toBe(1);
});

⚠️ Errores comunes

  • Llamar el factory selector en el template o en cada select: cada llamada crea un selector nuevo sin caché — recálculo en cada change detection.
  • Filtrar/mapear en el componente en vez de en un selector: pierdes memoización y duplicas lógica entre componentes.
  • Selectores que devuelven objetos literales nuevos por diseño: está bien en view models, pero recuerda que emiten referencia nueva en cada recálculo — no los uses como input de otros selectors con lógica pesada sin necesidad.
  • Feature selector con string equivocado: createFeatureSelector('product') vs 'products' devuelve undefined silenciosamente. Usa createFeature para tipar el nombre.

🎤 Preguntas de entrevista

P: ¿Cómo funciona la memoización de createSelector?
R: Guarda los últimos argumentos y el último resultado (caché de 1). Compara los nuevos inputs por referencia (===): si son iguales, devuelve el cacheado; si no, ejecuta el projector y actualiza la caché.
P: ¿Por qué la memoización depende de la inmutabilidad?
R: Porque compara por referencia. Si el reducer muta, la referencia no cambia y el selector devuelve el valor viejo cacheado aunque los datos hayan cambiado.
P: ¿Cómo testeas un selector sin montar el Store?
R: Con selector.projector(...inputs): ejecuta solo la función de proyección con datos fake — test de función pura.
P: ¿Cuál es el riesgo de los selectores parametrizados (factory)?
R: Cada invocación de la factory crea una instancia con su propia caché vacía. Hay que crearla una vez por componente/parámetro, o usar el mapa de entities e indexar.

🧩 Patrones Avanzados de NgRx

1. Facade Pattern

Un servicio "facade" encapsula el Store — los componentes nunca inyectan Store directamente. Esto desacopla los componentes de NgRx, facilita testing, y centraliza la lógica de dispatch/select. 

// product.facade.ts — encapsula TODO el acceso al Store
@Injectable({ providedIn: 'root' })
export class ProductFacade {
  private store = inject(Store);

  // Selectors expuestos como Signals
  readonly products = this.store.selectSignal(selectAllProducts);
  readonly loading = this.store.selectSignal(selectLoading);
  readonly error = this.store.selectSignal(selectError);
  readonly selectedProduct = this.store.selectSignal(selectSelectedProduct);

  // Actions encapsuladas como métodos
  loadProducts() {
    this.store.dispatch(ProductActions.loadProducts());
  }

  createProduct(product: CreateProductDto) {
    this.store.dispatch(ProductActions.createProduct({ product }));
  }

  deleteProduct(id: string) {
    this.store.dispatch(ProductActions.deleteProduct({ id }));
  }

  selectProduct(id: string) {
    this.store.dispatch(ProductActions.selectProduct({ id }));
  }
}

// En el componente — limpio y desacoplado de NgRx
@Component({ ... })
export class ProductListComponent {
  private facade = inject(ProductFacade);
  products = this.facade.products;
  loading = this.facade.loading;

  ngOnInit() { this.facade.loadProducts(); }
  onDelete(id: string) { this.facade.deleteProduct(id); }
}

2. Optimistic Updates

Actualizar la UI inmediatamente antes de que el API responda, y revertir si falla. Mejora la UX percibida. 

// Reducer — actualiza INMEDIATAMENTE al despachar deleteProduct
on(ProductActions.deleteProduct, (state, { id }) =>
  adapter.removeOne(id, state)   // ← optimista: borra antes de confirmar
),

// Si falla, re-agregar el producto
on(ProductActions.deleteProductFailure, (state, { product }) =>
  adapter.addOne(product, state) // ← revertir: lo vuelve a agregar
),

// Effect — intenta borrar en el API
deleteProduct$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.deleteProduct),
    mergeMap(({ id }) =>
      this.productService.delete(id).pipe(
        map(() => ProductActions.deleteProductSuccess({ id })),
        catchError(() => {
          // Necesitamos el producto original para revertir
          const product = this.store.selectSignal(selectProductById(id))();
          return of(ProductActions.deleteProductFailure({ product }));
        })
      )
    )
  )
);

3. Entity Pattern — Normalización

¿Por qué no usar arrays? Buscar un item en un array es O(n). Con EntityAdapter, los datos se guardan como { ids: string[], entities: { [id]: Entity } }, haciendo lookups O(1). Elimina duplicados y simplifica updates. 

// ❌ Con array — lento, propenso a duplicados
interface State {
  products: Product[];  // Buscar: products.find(p => p.id === id) → O(n)
}

// ✅ Con EntityAdapter — normalizado, O(1) lookups
interface State extends EntityState<Product> {
  // EntityState provee: ids: string[], entities: { [id: string]: Product }
  loading: boolean;
}

const adapter = createEntityAdapter<Product>();

// Operaciones CRUD limpias:
adapter.addOne(product, state);              // Agregar uno
adapter.addMany(products, state);            // Agregar varios
adapter.setAll(products, state);             // Reemplazar todos
adapter.updateOne({ id, changes }, state);  // Actualizar parcial
adapter.removeOne(id, state);                // Eliminar uno
adapter.upsertOne(product, state);           // Agregar o actualizar

// Selectors generados automáticamente:
const { selectAll, selectEntities, selectIds, selectTotal } =
  adapter.getSelectors();

5. Meta-reducers

Meta-reducers son middleware que interceptan TODAS las actions antes de que lleguen a los reducers. Útiles para logging, freeze de state en dev, y limpiar state al hacer logout. 

import { MetaReducer, ActionReducer } from '@ngrx/store';

// Meta-reducer para logging
function logger(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    console.group(action.type);
    console.log('State antes:', state);
    const nextState = reducer(state, action);
    console.log('State después:', nextState);
    console.groupEnd();
    return nextState;
  };
}

// Meta-reducer para freeze — previene mutaciones accidentales en dev
function freeze(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    const nextState = reducer(state, action);
    Object.freeze(nextState); // Lanza error si alguien intenta mutar
    return nextState;
  };
}

// Meta-reducer para limpiar state al hacer logout
function clearState(reducer: ActionReducer<any>): ActionReducer<any> {
  return (state, action) => {
    if (action.type === '[Auth] Logout') {
      return reducer(undefined, action); // Reset a initialState
    }
    return reducer(state, action);
  };
}

// Registrar meta-reducers
const metaReducers: MetaReducer[] = isDevMode()
  ? [logger, freeze]
  : [clearState];

// En app.config.ts:
provideStore({ router: routerReducer }, { metaReducers });

⚠️ Errores comunes

  • Facade como god-object: si el facade acumula métodos de 4 features distintas, se convierte en el nuevo acoplamiento. Un facade por feature, con API mínima.
  • Facade que oculta el modelo de eventos: métodos tipo facade.setProducts(list) convierten NgRx en un setter glorificado. Los métodos deben mapear 1:1 a eventos de dominio.
  • Optimistic update sin rollback: si borras de la UI antes de confirmar, la action de failure DEBE restaurar el item (y avisar al usuario). Sin plan de reversión, no hagas optimistic.
  • Lógica de dominio en meta-reducers: son infraestructura transversal (logging, reset, hydration). Si un meta-reducer conoce tus entidades de negocio, ese código pertenece a un reducer.

🎤 Preguntas de entrevista

R: Ventajas: componentes desacoplados de NgRx, testing sencillo (mockeas el facade), API de dominio clara. Desventajas: capa extra de indirección y riesgo de esconder el modelo de eventos detrás de métodos imperativos.
P: ¿Cuándo aplicarías optimistic updates?
R: En operaciones con alta probabilidad de éxito y bajo costo de reversión (likes, toggles, deletes recuperables). Nunca en pagos u operaciones críticas donde mostrar un estado falso es inaceptable.
P: ¿Qué es un meta-reducer y un caso de uso real?
R: Una función de orden superior que envuelve a los reducers (middleware). Caso típico: al despachar [Auth] Logout, delegar con state = undefined para resetear TODO el store al estado inicial.
P: ¿Para qué sirve @ngrx/router-store?
R: Sincroniza la ruta con el Store para que la URL sea parte del estado: puedes crear selectors que combinan route params con entities (p. ej. el detalle actual) sin inyectar ActivatedRoute.

📦 @ngrx/entity — EntityAdapter

EntityAdapter normaliza colecciones en un formato {{ '{' }} ids: [], entities: {{ '{' }}{{ '}' }} {{ '}' }}, eliminando arrays anidados y haciendo lookups O(1) por ID. 

import {{ '{' }} createEntityAdapter, EntityState {{ '}' }} from '@ngrx/entity';

export interface Product {{ '{' }}
  id: string;
  name: string;
  sustainabilityScore: number;
{{ '}' }}

// Crear el adapter
export const productAdapter = createEntityAdapter<Product>();

// Estado inicial normalizado
export interface ProductState extends EntityState<Product> {{ '{' }}
  loading: boolean;
  error: string | null;
  selectedId: string | null;
{{ '}' }}

export const initialState: ProductState = productAdapter.getInitialState({{ '{' }}
  loading: false,
  error: null,
  selectedId: null,
{{ '}' }});

// En el reducer — operaciones CRUD limpias
on(loadProductsSuccess, (state, {{ '{' }} products {{ '}' }}) =>
  productAdapter.setAll(products, {{ '{' }} ...state, loading: false {{ '}' }})
),
on(addProductSuccess, (state, {{ '{' }} product {{ '}' }}) =>
  productAdapter.addOne(product, state)
),

¿Por qué normalizar? Evita duplicados, simplifica updates, y los selectors son más eficientes.

selectId y sortComparer

El adapter acepta dos opciones de configuración: selectId (cómo extraer la clave — por defecto busca la propiedad id) y sortComparer (mantiene ids ordenado en cada inserción; sin él, el orden es el de llegada). 

// Así está configurado en este proyecto:
export const productAdapter = createEntityAdapter<Product>({
  selectId: (p) => p.id,                                // clave primaria
  sortComparer: (a, b) => a.name.localeCompare(b.name), // orden alfabético
});

// Si tu entidad usa otra clave (sku, uuid, code)... selectId es OBLIGATORIO:
const adapter = createEntityAdapter<Sku>({ selectId: (s) => s.sku });

Métodos CRUD del adapter

MétodoQué hace
addOne / addManyAgrega si NO existe (si el id ya está, no hace nada)
setOne / setManyAgrega o REEMPLAZA completo por id
setAllReemplaza TODA la colección (típico en Load Success)
updateOne / updateManyActualización PARCIAL: recibe { id, changes } (tipo Update<T>)
upsertOne / upsertManyInserta si no existe, mergea si existe
removeOne / removeMany / removeAllElimina por id(s) / vacía la colección
mapOne / mapTransforma una / todas las entidades con una función

Selectors generados — getSelectors()

// El adapter genera 4 selectors contra EntityState<Product>:
export const { selectAll, selectEntities, selectIds, selectTotal } =
  productAdapter.getSelectors();

// Se componen con el feature selector:
export const selectAllProducts = createSelector(selectProductState, selectAll);
export const selectProductTotal = createSelector(selectProductState, selectTotal);

// selectEntities (el diccionario) es la base para lookups O(1):
export const selectSelectedProduct = createSelector(
  selectSelectedId,
  selectProductEntities,
  (id, entities) => (id ? entities[id] ?? null : null)
);

⚠️ Errores comunes

  • Confundir addOne con setOne/upsertOne: addOne ignora silenciosamente si el id ya existe — el clásico "actualicé pero no se refleja".
  • Pasar la entidad completa a updateOne: espera { id, changes }, no el objeto plano. updateOne(product, state) compila raro y falla en runtime.
  • Olvidar selectId cuando la clave no se llama id: el adapter guarda entidades bajo undefined.
  • sortComparer en colecciones enormes con inserciones frecuentes: reordena en cada operación de escritura. Si el orden solo importa en una vista, ordena en un selector.
  • Iterar entities con Object.values() en cada componente: para listas usa selectAll, que ya está memoizado.

🎤 Preguntas de entrevista

P: ¿Qué estructura interna crea EntityAdapter y por qué?
R: { ids: string[], entities: { [id]: T } } — el array preserva el orden y el diccionario da lookup O(1). Es la forma normalizada estándar de Redux.
P: ¿Diferencia entre addOne, setOne y upsertOne?
R: addOne solo inserta si no existe; setOne inserta o reemplaza el objeto completo; upsertOne inserta o hace merge parcial con el existente.
P: ¿Qué recibe updateOne?
R: Un Update<T>: { id, changes } donde changes es un Partial<T> — permite actualizar solo los campos que cambiaron.
P: ¿Para qué sirve sortComparer?
R: Mantiene el array ids ordenado en cada mutación, de modo que selectAll siempre devuelve la colección ordenada sin trabajo extra en los selectors.

createFeature — menos strings mágicos

import { createFeature, createReducer } from '@ngrx/store';

export const productsFeature = createFeature({

  reducer: createReducer(initialState, /* ...on() */),

// Genera automáticamente el feature selector y un selector
// por cada propiedad top-level del state:
productsFeature.selectProductsState;  // ← feature selector

  • Olvidar provideStore() en el root: provideState() en una ruta lazy falla si no existe el Store raíz en app.config.ts (aunque sea vacío).
  • Registrar el feature state en el root "por si acaso": pierdes el lazy loading — el reducer y sus imports entran en el bundle inicial.
  • provideEffects a nivel de componente en vez de ruta: los effects se registran cada vez que el componente se instancia; a nivel de ruta se registran una sola vez.

P: ¿Qué aporta createFeature?
R: Une nombre + reducer en un objeto tipado, genera el feature selector y un selector por propiedad del state, y elimina los strings mágicos duplicados.

🛠️ @ngrx/store-devtools

Conecta el Store con la extensión Redux DevTools del navegador: log de actions, inspección del árbol de estado, diffs entre estados, time-travel y export/import de sesiones. Es la razón número uno por la que debuggear NgRx es más fácil que debuggear servicios con estado disperso. 

import { provideStoreDevtools } from '@ngrx/store-devtools';
import { isDevMode } from '@angular/core';

provideStoreDevtools({
  maxAge: 25,                 // últimas N actions en memoria (default: false = ilimitado)
  logOnly: !isDevMode(),      // en prod: solo lectura, sin time-travel
  autoPause: true,            // pausa la grabación si DevTools está cerrado
  trace: false,               // stack trace del origen de cada dispatch (costoso)
  traceLimit: 75,
  actionsBlocklist: ['[Mouse] Move'],   // filtra actions ruidosas del log
  connectInZone: true,        // integra con Zone.js para el timing correcto
})

Qué mirar en DevTools cuando algo falla

  • La action no aparece en el log → nunca se despachó (bug en el componente/facade).
  • La action aparece pero el diff está vacío → ningún reducer la maneja (falta el on()) o el handler devolvió el mismo objeto.
  • Aparece la action de trigger pero nunca Success/Failure → el effect no está registrado, o murió por un catchError mal puesto.
  • La misma action se repite sin fin → loop infinito entre effect y action.

⚠️ Errores comunes

  • DevTools completo en producción: mantiene el historial de estados en memoria y expone datos. Usa logOnly: !isDevMode() o no lo registres en prod.
  • State no serializable (Date, Map, Set, instancias): el inspector muestra objetos vacíos y el time-travel reconstruye estados corruptos.
  • maxAge sin límite en apps con muchas actions: el historial crece indefinidamente — memoria del tab por las nubes.
  • trace: true siempre activado: capturar el stack trace de cada dispatch tiene costo real de rendimiento; actívalo solo mientras cazas un bug.

🎤 Preguntas de entrevista

P: ¿Qué es time-travel debugging y qué lo hace posible?
R: Saltar a cualquier estado pasado re-aplicando la secuencia de actions con los reducers. Es posible porque los reducers son puros y el estado es inmutable y serializable: misma secuencia = mismo estado.
P: ¿Por qué el estado debe ser serializable?
R: DevTools serializa estado y actions a JSON para inspección, export y time-travel. Además habilita hidratación (persistir/restaurar el store) y los runtimeChecks de NgRx lo validan en dev.
P: ¿Cómo evitas que actions muy frecuentes ensucien el log?
R: Con actionsBlocklist (o actionsSafelist para lo inverso) en la configuración de provideStoreDevtools.

🚦 NgRx Store vs SignalStore (@ngrx/signals)

@ngrx/signals es la librería moderna del equipo NgRx construida 100% sobre Signals. En vez del ciclo action → reducer → selector, defines un store con estado, computeds y métodos — más directo, mucho menos boilerplate, pero sin el modelo de eventos global. 

import { signalStore, withState, withComputed, withMethods, patchState } from '@ngrx/signals';
import { rxMethod } from '@ngrx/signals/rxjs-interop';
import { tapResponse } from '@ngrx/operators';

export const ProductsStore = signalStore(
  { providedIn: 'root' },   // global; sin esto, se provee por componente (local)

  withState({
    products: [] as Product[],
    loading: false,
    error: null as string | null,
  }),

  withComputed((store) => ({
    total: computed(() => store.products().length),
    sustainable: computed(() =>
      store.products().filter((p) => p.sustainabilityScore >= 80)
    ),
  })),

  withMethods((store, api = inject(MockApiService)) => ({
    // rxMethod: side effects reactivos (equivale a un effect)
    loadProducts: rxMethod<void>(
      pipe(
        tap(() => patchState(store, { loading: true, error: null })),
        switchMap(() =>
          api.getProducts().pipe(
            tapResponse({
              next: (products) => patchState(store, { products, loading: false }),
              error: (err: Error) =>
                patchState(store, { error: err.message, loading: false }),
            })
          )
        )
      )
    ),
  }))
);

// En el componente — sin dispatch, sin selectors:
export class ProductListComponent {
  store = inject(ProductsStore);
  // template: store.products(), store.loading(), store.sustainable()
  ngOnInit() { this.store.loadProducts(); }
}
Criterio NgRx Store (Redux) SignalStore (@ngrx/signals)
Paradigma Event-driven: actions globales, indirección total Llamadas directas a métodos del store
Boilerplate Actions + reducer + selectors + effects Un archivo: withState/withComputed/withMethods
Trazabilidad Redux DevTools, time-travel, log de eventos Limitada (no hay log de actions nativo)
Reactividad Observables (+ selectSignal como puente) Signals nativos — integra con OnPush/zoneless gratis
Alcance Global por diseño Global o local (por componente/ruta)
Colecciones @ngrx/entity withEntities() de @ngrx/signals/entities
Extensibilidad Meta-reducers, router-store Custom features (withX composables)

¿Cuándo usar cada uno?

  • NgRx Store clásico: estado de dominio compartido por MUCHOS features, equipos grandes que necesitan trazabilidad de eventos, requisitos de auditoría/debugging con time-travel, o múltiples consumidores reaccionando al mismo evento.
  • SignalStore: estado de un feature o componente complejo, apps nuevas signal-first/zoneless, equipos pequeños que priorizan velocidad de desarrollo sobre el event log.
  • Pueden convivir: Store global para el dominio transversal (auth, carrito) + SignalStores locales por feature es una arquitectura perfectamente válida hoy.

⚠️ Errores comunes

  • Asumir que SignalStore "reemplaza" a NgRx Store: son trade-offs distintos; pierdes el modelo de eventos y el tooling de Redux.
  • Mutar el state en vez de usar patchState: los signals internos son readonly; toda escritura pasa por patchState/updateState.
  • Olvidar { providedIn: 'root' } cuando quieres un store global — sin él, cada componente que lo provea tiene SU PROPIA instancia.
  • Suscripciones manuales para side effects en vez de rxMethod, que gestiona la suscripción y la limpieza automáticamente.

🎤 Preguntas de entrevista

P: ¿Diferencia fundamental entre NgRx Store y SignalStore?
R: NgRx Store es event-driven: despachas actions y reducers/effects reaccionan (indirección + trazabilidad). SignalStore expone estado y métodos que llamas directamente: menos ceremonia, menos trazabilidad.
P: ¿Qué es rxMethod?
R: Un método reactivo que acepta valores, signals u observables como input y ejecuta un pipeline RxJS gestionando la suscripción — el equivalente de un effect dentro de un SignalStore.
P: ¿Pueden convivir en la misma app?
R: Sí. Es común mantener el Store Redux para dominio global existente e introducir SignalStores en features nuevas.
P: ¿SignalStore puede ser local?
R: Sí: si lo agregas a los providers de un componente, vive y muere con él — ideal para estado de UI complejo pero no global.

🧪 Testing NgRx

La gran ventaja de NgRx: casi todo es una función pura. Reducers y selectors se testean sin Angular; para componentes existe provideMockStore y para effects, provideMockActions.

Reducers — funciones puras

it('activa loading al despachar loadProducts', () => {
  const state = productReducer(initialState, ProductActions.loadProducts());
  expect(state.loading).toBe(true);
  expect(state).not.toBe(initialState);   // referencia NUEVA (inmutabilidad)
});

Selectors — projector()

it('selecciona el producto activo', () => {
  const entities = { '1': mockProduct };
  expect(selectSelectedProduct.projector('1', entities)).toEqual(mockProduct);
  expect(selectSelectedProduct.projector(null, entities)).toBeNull();
});

Effects — provideMockActions

import { provideMockActions } from '@ngrx/effects/testing';

it('emite success cuando el API responde', (done) => {
  const actions$ = of(ProductActions.loadProducts());
  TestBed.configureTestingModule({
    providers: [
      ProductEffects,
      provideMockActions(() => actions$),
      { provide: MockApiService, useValue: { getProducts: () => of([mockProduct]) } },
    ],
  });

  TestBed.inject(ProductEffects).loadProducts$.subscribe((action) => {
    expect(action).toEqual(
      ProductActions.loadProductsSuccess({ products: [mockProduct] })
    );
    done();
  });
});

Componentes — provideMockStore

import { provideMockStore, MockStore } from '@ngrx/store/testing';

TestBed.configureTestingModule({
  providers: [
    provideMockStore({
      selectors: [
        { selector: selectAllProducts, value: [mockProduct] },
        { selector: selectLoading, value: false },
      ],
    }),
  ],
});

const store = TestBed.inject(MockStore);

// Cambiar el valor de un selector a mitad de test:
store.overrideSelector(selectLoading, true);
store.refreshState();          // ← imprescindible para re-emitir
fixture.detectChanges();

// Verificar dispatches:
const spy = jest.spyOn(store, 'dispatch');
component.onDelete('1');
expect(spy).toHaveBeenCalledWith(ProductActions.deleteProduct({ id: '1' }));

⚠️ Errores comunes

  • Usar el Store real en tests de componente: acoplas el test a reducers/effects. provideMockStore aísla el componente por completo.
  • Olvidar refreshState() después de overrideSelector — el componente sigue viendo el valor anterior.
  • Testear effects con el servicio HTTP real en vez de un stub — lento y frágil.
  • No testear el caso de error del effect: es exactamente donde viven los bugs (catchError mal ubicado, mensaje perdido).

🎤 Preguntas de entrevista

P: ¿Por qué NgRx facilita el testing?
R: Reducers y selectors son funciones puras (input → output, sin mocks), los effects aíslan TODOS los side effects en un solo lugar, y provideMockStore permite testear componentes sin tocar la lógica de estado.
P: ¿Cómo testeas un componente que usa selectSignal?
R: Igual que con select: provideMockStore con selectors mockeados; selectSignal lee del MockStore. Para cambios en runtime, overrideSelector + refreshState.
P: ¿Qué provee provideMockActions?
R: Reemplaza el stream Actions por uno controlado por el test (Observable o Subject), para emitir exactamente las actions que quieres y verificar qué emite el effect.
