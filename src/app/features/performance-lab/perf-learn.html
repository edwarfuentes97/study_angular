1. OnPush + trackBy

La estrategia OnPush reduce drásticamente los ciclos de change detection. Con la estrategia Default, cada evento (click, timer, HTTP response) hace que Angular recorra todo el árbol de componentes comparando cada binding. Con OnPush, Angular salta el componente completo (y su subárbol) a menos que ocurra algo que lo marque como "dirty".

¿Qué dispara CD en un componente OnPush?

  • Cambio de referencia en un @Input(): Angular compara con ===. Mutar el objeto NO cuenta; asignar uno nuevo sí.
  • Un evento del propio template: (click), (input), etc. disparados dentro del componente o de sus hijos.
  • Un signal leído en el template cambia: los signals marcan el componente como dirty automáticamente — es la forma más ergonómica de trabajar con OnPush.
  • El async pipe emite un valor nuevo (internamente llama a markForCheck()).
  • Llamada manual a ChangeDetectorRef.markForCheck().

Inmutabilidad: el requisito no negociable

OnPush compara referencias, no contenido. Si mutas (user.name = 'X', array.push()) la referencia no cambia y la vista queda desactualizada silenciosamente. La regla: todo cambio de estado crea un objeto/array nuevo (spread, .map(), .filter()). Con signals, set() y update() te empujan naturalmente a este patrón. 

@Component({
  selector: 'app-user-list',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: \`
    @for (user of users(); track user.id) {
      <app-user-card [user]="user" />
    }
  \`
})
export class UserListComponent {
  users = signal<User[]>([]);

  rename(id: number, name: string) {
    // ✅ Inmutabilidad: nueva referencia de array Y de objeto
    this.users.update(list =>
      list.map(u => u.id === id ? { ...u, name } : u)
    );
  }
}

track en @for: por qué es obligatorio

En Angular moderno track es obligatorio en @for (con *ngFor el trackBy era opcional y se olvidaba). Sin una key estable, cuando la lista cambia Angular no sabe qué item es cuál: destruye y recrea todos los nodos DOM, pierde estado local (focus, inputs, animaciones) y re-ejecuta lifecycle hooks de cada componente hijo.

Cómo elegir una buena key:

  • ✅ Un identificador único y estable del dato: track item.id, track user.uuid. Sobrevive a reordenamientos, inserciones y borrados.
  • ⚠️ track $index solo es aceptable en listas estáticas (nunca se reordenan ni se insertan/borran items en medio). Si el orden cambia, Angular "recicla" nodos con datos equivocados.
  • ❌ track item (el objeto entero) falla si recreas los objetos (p. ej. tras un fetch): cada referencia nueva = nodo DOM nuevo.

⚠️ Errores comunes

  • Mutar estado con OnPush: this.items.push(x) no actualiza la vista. Usa this.items = [...this.items, x] o signals.
  • Poner OnPush solo en el padre: el beneficio real llega cuando todo el árbol es OnPush; un hijo Default dentro de un padre OnPush sí se revisa cuando el padre se revisa.
  • Confiar en setTimeout/librerías externas: si actualizan propiedades planas (no signals) dentro de un componente OnPush, necesitas markForCheck().
  • Usar detectChanges() en vez de markForCheck(): detectChanges() ejecuta CD síncrono inmediato (caro y puede causar errores de timing); markForCheck() solo marca el camino hasta la raíz para el próximo ciclo.

🎤 Preguntas de entrevista

  • ¿Qué dispara change detection en un componente OnPush? Cambio de referencia en un @Input, un evento del template, un signal leído en el template que cambia, el async pipe al emitir, o markForCheck() manual.
  • ¿Por qué OnPush exige inmutabilidad? Porque compara inputs con ===: una mutación conserva la referencia y el cambio pasa inadvertido.
  • ¿Diferencia entre markForCheck() y detectChanges()? markForCheck() marca el componente y sus ancestros como dirty para el próximo ciclo (barato); detectChanges() ejecuta CD síncrono inmediato sobre ese subárbol (caro).
  • ¿Qué pasa en un @for sin una key estable? Angular destruye y recrea todos los nodos DOM en cada cambio de lista: se pierde estado local y se dispara layout/paint innecesario.

2. @defer Blocks

Los bloques @defer (Angular 17+) permiten lazy-loading declarativo a nivel de template: el compilador extrae las dependencias del bloque (componentes, directivas, pipes standalone) a un chunk JavaScript separado que solo se descarga cuando se activa el trigger. Es code splitting sin tocar el router — ideal para contenido "below the fold", modales, gráficos y widgets pesados.

Triggers disponibles

TriggerCuándo cargaUso típico
on idleCuando el browser está ocioso (requestIdleCallback). Es el default si no especificas trigger.Contenido secundario no urgente
on viewportCuando el placeholder entra al viewport (IntersectionObserver)Secciones below the fold
on interactionClick o keydown sobre el placeholder (o un elemento referenciado)Tabs, acordeones, editores
on hoverMouseover / focusin sobre el placeholderMenús, tooltips ricos
on timer(3s)Después de un delay fijoBanners, contenido diferido temporal
when condiciónCuando la expresión pasa a true (una sola vez, no vuelve atrás)Carga controlada por estado

Los triggers on viewport / interaction / hover pueden observar otro elemento usando una template reference: @defer (on interaction(botonAbrir)).

prefetch: descargar antes, renderizar después

prefetch separa descargar el chunk de renderizarlo. Ejemplo: descargar el JS cuando el browser esté idle, pero instanciar el componente solo cuando el usuario haga scroll hasta él. Así la carga es instantánea cuando llega el momento. 

@defer (on viewport; prefetch on idle) {
  <app-heavy-chart [data]="chartData" />
} @placeholder (minimum 500ms) {
  <div class="skeleton">📊 Gráfico disponible al hacer scroll</div>
} @loading (after 100ms; minimum 300ms) {
  <div class="spinner">Cargando...</div>
} @error {
  <div class="error">Error al cargar el gráfico</div>
}

Los sub-bloques y sus parámetros

  • @placeholder (minimum 500ms) — contenido inicial antes de cargar. minimum evita flicker: se muestra al menos ese tiempo. Necesario para on viewport / interaction / hover (algo debe observarse).
  • @loading (after 100ms; minimum 300ms) — se muestra durante la descarga. after evita mostrar spinner en cargas rápidas; minimum evita el parpadeo spinner→contenido.
  • @error — si falla la carga del chunk (red caída, deploy que invalidó hashes). Sin este bloque, el fallo es silencioso.

⚠️ Errores comunes

  • El componente no se separa en chunk: pasa si además lo referencias fuera del @defer (en el mismo template, con @ViewChild, o si no es standalone). Verifica en el build que aparece un lazy chunk nuevo.
  • Deferir contenido above the fold: empeora LCP y provoca layout shift. @defer es para lo que NO se ve al cargar.
  • Placeholder sin dimensiones: cuando el contenido real aparece, empuja el layout (CLS). Dale al placeholder el mismo tamaño que el contenido final (skeleton).
  • Abusar de when esperando que se "des-cargue": una vez renderizado, el bloque no vuelve al placeholder aunque la condición sea false (para eso usa @if).

🎤 Preguntas de entrevista

  • ¿Qué diferencia hay entre @defer y lazy loading de rutas? El lazy loading de rutas divide el bundle por navegación; @defer divide dentro de una misma página a nivel de template, con triggers de UX.
  • ¿Para qué sirve prefetch? Descarga el chunk anticipadamente (p. ej. en idle) sin renderizarlo; al activarse el trigger principal el render es inmediato.
  • ¿Qué condiciones debe cumplir un componente para ser deferrable? Ser standalone y no estar referenciado fuera del bloque @defer en ese archivo.
  • ¿Cuál es el trigger por defecto? on idle.

3. Virtual Scrolling

El CDK ScrollingModule proporciona cdk-virtual-scroll-viewport, que solo renderiza los items visibles en el viewport (más un pequeño buffer). Una lista de 10,000 filas pasa de 10,000 nodos DOM a ~20: menos memoria, menos tiempo de paint y scroll fluido. El viewport calcula la altura total "virtual" con itemSize × total y va reciclando los nodos mientras el usuario hace scroll. 

<cdk-virtual-scroll-viewport itemSize="48" class="viewport">
  <div *cdkVirtualFor="let item of items; trackBy: trackById" class="item">
    {{ item.name }}
  </div>
</cdk-virtual-scroll-viewport>

// El viewport NECESITA altura fija en CSS:
// .viewport { height: 400px; }

Piezas clave

  • itemSize — altura fija (px) de cada fila con la estrategia FixedSizeVirtualScrollStrategy. Si el valor no coincide con la altura real, el scroll "salta" y el scrollbar miente.
  • minBufferPx / maxBufferPx — cuánto contenido extra renderizar fuera de pantalla para evitar huecos en scroll rápido.
  • trackBy en *cdkVirtualFor — igual de importante que en @for: el viewport recicla templates y sin key estable re-renderiza de más.
  • Para alturas variables existe autosize (en @angular/cdk-experimental) o estrategias custom implementando VirtualScrollStrategy.

¿Virtual scroll o paginación?

CriterioVirtual scrollPaginación
UXExploración continua (feeds, logs, chats)Navegación por bloques, resultados de búsqueda
DatosYa cargados en memoria (o infinite scroll + API)Se piden por página al servidor
SEO / linksMalo: no hay URL por posiciónBueno: cada página es enlazable
AlturasIdealmente fijas (itemSize)Indiferente
AccesibilidadCuidado: el DOM no contiene toda la lista (Ctrl+F, lectores)Más predecible

Regla práctica: < 100 items → lista normal con track; cientos/miles ya en memoria → virtual scroll; miles en servidor → paginación (o virtual scroll + fetch incremental).

⚠️ Errores comunes

  • Viewport sin altura CSS: si el cdk-virtual-scroll-viewport no tiene height, renderiza 0 items o crece infinito.
  • itemSize incorrecto: scrollbar impreciso y saltos al hacer scroll. Mide la fila real (incluye borders/padding).
  • Items con altura variable usando FixedSize: usa autosize o normaliza las alturas.
  • Usarlo para 30 items: agrega complejidad (posicionamiento absoluto, eventos de scroll) sin beneficio medible.

🎤 Preguntas de entrevista

  • ¿Cómo funciona internamente el virtual scroll? Renderiza solo los items del viewport + buffer, posiciona el contenido con transform y simula la altura total con un spacer calculado a partir de itemSize.
  • ¿Cuándo elegirías paginación en lugar de virtual scroll? Cuando los datos viven en el servidor, se necesita URL por página/SEO, o la UX es de "resultados" y no de feed.
  • ¿Qué pasa si los items tienen alturas distintas? FixedSize falla; se usa la estrategia autosize experimental o una VirtualScrollStrategy propia.

4. Bundle Analysis

El JavaScript inicial es el mayor enemigo del startup: se descarga, parsea y ejecuta antes de que la app sea interactiva. La estrategia tiene dos partes: dividir el código (lazy loading / code splitting) y medir qué ocupa espacio (analizadores de bundle + budgets que fallan el build si te pasas).

Lazy loading y code splitting

Cada loadComponent() / loadChildren() con import() dinámico genera un chunk separado que solo se descarga al navegar. Los bloques @defer hacen lo mismo dentro de una página. 

// app.routes.ts — cada feature es un chunk independiente
export const routes: Routes = [
  {
    path: 'reports',
    loadComponent: () =>
      import('./features/reports/reports.component')
        .then(m => m.ReportsComponent)
  },
  {
    path: 'admin',
    loadChildren: () => import('./features/admin/admin.routes')
  }
];

Analizar el bundle

// Opción A — source-map-explorer (treemap desde source maps)
// ng build --source-map
// npx source-map-explorer dist/app/browser/*.js

// Opción B — esbuild (builder "application", Angular 17+)
// ng build --stats-json
// npx esbuild-visualizer --metadata dist/app/stats.json --open

Qué buscar en el treemap: dependencias enormes (moment.js → date-fns, lodash completo → lodash-es con imports puntuales), código duplicado entre chunks, features que deberían ser lazy pero están en el bundle inicial (un import estático "accidental" desde un archivo eager arrastra todo el módulo al main).

Budgets en angular.json: presupuesto que rompe el build

// angular.json → projects.app.architect.build.configurations.production
"budgets": [
  {
    "type": "initial",
    "maximumWarning": "500kB",
    "maximumError": "1MB"
  },
  {
    "type": "anyComponentStyle",
    "maximumWarning": "4kB",
    "maximumError": "8kB"
  }
]

Los budgets convierten el tamaño del bundle en una regresión detectable en CI: si un PR agrega una dependencia pesada, el build avisa (warning) o falla (error) antes de llegar a producción.

⚠️ Errores comunes

  • Import estático que rompe el split: importar un símbolo de una feature lazy desde código eager (p. ej. un modelo o constante) mete todo el chunk en el main. Extrae lo compartido a shared/.
  • Analizar el build de desarrollo: sin tree shaking ni minificación los números no significan nada. Siempre analiza el build de producción.
  • Barrel files (index.ts) gigantes: re-exportar todo desde un barrel puede impedir el tree shaking efectivo y crear dependencias circulares.
  • Budgets tan holgados que nunca fallan: ajústalos al tamaño real actual + margen pequeño, y bájalos cuando optimices.

🎤 Preguntas de entrevista

  • ¿Cómo reduces el bundle inicial de una app Angular? Lazy loading de rutas, @defer, tree shaking con imports puntuales, revisar dependencias pesadas con un analizador y aplicar budgets para evitar regresiones.
  • ¿Qué diferencia hay entre lazy loading y tree shaking? Tree shaking elimina código muerto en build time; lazy loading difiere la descarga de código vivo hasta que se necesita.
  • ¿Cómo detectas qué dependencia infló el bundle? Build de producción con source maps o stats.json + source-map-explorer / esbuild-visualizer: el treemap muestra el peso por módulo.

5. Preloading Strategies

Las estrategias de precarga permiten cargar módulos lazy en background después del initial load. PreloadAllModules precarga todo automáticamente. Para control fino, crea una estrategia custom que precargue solo rutas marcadas con data: { preload: true }. 

provideRouter(routes, withPreloading(PreloadAllModules))

⚠️ Errores comunes

  • PreloadAllModules en apps grandes: precargar 40 features compite por ancho de banda con imágenes y datos del initial load. Usa una estrategia selectiva o basada en navigator.connection (no precargar en redes lentas).
  • Confundir preloading con eager loading: el preloading ocurre después del initial load, en background; no afecta el bundle inicial.

🎤 Preguntas de entrevista

  • ¿Preloading empeora el tiempo de carga inicial? No: se ejecuta después de que la app es estable; solo consume red en background.
  • ¿Cómo precargarías solo algunas rutas? Con una PreloadingStrategy custom que lea route.data['preload'] y decida llamar a load() o devolver of(null).

6. Web Workers

Los Web Workers ejecutan código en un hilo separado, evitando bloquear el main thread con operaciones pesadas (procesamiento de datos, cálculos, compresión). Angular CLI facilita su creación con ng generate web-worker. 

// Generar:
// ng generate web-worker my-worker

// app.component.ts
if (typeof Worker !== 'undefined') {
  const worker = new Worker(new URL('./my-worker.worker', import.meta.url));
  worker.onmessage = ({ data }) => {
    console.log('Resultado:', data);
  };
  worker.postMessage({ items: largeDataset });
}

// my-worker.worker.ts
addEventListener('message', ({ data }) => {
  const result = heavyComputation(data.items);
  postMessage(result);
});

⚠️ Errores comunes

  • Usar workers para tareas triviales: serializar los datos (structured clone) tiene costo; para cálculos de < 10-20 ms no compensa.
  • Intentar tocar el DOM o inyectar servicios Angular dentro del worker: el worker corre en otro contexto, sin DOM ni DI.
  • Enviar objetos gigantes por copia: para buffers grandes usa Transferable (ArrayBuffer) y evita la copia.

🎤 Preguntas de entrevista

  • ¿Cuándo mueves trabajo a un Web Worker? Cuando un cálculo síncrono bloquea el main thread > 50 ms (parsing masivo, orden/filtrado de datasets grandes, compresión) y degradaría INP.
  • ¿Cómo se comunican worker y componente? Solo por mensajes (postMessage / onmessage) con structured clone o Transferables; no comparten memoria (salvo SharedArrayBuffer).

7 · Estrategia completa de optimización

Checklist ordenado de mayor a menor impacto para optimizar una app Angular. Aplica cada nivel en orden: corregir change detection mal usado tiene más efecto que cualquier ajuste de red o SSR.

Nivel 1 — Change Detection (mayor impacto)

Objetivo: OnPush en el 100 % de componentes. Angular Default recorre todo el árbol en cada evento; OnPush lo limita a cambios de referencia en @Input(), eventos del template o markForCheck().

Migrar a OnPush sin romper nada:

  1. Reemplazar mutaciones por inmutabilidad: { ...obj, prop: newVal } en vez de obj.prop = newVal.
  2. Usar async pipe o signals para datos reactivos.
  3. Agregar changeDetection: ChangeDetectionStrategy.OnPush al componente.

Signal-based components ofrecen CD granular sin Zone.js. Para código que no necesita disparar CD (scroll listeners, animaciones de canvas), usa NgZone.runOutsideAngular(): 

@Component({ ... })
export class ScrollTrackerComponent {
  private ngZone = inject(NgZone);
  private scrollY = signal(0);

  constructor() {
    // Ejecuta fuera de Zone → no dispara CD en cada scroll
    this.ngZone.runOutsideAngular(() => {
      window.addEventListener('scroll', () => {
        this.scrollY.set(window.scrollY);
      });
    });
  }
}

Nivel 2 — Bundle Size

  • Lazy loading de TODAS las features: loadChildren() / loadComponent().
  • Analizar bundle: ng build --stats-json && npx source-map-explorer dist/**/*.js
  • Tree shaking: importar selectivamente import { map } from 'rxjs/operators', nunca import * as rx from 'rxjs'.
  • Remover dependencias innecesarias del package.json.
  • Preloading strategies: PreloadAllModules o una estrategia custom para rutas frecuentes.

Estrategia de preloading custom:

@Injectable({ providedIn: 'root' })
export class SelectivePreloadStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<any>): Observable<any> {
    // Solo precarga rutas marcadas con data.preload = true
    return route.data?.['preload'] ? load() : of(null);
  }
}

// app.routes.ts
export const routes: Routes = [
  {
    path: 'products',
    loadComponent: () => import('./features/products/products.component'),
    data: { preload: true }   // ← se precarga en background
  },
  {
    path: 'admin',
    loadComponent: () => import('./features/admin/admin.component')
    // sin preload → se carga solo cuando el usuario navega
  }
];

// app.config.ts
provideRouter(routes, withPreloading(SelectivePreloadStrategy))

Nivel 3 — Rendering

  • trackBy en TODOS los @for / *ngFor: evita re-crear nodos DOM al cambiar la lista.
  • Virtual scrolling para listas largas (100+ items): cdk-virtual-scroll-viewport.
  • @defer para componentes pesados below the fold.
  • Evitar funciones en templates: {{ getTotal() }} se recalcula en cada CD cycle → usar computed signal o un pipe puro.
  • ngClass / ngStyle con objetos cacheados, no funciones.

Nivel 4 — Network

  • shareReplay(1) para cachear HTTP responses compartidas.
  • HTTP caching con interceptor: ETag, If-None-Match.
  • Prefetch data con resolvers.
  • Pagination vs infinite scroll vs virtual scroll — elige según el UX.
  • Compress: gzip / brotli en servidor.

Interceptor de caché HTTP con ETag:

export function httpCacheInterceptor(): HttpInterceptorFn {
  const cache = new Map<string, { etag: string; body: any }>();

  return (req, next) => {
    if (req.method !== 'GET') return next(req);

    const cached = cache.get(req.urlWithParams);
    let headers = req.headers;
    if (cached) {
      headers = headers.set('If-None-Match', cached.etag);
    }

    return next(req.clone({ headers })).pipe(
      map(event => {
        if (event instanceof HttpResponse) {
          const etag = event.headers.get('ETag');
          if (etag) {
            cache.set(req.urlWithParams, {
              etag,
              body: event.body
            });
          }
        }
        return event;
      }),
      catchError(err => {
        // 304 Not Modified → devolver respuesta cacheada
        if (err.status === 304 && cached) {
          return of(new HttpResponse({
            body: cached.body,
            status: 200
          }));
        }
        return throwError(() => err);
      })
    );
  };
}

// app.config.ts
provideHttpClient(withInterceptors([httpCacheInterceptor()]))

Nivel 5 — SSR & Hydration

  • Server-Side Rendering mejora LCP y SEO.
  • Hydration (Angular 16+): no destruye y recrea el DOM del servidor, lo reutiliza.
  • provideClientHydration() en app.config.
  • afterNextRender() para código que solo debe correr en el browser (acceso a window, document, etc.).
// app.config.ts

}

  • ng generate web-worker my-worker
  • No pueden acceder al DOM.
  • Comunicación por mensajes: worker.postMessage() / worker.onmessage.

8 · Anti-patterns de Performance

Errores comunes que degradan el rendimiento de una app Angular. Cada uno muestra el código incorrecto (❌) y la corrección (✅).

1. Funciones en template — se ejecutan en cada CD cycle

// ❌ MAL — getTotal() se llama en CADA ciclo de change detection
@Component({
  template: \`<span>Total: {{ getTotal() }}</span>\`
})
export class OrderComponent {
  items = signal<Item[]>([]);
  getTotal(): number {
    return this.items().reduce((sum, i) => sum + i.price, 0);
  }
}

// ✅ BIEN — computed() solo recalcula cuando items() cambia
@Component({
  template: \`<span>Total: {{ total() }}</span>\`
})
export class OrderComponent {
  items = signal<Item[]>([]);
  total = computed(() =>
    this.items().reduce((sum, i) => sum + i.price, 0)
  );
}

¿Por qué? Angular ejecuta las expresiones del template en cada ciclo de CD. Un computed signal memoriza el resultado y solo recalcula cuando sus dependencias cambian.

2. Suscripciones manuales sin cleanup — memory leaks

// ❌ MAL — suscripción nunca se limpia → memory leak
@Component({ ... })
export class ChatComponent implements OnInit {
  messages: Message[] = [];

  ngOnInit() {
    this.chatService.messages$.subscribe(msgs => {
      this.messages = msgs;
    });
  }
}

// ✅ BIEN — Opción A: async pipe (se des-suscribe automáticamente)
@Component({
  template: \`
    @for (msg of messages$ | async; track msg.id) {
      <app-message [data]="msg" />
    }
  \`
})
export class ChatComponent {
  messages$ = inject(ChatService).messages$;
}

// ✅ BIEN — Opción B: DestroyRef + takeUntilDestroyed
@Component({ ... })
export class ChatComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    this.chatService.messages$
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(msgs => this.messages = msgs);
  }
}

¿Por qué? Las suscripciones manuales persisten después de destruir el componente. El async pipe o takeUntilDestroyed garantizan la limpieza automática.

3. Importar módulos enteros — bundle bloat

// ❌ MAL — importa TODO RxJS (~45 KB)
import * as rx from 'rxjs';
const result$ = rx.of(1).pipe(rx.map(x => x * 2));

// ✅ BIEN — tree-shakeable, solo importa lo usado (~2 KB)
import { of } from 'rxjs';
import { map } from 'rxjs/operators';
const result$ = of(1).pipe(map(x => x * 2));

¿Por qué? Los import con * impiden que el bundler elimine código no utilizado (tree shaking). Importar solo los símbolos necesarios reduce drásticamente el tamaño del bundle.

4. No usar trackBy — re-renderiza toda la lista

// ❌ MAL — sin track, Angular destruye y recrea todos los nodos DOM
@Component({
  template: \`
    @for (user of users(); track $index) {
      <app-user-card [user]="user" />
    }
  \`
})

// ✅ BIEN — track por ID: Angular reutiliza nodos existentes
@Component({
  template: \`
    @for (user of users(); track user.id) {
      <app-user-card [user]="user" />
    }
  \`
})

¿Por qué? Sin un track estable, Angular no puede identificar qué elementos cambiaron. Usar $index falla cuando el orden cambia. Siempre usa una propiedad única como id.

5. Mutación de objetos con OnPush — cambios invisibles

// ❌ MAL — mutar el array no cambia la referencia → OnPush no detecta
updateUser(index: number, name: string) {
  this.users[index].name = name;  // mutación
}

// ✅ BIEN — nueva referencia → OnPush detecta el cambio
updateUser(index: number, name: string) {
  this.users = this.users.map((u, i) =>
    i === index ? { ...u, name } : u
  );
}

// ✅ MEJOR — con signals, update() crea nueva referencia
users = signal<User[]>([]);

updateUser(index: number, name: string) {
  this.users.update(list =>
    list.map((u, i) => i === index ? { ...u, name } : u)
  );
}

¿Por qué? OnPush compara referencias, no contenido. Mutar un objeto/array mantiene la misma referencia, así que Angular ignora el cambio. Siempre crea nuevas referencias con spread o .map().

6. ExpressionChangedAfterItHasBeenCheckedError

// ❌ MAL — cambiar estado en ngAfterViewInit dispara el error
@Component({ ... })
export class BadComponent implements AfterViewInit {
  title = '';

  ngAfterViewInit() {
    this.title = 'Loaded';  // ERROR en dev mode
  }
}

// ✅ BIEN — usar afterNextRender o un signal con efecto
@Component({ ... })
export class GoodComponent {
  title = signal('');

  constructor() {
    afterNextRender(() => {
      this.title.set('Loaded');
    });
  }
}

¿Por qué? Angular verifica que los bindings no cambien entre el check y el re-check (en dev mode). Modificar estado en ngAfterViewInit viola esta regla. Usa afterNextRender o programa el cambio para el siguiente frame.

7. Observables sin compartir — múltiples HTTP calls

// ❌ MAL — cada suscripción dispara un HTTP request nuevo
@Injectable({ providedIn: 'root' })
export class UserService {
  getUser(id: number) {
    return this.http.get<User>(\`/api/users/${id}\`);
  }
}
// Si 3 componentes llaman getUser(1) → 3 HTTP requests

// ✅ BIEN — shareReplay cachea y comparte la última respuesta
@Injectable({ providedIn: 'root' })
export class UserService {
  private userCache = new Map<number, Observable<User>>();

  getUser(id: number): Observable<User> {
    if (!this.userCache.has(id)) {
      this.userCache.set(id,
        this.http.get<User>(\`/api/users/${id}\`).pipe(
          shareReplay({ bufferSize: 1, refCount: true })
        )
      );
    }
    return this.userCache.get(id)!;
  }
}

¿Por qué? Cada .subscribe() a un Observable HTTP frío crea una nueva petición. shareReplay con refCount: true comparte la respuesta y libera el caché cuando no hay suscriptores.

9. Memoización — computed signals y pure pipes

Memoizar = cachear el resultado de un cálculo y recalcular solo cuando cambian sus entradas. En Angular hay dos herramientas nativas: computed() (signals) y los pure pipes. El anti-patrón que ambas resuelven es el mismo: llamar métodos en el template.

¿Por qué los métodos en template son costosos?

Angular no puede saber si getTotal() es pura o depende de algo externo, así que la re-ejecuta en cada ciclo de change detection — que en apps con estrategia Default puede correr decenas de veces por segundo (cada click, keypress, respuesta HTTP...). Un filter().map().reduce() "inocente" dentro de un @for de 50 items se convierte en miles de ejecuciones por segundo. 

// ❌ Se ejecuta en CADA ciclo de CD
template: \`<p>Total: {{ getTotal() }}</p>\`

// ✅ computed: memoizado, recalcula SOLO si cambian sus dependencias
items = signal<Item[]>([]);
total = computed(() =>
  this.items().reduce((sum, i) => sum + i.price, 0)
);
template: \`<p>Total: {{ total() }}</p>\`

Pure pipes: memoización por argumentos

Un pipe con pure: true (el default) solo se re-ejecuta cuando cambia la referencia de sus argumentos. Es la opción idónea para transformar datos que llegan por @Input() o cuando la misma transformación se usa en muchos templates. 

@Pipe({ name: 'fileSize' })  // pure por defecto
export class FileSizePipe implements PipeTransform {
  transform(bytes: number): string {
    if (bytes < 1024) return \`${bytes} B\`;
    if (bytes < 1_048_576) return \`${(bytes / 1024).toFixed(1)} KB\`;
    return \`${(bytes / 1_048_576).toFixed(1)} MB\`;
  }
}

// template — solo se recalcula si doc.size cambia de valor
// {{ doc.size | fileSize }}

¿computed o pipe?

  • computed() — lógica derivada del estado del componente (signals). Encadenable, testeable, granular con OnPush/zoneless.
  • Pure pipe — transformación reutilizable y sin estado (formatos, unidades, filtros). Se comparte entre toda la app.
  • Impure pipe (pure: false) — casi nunca: se ejecuta en cada CD, igual de caro que un método en template.

⚠️ Errores comunes

  • computed con side effects: computed() debe ser puro; para efectos usa effect().
  • Pure pipe con arrays mutados: si mutas el array la referencia no cambia y el pipe no se re-ejecuta — el gemelo del problema de OnPush.
  • Getters en template creyendo que se cachean: get total() se ejecuta en cada CD igual que un método.
  • Crear objetos/arrays inline en el template ([options]="{ compact: true }"): nueva referencia en cada CD → el hijo OnPush se re-renderiza siempre.

🎤 Preguntas de entrevista

  • ¿Por qué no llamar métodos en el template? Angular los re-ejecuta en cada ciclo de CD porque no puede asumir que son puros; con listas y CD frecuente el costo escala multiplicativamente.
  • ¿Cuándo se re-ejecuta un pure pipe? Solo cuando cambia la referencia (o valor primitivo) de sus argumentos.
  • ¿Qué ventaja tiene computed() sobre un método? Memoiza el resultado y rastrea sus dependencias: recalcula únicamente cuando alguno de los signals leídos cambia.

10. NgOptimizedImage — imágenes optimizadas

Las imágenes suelen ser el elemento LCP y la primera causa de CLS. La directiva NgOptimizedImage (@angular/common) aplica las best practices automáticamente: lazy loading por defecto, exige width/height para reservar espacio (evita layout shift), genera srcset responsive y con priority emite fetchpriority="high" + <link rel="preload"> para la imagen crítica. 

import { NgOptimizedImage } from '@angular/common';

@Component({
  imports: [NgOptimizedImage],
  template: \`
    <!-- Imagen LCP (hero): priority = eager + preload + fetchpriority -->
    <img ngSrc="/assets/hero.webp" width="1200" height="600"
         priority alt="Dashboard" />

    <!-- Grid responsive: sizes genera el srcset adecuado -->
    <img ngSrc="/assets/products/p-101.webp" width="400" height="300"
         sizes="(max-width: 768px) 100vw, 33vw" alt="Producto" />

    <!-- Contenedor fluido: fill en lugar de width/height -->
    <div class="banner">  <!-- position: relative -->
      <img ngSrc="/assets/banner.webp" fill alt="" />
    </div>
  \`
})

Image loaders para CDNs

Con un loader, ngSrc se transforma en URLs del CDN con resize/format automático (Cloudinary, Imgix, ImageKit, Netlify o uno custom): 

// app.config.ts
providers: [
  provideImgixLoader('https://mi-cuenta.imgix.net'),
  // o custom:
  {
    provide: IMAGE_LOADER,
    useValue: (config: ImageLoaderConfig) =>
      \`https://cdn.midominio.com/${config.src}?w=${config.width}&fmt=webp\`
  }
]

Best practices

  • priority solo en la imagen LCP (1 por página, above the fold).
  • Formatos modernos: WebP/AVIF reducen 30-60 % vs JPEG.
  • width/height con el aspect ratio real del archivo — la directiva avisa si lo distorsionas.
  • Todo lo below the fold se queda con el lazy loading por defecto.

⚠️ Errores comunes

  • Olvidar priority en el hero: la imagen LCP se carga lazy y el LCP se dispara. NgOptimizedImage lo detecta y lanza warning en dev.
  • Poner priority en muchas imágenes: todas compiten por la red y ninguna llega antes.
  • fill sin position: relative en el contenedor: la imagen se posiciona respecto a otro ancestro.
  • Servir una imagen de 2000px en un slot de 300px: usa un loader/CDN que redimensione según el srcset.

🎤 Preguntas de entrevista

  • ¿Qué hace exactamente priority? Desactiva el lazy loading, marca fetchpriority="high" y genera un preload hint: la imagen crítica empieza a descargarse lo antes posible.
  • ¿Cómo evita CLS NgOptimizedImage? Obliga a declarar width/height (o fill) para que el browser reserve el espacio antes de descargar la imagen.
  • ¿Para qué sirve sizes? Le dice al browser qué ancho ocupará la imagen en cada breakpoint para que elija la variante correcta del srcset.

11. runOutsideAngular y Zoneless

Zone.js parchea las APIs asíncronas del browser (eventos, timers, promises, XHR) para saber "algo pasó → ejecuta change detection". El problema: eventos de alta frecuencia (scroll, mousemove, requestAnimationFrame, websockets con ticks) disparan CD decenas de veces por segundo aunque la UI no necesite actualizarse.

runOutsideAngular: escapar de la zona

@Component({ /* ... */ })
export class CanvasChartComponent {
  private ngZone = inject(NgZone);
  private fps = signal(0);

  constructor() {
    this.ngZone.runOutsideAngular(() => {
      // ✅ el loop de animación NO dispara CD en cada frame
      const loop = () => {
        this.drawFrame();
        requestAnimationFrame(loop);
      };
      requestAnimationFrame(loop);
    });
  }

  private onStatsReady(fps: number) {
    // Volver a la zona SOLO cuando la UI debe actualizarse
    this.ngZone.run(() => this.fps.set(fps));
  }
}

Casos de uso: listeners de scroll/resize/mousemove, animaciones canvas/WebGL, integración con librerías third-party (charts, mapas), polling. Regla: corre fuera lo frecuente, y re-entra con ngZone.run() (o actualiza un signal) solo cuando haya algo que pintar.

Zoneless: Angular sin Zone.js

El modo zoneless elimina Zone.js: la CD ya no se dispara "por si acaso", sino por notificaciones explícitas — signals leídos en templates, eventos de template, async pipe y markForCheck(). Beneficios: menos JS inicial (~13 KB de zone.js), sin parcheo global, stack traces limpios y CD proporcional a lo que realmente cambió. 

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    // y elimina 'zone.js' de "polyfills" en angular.json
  ]
};

// En zoneless, esto NO actualiza la UI:
setTimeout(() => { this.title = 'nuevo'; }, 1000);       // ❌

// Esto sí:
setTimeout(() => this.title.set('nuevo'), 1000);          // ✅ signal

⚠️ Errores comunes

  • Olvidar re-entrar a la zona: actualizas una propiedad plana desde un callback fuera de la zona y la vista no cambia. Usa ngZone.run() o signals (funcionan en ambos mundos).
  • Migrar a zoneless con estado en propiedades mutables: todo lo que la vista muestre debe fluir por signals / async pipe / eventos; lo demás queda congelado.
  • Registrar listeners fuera de la zona y no removerlos: sigue siendo un memory leak; guarda la referencia y limpia en DestroyRef.onDestroy().

🎤 Preguntas de entrevista

  • ¿Qué hace Zone.js y qué costo tiene? Parchea las APIs async para disparar CD tras cada tarea; el costo es CD excesiva con eventos de alta frecuencia y overhead global.
  • ¿Cuándo usarías runOutsideAngular? Para trabajo frecuente que no cambia la UI en cada tick: animaciones, scroll/mousemove, polling, librerías de terceros.
  • ¿Qué requisitos tiene una app zoneless? Que toda actualización de la vista pase por signals, async pipe, eventos de template o markForCheck() — nada de confiar en que "el setTimeout refresca solo".

12. Web Vitals: LCP, CLS e INP

Los Core Web Vitals son las métricas de experiencia de usuario que Google usa para medir (y rankear) páginas. Se miden sobre usuarios reales (field data / CrUX), no solo en Lighthouse.

MétricaQué mideBuenoPobre
LCP (Largest Contentful Paint)Cuándo se pinta el elemento más grande visible (hero, título)≤ 2.5 s> 4 s
CLS (Cumulative Layout Shift)Cuánto "salta" el layout durante la vida de la página≤ 0.1> 0.25
INP (Interaction to Next Paint)Latencia de la peor interacción (click/tecla → siguiente frame). Reemplazó a FID en 2024≤ 200 ms> 500 ms

Cómo mejorarlas en Angular

  • LCP: SSR (el HTML llega renderizado), NgOptimizedImage con priority en la imagen hero, bundle inicial pequeño (lazy loading + @defer), preconnect a CDNs y fuentes con font-display: swap.
  • CLS: dimensiones explícitas en imágenes/iframes/ads, skeletons con el tamaño real en @placeholder y estados de loading, no insertar banners encima del contenido ya pintado.
  • INP: handlers cortos (< 50 ms), OnPush/zoneless para que un click no revise toda la app, cálculos pesados a Web Workers o troceados (scheduler.yield/setTimeout), evitar métodos costosos en template.

Medir en usuarios reales (RUM)

// npm i web-vitals
import { onLCP, onCLS, onINP } from 'web-vitals';

export function initWebVitals() {
  const report = (metric: { name: string; value: number }) =>
    navigator.sendBeacon('/analytics', JSON.stringify(metric));

  onLCP(report);
  onCLS(report);
  onINP(report);
}

// main.ts
bootstrapApplication(AppComponent, appConfig)
  .then(() => initWebVitals());

⚠️ Errores comunes

  • Optimizar solo para Lighthouse local: tu máquina con caché caliente no es un móvil de gama media con 4G. Usa field data (CrUX, RUM).
  • Confundir FID con INP: FID medía solo el delay de la primera interacción; INP mide todas las interacciones e incluye el procesamiento del handler y el render.
  • Spinner que causa CLS: un loading pequeño que se reemplaza por contenido grande desplaza todo lo de abajo.

🎤 Preguntas de entrevista

  • ¿Qué mide INP y cómo lo mejorarías en Angular? La latencia de interacción hasta el siguiente paint; se mejora acortando handlers, con OnPush/zoneless, workers y evitando trabajo pesado en template.
  • ¿Cómo ayuda SSR al LCP? El HTML ya llega renderizado desde el servidor: el contenido principal se pinta sin esperar a descargar y ejecutar el JS.
  • ¿Qué provoca CLS en una SPA y cómo lo previenes? Contenido async sin espacio reservado; se previene con dimensiones explícitas y skeletons del tamaño final.

13. SSR, Hydration e Incremental Hydration

Con SSR (@angular/ssr) el servidor renderiza la ruta y envía HTML completo: el usuario ve contenido de inmediato (mejor LCP y SEO) mientras el JS se descarga en paralelo. Hydration es el paso siguiente: en lugar de destruir ese DOM y re-renderizar desde cero (comportamiento antiguo, causaba flicker), Angular reutiliza el DOM existente, adjunta listeners y restaura estado. 

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideClientHydration(
      withEventReplay(),           // captura clicks pre-hydration y los re-emite
      withIncrementalHydration()   // habilita hydrate triggers en @defer
    )
  ]
};

Reglas para que la hydration funcione

  • El HTML del servidor y el del cliente deben coincidir: nada de manipular DOM directo (innerHTML, jQuery) ni HTML inválido (<div> dentro de <p>, <tr> fuera de <table>).
  • Código browser-only (acceso a window, document, canvas) va en afterNextRender().
  • Componentes problemáticos pueden excluirse con ngSkipHydration (último recurso).
  • HttpClient con SSR cachea las respuestas del servidor y las transfiere al cliente (TransferState) para no repetir los requests.

Incremental hydration: hidratar solo lo que se usa

Con hydration completa, toda la página se hidrata al cargar el JS. La incremental hydration (Angular 19+) combina SSR con @defer: el HTML del bloque llega renderizado del servidor y se muestra desde el primer momento, pero su JavaScript no se descarga ni ejecuta hasta el trigger hydrate. Es "islands architecture" en Angular. 

@defer (hydrate on viewport) {
  <app-comments />   <!-- HTML visible desde el server-render;
                          JS solo cuando entra al viewport -->
}

@defer (hydrate on interaction) {
  <app-video-player />
}

@defer (hydrate never) {
  <app-static-footer />   <!-- nunca se hidrata: HTML puro -->
}

⚠️ Errores comunes

  • Acceder a window en el constructor/ngOnInit: revienta en el servidor (Node no tiene DOM). Usa afterNextRender().
  • HTML mal anidado: el browser lo "corrige" al parsear y ya no coincide con lo que Angular espera → hydration mismatch.
  • Renderizar contenido dependiente de datos no deterministas (fecha/aleatorio distinto en server y client) → mismatch visual.
  • Asumir que hydration = interactividad inmediata: hasta que el JS carga, los clicks se pierden — por eso existe withEventReplay().

🎤 Preguntas de entrevista

  • ¿Qué es hydration y qué problema resuelve? Reutilizar el DOM renderizado en el servidor en lugar de destruirlo y re-crearlo: elimina el flicker y mejora LCP/CLS.
  • ¿Qué aporta la incremental hydration? El HTML server-rendered se muestra ya, pero el JS de cada isla se difiere hasta su trigger — se hidrata solo lo que el usuario usa.
  • ¿Cómo evitas repetir requests HTTP tras SSR? Con el caching/TransferState de provideClientHydration: las respuestas del servidor viajan embebidas y el cliente las reutiliza.

14. Profiling con Angular DevTools y Chrome Performance

Regla de oro: medir antes de optimizar. Sin un profile, "optimizas" por intuición y sueles atacar el problema equivocado. Dos herramientas complementarias: Angular DevTools (qué hace Angular) y el panel Performance de Chrome (qué hace el browser).

Angular DevTools (extensión de Chrome/Firefox)

  • Components tab: inspecciona el árbol, inputs/outputs, signals y la estrategia de CD de cada componente.
  • Profiler tab: graba interacciones y muestra cada ciclo de change detection: qué lo disparó, qué componentes se revisaron y cuánto tardó cada uno (flame graph / bar chart).
  • Qué buscar: componentes que se re-revisan en cada ciclo sin cambiar (candidatos a OnPush), ciclos de CD demasiado frecuentes (eventos de alta frecuencia dentro de la zona) y componentes individuales lentos (lógica cara en getters/métodos de template).

Chrome Performance panel

  • Graba con CPU throttling 4x-6x para simular hardware real.
  • Long tasks (bloques rojos > 50 ms) bloquean el main thread y degradan INP: expande el flame chart para ver qué función los causa.
  • Distingue las fases: Scripting (JS/CD), Rendering (style/layout) y Painting. Muchos recálculos de layout intercalados con JS = layout thrashing (leer offsetHeight y escribir estilos en un loop).
  • La pestaña Lighthouse automatiza un diagnóstico inicial; el panel Performance explica el porqué.
// Marcar código propio para verlo en el timeline
performance.mark('filter-start');
const result = this.applyFilters(rows);
performance.mark('filter-end');
performance.measure('filter', 'filter-start', 'filter-end');
// → aparece en Performance panel → "Timings"

Workflow sugerido

  1. Reproduce la lentitud con una interacción concreta.
  2. Angular DevTools Profiler: ¿la CD corre demasiado o hay un componente lento?
  3. Chrome Performance: ¿el tiempo se va en scripting, layout o paint?
  4. Aplica el fix (OnPush, computed, defer, worker...) y vuelve a medir.

⚠️ Errores comunes

  • Perfilar en modo dev: los checks extra de desarrollo distorsionan; para números reales usa un build de producción (los profiles dev sirven para comparar relativo).
  • Perfilar sin throttling en una workstation potente: los usuarios no tienen tu máquina.
  • Optimizar sin baseline: guarda el profile "antes" para demostrar la mejora.

🎤 Preguntas de entrevista

  • ¿Cómo detectas qué componente ralentiza la app? Con el Profiler de Angular DevTools: grabar la interacción y ordenar los componentes por tiempo de CD por ciclo.
  • ¿Qué es una long task y por qué importa? Trabajo > 50 ms en el main thread; durante ese tiempo el browser no responde a input → INP alto.
  • ¿Qué es layout thrashing? Alternar lecturas (offsetWidth) y escrituras de estilos forzando recálculos de layout síncronos repetidos; se arregla agrupando lecturas y escrituras.

15. Memory Leaks — detección y prevención

Un memory leak en una SPA es memoria que debería liberarse al destruir un componente pero sigue referenciada. Como la app vive horas sin recargar, los leaks se acumulan: la app se vuelve lenta, el GC trabaja cada vez más y al final la pestaña crashea.

Las 5 fuentes clásicas en Angular

  • Suscripciones RxJS sin cleanup a streams que no completan (Subject de un servicio, interval, router events, store). El closure retiene el componente entero.
  • Event listeners globales: window.addEventListener('resize', ...) sin el removeEventListener correspondiente.
  • setInterval / requestAnimationFrame sin clearInterval / cancelAnimationFrame.
  • Librerías third-party sin destroy(): charts, mapas y editores que registran sus propios listeners.
  • Referencias en servicios root: guardar componentes/elementos DOM en un singleton (caches, registries) los mantiene vivos para siempre.
@Component({ /* ... */ })
export class DashboardComponent {
  private destroyRef = inject(DestroyRef);

  constructor() {
    // ✅ RxJS: se cancela al destruir el componente
    interval(5000)
      .pipe(takeUntilDestroyed())
      .subscribe(() => this.refresh());

    // ✅ Listener global con cleanup registrado
    const onResize = () => this.relayout();
    window.addEventListener('resize', onResize);
    this.destroyRef.onDestroy(() =>
      window.removeEventListener('resize', onResize)
    );

    // ✅ Librería externa con destroy()
    afterNextRender(() => {
      const chart = new Chart(this.canvas(), { /* ... */ });
      this.destroyRef.onDestroy(() => chart.destroy());
    });
  }
}

Detectar leaks con Chrome DevTools

  1. Sospecha: en Performance, graba con "Memory" activado: si el heap sube en escalera y el GC nunca lo baja al nivel inicial, hay leak.
  2. Técnica de los 3 snapshots (Memory → Heap snapshot): snapshot → navega para montar/desmontar el componente sospechoso varias veces → fuerza GC (icono 🗑) → snapshot. Compara: si instancias del componente siguen creciendo, hay leak.
  3. Buscar la causa: filtra por el nombre de la clase del componente y revisa Retainers: la cadena de referencias muestra quién lo mantiene vivo (un Subscriber, un listener, un array en un servicio...).
  4. Detached elements: nodos DOM fuera del documento pero retenidos por JS — en el snapshot se llaman "Detached <div>".

⚠️ Errores comunes

  • "HttpClient no necesita unsubscribe, entonces nada lo necesita": los observables HTTP completan solos, pero Subjects, intervals, router events y stores NO.
  • takeUntilDestroyed() fuera del injection context sin pasar DestroyRef: lanza error. En ngOnInit u otros métodos: takeUntilDestroyed(this.destroyRef).
  • Remover el listener con otra referencia de función: removeEventListener('resize', () => ...) no remueve nada; guarda la misma referencia.
  • Comparar snapshots sin forzar GC: memoria pendiente de recolección parece leak sin serlo.

🎤 Preguntas de entrevista

  • ¿Cómo confirmas un memory leak en una SPA? Heap snapshots repetidos tras montar/desmontar el componente con GC forzado: si las instancias crecen monotónicamente, hay leak; los Retainers muestran la causa.
  • ¿Qué observables requieren cleanup? Los que no completan: Subjects de servicios, interval/timer, fromEvent, router events, stores. HTTP completa solo.
  • ¿Qué mecanismos modernos de cleanup ofrece Angular? takeUntilDestroyed(), DestroyRef.onDestroy(), el async pipe y los effects (se limpian con su contexto).
