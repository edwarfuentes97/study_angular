1 · Smart / Dumb Component Pattern

Los Smart components (containers) gestionan el estado y la lógica de negocio: inyectan servicios, Store o signals y coordinan la comunicación con el backend. Los Dumb components (presentacionales) solo reciben datos vía @Input() y emiten eventos con @Output(), usando ChangeDetectionStrategy.OnPush para máximo rendimiento. Esta separación facilita testing, reutilización y mantenimiento. 

// ─── Smart container ───────────────────────────────
@Component({
  template: \`
    <app-user-list
      [users]="users()"
      (select)="onSelect($event)" />
  \`
})
export class UserContainerComponent {
  private store = inject(Store);
  users = toSignal(this.store.select(selectAllUsers));

  onSelect(user: User) {
    this.store.dispatch(selectUser({ user }));
  }
}

// ─── Dumb presentational ──────────────────────────
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class UserListComponent {
  @Input() users: User[] = [];
  @Output() select = new EventEmitter<User>();
}

// ─── Versión moderna con signal inputs (Angular 17.1+) ──
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class UserListComponent {
  users = input.required<User[]>();   // input() signal
  select = output<User>();            // output() function
}

Beneficios del patrón

  • Testing barato: un dumb component se prueba solo con inputs y outputs — sin mocks de servicios, Store ni HTTP. Los smart se prueban mockeando una sola dependencia (Store o facade).
  • Reutilización: el mismo presentacional sirve en varias pantallas porque no sabe de dónde vienen los datos.
  • Rendimiento: con OnPush, el dumb solo se re-renderiza cuando cambia la referencia de sus inputs.
  • Flujo unidireccional: datos hacia abajo, eventos hacia arriba. Depurar significa mirar un solo punto: el container.
// Test de un dumb component — sin mocks
it('emite select al hacer click en un usuario', () => {
  const fixture = TestBed.createComponent(UserListComponent);
  fixture.componentRef.setInput('users', [{ id: 1, name: 'Ana' }]);
  fixture.detectChanges();

  let selected: User | undefined;
  fixture.componentInstance.select.subscribe(u => (selected = u));

  fixture.nativeElement.querySelector('li').click();
  expect(selected?.id).toBe(1);
});

⚠️ Errores comunes

  • Inyectar servicios (HTTP, Store) dentro del presentacional — deja de ser dumb y pierde reutilización y testabilidad.
  • Mutar el objeto recibido por @Input dentro del hijo: rompe OnPush y crea efectos laterales invisibles para el padre.
  • Convertir TODO en dumb components: un container por cada 5-10 presentacionales es la proporción típica; el dogmatismo genera prop-drilling de 4 niveles.
  • Pasar callbacks como @Input en lugar de usar @Output: oculta el flujo de eventos y complica el testing.

🎤 Preguntas de entrevista

¿Cómo decides si un componente debe ser smart o dumb?

Si necesita saber de dónde vienen los datos (servicio, Store, ruta) es smart; si solo necesita saber qué pintar y qué eventos emitir, es dumb. Regla práctica: los componentes de página/ruta son smart; casi todo lo demás puede ser presentacional.

¿Por qué los dumb components se benefician de OnPush?

Porque todas sus fuentes de cambio son inputs (referencias) y eventos propios, exactamente lo que OnPush escucha. Angular puede saltarse el componente en cada ciclo de change detection si sus inputs no cambiaron.

¿El patrón sigue vigente con signals?

Sí. Cambia la sintaxis (input()/output() en lugar de decoradores) pero la separación estado/presentación sigue siendo el criterio arquitectónico. Con signals incluso mejora: los inputs signal integran naturalmente con computed() en el hijo.

2 · Comunicación entre Componentes

Angular ofrece varios mecanismos: @Input/@Output para flujos padre→hijo directos, servicios compartidos con BehaviorSubject para componentes hermanos o desacoplados, Signals como alternativa reactiva moderna, y NgRx Store para estado global predecible. Usa el más sencillo que resuelva el problema. 

// ─── Servicio con BehaviorSubject ──────────────────
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme$ = new BehaviorSubject<'light' | 'dark'>('dark');
  current$ = this.theme$.asObservable();

  toggle() {
    this.theme$.next(
      this.theme$.value === 'dark' ? 'light' : 'dark'
    );
  }
}

// ─── Mismo servicio con Signals ───────────────────
@Injectable({ providedIn: 'root' })
export class ThemeSignalService {
  theme = signal<'light' | 'dark'>('dark');

  toggle() {
    this.theme.update(t => t === 'dark' ? 'light' : 'dark');
  }
}

¿Qué canal usar?

  • input()/output(): relación padre-hijo directa. Siempre la primera opción: explícita, tipada y visible en el template.
  • Servicio compartido (signal o BehaviorSubject): hermanos o componentes lejanos que comparten un estado pequeño (tema, sidebar, toasts).
  • Store (NgRx): estado consumido por varias features, con necesidad de trazabilidad (DevTools) o efectos coordinados.
  • Router state (params/queryParams): cuando el estado debe sobrevivir a un refresh o ser compartible por URL (filtros, paginación, id seleccionado).

⚠️ Errores comunes

  • Exponer el BehaviorSubject directamente en lugar de asObservable(): cualquiera puede hacer next() y el estado se vuelve impredecible.
  • Exponer un signal escribible público cuando solo un módulo debería mutarlo — expón readonly() o computed() y métodos de mutación.
  • Saltar directo a NgRx para comunicar dos componentes hermanos: un servicio con signal resuelve el 80 % de los casos.
  • Duplicar en un Store estado que ya vive en la URL (query params) — dos fuentes de verdad que se desincronizan.

🎤 Preguntas de entrevista

¿Signal o BehaviorSubject para un servicio compartido?

Signal si el consumo es principalmente en templates y el valor es síncrono: menos boilerplate, sin async pipe, integración con computed(). BehaviorSubject/Observable si necesitas operadores de tiempo (debounce, switchMap) o interoperar con streams; siempre puedes puentear con toSignal()/toObservable().

¿Cuándo el estado debe vivir en la URL?

Cuando el usuario espera poder recargar, compartir el enlace o usar atrás/adelante y ver lo mismo: filtros de búsqueda, página actual, pestaña activa, id del detalle abierto. La URL es estado global gratuito y persistente.

3 · Estructura de Carpetas Enterprise

Organiza el proyecto en tres capas: core/ para servicios singleton, guards e interceptors; shared/ para componentes, pipes y directivas reutilizables; y features/ con un módulo por dominio. Cada capa expone un index.ts barrel export para imports limpios. 

src/app/
├── core/
│   ├── services/
│   │   ├── auth.service.ts
│   │   └── api.service.ts
│   ├── guards/
│   │   └── auth.guard.ts
│   ├── interceptors/
│   │   └── token.interceptor.ts
│   └── index.ts
├── shared/
│   ├── components/
│   │   └── button/button.ts
│   ├── pipes/
│   │   └── currency-format.ts
│   ├── directives/
│   │   └── tooltip.ts
│   └── index.ts
├── features/
│   ├── users/
│   │   ├── user-list.ts
│   │   ├── user-detail.ts
│   │   └── users.routes.ts
│   └── products/
│       ├── product-list.ts
│       └── products.routes.ts
└── app.routes.ts

⚠️ Errores comunes

  • Carpetas por tipo técnico a nivel raíz (components/, services/, pipes/ globales): al crecer, cada cambio de una feature toca 4 carpetas distintas. Agrupa por dominio.
  • Meter en shared/ componentes con lógica de negocio "porque dos features lo usan": si tiene reglas de dominio no es shared, extrae la lógica a un servicio de core o duplica el componente.
  • core/ convertido en cajón de sastre: si un servicio solo lo usa una feature, vive en esa feature.

🎤 Preguntas de entrevista

¿Qué va en core/ y qué en shared/?

core/: cosas que existen una sola vez en la app (auth, interceptors, configuración, ErrorHandler). shared/: piezas de UI sin estado y sin dependencias hacia arriba, reutilizables por cualquier feature (botones, pipes, directivas). Core es "infraestructura singleton", shared es "kit de UI".

Con standalone components ¿sigue teniendo sentido core/shared/features?

Sí. Los NgModules desaparecen pero las fronteras lógicas siguen: la estructura de carpetas + reglas de import (lint) reemplazan lo que antes forzaban los módulos.

📂 Estructura actual del proyecto:
  • core/ — Servicios singleton, interceptors, guards
  • shared/ — Componentes, directivas, modelos reutilizables
  • features/ — Módulos por dominio (ngrx-lab, rxjs-lab, etc.)

4 · SCAM → Standalone

El patrón SCAM (Single Component Angular Module) envolvía cada componente en su propio NgModule. Con standalone components de Angular 14+ ya no se necesita esa capa extra: el componente declara sus propios imports directamente. 

// ─── Antes: SCAM ──────────────────────────────────
@NgModule({
  declarations: [UserCardComponent],
  imports: [AutoGlossaryDirective, CommonModule],
  exports: [UserCardComponent]
})
export class UserCardModule {}

// ─── Después: Standalone ──────────────────────────
@Component({
  standalone: true,
  imports: [AutoGlossaryDirective, NgIf, NgFor],
  selector: 'app-user-card',
  template: \`...\`
})
export class UserCardComponent {}

⚠️ Errores comunes

  • Escribir standalone: true en Angular 19+: ya es el default; el flag explícito solo añade ruido (y standalone: false es lo que hay que marcar en código legacy).
  • Importar CommonModule completo cuando solo usas un pipe: importa AsyncPipe, DatePipe, etc. individualmente (o usa control flow @if/@for que no requiere imports).
  • Migrar a standalone y conservar NgModules "de agrupación" vacíos: elimina la capa, las rutas con loadComponent la reemplazan.

🎤 Preguntas de entrevista

¿Qué problema resolvía SCAM y por qué standalone lo elimina?

SCAM daba a cada componente un módulo propio para lograr imports granulares y tree-shaking fino. Standalone hace eso nativo: el componente declara sus propias dependencias en imports, sin módulo intermedio.

📂 Todos los componentes del proyecto son Standalone (Angular 20.3 default)

5 · Barrel Exports

Los archivos index.ts re-exportan los símbolos públicos de una carpeta, simplificando las rutas de importación y creando una API pública clara para cada capa del proyecto. 

// shared/index.ts
export * from './components/button/button';
export * from './pipes/currency-format';
export * from './directives/tooltip';

// Consumo limpio desde cualquier feature:
import {
  ButtonComponent,
  CurrencyFormatPipe,
  TooltipDirective
} from '../../shared';

⚠️ Errores comunes

  • Imports circulares: si a.ts importa desde el barrel y el barrel re-exporta a.ts, obtienes undefined en runtime. Dentro de la misma carpeta importa siempre por ruta directa, nunca desde el propio index.ts.
  • Barrels gigantes: un único index.ts para toda la app arrastra medio proyecto a cada chunk y degrada el tree-shaking y el build incremental. Un barrel por capa/feature, no uno global.
  • Re-exportar TODO (export *) incluyendo internos: el barrel es la API pública — exporta explícitamente solo lo que otras capas deben ver.

🎤 Preguntas de entrevista

¿Qué desventajas tienen los barrel exports?

Dependencias circulares difíciles de depurar, peor tree-shaking si el bundler no puede probar ausencia de side effects, y builds incrementales más lentos porque tocar un archivo invalida a todos los consumidores del barrel. Por eso se usan como API pública entre capas, no para cada carpeta interna.

6 · Arquitectura de una app Angular Enterprise

En aplicaciones grandes, la arquitectura define la mantenibilidad a largo plazo. La clave es separar responsabilidades en capas con reglas de dependencia estrictas.

│       └── users.routes.ts

Reglas de dependencia

  • core/ → importa de shared/, nunca de features/.
  • shared/ → NO importa de core/ ni de features/. Es 100 % autónoma.
  • features/ → importa de core/ y shared/, nunca de otros features.
  • Si dos features necesitan comunicarse → usan un servicio de core/ o un Store global (NgRx).
// ❌ MAL — feature importa directamente de otro feature
import { ProductService } from '../products/services/product.service';

// ✅ BIEN — servicio compartido vive en core/
import { CatalogService } from '../../core/services/catalog.service';

// ✅ BIEN — comunicación via Store global
import { selectCartItems } from '../../core/store/cart.selectors';

Barrel exports (index.ts)

Cada capa expone un index.ts que re-exporta solo la API pública. Esto simplifica imports y crea un contrato claro entre capas. 

// shared/components/index.ts
export { ButtonComponent } from './button/button.component';
export { CardComponent } from './card/card.component';
export { ModalComponent } from './modal/modal.component';

// core/index.ts
export { AuthService } from './services/auth.service';
export { ApiService } from './services/api.service';
export { AuthGuard } from './guards/auth.guard';

// En un feature:
import { ButtonComponent, CardComponent } from '@shared/components';
import { AuthService } from '@core';

Tip: configura paths en tsconfig.json para alias como @core, @shared, @features. 

// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@core/*": ["src/app/core/*"],
      "@shared/*": ["src/app/shared/*"],
      "@features/*": ["src/app/features/*"]
    }
  }
}

Cómo hacer cumplir las reglas (lint)

Las reglas de dependencia que no se verifican automáticamente se violan en el primer sprint con prisa. Con eslint-plugin-boundaries (o los module boundaries de Nx) el import prohibido rompe el CI: 

// eslint.config.js (extracto con eslint-plugin-boundaries)
settings: {
  'boundaries/elements': [
    { type: 'core', pattern: 'src/app/core/*' },
    { type: 'shared', pattern: 'src/app/shared/*' },
    { type: 'feature', pattern: 'src/app/features/*', capture: ['name'] }
  ]
},
rules: {
  'boundaries/element-types': ['error', {
    default: 'disallow',
    rules: [
      { from: 'shared', allow: ['shared'] },
      { from: 'core', allow: ['core', 'shared'] },
      // un feature NO puede importar de otro feature
      { from: 'feature', allow: ['core', 'shared', ['feature', { name: '${from.name}' }]] }
    ]
  }]
}

⚠️ Errores comunes

  • Definir las reglas de capas solo en un README: sin lint, la arquitectura se erosiona silenciosamente.
  • Imports entre features "solo por esta vez": crea un grafo de dependencias que impide extraer, lazy-loadear o borrar features de forma segura.
  • Path aliases sin regla de lint equivalente: @features/a importando @features/b sigue compilando feliz.

🎤 Preguntas de entrevista

Dos features necesitan el mismo dato, ¿qué haces?

Nunca importar una desde la otra. Opciones: (1) subir el estado a un servicio o Store en core/, (2) extraer el modelo/lógica compartida a una lib o carpeta de dominio compartido, (3) comunicarlas por la URL si es estado navegacional. El criterio: la dependencia siempre apunta "hacia abajo".

¿Cómo evitas que la arquitectura se degrade con el tiempo?

Automatización: lint de boundaries en CI, revisión de imports en PRs, budgets de bundle por lazy chunk, y ADRs (Architecture Decision Records) para que las excepciones queden documentadas y no se conviertan en precedente.

  • shared/ — Componentes, directivas, modelos reutilizables
  • features/ — Módulos por dominio lazy-loaded (ngrx-lab, rxjs-lab, etc.)

7 · Patrones de comunicación entre componentes

Angular ofrece 6 mecanismos de comunicación. Elegir el correcto depende de la relación entre componentes y la complejidad del estado compartido.

1. @Input / @Output — Padre ↔ Hijo directo

El mecanismo más simple y explícito. Limitado a la jerarquía directa padre-hijo. Ideal para componentes presentacionales. 

// Padre envía datos al hijo con @Input
// Hijo notifica al padre con @Output
@Component({
  selector: 'app-parent',
  template: \`
    <app-child
      [item]="selectedItem"
      (save)="onSave($event)" />
  \`
})
export class ParentComponent {
  selectedItem = signal({ id: 1, name: 'Angular' });

  onSave(updated: Item) {
    this.selectedItem.set(updated);
  }
}

@Component({
  selector: 'app-child',
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class ChildComponent {
  @Input() item!: Item;
  @Output() save = new EventEmitter<Item>();

  onSubmit() {
    this.save.emit({ ...this.item, name: 'Updated' });
  }
}

2. Shared Service + BehaviorSubject — Sin relación directa

Un servicio inyectable actúa como mediador entre componentes que no tienen relación padre-hijo. Usa BehaviorSubject para mantener y emitir el último valor. 

@Injectable({ providedIn: 'root' })
export class NotificationService {
  private notificationsSubject = new BehaviorSubject<string[]>([]);
  notifications$ = this.notificationsSubject.asObservable();

  add(message: string) {
    const current = this.notificationsSubject.value;
    this.notificationsSubject.next([...current, message]);
  }

  clear() {
    this.notificationsSubject.next([]);
  }
}

// Componente A — agrega notificaciones
@Component({ ... })
export class OrderComponent {
  private notif = inject(NotificationService);

  placeOrder() {
    this.notif.add('Pedido creado exitosamente');
  }
}

// Componente B — muestra notificaciones
@Component({
  template: \`
    @for (msg of notifications$ | async; track msg) {
      <div class="toast">{{ msg }}</div>
    }
  \`
})
export class ToastComponent {
  notifications$ = inject(NotificationService).notifications$;
}

3. Signals (Angular 17+) — Reactivo y síncrono

Un signal en un servicio es la alternativa moderna a BehaviorSubject. Más simple, síncrono y con mejor integración con el template. 

@Injectable({ providedIn: 'root' })
export class CartService {
  items = signal<CartItem[]>([]);
  total = computed(() =>
    this.items().reduce((sum, i) => sum + i.price * i.qty, 0)
  );

  addItem(item: CartItem) {
    this.items.update(list => [...list, item]);
  }

  removeItem(id: number) {
    this.items.update(list => list.filter(i => i.id !== id));
  }
}

// Cualquier componente lo consume directamente
@Component({
  template: \`
    <span class="badge">{{ cart.items().length }}</span>
    <span>Total: {{ cart.total() | currency }}</span>
  \`
})
export class NavbarComponent {
  cart = inject(CartService);
}

4. NgRx Store — Estado global predecible

Para apps complejas con muchos consumidores del mismo estado. Patrón Redux: Actions → Reducers → Selectors → Effects. 

// Actions
export const loadProducts = createAction('[Products] Load');
export const loadProductsSuccess = createAction(
  '[Products] Load Success',
  props<{ products: Product[] }>()
);

// Reducer
export const productsReducer = createReducer(
  initialState,
  on(loadProductsSuccess, (state, { products }) => ({
    ...state, products, loading: false
  }))
);

// Selector
export const selectAllProducts = createSelector(
  selectProductsState,
  state => state.products
);

// Componente consume con Store
@Component({ ... })
export class ProductListComponent {
  private store = inject(Store);
  products = toSignal(this.store.select(selectAllProducts));

  ngOnInit() {
    this.store.dispatch(loadProducts());
  }
}

5. @ViewChild / @ContentChild — Acceso directo al hijo

El padre accede a propiedades y métodos públicos del hijo. Acoplamiento alto, usar con precaución y solo cuando sea necesario (ej: controlar un reproductor de video). 

@Component({
  template: \`
    <app-video-player #player />
    <button (click)="player.play()">Play</button>
    <button (click)="player.pause()">Pause</button>
  \`
})
export class VideoPageComponent {
  @ViewChild('player') player!: VideoPlayerComponent;

  seekTo(seconds: number) {
    this.player.seek(seconds);
  }
}

6. Event Bus (no recomendado)

Un Subject como bus global de eventos. Es un anti-pattern en Angular porque es difícil de rastrear, depurar y tipar. Usa NgRx Store en su lugar. 

// ❌ Anti-pattern — difícil de rastrear y depurar
@Injectable({ providedIn: 'root' })
export class EventBus {
  private bus = new Subject<{ type: string; payload: any }>();

  emit(type: string, payload: any) {
    this.bus.next({ type, payload });
  }

  on(type: string) {
    return this.bus.pipe(filter(e => e.type === type));
  }
}

// ✅ Mejor alternativa: NgRx Store o un servicio tipado con signals

7. Router state — La URL como fuente de verdad

Params, query params y data de resolvers son un canal de comunicación de primera clase: sobreviven al refresh, son compartibles y activan el botón atrás/adelante. Con withComponentInputBinding() los params llegan directo como inputs. 

// app.config.ts
provideRouter(routes, withComponentInputBinding());

// Ruta: /products?category=books&page=2
@Component({ ... })
export class ProductListComponent {
  // query params → signal inputs automáticamente
  category = input<string>();
  page = input(1, { transform: numberAttribute });

  private router = inject(Router);

  setCategory(cat: string) {
    // escribir estado = navegar
    this.router.navigate([], {
      queryParams: { category: cat, page: 1 },
      queryParamsHandling: 'merge'
    });
  }
}

Tabla de decisión

EscenarioPatrón recomendado
Padre → Hijo@Input
Hijo → Padre@Output
Hermanos cercanosShared Service o Signal

Estado de UI localSignal en componente

⚠️ Errores comunes

  • Encadenar @Output a través de 3+ niveles (event bubbling manual): señal de que falta un servicio compartido o un container intermedio.
  • Usar @ViewChild para leer estado del hijo en cada ciclo: invierte el flujo de datos; el hijo debería emitirlo.
  • Olvidar desuscribirse de observables de servicios en componentes (usa takeUntilDestroyed(), async pipe o toSignal()).

🎤 Preguntas de entrevista

¿Por qué un Event Bus global se considera anti-pattern?

Porque desacopla en exceso: cualquier parte de la app puede emitir o escuchar cualquier evento, sin tipado fuerte ni trazabilidad. Depurar "quién emitió esto" es arqueología. Un Store con actions tipadas da el mismo desacoplamiento pero con contrato explícito y DevTools.

Dos componentes hermanos deben sincronizar un contador, ¿qué usas?

Un servicio con signal provisto en el ancestro común (o root si es app-wide). Input/Output vía el padre también funciona si el padre ya existe y son pocos niveles; NgRx sería sobredimensionado.

8 · State Management — Cuándo usar qué

Elegir la herramienta de estado correcta evita sobre-ingeniería en apps simples y caos en apps complejas. Sigue este árbol de decisión:

1. Estado local del componente → signal() o propiedad

Contadores, toggles, estado de formulario, visibilidad de UI. No necesita salir del componente. 

@Component({
  template: \`
    <button (click)="showPanel.set(!showPanel())">
      {{ showPanel() ? 'Ocultar' : 'Mostrar' }}
    </button>
    @if (showPanel()) {
      <div class="panel">Contenido</div>
    }
  \`
})
export class TogglePanelComponent {
  showPanel = signal(false);
}

2. Compartido entre pocos componentes → Service + signal() o BehaviorSubject

Tema oscuro/claro, estado del sidebar, preferencias de usuario. Un servicio singleton con signal es la solución más simple. 

@Injectable({ providedIn: 'root' })
export class SidebarService {
  isOpen = signal(true);

  toggle() {
    this.isOpen.update(v => !v);
  }
}

// Header component
@Component({
  template: \`<button (click)="sidebar.toggle()">☰</button>\`
})
export class HeaderComponent {
  sidebar = inject(SidebarService);
}

// Layout component
@Component({
  template: \`
    <aside [class.open]="sidebar.isOpen()">...</aside>
  \`
})
export class LayoutComponent {
  sidebar = inject(SidebarService);
}

3. Estado complejo de feature → NgRx ComponentStore o SignalStore

CRUD de una feature, estado de formulario complejo, filtros + paginación. El estado es más complejo que un solo signal pero no necesita ser global. 

// NgRx SignalStore — alternativa moderna a ComponentStore
import { signalStore, withState, withComputed, withMethods, patchState }
  from '@ngrx/signals';

interface ProductState {
  products: Product[];
  loading: boolean;
  filter: string;
}

export const ProductStore = signalStore(
  withState<ProductState>({
    products: [],
    loading: false,
    filter: ''
  }),
  withComputed(({ products, filter }) => ({
    filteredProducts: computed(() => {
      const f = filter().toLowerCase();
      return f
        ? products().filter(p => p.name.toLowerCase().includes(f))
        : products();
    }),
    count: computed(() => products().length)
  })),
  withMethods((store, api = inject(ProductApiService)) => ({
    async loadProducts() {
      patchState(store, { loading: true });
      const products = await firstValueFrom(api.getAll());
      patchState(store, { products, loading: false });
    },
    setFilter(filter: string) {
      patchState(store, { filter });
    },
    removeProduct(id: number) {
      patchState(store, {
        products: store.products().filter(p => p.id !== id)
      });
    }
  }))
);

// Uso en componente — proveer a nivel de feature
@Component({
  providers: [ProductStore],
  template: \`
    <input (input)="store.setFilter($event.target.value)" />
    <p>{{ store.count() }} productos</p>
    @if (store.loading()) {
      <app-spinner />
    }
    @for (p of store.filteredProducts(); track p.id) {
      <app-product-card [product]="p" />
    }
  \`
})
export class ProductListComponent {
  store = inject(ProductStore);

  constructor() {
    this.store.loadProducts();
  }
}

4. Estado global de la app → NgRx Store

Auth, carrito de compras, datos compartidos entre features. Necesario cuando requieres: time-travel debugging, logging, hydration SSR, o múltiples features consumen el mismo estado. 

// Estructura NgRx completa para auth global
// auth.actions.ts
export const login = createAction('[Auth] Login', props<{ credentials: Credentials }>());
export const loginSuccess = createAction('[Auth] Login Success', props<{ user: User }>());
export const logout = createAction('[Auth] Logout');

// auth.reducer.ts
export const authReducer = createReducer(
  { user: null, isAuthenticated: false },
  on(loginSuccess, (state, { user }) => ({
    ...state, user, isAuthenticated: true
  })),
  on(logout, () => ({ user: null, isAuthenticated: false }))
);

// auth.selectors.ts
export const selectIsAuthenticated = createSelector(
  selectAuthState,
  state => state.isAuthenticated
);

// auth.effects.ts
loginEffect = createEffect(() =>
  this.actions$.pipe(
    ofType(login),
    exhaustMap(action =>
      this.authApi.login(action.credentials).pipe(
        map(user => loginSuccess({ user })),
        catchError(() => of(loginFailure()))
      )
    )
  )
);

Árbol de decisión rápido

PreguntaSi síSi no
¿El estado sale del componente? → Siguiente pregunta signal()
¿Lo usan más de 2-3 componentes? → Siguiente pregunta Service + signal
¿Es cross-feature o necesita DevTools? NgRx Store SignalStore

⚠️ Errores comunes

  • Todo al Store global: meter estado de UI efímero (modal abierto, texto de un input) en NgRx genera boilerplate sin beneficio y acopla componentes a estado que no les pertenece.
  • Estado servidor tratado como estado cliente: cachear datos HTTP a mano en el Store sin invalidación ni refetch. El "server state" tiene su propio ciclo de vida (stale, revalidación) — modélalo explícitamente o usa httpResource / una capa de caché dedicada.
  • Duplicar el mismo dato en varios niveles (componente + servicio + Store): una sola fuente de verdad, lo demás son computed() derivados.
  • Elegir herramienta por moda y no por requisitos: migrar a NgRx "porque es enterprise" o quitarlo "porque signals" sin analizar quién consume el estado.

🎤 Preguntas de entrevista

¿Cuándo justificarías introducir NgRx en un proyecto?

Cuando varias features consumen y mutan el mismo estado, necesitas auditoría o time-travel debugging, hay efectos coordinados complejos, o el equipo es grande y el patrón único (action → reducer → selector) reduce fricción. Para estado local o de feature, signals/SignalStore es suficiente.

¿Qué diferencia hay entre estado de cliente y estado de servidor?

El estado de cliente (tema, sidebar, selección) nace y muere en el navegador: tú eres la fuente de verdad. El estado de servidor es una copia caché de datos remotos: puede quedar obsoleto, requiere sincronización, loading y errores. Confundirlos lleva a Stores llenos de flags loading duplicados.

¿Dónde debería vivir el estado de un formulario multi-paso?

En un servicio con signals provisto en la ruta padre del wizard (scope de feature, no root): sobrevive a la navegación entre pasos y se destruye al salir del flujo. Global solo si otro módulo lo necesita.

9 · Arquitectura por Capas y Feature Slicing

Hay dos formas de partir una app: en capas horizontales (presentación, aplicación, dominio, infraestructura — cada capa atraviesa toda la app) o en slices verticales por feature (cada dominio de negocio contiene todo lo que necesita). La práctica moderna combina ambas: primero se corta vertical por feature, y dentro de cada slice se aplican capas ligeras. Así los cambios de negocio quedan localizados en una carpeta y el acoplamiento entre dominios es explícito. 

// Slice vertical con capas internas ligeras
features/orders/
├── ui/                        # capa presentación
│   ├── order-list.component.ts     (smart)
│   └── order-row.component.ts      (dumb)
├── data/                      # capa acceso a datos
│   ├── order-api.service.ts        (HTTP + DTOs)
│   └── order.store.ts              (SignalStore del slice)
├── domain/                    # capa dominio (TS puro)
│   ├── order.model.ts
│   └── order.rules.ts              (cálculos, validaciones)
└── orders.routes.ts           # entrada lazy del slice

Reglas dentro del slice

  • ui/ depende de data/ y domain/; data/ depende de domain/; domain/ no depende de nadie (ni de Angular).
  • El slice expone una API mínima (rutas + quizá un facade); el resto es privado.
  • Lo compartido entre slices "baja" a shared/ (UI) o a una carpeta de dominio compartido — nunca se importa slice a slice.

⚠️ Errores comunes

  • Capas horizontales puras en apps grandes: para tocar "pedidos" hay que editar components/, services/, models/ y store/ globales — alto acoplamiento por cercanía técnica en vez de por dominio.
  • Slices que crecen sin capas internas: 40 archivos planos donde el HTTP, el estado y la UI se mezclan en los mismos ficheros.
  • Crear las 4 capas completas en un slice de 3 pantallas: la estructura debe ganarse su existencia; empieza plano y extrae capas cuando duela.

🎤 Preguntas de entrevista

¿Qué ventaja tiene el feature slicing sobre las capas horizontales?

Cohesión: un cambio de negocio toca una sola carpeta. Además habilita lazy loading natural (un slice = un chunk), ownership por equipo y borrado seguro de features. Las capas horizontales optimizan para la simetría técnica, no para el cambio.

¿Cuándo "baja" código de un feature a shared?

Cuando un segundo consumidor real lo necesita (regla de tres: con el tercero seguro). Bajarlo especulativamente crea una capa shared enorme que nadie puede modificar sin miedo.

10 · Lazy Loading como Frontera Arquitectónica

El lazy loading no es solo una optimización de rendimiento: es la frontera física que hace real la separación entre features. Cada loadChildren/loadComponent produce un chunk independiente; si dos features se importan entre sí, el bundler las fusiona y la frontera desaparece. El tamaño de los chunks es, de hecho, un test de arquitectura. 

// app.routes.ts — cada feature es una frontera
export const routes: Routes = [
  {
    path: 'orders',
    loadChildren: () =>
      import('./features/orders/orders.routes')
        .then(m => m.ORDERS_ROUTES)
  },
  {
    path: 'admin',
    canMatch: [adminGuard],          // ni siquiera descarga el chunk sin permiso
    loadChildren: () =>
      import('./features/admin/admin.routes')
        .then(m => m.ADMIN_ROUTES)
  }
];

// orders.routes.ts — providers con scope de feature
export const ORDERS_ROUTES: Routes = [{
  path: '',
  providers: [
    OrderStore,                      // vive y muere con el feature
    { provide: ORDER_CONFIG, useValue: defaultOrderConfig }
  ],
  children: [
    { path: '', loadComponent: () => import('./ui/order-list.component').then(m => m.OrderListComponent) },
    { path: ':id', loadComponent: () => import('./ui/order-detail.component').then(m => m.OrderDetailComponent) }
  ]
}];

Mejores prácticas

  • Providers a nivel de ruta: servicios y stores del feature se declaran en sus rutas, no en root — se cargan con el chunk y no contaminan el bundle inicial.
  • canMatch para features protegidas: evita hasta la descarga del código si el usuario no tiene acceso.
  • Preloading selectivo: PreloadAllModules o una estrategia custom (por ejemplo, precargar según rol o con requestIdleCallback) equilibra arranque rápido y navegación instantánea.
  • Budgets en angular.json: fija límites de tamaño por bundle para detectar cuando una frontera se rompió.

⚠️ Errores comunes

  • Importar un símbolo de otro feature "solo un enum": el bundler arrastra el chunk entero o lo duplica en un common chunk gigante.
  • Servicios de feature con providedIn: 'root' que referencian código pesado: acaban en el bundle inicial aunque el feature sea lazy.
  • Lazy loading de TODO, incluida la pantalla principal: añade un roundtrip a la ruta más visitada. La ruta crítica puede ir eager.

🎤 Preguntas de entrevista

¿Por qué lazy loading es una decisión arquitectónica y no solo de performance?

Porque obliga a que las features sean autocontenidas: define fronteras de compilación y despliegue, delimita el scope de los providers y hace medible el acoplamiento (si el chunk crece o se fusiona, alguien cruzó la frontera).

¿Diferencia entre canActivate y canMatch en rutas lazy?

canActivate bloquea la activación pero el chunk ya puede haberse descargado; canMatch descarta la ruta antes de resolver el import, así que el código ni se descarga. Para features por rol, canMatch.

11 · Librerías y Monorepos (Nx)

Cuando el proyecto (o la organización) crece, las carpetas dejan de ser suficientes como frontera y se pasa a librerías dentro de un monorepo. Nx es el estándar de facto: cada lib tiene su API pública (un barrel), tags que declaran su tipo y dominio, y reglas de lint que prohíben dependencias ilegales. Además Nx cachea builds y solo re-testea lo afectado por un cambio (nx affected). 

// Taxonomía típica de libs en Nx
libs/
├── orders/
│   ├── feature-list/        # smart components + rutas    [type:feature]
│   ├── ui-order-card/       # dumb components             [type:ui]
│   ├── data-access/         # servicios HTTP + estado     [type:data-access]
│   └── domain/              # modelos + lógica pura       [type:util]
├── shared/
│   ├── ui-design-system/
│   └── util-formatters/
└── ...

// Reglas de dependencia por tipo:
// feature      → puede usar: feature, ui, data-access, util
// ui           → puede usar: ui, util
// data-access  → puede usar: data-access, util
// util         → puede usar: util
// eslint — module boundaries con tags
"@nx/enforce-module-boundaries": ["error", {
  "depConstraints": [
    { "sourceTag": "type:ui",
      "onlyDependOnLibsWithTags": ["type:ui", "type:util"] },
    { "sourceTag": "type:data-access",
      "onlyDependOnLibsWithTags": ["type:data-access", "type:util"] },
    { "sourceTag": "scope:orders",
      "onlyDependOnLibsWithTags": ["scope:orders", "scope:shared"] }
  ]
}]

Mejores prácticas

  • Dos dimensiones de tags: type:* (qué es) y scope:* (a qué dominio pertenece). Las reglas cruzan ambas.
  • Libs pequeñas y muchas: el grafo (nx graph) se vuelve la documentación viva de la arquitectura.
  • Apps delgadas: la app solo compone libs y define configuración; toda la lógica vive en libs reutilizables.

⚠️ Errores comunes

  • Monorepo sin reglas de boundaries: acabas con un "monolito distribuido en carpetas" donde todo importa de todo.
  • Una lib gigante shared/utils que depende de media app: rompe el cacheo y el affected (todo cambio la "afecta").
  • Adoptar Nx en un equipo de 2 personas con una sola app pequeña: el overhead de estructura supera el beneficio; la estructura core/shared/features basta.

🎤 Preguntas de entrevista

¿Qué aporta un monorepo Nx frente a multi-repo?

Cambios atómicos cross-proyecto (un PR toca lib y consumidores), una sola versión de dependencias, refactors seguros con tipado end-to-end, y CI inteligente que solo construye/testea lo afectado. El costo: exige disciplina de boundaries y tooling compartido.

¿Para qué sirven los tags de Nx?

Codifican la arquitectura en metadatos verificables: cada lib declara su tipo y scope, y el linter convierte el diagrama de capas en errores de compilación. Sin tags, las fronteras son solo convención.

12 · Patrón Facade

Un facade es un servicio que expone una API simple y orientada al dominio, ocultando la maquinaria de estado que hay detrás (NgRx, signals, varios servicios coordinados). Los componentes dependen del facade, no del Store: si mañana migras de NgRx a SignalStore, los componentes no se enteran. 

@Injectable({ providedIn: 'root' })
export class ProductsFacade {
  private store = inject(Store);

  // ── Lectura: el componente no conoce selectors ──
  products = toSignal(this.store.select(selectAllProducts),
    { initialValue: [] as Product[] });
  loading = toSignal(this.store.select(selectLoading),
    { initialValue: false });

  // ── Escritura: el componente no conoce actions ──
  load() {
    this.store.dispatch(ProductActions.load());
  }
  addToCart(product: Product) {
    this.store.dispatch(CartActions.add({ product }));
  }
}

// El componente queda trivial (y trivial de testear)
@Component({ ... })
export class ProductPageComponent {
  facade = inject(ProductsFacade);

  constructor() {
    this.facade.load();
  }
}

// Test: se mockea UNA dependencia con una API plana
const facadeMock = {
  products: signal([{ id: 1, name: 'Test' }]),
  loading: signal(false),
  load: jasmine.createSpy()
};
TestBed.configureTestingModule({
  providers: [{ provide: ProductsFacade, useValue: facadeMock }]
});

Mejores prácticas

  • API en lenguaje de dominio (addToCart), no de implementación (dispatch).
  • Un facade por feature/caso de uso, no un facade global.
  • El facade puede coordinar varios servicios (API + Store + router) y ofrecer una sola operación de alto nivel al componente.

⚠️ Errores comunes

  • Facade "dios" que agrega todo el estado de la app: mismo problema que un componente inflado, pero en un servicio.
  • Pass-through anémico obligatorio: si el facade solo renombra métodos de un único servicio sin añadir nada, es indirection gratuita — el patrón se aplica cuando hay complejidad que ocultar.
  • Filtrar tipos de la implementación (exponer Action o MemoizedSelector en la firma pública): rompe el aislamiento que justificaba el facade.

🎤 Preguntas de entrevista

¿Qué problema resuelve un facade sobre NgRx?

Desacopla componentes del mecanismo de estado: reduce el API surface que los componentes conocen (ni actions ni selectors), simplifica los tests (un mock plano) y permite cambiar la implementación de estado sin tocar la UI.

¿Cuándo NO usarías un facade?

Cuando no hay nada que ocultar: un componente que consume un único servicio simple no gana nada con una capa más. El facade se justifica con múltiples fuentes coordinadas o una librería de estado que quieres poder cambiar.

13 · Patrón Repository

El repository abstrae el acceso a datos detrás de un contrato: quien lo consume pide "productos" sin saber si vienen de HTTP, IndexedDB, un mock o GraphQL. En Angular, como las interfaces de TypeScript desaparecen en runtime, el contrato se materializa con una clase abstracta (que sí puede ser token de DI). 

// ── Contrato (dominio) ────────────────────────────
export abstract class ProductRepository {
  abstract getAll(): Observable<Product[]>;
  abstract getById(id: string): Observable<Product>;
  abstract save(product: Product): Observable<Product>;
}

// ── Implementación HTTP (infraestructura) ─────────
@Injectable()
export class HttpProductRepository extends ProductRepository {
  private http = inject(HttpClient);

  getAll() {
    return this.http.get<ProductDto[]>('/api/products')
      .pipe(map(dtos => dtos.map(toProduct)));
  }
  getById(id: string) {
    return this.http.get<ProductDto>(\`/api/products/${id}\`)
      .pipe(map(toProduct));
  }
  save(product: Product) {
    return this.http.post<ProductDto>('/api/products', toDto(product))
      .pipe(map(toProduct));
  }
}

// ── Wiring por DI: la app elige la implementación ──
providers: [
  { provide: ProductRepository, useClass: HttpProductRepository }
]

// En tests o demos: sin tocar a los consumidores
providers: [
  { provide: ProductRepository, useClass: InMemoryProductRepository }
]

Mejores prácticas

  • El repository devuelve modelos de dominio, nunca DTOs crudos: el mapeo es su responsabilidad.
  • Métodos con semántica de dominio (findActive()) mejor que un genérico query(params) que filtra la UI.
  • Combínalo con facade/Store: el effect o el store llaman al repository, los componentes nunca lo tocan directamente.

⚠️ Errores comunes

  • Usar una interface TS como token de provider: no existe en runtime; usa clase abstracta o InjectionToken.
  • Repository que expone detalles HTTP (HttpResponse, headers, códigos de estado): el contrato debe ser agnóstico del transporte.
  • Crear repositories para una app que solo hará GET a 3 endpoints estables: un servicio API simple es suficiente; el patrón paga cuando hay múltiples fuentes o testing intensivo.

🎤 Preguntas de entrevista

¿Por qué una clase abstracta y no una interface para el contrato?

Las interfaces de TypeScript se borran al compilar, así que no pueden ser tokens de inyección. Una clase abstracta existe en runtime, sirve como token y además obliga a implementar el contrato. La alternativa es InjectionToken<MiInterface>.

¿Qué gana el testing con repositories?

Los tests de lógica de negocio usan una implementación in-memory determinista en lugar de HttpTestingController: menos setup, más rápidos, y los tests no se rompen cuando cambia la forma de la API.

14 · Patrón Adapter

El adapter traduce entre dos contratos incompatibles. En frontend tiene dos usos estrella: (1) anti-corruption layer — traducir respuestas del backend a tus modelos para que sus cambios no se propaguen por la app; y (2) envolver librerías de terceros (charts, analytics, SDKs) detrás de una interfaz propia para poder reemplazarlas. 

// ── 1. Adapter de API (anti-corruption) ───────────
// Lo que manda el backend (snake_case, strings)
interface UserApiResponse {
  user_id: number;
  full_name: string;
  created_at: string;          // "2026-07-10T12:00:00Z"
  subscription_tier: 'FREE' | 'PRO' | 'ENT';
}

// Tu modelo de dominio
interface User {
  id: number;
  name: string;
  createdAt: Date;
  isPremium: boolean;
}

// El adapter: única pieza que conoce ambos mundos
export function toUser(raw: UserApiResponse): User {
  return {
    id: raw.user_id,
    name: raw.full_name,
    createdAt: new Date(raw.created_at),
    isPremium: raw.subscription_tier !== 'FREE'
  };
}

// ── 2. Adapter de librería de terceros ────────────
export abstract class AnalyticsService {
  abstract track(event: string, data?: Record<string, unknown>): void;
}

@Injectable()
export class MixpanelAnalytics extends AnalyticsService {
  track(event: string, data?: Record<string, unknown>) {
    mixpanel.track(event, data);   // API específica encapsulada
  }
}
// Cambiar de proveedor = escribir otro adapter, cero cambios en la app

⚠️ Errores comunes

  • Usar el tipo de la respuesta API directamente en componentes y templates: cuando el backend renombra un campo, el grep recorre toda la app.
  • Adapters con lógica de negocio dentro: el adapter traduce; las reglas viven en el dominio.
  • Envolver librerías estables y triviales "por si acaso": abstraer Date o lodash rara vez paga; abstrae lo que es plausible reemplazar (analytics, pagos, charts).

🎤 Preguntas de entrevista

¿Qué es un anti-corruption layer en frontend?

Una franja de adapters/mappers en el borde de la app que traduce los contratos externos (API, SDKs) a modelos propios. "Corrupción" es que las convenciones ajenas (snake_case, fechas string, enums crípticos) se infiltren en tu dominio y UI.

¿Diferencia entre Adapter y Facade?

El adapter traduce entre dos contratos existentes e incompatibles (misma funcionalidad, distinta forma). El facade simplifica: expone una API reducida sobre un subsistema complejo propio. Suelen convivir: el facade usa repositories que internamente usan adapters.

15 · API Layer: DTOs vs Modelos de Dominio

Un DTO (Data Transfer Object) describe exactamente lo que viaja por la red; un modelo de dominio describe lo que tu app necesita para trabajar. Mantenerlos separados —con un mapper entre ambos— aísla a toda la aplicación de los cambios del backend y concentra en un solo punto las conversiones (fechas, nombres, campos derivados, defaults). 

// ── DTO: espejo del contrato HTTP (solo en la capa API) ──
export interface InvoiceDto {
  id: string;
  customer_name: string;
  issued_at: string;             // ISO string
  line_items: { desc: string; cents: number }[];
  status: 'DRAFT' | 'SENT' | 'PAID' | 'VOID';
}

// ── Modelo de dominio: lo que la app quiere usar ──
export interface Invoice {
  id: string;
  customerName: string;
  issuedAt: Date;
  lines: InvoiceLine[];
  total: number;                 // derivado, calculado una vez
  isPayable: boolean;            // regla de negocio explícita
}

// ── Mapper: único punto de traducción ─────────────
export function toInvoice(dto: InvoiceDto): Invoice {
  const lines = dto.line_items.map(li => ({
    description: li.desc,
    amount: li.cents / 100
  }));
  return {
    id: dto.id,
    customerName: dto.customer_name,
    issuedAt: new Date(dto.issued_at),
    lines,
    total: lines.reduce((s, l) => s + l.amount, 0),
    isPayable: dto.status === 'SENT'
  };
}

// ── Servicio API: los DTOs no salen de aquí ───────
@Injectable({ providedIn: 'root' })
export class InvoiceApiService {
  private http = inject(HttpClient);

  getInvoices(): Observable<Invoice[]> {
    return this.http.get<InvoiceDto[]>('/api/invoices')
      .pipe(map(dtos => dtos.map(toInvoice)));
  }
}

Mejores prácticas

  • Los DTOs viven junto al servicio API y no se exportan fuera de la capa de datos.
  • Mappers como funciones puras: triviales de testear y componibles.
  • En fronteras críticas, valida además en runtime (zod/valibot): el get<T> de HttpClient es solo una promesa de tipos, no una comprobación.
  • Fechas siempre a Date (o a tu tipo de fecha) en el mapper — nunca strings ISO paseándose por templates.

⚠️ Errores comunes

  • Usar el DTO como modelo "para no duplicar interfaces": ahorras 10 líneas hoy y acoplas cada template al naming del backend para siempre.
  • Confiar en el genérico de http.get<T>() como validación: si el backend manda otra cosa, el error explota lejos, en un template.
  • Mapear en el componente: cada consumidor repite (y desincroniza) la conversión.

🎤 Preguntas de entrevista

El backend renombra un campo, ¿cuántos archivos cambias?

Con capa de DTOs + mapper: dos (el DTO y el mapper). Sin ella: cada componente, selector y test que tocaba ese campo. Esa es la métrica que justifica la capa.

¿TypeScript no valida ya la respuesta HTTP?

No. Los tipos se borran en compilación; get<InvoiceDto> es un cast implícito. Para garantías reales en runtime se valida el JSON en la frontera (schemas con zod/valibot) o al menos se defiende el mapper con defaults.

16 · Manejo Global de Errores

Una app profesional tiene una estrategia de errores en dos niveles: un interceptor HTTP para errores de red/API (mapear códigos, reintentos, redirigir en 401, notificar) y un ErrorHandler global como red de seguridad para todo lo no capturado (bugs, errores en efectos). Así los componentes solo manejan los errores que pueden resolver. 

// ── Interceptor funcional: errores HTTP ───────────
export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const router = inject(Router);
  const toast = inject(ToastService);

  return next(req).pipe(
    retry({ count: 1, delay: 300 }),          // reintento leve en GET fallidos
    catchError((error: HttpErrorResponse) => {
      switch (error.status) {
        case 401:
          router.navigate(['/login']);
          break;
        case 403:
          toast.error('No tienes permisos para esta acción');
          break;
        case 0:
          toast.error('Sin conexión con el servidor');
          break;
        default:
          toast.error('Ocurrió un error inesperado');
      }
      // Re-lanza un error de dominio, no el HttpErrorResponse crudo
      return throwError(() => new AppError(error.status, req.url));
    })
  );
};

// ── ErrorHandler global: red de seguridad ─────────
@Injectable()
export class GlobalErrorHandler implements ErrorHandler {
  private logger = inject(LoggingService);

  handleError(error: unknown): void {
    this.logger.report(error);               // Sentry, DataDog...
    console.error(error);                    // nunca tragarse el error
  }
}

// ── Registro en app.config.ts ─────────────────────
export const appConfig: ApplicationConfig = {
  providers: [
    provideBrowserGlobalErrorListeners(),    // captura errores fuera de Angular
    provideHttpClient(withInterceptors([errorInterceptor])),
    { provide: ErrorHandler, useClass: GlobalErrorHandler }
  ]
};

Mejores prácticas

  • Clasifica: errores esperables (validación, 404 de un recurso) se manejan localmente con UI específica; los inesperados suben al handler global.
  • El interceptor traduce HttpErrorResponse a errores de dominio tipados — los componentes nunca inspeccionan status.
  • Reporta a un servicio de observabilidad (Sentry/DataDog) desde el ErrorHandler, con contexto (usuario, ruta, versión).
  • En streams, coloca catchError en el flujo interno para no matar el observable fuente (típico en effects y polling).

⚠️ Errores comunes

  • catchError(() => of([])) silencioso por defecto: el usuario ve una lista vacía y nadie sabe que la API está caída. Silenciar debe ser una decisión explícita, no el patrón por defecto.
  • ErrorHandler que hace navegación o llamadas que pueden fallar sin try/catch: un error dentro del handler produce loops.
  • Mostrar toasts desde interceptor Y componente para el mismo error: define quién es dueño de la notificación (regla común: interceptor notifica lo global, el componente lo contextual, nunca ambos).
  • Olvidar que ErrorHandler corre fuera de la zona en algunos casos: si el handler muta estado de UI, puede necesitar NgZone.run (o mejor, signals).

🎤 Preguntas de entrevista

¿Interceptor o ErrorHandler — cuál captura qué?

El interceptor solo ve fallos de peticiones HTTP y actúa dentro del stream (retry, mapear, redirigir). El ErrorHandler recibe cualquier excepción no capturada en la app (templates, listeners, promesas de Angular). Son complementarios: red de negocio vs red de seguridad.

¿Dónde reintentarías una petición fallida?

En el interceptor para políticas globales (reintento único en errores de red idempotentes), o en el servicio/effect para políticas específicas (retry con backoff exponencial). Nunca reintentar POST no idempotentes a ciegas.

¿Cómo evitas que un error HTTP mate un effect de NgRx?

Poniendo el catchError dentro del observable interno del switchMap/exhaustMap y devolviendo una action de fallo. Si el catchError va en el stream exterior, el effect se completa y deja de escuchar actions.
