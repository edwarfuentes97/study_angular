Introducción

Playwright es un framework moderno de pruebas end-to-end creado por Microsoft. Permite testear aplicaciones web de forma confiable, rápida y cross-browser, ejecutando flujos reales del usuario en Chromium, Firefox y WebKit.

Para aplicaciones Angular esta herramienta es ideal porque automatiza la espera de acciones, maneja múltiples contextos aislados y soporta pruebas en paralelo, todo sin depender de un servidor real en cada ejecución.

¿Por qué elegir Playwright?

  • Compatibilidad con múltiples navegadores: Chromium, Firefox y WebKit.
  • Auto-waiting integrado: espera que los elementos sean interactuables antes de actuar.
  • Control de red y contexto: intercepte requests, use múltiples tabs y orígenes separados.
  • Ejecuta pruebas aisladas: cada test obtiene un contexto fresco en el mismo proceso.
  • Depuración avanzada: inspector, grabador y trace viewer ayudan a diagnosticar fallas.
  • Paralelismo eficiente: ejecuta tests en workers simultáneos y reduce tiempos de ejecución.

Diferencias con Cypress

Cypress corre dentro del navegador, lo que entrega una experiencia de depuración muy fluida. Sin embargo, su sandbox limita escenarios como múltiples pestañas, cross-origin y control externo del navegador.

Playwright corre fuera del navegador y controla su interno, lo que lo hace más flexible para pruebas complejas, aunque a veces la configuración inicial puede ser un poco más detallada.

⚠️ Errores comunes

  • Actualizar el paquete npm sin volver a ejecutar npx playwright install: los binarios quedan desincronizados y los tests fallan con errores de "Executable doesn't exist".
  • Olvidar --with-deps en CI Linux: faltan librerías del sistema y el navegador no arranca.

🎤 Preguntas de entrevista

  • ¿Qué hace npx playwright install --with-deps? Instala los binarios de los navegadores y además las dependencias del sistema operativo (fuentes, librerías gráficas) que necesitan en entornos CI Linux.

playwright.config.ts en profundidad

El archivo de configuración es el contrato de toda la suite. Este repo tiene uno real en la raíz (playwright.config.ts) que ilustra las opciones más importantes:

Opciones clave

  • baseURL: permite escribir page.goto('/lab/playwright') con rutas relativas; cambiar de entorno (local, staging) es cambiar una sola línea o una variable de entorno.
  • projects: cada project es una combinación de navegador + opciones. Sirven para cross-browser, viewports móviles, o para modelar dependencias (proyecto de setup de autenticación).
  • webServer: levanta la app antes de la suite y espera a que responda la URL. reuseExistingServer: !process.env.CI reutiliza tu ng serve local pero garantiza un server limpio en CI.
  • retries: reintenta tests fallidos. Útil en CI (retries: 2) para absorber fallas de infraestructura, pero un test que necesita retries para pasar es un test flaky que hay que arreglar.
  • workers: procesos paralelos. Local usa el default (mitad de los cores); este repo usa workers: 1 en CI porque los runners compartidos tienen poca CPU.
  • trace / screenshot / video: artefactos de diagnóstico. La combinación trace: 'on-first-retry' + screenshot: 'only-on-failure' da evidencia sin penalizar los runs exitosos.

⚠️ Errores comunes

  • Poner reuseExistingServer: true también en CI: puedes terminar testeando contra un server zombie con código viejo.
  • Subir workers en CI sin medir: más workers que cores reales provoca timeouts en cascada.
  • Usar trace: 'on' siempre: los traces pesan y ralentizan; guárdalos solo en retry o failure.
  • Hardcodear URLs absolutas en los tests en vez de confiar en baseURL.

🎤 Preguntas de entrevista

  • ¿Qué diferencia hay entre timeout y expect.timeout? El primero limita la duración total de cada test (30s default); el segundo, cuánto reintenta cada assertion web-first (5s default).
  • ¿Para qué sirven los projects además de cross-browser? Para variar cualquier opción de use (viewport, locale, storageState) y para encadenar dependencias, como un project de setup que autentica antes que el resto.
  • ¿Qué hace forbidOnly? Falla el build en CI si alguien dejó un test.only, evitando que la suite corra incompleta sin que nadie lo note.

Cómo estructurar las pruebas

Organiza tus archivos en carpetas claras. Una convención útil es:

  • tests/shared/ para utilidades y elementos reutilizables.
  • tests/pages/ para page objects del Page Object Model.
  • tests/fixtures/ para extensiones de fixtures personalizadas.
  • tests/*.spec.ts para los casos de prueba.

De esta forma cada prueba queda más legible y el mantenimiento se concentra en los objetos de página.

Escribir tu primer test E2E

Playwright sincroniza automáticamente las acciones, por lo que no necesitas timeouts manuales en la mayoría de los casos.

Jerarquía de locators recomendada

La documentación oficial recomienda un orden claro: getByRole > getByLabel > getByText > getByTestId. La razón: cuanto más se parezca el locator a cómo un usuario (o un lector de pantalla) encuentra el elemento, más resistente será a refactors de CSS y de estructura del DOM.

  • getByRole: usa el árbol de accesibilidad (ARIA). Si un botón cambia de clase, de tag o de posición, el rol y su nombre accesible suelen mantenerse. Además valida gratis que tu UI es semántica.
  • getByLabel: el estándar para formularios. Si falla, probablemente al input le falta un label, lo cual también es un bug de accesibilidad.
  • getByText: válido para contenido visible único; frágil si el copy cambia con frecuencia o está internacionalizado.
  • getByTestId: estable pero opaco: no valida nada de la experiencia real del usuario. Resérvalo para elementos sin semántica (divs decorativos, ítems generados).

⚠️ Errores comunes

  • Selectores CSS acoplados a la estructura: div.container > div:nth-child(3) span se rompe con cualquier refactor.
  • Abusar de .nth(i): el orden de los elementos es un detalle de implementación; prefiere .filter() con texto o con otro locator.
  • Usar XPath por costumbre: casi siempre existe un locator semántico equivalente más legible.
  • Poner data-testid en todo "por si acaso": pierdes la validación semántica que dan los locators por rol.

🎤 Preguntas de entrevista

  • ¿Por qué getByRole es preferible a un selector CSS? Porque se basa en el árbol de accesibilidad, no en la implementación: sobrevive a cambios de estilos y verifica de paso la semántica ARIA.
  • ¿Qué es un locator en Playwright? Una descripción lazy de cómo encontrar un elemento; no busca en el DOM hasta que ejecutas una acción o assertion, y en ese momento re-resuelve el elemento (evita stale references).
  • ¿Qué es el strict mode de locators? Si un locator usado en una acción resuelve más de un elemento, Playwright lanza error en vez de actuar sobre el primero; te obliga a desambiguar.

Auto-waiting y web-first assertions

Antes de cada acción, Playwright ejecuta actionability checks: espera a que el elemento esté adjunto al DOM, visible, estable (sin animarse), habilitado y que reciba los eventos (no tapado por otro elemento). Por eso casi nunca necesitas esperas manuales.

Las web-first assertions (await expect(locator)...) complementan esto: no leen el estado una vez, sino que hacen polling reintentando hasta que la condición se cumple o se agota el timeout. Son la herramienta central contra el flakiness en apps Angular, donde el render es asíncrono.

Mejores prácticas

  • Prefiere toBeVisible, toHaveText, toHaveURL, toHaveCount sobre extraer valores con textContent() y comparar a mano.
  • Después de una navegación SPA, asserta sobre la URL o sobre contenido nuevo; no uses waitForLoadState('networkidle') como comodín (está desaconsejado).
  • Usa expect.poll o expect.toPass para condiciones que no son locators (respuestas de API, estado de un store).

⚠️ Errores comunes

  • Olvidar el await en la assertion: expect(locator).toBeVisible() sin await no espera nada y el test sigue de largo.
  • Mezclar expect(await locator.isVisible()).toBe(true): convierte una assertion con polling en una lectura única y flaky.
  • Añadir waitForTimeout "para estar seguros": suma minutos a la suite y esconde la condición real que había que esperar.

🎤 Preguntas de entrevista

  • ¿Qué checks hace Playwright antes de un click? Visible, estable, adjunto al DOM, habilitado y que el punto de click no esté interceptado por otro elemento; reintenta hasta el timeout.
  • ¿Diferencia entre una assertion web-first y una de Jest? La web-first recibe un locator y reintenta con polling; la clásica evalúa un valor ya resuelto una sola vez.
  • ¿Cuándo sí usarías una espera explícita? Para eventos no ligados a un elemento: waitForURL tras navegación, waitForResponse para una llamada de red concreta.

Page Object Model

El patrón Page Object Model (POM) abstrae selectores y flujos de la aplicación en clases, dejando los tests enfocados en el comportamiento y no en la implementación de la UI.

Un Page Object representa una página o una parte relevante de la interfaz, y expone métodos de alto nivel que describen acciones de usuario, en lugar de exponer selectores directamente.

¿Por qué usar POM?

  • Reduce la duplicación de selectores y flujos.
  • Hace los tests más legibles y fáciles de entender.
  • Protege los tests de cambios frecuentes en la UI.
  • Permite centralizar el mantenimiento de locators en un solo lugar.

Cuándo usarlo

  • Cuando una página tiene múltiples interacciones recurrentes.
  • Cuando quieres encapsular validaciones comunes.
  • Cuando tu suite crece y necesitas mantener tests escalables.

Estructura recomendada

Una convención típica es usar carpetas como:

tests/pages/dashboard.page.ts
tests/pages/login.page.ts
tests/pages/product.page.ts

También puedes usar un archivo tests/pages/base.page.ts para compartir utilidades comunes.

Buenas prácticas

  • Guarda los locators como propiedades privadas o readonly.
  • Expón solo métodos de alto nivel que representen comportamientos.
  • No pongas assertions complejas dentro del Page Object; deja las expectativas en el test.
  • Evita lógica de negocio en los Page Objects; solo encapsula interacción y consultas de UI.
  • Si una pantalla crece, divide la clase en page components (subobjetos) para mantenerla manejable.

Cuándo NO usarlo

  • Suites pequeñas (menos de ~10 tests sobre pocas pantallas): la capa extra solo agrega indirección.
  • Cuando el Page Object se convierte en una "god class" con decenas de métodos: divide o reemplaza por helpers puntuales.
  • Como sustituto de fixtures: el POM organiza páginas; el entorno (instancias, datos, auth) se organiza mejor con fixtures. En este repo ambos conviven: los page objects viven en tests/pages/ y se inyectan vía fixtures desde tests/fixtures/app-fixtures.ts.

Page Object real de este repo

⚠️ Errores comunes

  • Meter expect de negocio dentro del Page Object: oculta qué verifica cada test y acopla la clase al framework de assertions.
  • Exponer locators públicos crudos por todas partes: los tests vuelven a acoplarse a la UI y el patrón pierde sentido.
  • Devolver ElementHandle en lugar de Locator: los handles se vuelven stale; los locators se re-resuelven solos.
  • Crear jerarquías de herencia profundas entre pages: prefiere composición (page components).

🎤 Preguntas de entrevista

  • ¿Qué problema resuelve el POM? Centraliza selectores y flujos de UI: un cambio de interfaz se arregla en un solo archivo en vez de en cada test.
  • ¿Deben ir las assertions en el Page Object? Regla práctica: assertions de negocio en el test; el page object puede exponer estados o assertions genéricas de "página cargada".
  • ¿Qué alternativa moderna existe al POM clásico? Fixtures de Playwright que inyectan pages o helpers ya inicializados; combinan bien: la fixture crea, el POM encapsula.

  • ¿Cómo comparte una fixture datos con el test? Mediante el callback use(valor): lo previo es setup, el argumento es lo inyectado y lo posterior es teardown.

Fixtures: scope de test vs worker

Cada fixture tiene un scope. Con scope: 'test' (default) se crea y destruye por cada test: máximo aislamiento. Con scope: 'worker' se crea una sola vez por proceso worker y se comparte entre todos los tests que ese worker ejecute: ideal para setups caros (tokens, contenedores, seeds de base de datos).

Override de fixtures

También puedes sobrescribir fixtures integradas o propias para cambiar el comportamiento de toda una suite:

⚠️ Errores comunes

  • Guardar estado mutable en una fixture worker-scoped: un test lo modifica y contamina a los siguientes del mismo worker.
  • Hacer que una fixture worker-scoped dependa de una test-scoped: es imposible (viven distinto tiempo) y Playwright lo rechaza.
  • Usar worker scope "para ir más rápido" en cosas baratas: sacrificas aislamiento sin ganancia real.

🎤 Preguntas de entrevista

  • ¿Cuándo eliges worker scope? Cuando el setup es costoso e inmutable durante los tests: autenticarse contra una API, levantar un mock server, poblar datos base.
  • ¿Qué significa auto: true en una fixture? Que se inicializa en todos los tests aunque ninguno la pida en sus argumentos; útil para hooks transversales (logging, documentación).

Autenticación persistente

Para evitar logins redundantes en cada prueba, guarda el estado de autenticación en un archivo. Un proyecto de setup puede iniciar sesión y escribir storageState.

Luego el config de Playwright puede reutilizar ese archivo para pruebas autenticadas.

Patrón recomendado: setup project + storageState

storageState serializa cookies y localStorage del contexto. El patrón oficial es un setup project que hace login una vez y guarda el archivo; los demás projects lo declaran como dependencies y arrancan cada test ya autenticados:

Mejores prácticas

  • Agrega playwright/.auth/ a .gitignore: contiene credenciales de sesión.
  • Antes de guardar el estado, asserta algo que pruebe que el login terminó (evita guardar un estado a medias).
  • Para tests que necesitan varios roles, genera un archivo por rol (admin.json, user.json) y selecciónalo con test.use.
  • Si los tests mutan la sesión (logout, cambio de contraseña), aíslalos con un login propio.

⚠️ Errores comunes

  • Hacer login por UI en cada test: multiplica minutos de suite y añade un punto de flakiness por test.
  • Reusar un storageState expirado: si el token caduca, media suite falla con síntomas confusos; regenera el archivo en el setup de cada run de CI.
  • Guardar el estado sin esperar la redirección post-login: el archivo queda sin las cookies finales.

🎤 Preguntas de entrevista

  • ¿Qué guarda exactamente storageState? Cookies del contexto y localStorage por origen (no sessionStorage, que requiere un truco aparte).
  • ¿Cómo autenticas con distintos roles en la misma suite? Un setup project que genera un storageState por rol y test.use({ storageState: '...' }) por describe o archivo.

Mock de red con page.route

page.route(pattern, handler) intercepta las requests que hace la página. Con route.fulfill() respondes tú mismo (mock), con route.abort() cancelas la request y con route.continue() / route.fetch() la dejas pasar (opcionalmente modificada). Es la herramienta clave para probar estados difíciles de reproducir: errores 500, listas vacías, respuestas lentas o payloads enormes.

Esperar respuestas reales

Cuando no quieres mockear sino sincronizarte con la red real, usa waitForResponse registrando la espera antes de disparar la acción:

Mejores prácticas

  • Decide una estrategia por suite: E2E "full stack" contra backend real para flujos críticos, y mocks para estados excepcionales y edge cases.
  • Registra las rutas antes de page.goto(); si la app dispara la request al cargar, un route tardío no la intercepta.
  • Mantén los payloads mock en archivos/factories compartidos para no duplicar JSON en cada spec.
  • Usa page.unroute() o el helper del contexto si necesitas restaurar el comportamiento.

⚠️ Errores comunes

  • Patrones glob que no coinciden (olvidar **/ al inicio): el mock nunca aplica y el test pega al backend real.
  • Mockear TODO: terminas testeando tus mocks y no la integración; deja al menos un smoke test end-to-end real.
  • Poner waitForResponse después del click: si la respuesta llega antes de registrar la espera, el test se cuelga.
  • Olvidar contentType: 'application/json' en el fulfill: algunos clients HTTP no parsean el body.

🎤 Preguntas de entrevista

  • ¿Diferencia entre route.fulfill, route.continue y route.abort? fulfill responde sin tocar el servidor; continue deja pasar la request (con posibles modificaciones de headers/postData); abort la falla como error de red.
  • ¿Cómo probarías el estado de error de una pantalla? Interceptando su endpoint con route.fulfill({ status: 500 }) y assertando el mensaje de error visible.
  • ¿Qué es HAR replay? Grabar el tráfico real en un archivo HAR (routeFromHAR) y reproducirlo en los tests para tener red determinista sin escribir mocks a mano.

Tracing, Trace Viewer y UI Mode

El trace es la caja negra de un test: graba screenshots por acción, snapshots del DOM (antes/durante/después), requests de red, logs de consola y el código fuente ejecutado. Con trace: 'on-first-retry' (la opción de este repo) obtienes esa evidencia exactamente cuando un test falla en CI, sin costo en los runs verdes.

¿Qué herramienta usar para cada problema?

  • UI Mode (--ui): desarrollo diario; corre tests en watch, permite "time travel" por cada acción y probar locators en vivo.
  • Inspector (--debug): avanzar paso a paso una prueba concreta con el navegador visible.
  • Trace Viewer: diagnóstico post-mortem de fallas en CI; descarga el trace del artifact y ábrelo con show-trace (o en trace.playwright.dev).
  • Codegen: punto de partida para descubrir locators; el código generado se refactoriza después (POM, fixtures), no se pega tal cual.

⚠️ Errores comunes

  • Depurar fallas de CI a base de console.log y re-runs: un trace responde en segundos qué pasó.
  • Dejar trace: 'on' permanente: artefactos gigantes y suite más lenta.
  • Commitear directamente el output de codegen: suele usar el contexto equivocado y carece de assertions.

🎤 Preguntas de entrevista

  • ¿Cómo depuras un test que solo falla en CI? Con trace: 'on-first-retry', subiendo test-results/ como artifact y abriendo el trace en el Trace Viewer: incluye DOM, red y consola del momento exacto de la falla.
  • ¿Qué contiene un trace además de screenshots? Snapshots interactivos del DOM por acción, timeline, network, consola, código del test y metadatos del run.

Paralelización

Playwright ejecuta archivos de prueba en paralelo por defecto, usando múltiples workers. Dentro de cada archivo los tests se ejecutan de forma secuencial a menos que actives:

Para desactivar el paralelismo usa --workers=1 o workers: 1 en la configuración. Este repo usa fullyParallel: true en local y workers: 1 en CI (runners con poca CPU).

Paralelismo y sharding en CI

El paralelismo con workers escala dentro de una máquina; el sharding escala entre máquinas: divide la suite en N porciones y cada job de CI ejecuta una con --shard=i/N. Combinado con una matrix de GitHub Actions, una suite de 40 minutos puede bajar a 10.

Mejores prácticas

  • El requisito número uno es que los tests sean independientes: sin orden implícito ni datos compartidos.
  • Usa fail-fast: false en la matrix para que un shard fallido no cancele los demás y veas el panorama completo.
  • Con sharding usa el blob reporter y merge-reports para un único reporte HTML final.
  • Ajusta retries en CI (aquí: 2) para absorber fallas de infraestructura, y monitorea qué tests las consumen.

⚠️ Errores comunes

  • Tests que comparten un mismo usuario/registro de datos: en paralelo se pisan entre sí de forma intermitente.
  • Usar test.describe.configure({ mode: 'serial' }) para encadenar tests dependientes: si uno falla, arrastra a los demás y bloquea el sharding eficiente.
  • Asumir que los shards tardan lo mismo: distribuye archivos pesados o usa más shards pequeños.

🎤 Preguntas de entrevista

  • ¿Diferencia entre workers y shards? Workers son procesos paralelos en una máquina; shards son particiones de la suite entre varias máquinas. Se combinan.
  • ¿Cómo garantiza Playwright el aislamiento entre tests paralelos? Cada test recibe un BrowserContext nuevo (cookies, storage y cache limpios) dentro de un mismo browser por worker, lo cual es barato de crear.

Tags y grep: @smoke y @regression

Los tags permiten ejecutar subconjuntos de la suite según el contexto: un set @smoke pequeño y rápido que corre en cada push, y un set @regression amplio para pipelines nocturnos o pre-release. Se filtran con --grep y --grep-invert.

Mejores prácticas

  • Mantén @smoke corto (minutos, no decenas de minutos): son los flujos que, si fallan, bloquean todo lo demás.
  • Define un vocabulario de tags acordado por el equipo (@smoke, @regression, @slow, @a11y) y documéntalo.
  • Expón los filtros como scripts npm, como hace este repo (test:e2e:smoke, test:e2e:regression), para que CI y humanos usen los mismos comandos.

⚠️ Errores comunes

  • Escribir el tag con typo (@somke): el grep simplemente no lo encuentra y el test deja de correr sin aviso. Revisa el conteo de tests ejecutados.
  • Meter todo en @smoke "porque es importante": el set pierde su valor de feedback rápido.
  • Usar grep sobre títulos libres en vez de tags consistentes: los filtros se vuelven frágiles.

🎤 Preguntas de entrevista

  • ¿Cómo estructurarías la ejecución de E2E en un pipeline? Smoke en cada PR (rápido, bloqueante), regression completa nightly o pre-release, con sharding si crece.
  • ¿Cómo excluyes tests lentos de un run? Tagéalos (@slow) y usa --grep-invert @slow, o sepáralos en un project propio.

CI/CD

Las pruebas Playwright pueden ejecutarse en cualquier proveedor de CI. Muchas integraciones usan el mismo comando de terminal y agregan la instalación de navegadores.

Un workflow típico de GitHub Actions solo necesita ejecutar npm install y npx playwright test.

Extensión de VS Code

La extensión Playwright Test para VS Code ofrece un panel de tests, grabador de locators y depuración desde el editor. También puede generar workflows de GitHub Actions y ayuda a instalar navegadores.

Métodos Principales de Localización

page.goto(url)

Navega a una URL específica y espera automáticamente a que la página cargue.

await page.goto('/lab/ngrx');

Cuándo usar: Para iniciar navegación a páginas específicas.

page.locator(selector)

Localiza elementos usando selectores CSS, XPath o texto.

Cuándo usar: Para elementos específicos con selectores CSS o texto visible.

page.getByRole(role, options)

Localiza elementos por su rol semántico (ARIA).

Roles comunes: button, link, heading, textbox, checkbox, radio, etc.

Cuándo usar: Para elementos interactivos o semánticos. Mejor que selectores CSS porque es más resistente a cambios de estilo.

page.getByText(text, options)

Localiza elementos por su texto visible.

Cuándo usar: Cuando el texto es único y visible al usuario.

page.getByLabel(text)

Localiza elementos por su etiqueta asociada (label).

page.getByLabel('Nombre del producto');

Cuándo usar: Para inputs con labels explícitos.

page.getByPlaceholder(text)

Localiza inputs por su placeholder.

page.getByPlaceholder('Buscar...');

Cuándo usar: Para inputs con placeholders únicos.

page.getByTestId(id)

Localiza elementos por data-testid.

page.getByTestId('submit-button');

Cuándo usar: Para elementos sin texto o rol claro, cuando se necesita estabilidad.

page.waitForURL(url, options)

Espera a que la URL cambie a un patrón específico.

await page.waitForURL(/.*\/lab\/ngrx/);
await page.waitForURL('https://example.com/dashboard');

Cuándo usar: Después de clics que cambian la ruta, para asegurar navegación completa.

page.waitForLoadState(state)

Espera a que la página alcance un estado de carga.

await page.waitForLoadState('networkidle');

Estados: 'load' (default), 'domcontentloaded', 'networkidle'

Cuándo usar: Cuando se necesita esperar a que todos los recursos carguen.

Acciones en Elementos

locator.click()

Hace clic en el elemento y espera automáticamente a que sea clickable.

await button.click();

locator.fill(text)

Llena un input con texto (borra contenido previo).

await input.fill('nuevo valor');

locator.type(text, options)

Escribe texto carácter por carácter (simula usuario real).

locator.check()

Marca un checkbox.

await checkbox.check();

locator.selectOption(value)

Selecciona una opción en un select.

await select.selectOption('option1');

locator.hover()

Mueve el mouse sobre el elemento.

await element.hover();

locator.focus()

Enfoca el elemento.

await input.focus();

Assertions (Verificaciones)

expect(locator).toBeVisible()

Verifica que el elemento sea visible.

await expect(page.locator('h2')).toBeVisible();

expect(locator).toHaveText(text)

Verifica el texto exacto del elemento.

await expect(button).toHaveText('Guardar');

expect(locator).toContainText(text)

Verifica que el elemento contenga el texto.

await expect(div).toContainText('Error');

expect(page).toHaveURL(url)

Verifica la URL actual.

await expect(page).toHaveURL(/.*\/dashboard/);

expect(locator).toHaveCount(count)

Verifica el número de elementos encontrados.

await expect(cards).toHaveCount(10);

expect(locator).toBeChecked()

Verifica que un checkbox esté marcado.

await expect(checkbox).toBeChecked();

expect(locator).toBeDisabled()

Verifica que un elemento esté deshabilitado.

await expect(button).toBeDisabled();

Atributos de Datos en HTML

Para hacer tests más robustos, agrega atributos data-* en el HTML:

data-testid

Para elementos sin texto único o rol claro.

<button data-testid="submit-btn">Enviar</button>

Uso en test: page.getByTestId('submit-btn')

data-cy (estilo Cypress)

Similar a data-testid, popular en algunos equipos.

<input data-cy="username" />

data-inputid

Para inputs específicos.

<input data-inputid="product-name" />

data-role

Para roles personalizados.

<div data-role="notification">...</div>

data-state

Para estados dinámicos.

<button data-state="loading">Cargando...</button>

Problemas comunes

  • Elemento no encontrado: Verifica selectores y espera a que cargue.
  • Timeout: Aumenta timeout o usa esperas más específicas.
  • Flaky tests: Evita dependencias entre tests, usa selectores robustos.
  • Cross-browser: Prueba en todos los navegadores soportados.

Resumen

Playwright convierte las pruebas end-to-end en una experiencia moderna y escalable para Angular. Su fortaleza está en la mezcla de confiabilidad, compatibilidad y herramientas de depuración.
