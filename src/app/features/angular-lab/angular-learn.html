1. Change Detection

¿Cómo funciona?

Angular usa Zone.js para interceptar todos los eventos asíncronos (clicks, setTimeout, HTTP requests, Promises) y dispara un ciclo de Change Detection (CD) desde la raíz del árbol de componentes hacia abajo. En cada ciclo, Angular compara los valores actuales del template con los anteriores y actualiza el DOM si hay diferencias.

Zone.js por dentro

Zone.js hace monkey-patching de las APIs asíncronas del navegador (addEventListener, setTimeout, Promise.then, XMLHttpRequest/fetch). Angular ejecuta la app dentro de la NgZone: cuando la última tarea asíncrona termina, la zona emite onMicrotaskEmpty y ApplicationRef.tick() inicia un ciclo de CD desde la raíz. Por eso "todo funciona mágicamente" sin avisarle a Angular… y por eso también se revisa de más. 

// Simplificación conceptual de lo que hace Zone.js
const originalSetTimeout = window.setTimeout;
window.setTimeout = (fn, delay) => originalSetTimeout(() => {
  fn();                          // 1. ejecuta tu callback
  ngZone.notifyTaskDone();       // 2. avisa a Angular
  // 3. onMicrotaskEmpty → ApplicationRef.tick() → CD en TODO el árbol
}, delay);

// Escapar de la zona: trabajo intensivo que NO debe disparar CD
export class CanvasComponent {
  private ngZone = inject(NgZone);

  startAnimation() {
    this.ngZone.runOutsideAngular(() => {
      setInterval(() => this.drawFrame(), 16); // 60fps sin ciclos de CD
    });
  }

  // Volver a entrar cuando sí necesitas actualizar la UI
  finish() {
    this.ngZone.run(() => this.status = 'done');
  }
}

El árbol de Change Detection

CD recorre el árbol de componentes top-down (de la raíz hacia las hojas), en una sola pasada y de forma unidireccional: un padre se revisa siempre antes que sus hijos. Con Default se visita todo el árbol; con OnPush las ramas "limpias" se saltan completas (incluidos sus descendientes). Por eso markForCheck() marca también a los ancestros: si el camino desde la raíz no está marcado, el ciclo nunca llegaría hasta el componente. En dev mode Angular ejecuta una segunda pasada de verificación — el origen del famoso ExpressionChangedAfterItHasBeenCheckedError. 

AppComponent (Default)            ← tick() siempre empieza aquí
├── HeaderComponent (OnPush)      ← limpio → se SALTA (y todo lo que cuelga de él)
├── ListComponent (OnPush) ✔      ← @Input cambió por referencia → se revisa
│   └── ItemComponent (OnPush)    ← se evalúan sus bindings porque el padre lo revisa
└── FooterComponent (Default)     ← se revisa en cada ciclo, cambie o no

Default Strategy

Revisa TODOS los componentes en cada ciclo de CD, sin importar si cambiaron o no. Simple pero costoso en apps grandes — un solo click puede causar que cientos de componentes se revisen innecesariamente. 

// Default — se revisa en CADA ciclo de CD, incluso si nada cambió
@Component({
  selector: 'app-default',
  template: \`<p>{{ heavyComputation() }}</p>\`
  // No se especifica changeDetection → usa Default
})
export class DefaultComponent {
  heavyComputation() {
    console.log('Se ejecuta en CADA ciclo de CD');
    return this.items.filter(i => i.active).length;
  }
}

OnPush Strategy

Solo revisa el componente cuando ocurre una de estas condiciones:

  • (a) Un @Input() cambia por referencia (no mutación)
  • (b) Un evento DOM se dispara en el componente o en un hijo
  • (c) Un async pipe emite un nuevo valor
  • (d) Un Signal leído en el template cambia
  • (e) Se llama markForCheck() manualmente
@Component({
  selector: 'app-onpush',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: \`
    <!-- (a) Input por referencia -->
    <p>{{ user.name }}</p>

    <!-- (b) Evento DOM -->
    <button (click)="increment()">+1</button>

    <!-- (c) Async pipe -->
    <p>{{ data$ | async }}</p>

    <!-- (d) Signal -->
    <p>{{ count() }}</p>
  \`
})
export class OnPushComponent {
  @Input() user!: { name: string };
  count = signal(0);
  data$ = inject(DataService).getData();

  increment() { this.count.update(v => v + 1); }
}

ChangeDetectorRef

Permite controlar manualmente el ciclo de CD: 

export class ManualCDComponent {
  private cdr = inject(ChangeDetectorRef);

  // Fuerza una revisión inmediata de este componente y sus hijos
  forceCheck() {
    this.cdr.detectChanges();
  }

  // Marca el componente como "dirty" — se revisará en el próximo ciclo
  markDirty() {
    this.cdr.markForCheck();
  }

  // Desconecta el componente del árbol de CD — nunca se revisa automáticamente
  detachFromCD() {
    this.cdr.detach();
  }

  // Reconecta al árbol de CD
  reattach() {
    this.cdr.reattach();
  }
}

markForCheck() vs detectChanges()

Aspecto markForCheck() detectChanges()
Qué hace Marca el componente y sus ancestros como dirty Ejecuta CD síncronamente en el componente y sus hijos
Cuándo corre la CD En el próximo ciclo (se agrupa/coalesce con otros cambios) Inmediatamente, fuera del ciclo normal
Riesgos Prácticamente ninguno ExpressionChanged...Error, ciclos infinitos, coste si se abusa
Uso típico OnPush + datos que llegan fuera de Angular (WebSocket, callbacks de libs de terceros) Componentes con detach(), forzar render en tests

Regla práctica: el 95% de las veces quieres markForCheck() (o mejor: un Signal, que lo hace por ti). detectChanges() es un martillo para casos muy puntuales.

Zoneless con provideZonelessChangeDetection()

Angular ya soporta un modelo sin Zone.js: en vez de interceptar todas las APIs asíncronas, la CD se programa solo cuando Angular recibe una notificación explícita. Disparan CD en modo zoneless: escribir un Signal leído en un template, markForCheck() (el async pipe lo llama internamente), eventos de template ((click), etc.) y crear/destruir vistas dinámicas. Beneficios: menos overhead, sin polyfill de Zone.js en el bundle, stack traces limpios y mejor interop con librerías externas. 

// app.config.ts — activar Zoneless (estable desde Angular 20;
// en v18/19 se llamaba provideExperimentalZonelessChangeDetection)
export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    // Quita 'zone.js' del array "polyfills" en angular.json
  ]
};

// Con Zoneless, esto NO actualiza la UI (nadie notifica a Angular):
setTimeout(() => { this.value = 'nuevo'; }, 1000);   // ❌

// Esto SÍ funciona:
setTimeout(() => this.valueSignal.set('nuevo'), 1000); // ✅ Signal
this.cdr.markForCheck();                                // ✅ notificación explícita
data$ | async                                           // ✅ usa markForCheck por dentro

// Checklist de migración a zoneless:
// 1. Todos los componentes OnPush (o basados en signals)
// 2. Estado en Signals / async pipe — nada de mutar campos "a mano"
// 3. Nada que dependa de NgZone.onStable / tick implícito

🧠 Diagrama mental

Default = "revísalo TODO siempre" — fácil pero lento.
OnPush = "no me molestes a menos que algo cambie realmente" — rápido pero requiere inmutabilidad.
Zoneless = "solo los Signals mandan" — el futuro de Angular.

⚠️ Errores comunes

  • Mutar objeto con OnPush: this.user.name = 'Nuevo' NO dispara CD porque la referencia no cambió. Solución: this.user = { ...this.user, name: 'Nuevo' }
  • ExpressionChangedAfterItHasBeenCheckedError: Ocurre cuando un valor del template cambia durante el mismo ciclo de CD (por ejemplo, en ngAfterViewInit). Soluciones modernas: mover el estado a un Signal, usar afterNextRender(), o como último recurso diferir con setTimeout/queueMicrotask.
  • Llamar funciones en el template: {{ total() }} (método normal, no signal) se re-ejecuta en cada ciclo de CD. Solución: computed(), pipes puros o precalcular el valor.
  • Suscribirse y mutar un campo con OnPush: el callback de subscribe() corre dentro de la zona, pero el componente OnPush no está dirty → la vista no se actualiza. Solución: async pipe, toSignal() o markForCheck().
  • Poner OnPush "a medias": si el padre es OnPush y se salta, los hijos Default que cuelgan de él tampoco se revisan. La estrategia debe pensarse por ramas, no por componentes sueltos.

✅ Mejores prácticas

  • Usa OnPush por defecto en componentes nuevos (configúralo en los schematics de angular.json).
  • Trata los @Input() como inmutables: nuevas referencias, nunca mutación.
  • Prefiere Signals o async pipe antes que ChangeDetectorRef manual.
  • Trabajo intensivo que no toca la UI → NgZone.runOutsideAngular().
  • Diseña hoy pensando en zoneless: OnPush + Signals hace la migración trivial.

🎤 Preguntas de entrevista

  • ¿Qué es Zone.js y cómo lo usa Angular? Una librería que hace monkey-patching de las APIs asíncronas del navegador. Angular ejecuta la app dentro de NgZone y dispara ApplicationRef.tick() cuando la zona queda vacía de tareas, revisando todo el árbol de componentes.
  • ¿Qué dispara CD en un componente OnPush? Cambio de referencia en un @Input, un evento de template en el componente o sus hijos, una emisión del async pipe, un Signal leído en el template que cambia, o markForCheck().
  • ¿Diferencia entre markForCheck() y detectChanges()? markForCheck() solo marca dirty hacia arriba y espera al próximo ciclo; detectChanges() ejecuta CD síncrona inmediata sobre el componente y sus hijos.
  • ¿Cómo prepararías una app para zoneless? Migrar todo a OnPush, mover el estado a Signals (o async pipe), eliminar dependencias implícitas de Zone.js (mutaciones en setTimeout, onStable) y activar provideZonelessChangeDetection().
  • ¿Por qué OnPush exige inmutabilidad? Porque la comprobación del input es por referencia (===): si mutas el objeto, la referencia no cambia y Angular asume que no hay nada nuevo que renderizar.

2. Signals

2.1 ¿Qué son los Signals?

Los Signals son una primitiva reactiva síncrona introducida en Angular 16 que permite rastrear estado de forma granular. A diferencia de RxJS (que es asíncrono y basado en push/subscribe), los Signals son síncronos y basados en pull: lees el valor cuando lo necesitas llamándolo como función.

Angular los usa internamente para habilitar Change Detection granular y eventualmente eliminar la dependencia en Zone.js (Zoneless Angular). Cuando un Signal cambia, Angular sabe exactamente qué componentes necesitan actualizarse.

2.2 signal() — Estado mutable

signal() crea un valor reactivo mutable. Se lee llamándolo como función, se escribe con .set() o .update(). 

import { signal } from '@angular/core';

// Crear un signal con valor inicial
const count = signal(0);

// Leer el valor (como función)
console.log(count()); // 0

// Escribir un nuevo valor
count.set(5);
console.log(count()); // 5

// Actualizar basado en el valor anterior
count.update(v => v + 1);
console.log(count()); // 6

// Con objetos (tipado explícito)
interface User {
  name: string;
  age: number;
}
const user = signal<User>({ name: 'Ed', age: 30 });

// ⚠️ mutate() fue removido en Angular 17
// Ahora se usa update() con spread para objetos:
user.update(u => ({ ...u, age: 31 }));

// Equality function — controla cuándo se notifica a los consumidores
const position = signal(
  { x: 0, y: 0 },
  { equal: (a, b) => a.x === b.x && a.y === b.y }
);
// Si set() produce un valor "equal", NO se notifica

2.3 computed() — Derivaciones

computed() crea un Signal de solo lectura derivado de otros Signals. Se recalcula solo cuando sus dependencias cambian (memoizado automáticamente). Es lazy: no se calcula hasta que alguien lo lee. Y es glitch-free: siempre muestra valores consistentes, nunca intermedios. 

import { signal, computed } from '@angular/core';

const firstName = signal('Eduardo');
const lastName = signal('García');

// computed — se recalcula solo cuando firstName o lastName cambian
const fullName = computed(() => firstName() + ' ' + lastName());
console.log(fullName()); // 'Eduardo García'

firstName.set('Carlos');
console.log(fullName()); // 'Carlos García' — se recalculó automáticamente

// Ejemplo complejo: filtrado + ordenamiento + paginación
const items = signal<Product[]>([]);
const searchTerm = signal('');
const sortBy = signal<'name' | 'price'>('name');
const page = signal(1);
const pageSize = 10;

const filteredItems = computed(() =>
  items().filter(item =>
    item.name.toLowerCase().includes(searchTerm().toLowerCase())
  )
);

const sortedItems = computed(() => {
  const field = sortBy();
  return [...filteredItems()].sort((a, b) =>
    a[field] > b[field] ? 1 : -1
  );
});

const paginatedItems = computed(() => {
  const start = (page() - 1) * pageSize;
  return sortedItems().slice(start, start + pageSize);
});

const totalPages = computed(() =>
  Math.ceil(filteredItems().length / pageSize)
);

2.4 effect() — Side effects

effect() ejecuta código cuando los Signals que lee cambian. NO se debe usar para cambiar otros Signals (anti-pattern). Casos de uso: logging, localStorage, analytics, manipulación manual del DOM.

Injection context: se debe crear en el constructor, como campo de clase, o usando runInInjectionContext(). 

import { signal, effect, inject, Injector, runInInjectionContext } from '@angular/core';

// Ejemplo: persistir en localStorage
const count = signal(0);
effect(() => {
  localStorage.setItem('count', count().toString());
  console.log('Count guardado:', count());
});

// Cleanup function — se ejecuta antes de cada re-ejecución y al destruirse
effect((onCleanup) => {
  const id = setInterval(() => console.log('tick', count()), 1000);
  onCleanup(() => {
    clearInterval(id);
    console.log('Interval limpiado');
  });
});

// Fuera del injection context — usar runInInjectionContext
export class MyComponent {
  private injector = inject(Injector);
  count = signal(0);

  ngOnInit() {
    // ❌ effect(() => ...); // Error: no injection context
    // ✅ Correcto:
    runInInjectionContext(this.injector, () => {
      effect(() => console.log('Count:', this.count()));
    });
  }
}

2.5 linkedSignal() — Angular 19+

linkedSignal() crea un Signal que se resetea automáticamente cuando su fuente cambia, pero también se puede escribir manualmente. Ideal para formularios dependientes y selecciones que dependen de un contexto. 

import { signal, linkedSignal } from '@angular/core';

const items = signal(['Angular', 'React', 'Vue']);

// selected se resetea al primer item cuando items cambia
const selected = linkedSignal(() => items()[0]);
console.log(selected()); // 'Angular'

// Se puede escribir manualmente
selected.set('Vue');
console.log(selected()); // 'Vue'

// Cuando items cambia, selected se resetea
items.set(['Svelte', 'Solid', 'Qwik']);
console.log(selected()); // 'Svelte' — se reseteó automáticamente

// Caso de uso real: país → ciudad
const countries = signal(['México', 'España', 'Argentina']);
const selectedCountry = signal('México');
const cities = computed(() => {
  const cityMap: Record<string, string[]> = {
    'México': ['CDMX', 'Guadalajara', 'Monterrey'],
    'España': ['Madrid', 'Barcelona', 'Valencia'],
    'Argentina': ['Buenos Aires', 'Córdoba', 'Rosario'],
  };
  return cityMap[selectedCountry()] ?? [];
});
// Cuando el país cambia, la ciudad seleccionada se resetea a la primera
const selectedCity = linkedSignal(() => cities()[0]);

2.6 resource() — Angular 19+

resource() combina datos asíncronos con Signals, proporcionando .value(), .status() y .error() automáticamente. Status puede ser: idle, loading, resolved, error. 

import { resource, signal } from '@angular/core';

// resource básico
const products = resource({
  loader: () => fetch('/api/products').then(r => r.json())
});

// Usar en template o código:
products.value();   // Product[] | undefined
products.status();  // 'idle' | 'loading' | 'resolved' | 'error'
products.error();   // error object o undefined

// resource con dependencias — se recarga cuando selectedId cambia
const selectedId = signal('abc-123');
const productDetail = resource({
  request: () => ({ id: selectedId() }),
  loader: ({ request }) =>
    fetch(\`/api/products/${request.id}\`).then(r => r.json())
});

// Cambiar el ID recarga automáticamente
selectedId.set('xyz-456'); // productDetail se recarga

2.7 Interop con RxJS

Angular provee funciones para convertir entre Signals y Observables. toSignal() se auto-desuscribe al destruirse el componente. 

import { toSignal, toObservable } from '@angular/core/rxjs-interop';
import { signal } from '@angular/core';

// Observable → Signal
const products$ = this.http.get<Product[]>('/api/products');
const products = toSignal(products$, { initialValue: [] });
// products() devuelve Product[] — síncrono, sin async pipe

// Signal → Observable
const searchTerm = signal('');
const searchTerm$ = toObservable(searchTerm);
// searchTerm$ emite cada vez que searchTerm cambia

// toSignal se auto-desuscribe cuando el componente se destruye
// No necesitas takeUntilDestroyed() ni unsubscribe manual

¿Cuándo usar cada uno? Signals para estado de UI (contadores, formularios, selecciones). RxJS para streams de eventos (WebSocket, event buses, operaciones complejas con timing como debounce, retry, race).

2.8 Signals vs BehaviorSubject

Característica Signal BehaviorSubject
Síncrono ✅ Sí ❌ Async (schedulers)
Lectura signal() .getValue() o .subscribe()
Change Detection Automático (granular) Necesita async pipe o markForCheck
Composición computed() combineLatest, map
Memory leaks No Sí (si no unsubscribe)
Side effects effect() tap()

2.9 Patrones avanzados

Signal Store (NgRx SignalStore): Alternativa más simple al NgRx Store clásico. Usa Signals internamente y reduce el boilerplate de Actions/Reducers/Effects.

Component state con Signals: Reemplaza BehaviorSubject en servicios compartidos. Los componentes leen directamente service.items() sin async pipe.

Zoneless Angular: Los Signals habilitan Change Detection sin Zone.js. Angular detecta exactamente qué cambió y actualiza solo esos componentes. 

// Servicio con Signals en vez de BehaviorSubject
@Injectable({ providedIn: 'root' })
export class CartService {
  // Antes: private itemsSubject = new BehaviorSubject<Item[]>([]);
  // Ahora:
  private items = signal<Item[]>([]);

  readonly allItems = this.items.asReadonly();
  readonly total = computed(() =>
    this.items().reduce((sum, i) => sum + i.price, 0)
  );
  readonly count = computed(() => this.items().length);

  addItem(item: Item) {
    this.items.update(list => [...list, item]);
  }

  removeItem(id: string) {
    this.items.update(list => list.filter(i => i.id !== id));
  }
}

2.10 untracked() — Leer sin crear dependencia

Dentro de un computed() o effect(), toda lectura de un Signal crea una dependencia reactiva. untracked() permite leer un Signal sin suscribirse a él: el efecto no se re-ejecutará cuando ese Signal cambie. 

import { signal, effect, untracked } from '@angular/core';

const counter = signal(0);
const currentUser = signal('Ed');

effect(() => {
  // Se re-ejecuta cuando counter cambia...
  const value = counter();

  // ...pero NO cuando currentUser cambia (solo se lee su valor actual)
  const user = untracked(currentUser);
  console.log(`${user} llevó el contador a ${value}`);
});

// También sirve para invocar funciones que leen signals internamente
effect(() => {
  const value = counter();
  untracked(() => this.analytics.track('counter_changed', value));
});

2.11 input(), input.required(), model() y output()

Desde Angular 17.1+ los inputs pueden ser Signals: son de solo lectura, participan en computed(), y input.required() obliga al padre a proveer el valor (error de compilación si falta). model() crea un Signal escribible con two-way binding [(...)], y output() reemplaza a @Output() + EventEmitter con una API más simple. 

import { Component, input, model, output, computed } from '@angular/core';

@Component({
  selector: 'app-product-card',
  template: `
    <h3>{{ product().name }}</h3>
    <p>{{ priceWithTax() | currency }}</p>
    <input type="checkbox" [checked]="selected()"
           (change)="selected.set(!selected())" />
    <button (click)="addToCart.emit(product().id)">Agregar</button>
  `
})
export class ProductCardComponent {
  // input requerido — el padre DEBE pasarlo o no compila
  product = input.required<Product>();

  // input opcional con valor por defecto y transform
  discount = input(0, { transform: (v: number | string) => Number(v) });

  // model() — two-way binding: <app-product-card [(selected)]="isSelected" />
  selected = model(false);

  // output() — sin EventEmitter
  addToCart = output<string>();

  // Los inputs son signals → componen con computed()
  priceWithTax = computed(() =>
    this.product().price * (1 - this.discount()) * 1.16
  );
}

// En el padre:
// <app-product-card [product]="p" [discount]="0.1"
//                   [(selected)]="isSelected"
//                   (addToCart)="onAdd($event)" />

Ventajas sobre @Input() clásico: tipado de required en compile-time, derivaciones con computed() en vez de ngOnChanges, funcionan de forma nativa con OnPush y zoneless, y son de solo lectura (el hijo no puede pisar el valor del padre por accidente).

✅ Mejores prácticas

  • Estado derivado → computed(), nunca effect() + set(). El anti-pattern más común es sincronizar signals con efectos.
  • Expón signals privados y escribibles hacia fuera como asReadonly().
  • Reserva effect() para el mundo exterior: localStorage, analytics, DOM, logging.
  • Para objetos, define una equal function o mantén inmutabilidad estricta con update() + spread.
  • Combina con RxJS donde aporte: toObservable() → debounceTime/switchMap → toSignal().

⚠️ Errores comunes

  • Mutar en vez de reemplazar: items.update(l => { l.push(x); return l; }) devuelve la misma referencia → con la igualdad por defecto (===) nadie se entera. Usa [...l, x].
  • Crear effect() en ngOnInit: lanza error de injection context. Créalo como campo de clase, en el constructor, o pasa { injector }.
  • Escribir un Signal dentro de computed(): prohibido — los computed deben ser funciones puras sin side effects.
  • Bucles con effect(): leer y escribir el mismo Signal dentro de un efecto crea un ciclo. Suele indicar que necesitabas computed() o linkedSignal().
  • Olvidar los paréntesis en el template: {{ count }} imprime la función; es {{ count() }}.

🎤 Preguntas de entrevista

  • ¿Signal vs Observable — cuándo usar cada uno? Signals: estado síncrono de UI con lectura directa y CD granular. Observables: eventos en el tiempo, cancelación, operadores de timing (debounce, retry, race). Se combinan con toSignal()/toObservable().
  • ¿Qué significa que computed() sea "glitch-free" y lazy? Nunca expone valores intermedios inconsistentes (se recalcula de forma atómica) y no se evalúa hasta que alguien lo lee; además memoiza el resultado.
  • ¿Por qué computed() en vez de effect() para derivar estado? computed() es declarativo, memoizado y sin timing issues; effect() + set() introduce escrituras asíncronas, posibles ciclos y estados intermedios inválidos.
  • ¿Qué hace untracked()? Lee un Signal dentro de un contexto reactivo sin registrarlo como dependencia, evitando re-ejecuciones no deseadas.
  • ¿input() vs @Input()? input() devuelve un Signal readonly: required en compile-time, componible con computed(), y notifica CD granular. @Input() es un campo mutable que requiere ngOnChanges para reaccionar.
  • ¿Cómo relacionas Signals con zoneless? Los Signals notifican a Angular exactamente qué vista depende de qué valor, eliminando la necesidad de Zone.js para saber "algo pudo cambiar".

3. Reactive Forms

Reactive Forms ofrecen un modelo explícito e inmutable del formulario en la clase del componente: el estado (valor, validez, dirty, touched, pending) vive en objetos FormControl/FormGroup/FormArray y el template solo se "engancha" con [formGroup] y formControlName. Esto los hace predecibles, testeables sin DOM y componibles con RxJS.

3.1 Formularios tipados (Angular 14+)

Desde Angular 14 los formularios son estrictamente tipados: form.value conoce la forma exacta de los datos. Ojo: value es Partial<T> (los controles disabled se excluyen), mientras que getRawValue() devuelve todo. Con NonNullableFormBuilder evitas el molesto string | null (al hacer reset() vuelve al valor inicial, no a null). 

import { NonNullableFormBuilder, Validators } from '@angular/forms';

export class ProfileFormComponent {
  private fb = inject(NonNullableFormBuilder);

  form = this.fb.group({
    name:  ['', [Validators.required, Validators.minLength(3)]],
    email: ['', [Validators.required, Validators.email]],
    age:   [18, Validators.min(0)],
    address: this.fb.group({
      street: [''],
      city:   [''],
    }),
  });

  save() {
    if (this.form.invalid) {
      this.form.markAllAsTouched(); // muestra todos los errores
      return;
    }
    // Tipado: { name: string; email: string; age: number; address: ... }
    const data = this.form.getRawValue();
    this.api.saveProfile(data);
  }
}

// Acceso tipado a controles y errores
this.form.controls.email.errors;   // { required: true } | { email: true } | null
this.form.controls.email.touched;  // boolean
this.form.controls.age.setValue(30);      // solo acepta number ✅
// this.form.controls.age.setValue('30'); // ❌ error de compilación

form = this.fb.group({

3.3 Validadores síncronos custom

Un validador es una función pura: recibe el control y devuelve null (válido) o un objeto de errores. Para validación cross-field (comparar dos campos) se aplica al FormGroup, que recibe el grupo completo. 

// Validador parametrizable (factory)
export function forbiddenName(name: string): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null =>
    control.value === name
      ? { forbiddenName: { value: control.value } }
      : null;
}

// Cross-field: password === confirmPassword (se aplica al grupo)
export const passwordsMatch: ValidatorFn = (group: AbstractControl) => {
  const pass = group.get('password')?.value;
  const confirm = group.get('confirmPassword')?.value;
  return pass === confirm ? null : { passwordsMismatch: true };
};

form = this.fb.group(
  {
    username: ['', [Validators.required, forbiddenName('admin')]],
    password: ['', Validators.minLength(8)],
    confirmPassword: [''],
  },
  { validators: passwordsMatch }  // ← validador a nivel de grupo
);

// Template: @if (form.hasError('passwordsMismatch')) { ... }

3.4 Validadores asíncronos

Los AsyncValidatorFn devuelven Observable o Promise y solo corren después de que los validadores síncronos pasan. Mientras esperan, el control queda en estado pending. Buenas prácticas: debounce con timer() y cancelar la petición anterior con switchMap. 

@Injectable({ providedIn: 'root' })
export class UsernameValidator {
  private api = inject(UserApiService);

  unique(): AsyncValidatorFn {
    return (control: AbstractControl): Observable<ValidationErrors | null> =>
      timer(300).pipe(                       // debounce: espera a que dejen de teclear
        switchMap(() => this.api.checkUsername(control.value)),
        map(taken => (taken ? { usernameTaken: true } : null)),
        catchError(() => of(null)),          // si la API falla, no bloquees el form
      );
  }
}

username = this.fb.control('', {
  validators: [Validators.required],
  asyncValidators: [inject(UsernameValidator).unique()],
  updateOn: 'blur',   // valida al salir del campo, no en cada tecla
});

// Template:
// @if (username.pending) { <span>Verificando…</span> }
// @if (username.hasError('usernameTaken')) { <span>Ya existe</span> }

3.5 valueChanges + operadores RxJS

Todo control expone valueChanges y statusChanges como Observables. El patrón clásico de búsqueda reactiva: 

search = this.fb.control('');

results$ = this.search.valueChanges.pipe(
  debounceTime(300),           // espera 300ms de silencio
  distinctUntilChanged(),      // ignora si el valor no cambió
  filter(term => term.length >= 2),
  switchMap(term => this.api.search(term)),  // cancela la request anterior
);

// O como Signal, sin async pipe:
results = toSignal(this.results$, { initialValue: [] });

// Reaccionar a un campo para habilitar/deshabilitar otro
constructor() {
  this.form.controls.hasDiscount.valueChanges
    .pipe(takeUntilDestroyed())   // cleanup automático
    .subscribe(has => {
      const code = this.form.controls.discountCode;
      has ? code.enable() : code.disable();
    });
}

3.6 updateOn — cuándo se propaga el valor

Por defecto los controles se actualizan en cada tecla ('change'). updateOn: 'blur' retrasa valor + validación hasta salir del campo, y 'submit' hasta el envío — clave para no disparar validadores asíncronos (HTTP) en cada pulsación. 

// Por control
email = new FormControl('', {
  validators: [Validators.email],
  updateOn: 'blur',
});

// Por grupo (aplica a todos los hijos que no lo sobrescriban)
form = new FormGroup({ ... }, { updateOn: 'submit' });

✅ Mejores prácticas

  • Usa NonNullableFormBuilder y deja que TypeScript infiera los tipos del form.
  • patchValue() para updates parciales, setValue() cuando debes proveer todo (te protege de olvidar campos).
  • En el submit: form.markAllAsTouched() si es inválido, y deshabilita el botón con form.pending en cuenta.
  • Suscripciones a valueChanges siempre con takeUntilDestroyed() o conviértelas con toSignal().
  • Extrae validadores a archivos propios y testéalos como funciones puras.

⚠️ Errores comunes

  • Usar form.value con controles disabled: desaparecen del objeto (es Partial). Si los necesitas, usa getRawValue().
  • Async validator sin debounce: una petición HTTP por tecla. Combina timer + switchMap o updateOn: 'blur'.
  • Poner el validador cross-field en un control: desde un control no ves a sus hermanos de forma fiable; va en el FormGroup.
  • Suscribirse a valueChanges y hacer setValue() del mismo control: bucle infinito. Usa { emitEvent: false }.
  • Olvidar type="button" en botones de agregar/quitar dentro del <form>: hacen submit accidental.

🎤 Preguntas de entrevista

  • ¿Reactive vs Template-driven? Reactive: modelo en la clase, síncrono, tipado, testeable, escala a forms complejos. Template-driven: ngModel en el HTML, asíncrono, válido para forms triviales.
  • ¿Cómo validas que dos campos coincidan? Con un ValidatorFn aplicado al FormGroup, que lee ambos controles con get() y devuelve el error a nivel de grupo.
  • ¿Qué es el estado pending? El control tiene validadores asíncronos en vuelo; ni válido ni inválido todavía. Útil para spinners y para bloquear el submit.
  • ¿setValue vs patchValue? setValue exige el objeto completo y falla si falta un campo; patchValue acepta parciales e ignora claves extra.
  • ¿Qué aportan los typed forms? Autocompletado y errores de compilación en value, setValue, controls.x — antes todo era any y los typos explotaban en runtime.

4. Lazy Loading

Lazy loading divide la app en chunks que el navegador descarga bajo demanda, reduciendo el bundle inicial (y el Time to Interactive). El bundler detecta los import() dinámicos y genera un archivo JS separado por cada uno. Con loadComponent se carga un componente standalone en una ruta; con loadChildren, un árbol de rutas completo (una feature). 

import { Routes } from '@angular/router';

export const routes: Routes = [
  // Lazy load de un componente standalone
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./dashboard/dashboard').then(m => m.DashboardComponent)
  },
  // Lazy load de una feature completa (rutas hijas)
  {
    path: 'admin',
    loadChildren: () =>
      import('./admin/admin.routes').then(m => m.ADMIN_ROUTES)
  },
  // Con default export no hace falta el .then()
  {
    path: 'about',
    loadComponent: () => import('./about/about')  // export default class...
  },
];

Preloading strategies

Sin preloading, el chunk se descarga al navegar (el usuario espera). Las estrategias descargan chunks en segundo plano después del arranque: PreloadAllModules (todos), NoPreloading (default, ninguno) o una estrategia custom para precargar solo lo probable. 

// Estrategia selectiva: precarga solo rutas marcadas con data.preload
@Injectable({ providedIn: 'root' })
export class SelectivePreloadStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>): Observable<unknown> {
    return route.data?.['preload'] ? load() : of(null);
  }
}

export const routes: Routes = [
  { path: 'checkout', loadChildren: () => import('./checkout/routes'),
    data: { preload: true } },   // ← crítico, precárgalo
  { path: 'reports',  loadChildren: () => import('./reports/routes') }, // solo on-demand
];

// app.config.ts
provideRouter(routes, withPreloading(SelectivePreloadStrategy));

// Alternativa built-in: precargar cuando el link entra en el viewport / hover
// provideRouter(routes, withComponentInputBinding(), withPreloading(...))

Guards funcionales en rutas lazy

Detalle importante: con canActivate, la ruta ya "matcheó", así que el chunk se descarga igual aunque el guard rechace. canMatch corre antes del match: si devuelve false, el Router sigue probando otras rutas y el bundle nunca se descarga — ideal para rutas por rol. 

export const adminOnly: CanMatchFn = () =>
  inject(AuthService).hasRole('admin');

export const routes: Routes = [
  // Misma URL, distinto bundle según el rol — canMatch decide cuál matchea
  { path: 'home', canMatch: [adminOnly],
    loadComponent: () => import('./admin-home') },
  { path: 'home',
    loadComponent: () => import('./user-home') },
];

✅ Mejores prácticas

  • Lazy-load por feature (con loadChildren + archivo de rutas), no componente a componente.
  • La ruta inicial y el shell no se lazy-loadean: los necesitas ya.
  • Precarga selectiva de las rutas con más probabilidad de visita.
  • Verifica el split real con ng build --stats-json + source-map-explorer / esbuild analyzer.
  • Para diferir trozos de una misma página usa @defer (ver card de Control Flow), no rutas.

⚠️ Errores comunes

  • Romper el code-splitting con un import estático: si cualquier archivo del bundle inicial hace import { AdminComponent } from ..., ese código se cuela en el chunk principal aunque la ruta sea lazy.
  • Servicios providedIn: 'root' vs providers de ruta: proveer un servicio en las rutas lazy crea una instancia distinta por inyector de ruta — a veces deseado, a veces un bug de "estado que se pierde".
  • Usar canActivate esperando ahorrar la descarga: el chunk ya se pidió; para eso es canMatch.

🎤 Preguntas de entrevista

  • ¿Cómo funciona lazy loading por debajo? import() dinámico → el bundler corta un chunk separado → el Router lo descarga y ejecuta al activar la ruta (o antes, con preloading).
  • ¿loadComponent vs loadChildren? loadComponent carga un componente standalone individual; loadChildren carga un array de rutas (una feature entera con sus hijos, guards y providers).
  • ¿canMatch vs canActivate en rutas lazy? canMatch impide el match y la descarga del bundle, y permite fallback a otra ruta con el mismo path; canActivate solo bloquea la activación.
  • ¿Cuándo NO conviene PreloadAllModules? Apps grandes en redes lentas o móviles: saturas la red descargando features que quizá nunca se usen. Mejor una estrategia selectiva.

5. Interceptors & Guards

Desde Angular 15+, interceptors y guards se escriben como funciones en vez de clases: menos boilerplate, tree-shakables y con acceso a servicios vía inject(). Un HttpInterceptorFn recibe req y next y forma una cadena: cada interceptor puede transformar la request (inmutable → siempre clone()) y/o la response. Se registran con provideHttpClient(withInterceptors([...])) y se ejecutan en el orden del array (y en orden inverso para la respuesta).

5.1 Auth token + manejo de 401

import { HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  // ⚠️ inject() SOLO en el cuerpo síncrono — captura los servicios aquí,
  // no dentro de los callbacks de RxJS (ahí ya no hay injection context)
  const auth = inject(AuthService);
  const router = inject(Router);

  const token = auth.getToken();
  const cloned = token
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;

  return next(cloned).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) {
        auth.logout();
        router.navigate(['/login']);
      }
      return throwError(() => err);
    })
  );
};

5.2 Retry con backoff exponencial

export const retryInterceptor: HttpInterceptorFn = (req, next) => {
  // Solo reintenta GETs (los POST no son idempotentes)
  if (req.method !== 'GET') return next(req);

  return next(req).pipe(
    retry({
      count: 3,
      delay: (error: HttpErrorResponse, retryCount) => {
        // No reintentes errores del cliente (4xx)
        if (error.status >= 400 && error.status < 500) {
          return throwError(() => error);
        }
        // Backoff: 1s, 2s, 4s
        return timer(1000 * Math.pow(2, retryCount - 1));
      },
    })
  );
};

5.3 Logging con duración

export const loggingInterceptor: HttpInterceptorFn = (req, next) => {
  const started = performance.now();
  let status = 'failed';

  return next(req).pipe(
    tap(event => {
      if (event.type === HttpEventType.Response) status = `${event.status}`;
    }),
    finalize(() => {
      const elapsed = Math.round(performance.now() - started);
      console.log(`${req.method} ${req.urlWithParams} → ${status} en ${elapsed}ms`);
    })
  );
};

5.4 Caching de GETs

@Injectable({ providedIn: 'root' })
export class HttpCache {
  readonly store = new Map<string, HttpResponse<unknown>>();
}

export const cacheInterceptor: HttpInterceptorFn = (req, next) => {
  if (req.method !== 'GET') return next(req);

  const cache = inject(HttpCache);
  const cached = cache.store.get(req.urlWithParams);
  if (cached) return of(cached.clone());   // respuesta síncrona desde cache

  return next(req).pipe(
    tap(event => {
      if (event instanceof HttpResponse) {
        cache.store.set(req.urlWithParams, event.clone());
      }
    })
  );
};

// Registro — el ORDEN importa: logging ve lo que auth ya modificó
provideHttpClient(
  withInterceptors([authInterceptor, retryInterceptor, cacheInterceptor, loggingInterceptor])
);

5.5 Guards funcionales

Los guards son funciones que devuelven boolean | UrlTree (o Observable/Promise de ellos). Devolver un UrlTree es la forma correcta de redirigir — sin efectos colaterales de navigate() dentro del guard. Tipos: CanActivateFn (entrar), CanActivateChildFn (hijos), CanDeactivateFn (salir — cambios sin guardar), CanMatchFn (matchear/descargar) y ResolveFn (pre-cargar datos). 

// CanActivateFn — con redirect vía UrlTree
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn()
    ? true
    : router.createUrlTree(['/login'], {
        queryParams: { returnUrl: state.url },  // volver después del login
      });
};

// CanDeactivateFn — evita perder cambios sin guardar
export interface HasUnsavedChanges {
  hasUnsavedChanges(): boolean;
}
export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> =
  (component) =>
    !component.hasUnsavedChanges() || confirm('¿Salir sin guardar?');

// ResolveFn — datos listos antes de renderizar la ruta
export const userResolver: ResolveFn<User> = (route) =>
  inject(UserService).getById(route.paramMap.get('id')!);

export const routes: Routes = [
  {
    path: 'profile/:id',
    canActivate: [authGuard],
    canDeactivate: [unsavedChangesGuard],
    resolve: { user: userResolver },
    loadComponent: () => import('./profile/profile'),
  },
];

✅ Mejores prácticas

  • Redirige desde guards devolviendo UrlTree, no llamando router.navigate().
  • Captura los servicios con inject() al inicio de la función, antes de cualquier callback asíncrono.
  • Interceptors pequeños y de responsabilidad única; el orden de registro es parte del diseño.
  • Excluye endpoints (login, assets) con condiciones sobre req.url o con HttpContextToken.
  • Guards con lógica compleja → delega en un servicio y testéalo con TestBed.runInInjectionContext().

⚠️ Errores comunes

  • inject() dentro de un callback RxJS (ej. en catchError): lanza NG0203 porque el injection context ya terminó. Captura las dependencias fuera del pipe.
  • Intentar mutar la request: HttpRequest es inmutable; req.headers.set(...) devuelve una copia que estás descartando. Siempre req.clone({...}).
  • Reintentar o cachear POSTs: duplicas operaciones de escritura. Limita retry/cache a métodos idempotentes.
  • Bucle infinito de refresh token: el interceptor de auth intercepta también la petición de refresh. Márcala con un HttpContextToken para saltarla.

🎤 Preguntas de entrevista

  • ¿Por qué req.clone()? Las requests son inmutables para que un retry/reintento reciba siempre la request original sin efectos acumulados de otros interceptors.
  • ¿En qué orden corren los interceptors? Request: en el orden del array de withInterceptors. Response: en orden inverso (como capas de cebolla).
  • ¿Interceptor funcional vs de clase? Mismo poder; el funcional es menos código, tree-shakable y usa inject(). Los de clase siguen soportados vía withInterceptorsFromDi() (legacy).
  • ¿Cómo redirige un guard correctamente? Devolviendo router.createUrlTree([...]); el Router cancela la navegación actual y ejecuta la nueva de forma atómica.
  • ¿Resolver vs cargar datos en ngOnInit? El resolver garantiza datos antes de renderizar (sin estados "cargando" parciales), pero bloquea la navegación; para UX fluida suele preferirse skeleton + carga en el componente.

  • core/interceptors/auth.interceptor.ts — HttpInterceptorFn funcional
  • core/guards/auth.guard.ts — CanActivateFn funcional
  • app.config.ts — provideHttpClient(withInterceptors([...]))

6. Dependency Injection Avanzada

El sistema de DI de Angular resuelve dependencias consultando una jerarquía de inyectores. Entender esa jerarquía —y las formas de configurar providers— distingue a un dev senior: controla el scope (singleton vs instancia por componente), el tree-shaking y la testabilidad.

6.1 providedIn y providers tree-shakables

providedIn: 'root' registra el servicio en el inyector raíz como singleton y lo hace tree-shakable: si nadie lo inyecta, no entra al bundle. Alternativas: 'platform' (compartido entre varias apps Angular en la misma página) y declararlo en el array providers de un componente o una ruta (instancia por inyector). 

// Singleton global, tree-shakable — el caso del 95%
@Injectable({ providedIn: 'root' })
export class CartService { ... }

// Instancia POR COMPONENTE (y sus hijos): estado aislado por instancia
@Component({
  selector: 'app-wizard',
  providers: [WizardStateService],  // cada <app-wizard> tiene el suyo
})
export class WizardComponent { ... }

// Instancia por FEATURE lazy (providers a nivel de ruta)
export const routes: Routes = [{
  path: 'admin',
  providers: [AdminAuditService],   // vive mientras la ruta esté activa
  loadChildren: () => import('./admin/admin.routes'),
}];

6.2 InjectionToken + useValue / useClass / useFactory / useExisting

Para inyectar cosas que no son clases (configs, funciones, primitivas) o para elegir implementación, se usa InjectionToken con las cuatro "recetas" de provider: 

// Token tipado para configuración
export interface AppConfig {
  apiUrl: string;
  featureFlags: Record<string, boolean>;
}
export const APP_CONFIG = new InjectionToken<AppConfig>('app.config');

// useValue — un objeto literal
{ provide: APP_CONFIG, useValue: { apiUrl: '/api', featureFlags: {} } }

// useClass — sustituir implementación (ej. en tests o por entorno)
{ provide: LoggerService, useClass: environment.production
    ? RemoteLoggerService
    : ConsoleLoggerService }

// useFactory — construcción con lógica y dependencias
{
  provide: STORAGE,
  useFactory: () => {
    const platformId = inject(PLATFORM_ID);
    return isPlatformBrowser(platformId) ? localStorage : new MemoryStorage();
  },
}

// useExisting — alias: dos tokens, UNA misma instancia
{ provide: BasicLogger, useExisting: LoggerService }

// Token con factory por defecto (no necesita provider explícito)
export const WINDOW = new InjectionToken<Window>('window', {
  providedIn: 'root',
  factory: () => window,
});

// Consumo
export class ApiService {
  private config = inject(APP_CONFIG);
  private win = inject(WINDOW);
}

6.3 Jerarquía de inyectores

Hay dos árboles paralelos: el EnvironmentInjector (root → providers de rutas lazy) y el ElementInjector (por elemento DOM con providers: componentes/directivas). Al resolver, Angular sube por el ElementInjector del componente; si no encuentra el token, salta al EnvironmentInjector y sube hasta root. Si nada lo provee: NullInjectorError: No provider for X. 

Resolución de inject(TokenX) desde un componente:

ElementInjector del componente
  → ElementInjector del padre → ... → raíz del árbol de elementos
    → EnvironmentInjector de la ruta (providers de ruta lazy)
      → EnvironmentInjector root (providedIn: 'root', app.config.ts)
        → PlatformInjector → NullInjector ❌ (error)

Modificadores de resolución con inject(): 

// No falles si no existe → devuelve null
private analytics = inject(AnalyticsService, { optional: true });

// Solo busca en el inyector del propio elemento
private local = inject(TabsService, { self: true, optional: true });

// Empieza a buscar en el PADRE (patrón: componentes anidados recursivos)
private parent = inject(MenuComponent, { skipSelf: true, optional: true });

// No pases del host (útil en directivas dentro de componentes)
private form = inject(ControlContainer, { host: true });

6.4 inject() vs constructor

// Antes (constructor injection)
constructor(
  private http: HttpClient,
  @Inject(APP_CONFIG) private config: AppConfig,
  @Optional() private logger?: LoggerService,
) {}

// Ahora (inject) — sin decoradores, tipos inferidos, funciona en funciones
private http = inject(HttpClient);
private config = inject(APP_CONFIG);          // tipo AppConfig inferido
private logger = inject(LoggerService, { optional: true });

// Habilita helpers reutilizables (composición, estilo hooks)
export function injectQueryParam(name: string): Signal<string | null> {
  const route = inject(ActivatedRoute);
  return toSignal(route.queryParamMap.pipe(map(p => p.get(name))),
    { initialValue: null });
}

Regla: inject() solo funciona en injection context: inicializadores de campos, constructores, factories de providers, guards/interceptors funcionales y runInInjectionContext(). En ngOnInit, callbacks o setTimeout lanza NG0203.

⚠️ Errores comunes

  • Doble instancia accidental: proveer en providers de un componente un servicio que ya es providedIn: 'root' crea otra instancia para ese subárbol — y "misteriosamente" el estado no se comparte.
  • inject() fuera de contexto: llamarlo en ngOnInit o dentro de un callback. Muévelo a un campo de clase o usa runInInjectionContext(injector, ...).
  • Token sin provider ni factory por defecto: NullInjectorError en runtime. Da al token una factory o márcalo optional con fallback.
  • Lógica pesada en factories: los providers se instancian lazy pero de forma síncrona en la primera inyección — nada de HTTP en un useFactory.

🎤 Preguntas de entrevista

  • ¿Qué es un InjectionToken y cuándo lo usas? Un identificador único y tipado para inyectar valores que no son clases (config, window, callbacks) o para desacoplar interfaz de implementación (las interfaces TS no existen en runtime).
  • ¿providedIn: 'root' vs providers de componente? Root: singleton global tree-shakable. Componente: una instancia por instancia del componente, destruida con él — ideal para estado local (wizard, tabs).
  • ¿Cómo resuelve Angular una dependencia? Sube por el árbol de ElementInjectors del componente, luego por los EnvironmentInjectors (ruta → root → platform); el primer provider encontrado gana, o NullInjectorError.
  • ¿Ventajas de inject() sobre constructor? Tipado sin decoradores, permite funciones helper reutilizables, evita problemas de herencia (no hay que repetir el constructor) y es la única opción en guards/interceptors funcionales.
📂 app.config.ts — providers a nivel de aplicación (provideRouter, provideHttpClient)

7. Control Flow Moderno

Angular 17 introdujo la sintaxis de control flow nativa (@if/@for/@switch/@defer): integrada en el compilador (sin importar directivas), con mejor type-narrowing, hasta 90% más rápida en listas que *ngFor y con soporte de carga diferida declarativa. Es la sintaxis recomendada (hay migración automática: ng generate @angular/core:control-flow).

7.1 @if / @else

@if (user(); as u) {
  <p>Hola, {{ u.name }}</p>      <!-- "as" guarda el valor ya narrowed -->
} @else if (isLoading()) {
  <app-spinner />
} @else {
  <a routerLink="/login">Inicia sesión</a>
}

7.2 @for con track obligatorio

track es obligatorio (en *ngFor el trackBy era opcional y casi nadie lo usaba). Le dice a Angular la identidad de cada item para reusar nodos DOM en vez de destruir y recrear al cambiar la lista. 

@for (product of products(); track product.id) {
  <app-product-card [product]="product" />

  <!-- Variables de contexto disponibles -->
  <span>{{ $index }} de {{ $count }}</span>
  @if ($first) { <span>⭐ primero</span> }
  <!-- también: $last, $even, $odd -->
} @empty {
  <p>No hay productos que coincidan.</p>   <!-- sin ngIf extra -->
}

7.3 @switch

@switch (status()) {
  @case ('loading') { <app-skeleton /> }
  @case ('error')   { <app-error [detail]="error()" /> }
  @case ('empty')   { <app-empty-state /> }
  @default          { <app-results [items]="items()" /> }
}

7.4 @defer — lazy loading dentro del template

@defer corta en un chunk separado los componentes usados dentro del bloque y los carga según un trigger. Es lazy loading a nivel de vista, sin tocar el router. Bloques asociados: @placeholder (antes de cargar), @loading (durante) y @error. 

@defer (on viewport) {
  <app-comments [postId]="postId()" />   <!-- chunk aparte -->
} @placeholder {
  <div class="comments-skeleton"></div>  <!-- requerido por "on viewport" -->
} @loading (minimum 300ms; after 100ms) {
  <app-spinner />                        <!-- evita flicker -->
} @error {
  <p>No se pudieron cargar los comentarios.</p>
}

<!-- Triggers disponibles -->
@defer (on idle) { ... }                <!-- default: requestIdleCallback -->
@defer (on interaction) { ... }         <!-- click/keydown en el placeholder -->
@defer (on hover) { ... }
@defer (on timer(3s)) { ... }
@defer (on immediate) { ... }
@defer (when panelOpen()) { ... }       <!-- condición custom (one-shot) -->

<!-- prefetch: descarga antes, renderiza después -->
@defer (on interaction; prefetch on idle) {
  <app-heavy-chart />
}

✅ Mejores prácticas

  • track item.id con IDs estables; track $index solo para listas estáticas que nunca se reordenan.
  • Usa @empty en vez de un @if extra para listas vacías.
  • @defer para lo pesado y "below the fold": charts, editores, mapas, comentarios.
  • Combina @loading (minimum ...) para evitar parpadeos de spinner en conexiones rápidas.

⚠️ Errores comunes

  • track $index en listas que se reordenan/filtran: Angular reasigna datos a los DOM nodes equivocados (inputs descolocados, animaciones rotas). Usa un ID estable.
  • @defer (on viewport) sin @placeholder: error de compilación — el trigger necesita un elemento que observar con IntersectionObserver.
  • Esperar que @defer "des-renderice": una vez disparado, el contenido queda cargado; when no es un @if reactivo.
  • Componentes del bloque @defer referenciados también fuera: si algo más los importa estáticamente, no habrá chunk separado.

🎤 Preguntas de entrevista

  • ¿Ventajas del nuevo control flow sobre *ngIf/*ngFor? Nativo del compilador (sin imports de CommonModule), type-narrowing real en @if, track obligatorio (mejor performance por defecto), @empty, y @defer que las directivas estructurales no pueden ofrecer.
  • ¿Qué hace exactamente track? Define la clave de identidad de cada item para el algoritmo de diffing: con ella Angular mueve/reusa nodos DOM existentes en vez de destruirlos y recrearlos.
  • ¿Qué genera @defer en build? Un chunk JS independiente con los componentes/directivas/pipes usados solo dentro del bloque; se descarga cuando el trigger se cumple (o antes, con prefetch).
  • ¿@defer vs lazy loading de rutas? Rutas: divide por navegación (páginas). @defer: divide dentro de una misma página por visibilidad/interacción. Se complementan.

8. Hydration y SSR

Con SSR (Server-Side Rendering, paquete @angular/ssr) el servidor renderiza el HTML de la ruta y el navegador muestra contenido inmediatamente, mejorando FCP/LCP y SEO. Después, la app cliente arranca y hidrata ese HTML: en vez de destruirlo y re-renderizar (el comportamiento antiguo, que causaba parpadeo), Angular reutiliza el DOM existente y solo le adjunta listeners y estado.

8.1 Modos de renderizado

Modo Dónde se genera el HTML Ideal para
CSR (default) Navegador, en runtime Dashboards, apps internas tras login
SSR Servidor, por request Contenido dinámico público (e-commerce, feeds)
SSG / Prerender Servidor, en build time Landing pages, blogs, docs (contenido estable)

8.2 Activar hydration + event replay

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideClientHydration(
      withEventReplay(),      // captura clicks ANTES de hidratar y los re-emite
      withHttpTransferCacheOptions({ includePostRequests: false }),
      // ↑ las respuestas HTTP del servidor viajan embebidas en el HTML:
      //   el cliente NO repite las mismas requests al hidratar
    ),
  ],
};

// Configuración por ruta (Angular 19+): elegir modo por página
export const serverRoutes: ServerRoute[] = [
  { path: 'landing',   renderMode: RenderMode.Prerender },
  { path: 'products/**', renderMode: RenderMode.Server },
  { path: 'admin/**',  renderMode: RenderMode.Client },
];

8.3 Incremental hydration (Angular 19+)

Combina hydration con @defer: el HTML llega completo del servidor, pero el JavaScript de cada bloque se descarga e hidrata solo cuando hace falta. Menos JS inicial, misma UI visible. 

// requiere provideClientHydration(withIncrementalHydration())
@defer (hydrate on viewport) {
  <app-reviews />    <!-- HTML visible desde el server;
                          su JS se hidrata al entrar en pantalla -->
}
@defer (hydrate on interaction) { <app-configurator /> }
@defer (hydrate never) { <app-static-footer /> }  <!-- nunca se hidrata -->

8.4 Código seguro para el servidor

// ❌ En el servidor no existen window/document/localStorage
ngOnInit() { this.width = window.innerWidth; } // ReferenceError en SSR

// ✅ Opción 1: afterNextRender — solo corre en el navegador, post-render
constructor() {
  afterNextRender(() => {
    this.width = window.innerWidth;
    this.chart = new Chart(this.canvas.nativeElement, ...);
  });
}

// ✅ Opción 2: comprobar la plataforma
private platformId = inject(PLATFORM_ID);
ngOnInit() {
  if (isPlatformBrowser(this.platformId)) {
    this.restoreFromLocalStorage();
  }
}

✅ Mejores prácticas

  • Todo acceso a APIs del navegador → afterNextRender() o guardas de plataforma.
  • Activa withEventReplay(): sin él, los clicks previos a la hidratación se pierden.
  • Aprovecha el TransferCache de HTTP para no duplicar requests servidor/cliente.
  • HTML válido estricto: la estructura que renderiza el servidor debe coincidir con la del cliente.
  • Como escape hatch puntual: atributo ngSkipHydration en el host del componente problemático.

⚠️ Errores comunes

  • Manipulación directa del DOM (document.querySelector, innerHTML manual, o librerías jQuery-style): el DOM del cliente deja de coincidir con el del servidor → hydration mismatch (errores NG0500).
  • HTML inválido en templates: un <div> dentro de <p>, o <tr> sin <tbody> — el navegador "recoloca" nodos y la estructura ya no coincide al hidratar.
  • Renderizar contenido no determinista (fechas Date.now(), aleatorios) distinto en servidor y cliente: mismatch garantizado.
  • Usar localStorage/window en constructores o ngOnInit: en SSR ese código corre en Node y explota.

🎤 Preguntas de entrevista

  • ¿Qué problema resuelve la hydration no destructiva? Antes Angular descartaba el HTML del servidor y re-renderizaba todo (flicker + trabajo duplicado); ahora reutiliza el DOM existente y solo adjunta listeners y estado.
  • ¿Qué es incremental hydration? Hidratar bloques @defer bajo demanda (hydrate on viewport/interaction): el HTML es visible desde el inicio pero su JS se carga solo cuando se necesita.
  • ¿Cómo evitas "window is not defined"? Moviendo el código de navegador a afterNextRender(), usando isPlatformBrowser(PLATFORM_ID), o inyectando abstracciones (DOCUMENT, tokens custom para window).
  • ¿SSR siempre es mejor? No: añade infraestructura (servidor Node), latencia por request y complejidad. Para apps tras login sin SEO, CSR (o prerender del shell) suele ser más simple y suficiente.
